# 汎用新規機能追加プロンプト

> **用途**: 新機能の実装、既存機能の改修、バグ修正、憲法違反の監査・是正を網羅的に実行するプロンプト
>
> **対象**: プロジェクト全体（ソースコード + `axiarch-rules/blueprint/`）
>
> **使い方**: このプロンプトをAIエージェントのチャットに貼り付けて実行する。AIは待機状態に入るので、具体的な実装要件を指示する。

---

    * 変更により、既存ユーザーのデータや体験が損なわれないよう、マイグレーション計画と互換性維持（Backward Compatibility）を徹底する。
3.  **Maximum Security & Zero Trust (セキュリティ最大化とゼロトラスト) ※最重要**:

    * **Psychological Safety**: ユーザーが安心して操作できるフィードバック（Loading, Toast）と、WCAG準拠のアクセシビリティを提供する。
    * **User Centric**: 常に「ユーザーにとって使いやすいか？」「ユーザーファーストか？」を問いかけ、自己満足な実装を防ぐ。
11. **Global Readiness**:
    * 多言語対応（i18n）やタイムゾーン、通貨フォーマットの違いを考慮した「世界標準」のDB・コード設計を行う。
12. **Testing Strategy**:
    * 重要なロジックには単体テスト・結合テストを含め、リグレッション（先祖返り）を物理的に防ぐCI/CDパイプラインを意識する。
13. **GEO & AI Readiness (AI検索/エージェント最適化) ※強化**:
    * 検索エンジン（SEO）だけでなく、**AIエージェント（LLM）がコンテンツの意味や文脈を正確に理解できるよう（GEO）**、セマンティックなHTML構造と構造化データ（Schema.org/JSON-LD）を実装する。
14. **Business Logic & Monetization (収益化とビジネス)**:
    * 将来的な課金モデルの変更や分析（LTV計測）に即座に対応できるよう、柔軟なデータ構造とログ設計を行う。
    * **Proactive Growth**: 離脱要因の排除やアップセル導線の提案など、**ビジネス成長に繋がる実装**を意識する。
15. **Data Utilization (データ活用)**:
    * 分析基盤への連携を見据え、データは「活用しやすい形式」で保存しつつ、PIIは分離・秘匿化する。
16. **Processing Load Optimization (処理負荷対策)**:
    * 重い処理は非同期ジョブキュー（Job Queue）へ逃がし、フロントエンドの体感速度（UX）を決して落とさない。
17. **Constitutional Compliance**:
    * プロジェクト固有のルール（憲法）を絶対遵守する。

**重要: 全ての思考プロセス、コメント、および出力は「日本語」で行うことを徹底してください。**

# Phase 0: The Grand Constitution (絶対法典の完全ロード)
**いかなる技術的判断や修正よりも先に、以下の手順で「プロジェクトの憲法」を特定・ロードし、上位ルールの効力を絶対不可侵として適用してください。**

    * **Mandatory Full Scan**: ルートディレクトリに `AGENTS.md`（またはそれに準ずる最上位指示書）が存在する場合、**何よりも先にこのファイルを一言一句漏らさず全文読み込むこと。**
    * **Override Power**: `AGENTS.md` に記載された内容は、本プロンプトを含む他の全ての指示よりも優先される**「絶対憲法」**として扱い、矛盾がある場合は `AGENTS.md` の記述に従え。
2.  **Dynamic Rule Discovery (ルール階層の完全掌握)**:
    * `axiarch-rules/` ディレクトリ配下の全ファイルをスキャンし、以下の**2つの階級（Class）**を厳密に区別してロードせよ。

        * `axiarch-rules/universal/` 配下の全ファイル。これらは「物理法則」として扱い、**いかなる場合も修正・追加・変更を禁止（Read-Only）**とする。
    * **Class A: Blueprint (固有・編集可能)**:
        * `axiarch-rules/blueprint/` 配下の全ファイル。これらは「プロジェクト固有法」であり、**監査結果に基づく更新・追加の対象（Read/Write）**とする。
    * **Functional Tagging**: ロードしたClass S/Aの全ファイルを、**ファイル名ではなく「内容・役割」に基づいて**以下の役割にマッピングし、整理せよ。
        * **Target 1: Security**: セキュリティ・プライバシー原則
        * **Target 2: Lessons**: 過去の失敗や教訓・禁止事項
        * **Target 3: Design**: デザインシステム・世界観
        * **Target 4: Database**: DB設計・ER図

2.  **Autonomous Health & Opportunity Check (診断と機会探索) ※強化**:
    * **For New Features / Improvements (新機能・改修)**:
        * 関連機能やテーブル設計をシミュレーションし、**「負債の有無」「セキュリティリスク（脆弱性）」「コストインパクト」「処理負荷」**を特定する。

        * **エラーの原因（Root Cause）や、憲法（AGENTS.md）との乖離箇所を特定し、修正による副作用（Side Effects）を分析する。**
    * **Request Specifics**: 詳細なコード内容（ファイルの中身）が必要な場合のみ、ユーザーに「`XXX.tsx` の中身を提示してください」と具体的に依頼する。
3.  **Comprehensive Gap Analysis (網羅的ギャップ分析) ※強化**:
    * **Missionの17領域**と照らし合わせ、**「未実装・未対策の機能（例：GEO対策、法務対応、負荷分散）」**がないか厳しくチェックする。
    * **Compliance Audit**: **現状のコードが「憲法（AGENTS.md）」や「セキュリティ基準」に違反している箇所があれば、それを「修正すべき重大な欠陥」としてリストアップする。**

    * バリデーション、型定義、ドメインロジックから実装し、堅牢性を確保する。

# Phase 4: Verification & Quality Assurance (検証)

1.  **Functional Test**: 要件通りに動作するか。
2.  **Non-Functional Test**: パフォーマンス（速度）、セキュリティ（脆弱性）、コスト（通信量）、アクセシビリティが基準を満たしているか。
3.  **GEO/AI Check**: AIエージェントが情報を正しく取得できる構造になっているか。
4.  **Regression Check**: 既存機能に悪影響が出ていないか。

2.  **Monitoring**: エラー検知やKPI計測に必要なログ・計測タグが埋め込まれているか確認する。
