# 憲法遵守スキャンプロンプト

> **用途**: 7つの重大な憲法違反（セキュリティ・アーキテクチャ収益化統合・型安全・最適化・ハリボテ・根本原因）を軸とした深層コンプライアンス是正プロンプト
>
> **対象**: プロジェクト全体または指定Focus Area（全ファイル・全機能が対象）
>
> **使い方**: このプロンプトをAIエージェントのチャットに貼り付けて実行する。AIは待機状態に入るので、重点監査領域（Focus Area）と監査対象のコードまたはファイルパスを指示する。

# Phase 1: Deep Constitutional Compliance Scan (深層憲法遵守スキャン)
全ファイル・全機能を対象に、読み込んだ「法」に照らし合わせ、以下の**7つの重大な憲法違反（The 7 Major Constitutional Violations）**がないかを徹底的に調査してください。いかなる妥協も許しません。

## 1. Supreme Directive Violation (最高法規および絶対原則の違反)
* **Target**: `AGENTS.md` および Class S 規約への抵触。
* **Audit**:
    * プロジェクトの根本的な指示や、絶対に守るべき普遍的なコーディング規約・禁止事項が破られていないか？

## 2. Security & Privacy Law Violation (セキュリティ・プライバシー保護法違反)
* **Target**: 認証・認可ルールの回避、個人情報の不適切な取り扱い。
* **Audit**:
    * 「個人情報保護とセキュリティの最大化」という最優先ルールに反し、フロントのUI分岐（`if (isAdmin)` 等）だけで守り、バックエンド/APIのミドルウェアやポリシーレベルでの厳格な権限検証を怠っている無防備なAPIはないか？
    * 全てのデータアクセスにおいて、認証セッション（User Context）が正しく検証されるルールが守られているか？
    * PII（個人情報）が不要にログ出力されていないか？ 取得範囲は最小限(Minimization)という絶対ルールが守られているか？

## 3. Architecture & Monetization Standard Violation (アーキテクチャ・収益化戦略違反)
* **Target**: ユニコーン基準の設計パターン、データ販売（API Sales）戦略からの逸脱。
* **Audit**:
    * **Data Gateway, CQRS, 階層型キャッシュ, Select Spec** などの指定されたアーキテクチャパターンが無視され、ビジネスロジックがUIにベタ書きされる規約違反はないか？
    * 将来の**データを外部へAPI販売 (Monetization)** する戦略を見据えた際、機密情報が自動的に除外されるシリアライズ設計（DTOパターンの徹底）ルールが破られていないか？ AI/GEO向けにセマンティックな設計とするルールが無視されていないか？

## 4. Type Safety & Data Synchronization Violation (型安全・データ同期義務違反)
* **Target**: TypeScriptの型定義ルール、DBスキーマとの同期不備。
* **Audit**:
    * `any` 型の使用や `as unknown as ...` による無理やりなキャストなど、型安全性のルールを破っていないか？ APIレスポンスが推論任せになっていないか？
    * 管理者権限を持つクライアント（Admin SDK等）の特権操作において、厳格な型安全ルールを無視していないか？
    * DBスキーマとコード定義（幽霊カラム、Nullableの不整合など）を完全に同期させる義務を怠っていないか？

## 5. Optimization & Performance Mandate Violation (最適化・パフォーマンス厳守命令違反)
* **Target**: 画像/スライダー最適化、SSR厳守、SEO/GEO要件ルールの無視。
* **Audit**:
    * LCP増やCLS悪化を防ぐ画像の遅延読み込みやCDN最適化ルールが無視され、メインスレッドのブロックを引き起こしていないか？
    * 検索エンジン/AIクローラー向けに「サーバーサイドレンダリング厳守」と定められた重要なデータ取得において、クライアントサイドレンダリングに逃げている違反箇所はないか？

## 6. Deceptive Implementation (偽装・ハリボテ実装の摘発)
* **Target**: ルールを守っているように見せかけた悪質な実装。
* **Audit**:
    * 本番ロジックに `const data = [...]` のようなダミーデータを混入させて動作を偽装していないか？
    * DB書き込みを行わず `console.log` 等で終了するフェイクアクションはないか？
    * パフォーマンス低下やバグの真の原因を `catch (e) {}` で握り潰して正常系を装っていないか？

## 7. Rule-Breaking Root Cause (ルール逸脱の根本原因と「最適化の抜け穴」)
* **Target**: ルール違反が特定の箇所で発生している構造的欠陥。
* **Audit**:
    * ある機能ではルール（画像最適化やSSR対応など）が完璧に守られているのに、**別のファイルや機能ではそのルール適用がスッポリ抜け落ちて、最適化した内容と同様の悪化状態に陥っている箇所**はないか？
    * なぜそのルール違反（抜け漏れ）が発生したのか、全ファイルを徹底的に時間かけてでも深く分析・解析し、深く思考し、**根本原因(Root Cause)**を突き止めて完全に是正せよ。

# Execution Protocol (実行手順)

1.  **Analyze (憲法違反の全機能深層解析)**:
    * 全ファイルを走査し、読み込んだ「法（ルール）」と上記「7つの重大な憲法違反」に照らし合わせて、リスクとルール逸脱を徹底的にリストアップする。
    * 表面的なバグ探しではなく、「時間をかけてでも深く分析・解析し、深く思考する」という基準に従い、なぜルールが破られたのかという**根本原因(Root Cause)**を洗い出す。

2.  **Report & Plan (違反報告と是正計画)**:
    * 発見された違反を**Critical / High / Medium**の優先度で分類し、「どのファイルが、どのルール（憲法）に違反しているか」およびその根本原因を明確に報告する。
    * **未実装・未対策**の機能や、**コストパフォーマンス・処理負荷**に関するルール違反箇所について、ROI（投資対効果）を含めた能動的な是正・ブラッシュアップ案を提示する。

3.  **Refactor & Fix (ルールの強制執行と是正)**:
    * **Enforcement**: 画像/SSR最適化、セキュリティルール、型安全、ユニコーンアーキテクチャの適用を、違反箇所に対して一貫して強制執行する。
    * 表面的な対処ではなく、ルールが破られた根本原因を断つ修正を行う。

4.  **Final Verify (最終適法確認)**:
    * 修正後、ビルドと型チェックがエラーゼロで通過することを確認する。
    * LCP、SSR要件、セキュリティ等のルールが完璧に満たされ、システム全体が「法」に完全準拠していることを保証する。

# Output Format (出力形式)

**応答は必ず以下の構造で行ってください。**

1.  **Compliance Audit & Root Cause Report (憲法遵守監査・根本原因解析報告書)**:
    * 修正対象ファイル一覧と、それぞれの「違反内容（どの憲法・ルールに抵触したか）」、「根本原因（なぜ違反が発生したか）」、「優先度（Critical/High/Medium）」、「是正方針」。
    * **※Strategic Proposals (ブラッシュアップ提案)**:
        * **ルール準拠に向けた指摘**: 「憲法（GEO対策）に従い、この機能のSSR対応が必須です」「外部API販売ルールを見据えてDTOを分離すべきです」など、**指示待ちにならずExecution Standardsに基づき能動的に提案**する。
        * **コスト/負荷対策**: 「この処理はコスト削減ルールに反しています。階層型キャッシュを適用すべきです」等の提案。
2.  **Refactored Code (是正コード)**:
    * 修正後のコードブロック。必ずファイルパスを明記すること。
    * ※変更点だけでなく、文脈がわかる範囲で提示すること。
3.  **Updated Rules (法典改定案)**:
    * **Class A (Project Mutable Bylaws)** 内の特定ファイルへの追記・修正内容（Diff形式または追記文）。
    * **※重要: 更新対象としたファイルパスを明記し、`axiarch-rules/CRYSTALLIZATION_PROTOCOL.md` の手順に従って記録すること。**
    * **※結晶化ガード**: 結晶化する教訓は「今回の監査でコードベースから実際に発見された問題」に限定すること。AIがユーザーの明示的指示なしにコードベースと無関係な「一般的ベストプラクティス」を独自リサーチで追加することは禁止。`universal/` に既に同様のルールが存在しないかも確認すること。
    * **Domain Distribution**: 教訓ログ（`core/010_project_lessons_log.md`）は一時蓄積場所であり最終目的地ではない。関連するドメイン別のBlueprintファイルに適切に分散配置し、ルールとして昇格させること。**「教訓ログに書いたから終わり」は禁止。**
    * **新規作成**: 適切な既存ファイルがない場合は、**3桁Sparse Numbering（間隔採番）**に従い、同ディレクトリ内に新規ファイルの作成案を提示すること。

# Critical Constraint (絶対遵守事項)

> [!CRITICAL]
> **1. SECURITY & PRIVACY SUPREMACY（セキュリティ・プライバシー至上主義）**
> * 個人情報の漏洩、権限昇格、データの不整合は物理的に防ぐ設計にせよ。疑わしきは許可せず（Zero Trust）。

> [!CRITICAL]
> **2. CONSTITUTIONAL VIOLATION REPORTING（憲法違反の報告義務）**
> * 「憲法違反」「セキュリティリスク」「法的不備」が見つかった場合は、ユーザーに報告し修正の承認を得る（※自明なバグ修正は即時実行可）。

> [!CRITICAL]
> **3. DO NOT BREAK LEGACY（既存保護）**
> * 既存のユーザーデータや機能を破壊することは絶対に許されない。必ず**後方互換性**を維持せよ。

> [!CRITICAL]
> **4. COST & PERFORMANCE AWARENESS（FinOps）**
> * クラウド破産やユーザーへの過剰課金を防ぐため、「通信量」「DB読み書き回数」「計算リソース」を最小化する設計を選べ。
