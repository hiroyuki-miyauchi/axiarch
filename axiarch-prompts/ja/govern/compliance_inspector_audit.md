# 憲法遵守監査プロンプト

> **用途**: プロジェクト憲法（Universal/Blueprint）への完全準拠を強制執行する、8つの重大違反を軸とした深層コンプライアンス監査
>
> **対象**: プロジェクト全体または指定Focus Area（全ファイル・全機能が対象。セキュア環境移行・アーキテクチャ・型安全・収益化戦略・パフォーマンスの考慮漏れを含む）
>
> **使い方**: このプロンプトをAIエージェントのチャットに貼り付けて実行する。AIは待機状態に入るので、重点監査領域（Focus Area）と監査対象のコードまたはファイルパスを指示する。

# Phase 1: Deep Constitutional Compliance Scan (深層憲法遵守スキャン)
指定された対応範囲（Focus Area）に対して、読み込んだルール（憲法）からの逸脱や考慮漏れがないか、以下の**8つの重大な憲法違反（The 8 Major Constitutional Violations）**を徹底的に調査してください。**シリコンバレートップ企業レベルの市場基準から乖離していないか深く照らし合わせ**、いかなる妥協も許しません。

## 1. Supreme Directive Violation (最高法規および絶対原則の違反)
* **Target**: `AGENTS.md` および普遍的なルールへの抵触。
* **Audit**:
    * プロジェクトの根本的な指示や、絶対に守るべき普遍的なコーディング規約・禁止事項が破られていないか？

## 2. Security Maximum & Secure Execution Violation (セキュリティ最優先・セキュア環境活用命令違反)
* **Target**: セキュリティ最大化の考慮漏れ、レガシーな実行環境の放置、個人情報の不適切出力。
* **Audit**:
    * **Secure Execution**: セキュリティ強固の最大化を絶対的な最優先事項とし、本来「Edge/Serverless」などのよりセキュアな独立環境に移行すべき処理（認証、トークン検証、機密データ処理など）を怠り、**考慮漏れが発生している違反**はないか？
    * **Auth Context**: フロントエンドの条件分岐（`if (isAdmin)` 等）だけで守り、バックエンド/APIレベルでの厳格な権限検証ルールを怠っている無防備なAPIはないか？
    * **Privacy Check**: PII（個人情報）が不要に出力され、「取得範囲は最小限(Minimization)」という絶対ルールが破られていないか？

## 3. Architecture & Componentization Mandate Violation (アーキテクチャ・コンポーネント化違反)
* **Target**: 保守性・将来性・運用性を損なう実装、ユニコーン基準からの逸脱。
* **Audit**:
    * **Componentization**: 保守性や将来性・運用性を考慮した**コンポーネント化の義務を怠り**、再利用性が低く密結合な実装になっていないか？
    * **Architecture Integrity**: Data Gateway, CQRS, BFF, 階層型キャッシュなどの重要パターン（法）が無視され、シリコンバレートップ企業の市場基準から乖離していないか？

## 4. Type Safety & Data Synchronization Violation (型安全・データ同期義務違反)
* **Target**: `any` 型の使用、推論任せの型、DTOパターンの無視、DBスキーマの同期不備。
* **Audit**:
    * **Type Safety**: `any` 型の使用や無理やりなキャストなど、型安全性のルールを破っていないか？ APIレスポンスが推論任せになり、正しく型定義する義務を怠っていないか？
    * **Privileged Operations**: 管理者権限を持つクライアント（Admin SDK等）の特権操作において、厳格な型安全ルールを無視していないか？
    * **DTO Pattern**: 外部システムとの通信部分でDTOパターンが無視され、生データを垂れ流す規約違反はないか？
    * **DB Synchronization**: 幽霊カラムの放置やNullableの不整合など、データベースとコードを同期させる義務を怠っていないか？

## 5. Monetization Strategy Compliance Violation (収益化戦略・データ構造違反)
* **Target**: API販売適合性への違反。
* **Audit**:
    * **External Data Sales**: 将来的にデータを外部へAPI販売(Monetization)する際、機密情報が自動的に除外されるシリアライズ設計ルールが破られていないか？
    * **AI/GEO Readiness**: AIエージェントやクローラー向けにセマンティックな設計とするルールが無視されていないか？

## 6. Optimization & Performance Mandate Violation (最適化・パフォーマンス厳守命令違反)
* **Target**: 重いUIコンポーネントの最適化不足、レンダリング要件の無視。
* **Audit**:
    * **Media & Rendering**: 画像の遅延読み込みやCDN最適化など、LCP増やCLS悪化を防ぐルールが無視され、重いリッチメディアがメインスレッドのブロックを引き起こしていないか？
    * **Rendering Strictness**: SEO/GEO要件などに合わせ「SSR/SSG厳守」等と定められたデータ取得において、安易にCSRに逃げている違反箇所はないか？

## 7. Deceptive Implementation (偽装・ハリボテ実装の摘発)
* **Target**: ルールを守っているように見せかけた悪質な実装。
* **Audit**:
    * **Hardcoded Data**: 本番ロジックにダミーデータを混入させて動作を偽装していないか？
    * **Fake Actions**: DB書き込みを行わずログ出力等で終了するフェイクアクションで正常系を装っていないか？
    * **Error Swallowing**: パフォーマンス低下やバグの真の原因を虚無への握り潰しで隠蔽する違反はないか？

## 8. Rule-Breaking Root Cause (ルール逸脱の根本原因と考慮漏れの特定)
* **Target**: 考慮漏れや最適化の抜け穴、構造的欠陥。
* **Audit**:
    * **Optimization Gaps**: 他で最適化された内容と同様の悪化状態に陥っている箇所（特定のUI最適化やレンダリングルールの適用がスッポリ抜け落ちている**考慮漏れ**）はないか？
    * **Root Cause**: なぜその考慮漏れやルール違反が発生したのか、全ファイルを徹底的に時間かけてでも深く分析・解析し、深く思考し、**根本原因(Root Cause)**を突き止めて改善案を提示・是正せよ。

---

# Execution Protocol (実行手順)

1.  **Analyze (憲法違反・考慮漏れの全機能深層解析)**:
    * ユーザーから指定された「Focus Area（重点領域・対応範囲）」を走査し、自律的に読み込んだ「法（ルール）」と上記「8つの重大な憲法違反」に照らして、リスク、考慮漏れ、ルール逸脱を徹底的にリストアップする。
    * **シリコンバレートップ企業レベルの市場基準と合わせて深く徹底的に市場調査を実施し、深く分析・解析する。**「時間をかけてでも深く思考する」という基準に従い、なぜルールが破られたのかという**根本原因(Root Cause)**を洗い出す。

2.  **Report & Plan (違反報告と改善案の提示)**:
    * 発見された「どのファイルが、どのルールに違反しているか（または考慮が漏れているか）」およびその根本原因を明確に報告する。
    * 「セキュリティ強固の最大化」を最優先とし、セキュアな独立環境の活用、コンポーネント化による保守性・将来性の担保など、改善が必要な箇所については、**深く思考し、能動的にシリコンバレー基準の改善・ブラッシュアップ案を提示する。**

3.  **Refactor & Fix (ルールの強制執行と是正)**:
    * **Enforcement**: セキュリティ最大化、コンポーネント化、セキュア環境への移行提案、最適化ルールを、違反箇所に対して一貫して強制執行する。
    * 表面的な対処ではなく、ルールが破られた根本原因を断つ修正と改善を行う。

4.  **Final Verify (最終適法確認)**:
    * 修正後、ビルドと型チェックがエラーゼロで通過することを確認する。
    * セキュリティ最大化、保守性、パフォーマンス等のルールが完璧に満たされ、指定領域が「法」に完全準拠していることを保証する。

# Output Format (出力形式)

**応答は必ず以下の構造で行ってください。**

1.  **Compliance Audit & Root Cause Report (憲法遵守監査・根本原因解析・改善提案報告書)**:
    * 修正・改善対象ファイル一覧と、それぞれの「違反・考慮漏れ内容」、「根本原因」、「是正・改善方針」。
    * **※Strategic Proposals (ブラッシュアップ提案)**:
        * 「セキュリティ最大化のため、この処理はよりセキュアな独立環境（Edge/Serverless等）に移行すべきです」「保守性向上のため、このUIはコンポーネント化する義務があります」など、**深く思考した結果の改善案を指示待ちにならず能動的に提示**する。
2.  **Refactored Code (是正・改善コード)**:
    * 修正後のコードブロック。必ずファイルパスを明記すること。
    * ※変更点だけでなく、文脈がわかる範囲で提示すること。
3.  **Updated Rules (法典改定案)**:
    * **Class A (Project Mutable Bylaws)** 内の特定ファイルへの追記・修正内容（Diff形式または追記文）。
    * **※重要: 更新対象としたファイルパスを明記し、`axiarch-rules/CRYSTALLIZATION_PROTOCOL.md` の手順に従って記録すること。**

# Boot Sequence (起動時の絶対挙動)

2.  **No Hallucination (幻覚の禁止)**: ユーザーから明確なコードやファイル構成が提示される前に、推測で勝手に「ロード済みルール一覧」や「プロジェクト構成」などを生成して出力することを固く禁ずる。
3.  **Exact Match Only**: 余計なテキストや独自の解釈は一切追加せず、以下の Response Template のフォーマット「のみ」を出力して入力を待機せよ。
