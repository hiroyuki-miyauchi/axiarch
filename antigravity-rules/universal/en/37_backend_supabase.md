# 37. Backend Engineering (Supabase & PostgreSQL)

## 1. Hybrid Stack Responsibility
*   **The Hybrid Stack**:
    *   **Edge (Cloudflare)**: **Shield & Optimize**. Handles DDoS protection, WAF, image resizing, and static asset caching. Contains no logic.
    *   **Frontend (Vercel)**: **Router & Renderer**. Handles stateless UI rendering and API routing. High CPU load tasks (image processing, AI waiting) are prohibited.
    *   **Backend (Supabase)**: **Source of Truth**. Handles DB, Auth, Storage (Origin), and Async Jobs (Edge Functions).

## 2. Database Design (PostgreSQL)
*   **Data Integrity & Ownership**:
    *   **RLS Mandate**: Row Level Security (RLS) is absolute. Direct use of `service_role` key is prohibited except for batch jobs. All queries must pass through RLS.
    *   **Explicit Ownership**: Define `user_id` (or appropriate owner column) on table creation and verify `auth.uid() = user_id` in RLS policies. No "Publicly Readable" tables allowed except for master data.
    *   **Shared Resource Pattern**: For resources shared by families or teams (e.g., `projects`, `teams`), manage access via intermediate tables (e.g., `project_members`) and design so it doesn't end with simple "Owner Only" RLS.
*   **Type Safety**:
    *   **Database Types**: Use `supabase gen types` to auto-generate TypeScript definitions (`database.types.ts`) from DB schema. Manual typing is prohibited.
*   **Schema Design Standards**:
    *   **Schema Separation**: Strictly separate `public` (App Data), `extensions` (PostGIS, etc.), and `admin` (Audit Logs/Admin Views). Installing `extensions` into `public` is prohibited.
    *   **Constraints**:
        *   **Identity**: Use `GENERATED BY DEFAULT AS IDENTITY` (Standard SQL) instead of `SERIAL` for auto-increment.
        *   **Money**: Use `numeric`/`decimal` or `integer` (smallest unit) for monetary calculations. `float` is strictly prohibited.
        *   **Boolean**: `boolean` columns must be `NOT NULL DEFAULT false` to prevent 3-valued logic (`null` contamination).

## 3. Integrity & CMS Architecture
*   **Soft Delete Strategy**:
    *   Use `deleted_at` timestamp (Soft Delete) for primary data deletion. Apply `WHERE deleted_at IS NULL` (Partial Index) to unique constraints to allow re-registration.
*   **CMS Data Strategy (Triple Write Protocol)**:
    *   **The Holy Trinity**: Rich text data mandates **Atomic Update** to the following 3 columns:
        1.  **`_json` (JSONB)**: Structured data for editing (Editor Source).
        2.  **`content` (HTML)**: HTML for display (View Source). Zero parsing load.
        3.  **`search_text` (TEXT)**: Plain text for search (Search Source). Tags removed.
    *   **JSON Maintenance**: String replacement within JSON via SQL `REPLACE` is prohibited. Must be parsed and safely replaced on the application side.

## 3. Performance & Scalability
*   **Infinite Scalability**:
    *   **No Select All**: `select('*')` is banned in production. Specify columns explicitly.
    *   **Pagination**: Queries without 'Limit' are **strictly prohibited**. Always implement pagination or infinite scroll.
*   **Index Strategy**:
    *   Always index Foreign Keys and columns used for search/filtering.
    *   **Full-Text Search**: Use `pg_search` (tsvector) for full-text search to avoid performance degradation of simple `LIKE` searches.
*   **Scalability Strategy**:
    *   **Partitioning**: When a single table is projected to exceed **10 Million (10M)** records, introduce partitioning (`pg_partman`) based on time or ID range.
    *   **Read Replica**: Route heavy analytical queries (e.g., Admin Dashboard) to Read Replicas to prevent impact on user-facing transactions.
    *   **Archival**: Data "not accessed for over 1 year" must be detached from active DB and offloaded to **Object Storage (CSV/Parquet)**.

## 4. Auth & Security
*   **Gotrue**:
    *   Delegate auth flow completely to Supabase Auth (Gotrue). No custom password hashing.

*   **Automation via Triggers**:
    *   Auto-update of `updated_at` and creation of notifications must be implemented via DB Triggers, not app code, to guarantee data integrity.
*   **Notification Architecture**:
    *   **Aggregation Logic**: Duplicate actions on the same resource (e.g., multiple likes) must be "Aggregated" in the notification table to prevent notification bombing.
    *   **Delivery Channels**: Instant notifications via WebSocket (Realtime), emails via Async Jobs (`pgmq`) to balance UX and reliability.

## 5. Storage & Delivery ("The Shield")
*   **Centralized Storage**:
    *   Supabase Storage is the "Single Source of Truth". Dynamic saving to Vercel local storage or `public` folder is prohibited.
*   **Cloudflare Proxy Shield**:
    *   Image delivery must go through Cloudflare (`<DOMAIN>/cdn-cgi/image/...`) to reduce direct load on Supabase.
    *   **Cloudflare Standard (Hybrid Optimization Protocol)**:
        *   **Admin (Original Preservation)**: Prohibit app-side compression/conversion for admin uploads to strictly maintain "Master Data" integrity. Convert **Multibyte/Non-ASCII** filenames (e.g., Japanese) to Romaji/ASCII (`wanakana`) for SEO.
        *   **User (UX & Capacity)**: For UGC, perform client-side resizing (Max 1920px rec.) to prevent massive uploads and **MANDATORY EXIF Stripping (Location Data)**. Ban auto-cropping; provide a Crop UI.
        *   **Edge Resizing**: Delegate final delivery format conversion (WebP/AVIF) entirely to the CDN Edge (Cloudflare).
    *   **Bucket Separation**:
        *   **General**: `bucket-main-app` (Public: Posts/Avatars / Private: Invoices).
        *   **Ads**: `bucket-ads-assets` (Ad Banners). Fully Public, maximize CDN cache.
        *   Do not mix assets with different lifecycles and access control levels.
    *   **Sensitive Document Protocol (Medical/Financial)**:
        *   **Private Bucket Mandatory**: Sensitive documents like **passports, financial reports**, and identity docs MUST be stored in a **private bucket** (e.g., `private-secure-docs`). Public buckets are strictly prohibited.
        *   **Signed URL Access**: Access to files is permitted ONLY via **Signed URLs** issued after verifying RLS permissions on the server side. Set short expiration (e.g., 15 mins).

## 6. Operations & Migration
*   **Migration Protocol**:
    *   **Remote Execution**: To avoid CLI instability, executing SQL via **Supabase Dashboard SQL Editor** is recommended for schema changes.
    *   **Version Control**: However, executed SQL MUST be saved as a file in `supabase/migrations/` and managed in Git. Traceability is key.
    *   **Code-First Migration**: Creating/Modifying tables via Supabase Dashboard (Table Editor) is **Strictly Prohibited**. Always create migration files (`supabase/migrations/*.sql`).
    *   **Preview Evacuation Protocol**: If destructive integrity errors (`duplicate key` etc.) occur in Preview environment migrations, **do not fix; migrate to a new branch immediately (Evacuation)**. Reduce repair cost of disposable environments to zero.
*   **Seed Data (Determinism)**:
    *   Dev data generated by `seed.sql` must use **Fixed IDs/Values**, not random numbers, to ensure the same data state is always reproduced (Determinism). Improves debug efficiency.
*   **Type Safety Protocol**:
    *   **Internal Casting**: For complex JOINs or Views (Manual type definition is deleted as risk management; prohibited in principle).
 
## 7. Maintenance & Security
*   **Hardening Protocols**:
    *   **Public Schema Guard**: Execute `REVOKE CREATE ON SCHEMA public FROM PUBLIC;` to prevent unauthorized table creation by attackers.
    *   **View Security**: Set `security_invoker = true` when creating Views to prevent privilege escalation (Definer View Vulnerability).
    *   **Cache Reload**: Execute `NOTIFY pgrst, 'reload schema';` after schema changes to prevent API cache inconsistency.

## 8. Advanced Database Protocols

### 5.1. The Database Mutation Protocol (No Console Ops)
*   **Law**: Manual record creation/edit/delete via DB Dashboard (Table Editor) is **strictly prohibited** as it leaves no history.
*   **Action**:
    *   **Local Migration**: DB changes (e.g., adding columns) must be done via local Migration files, reviewed via GitHub PR, and applied via CI/CD.
    *   **Seed Data**: Manage initial data via `supabase/seed.sql`.

### 5.2. The Strict Type Safety Protocol (No Any)
*   **Law**: `as any` is an invitation to technical debt. Neglecting type definitions, especially in RPCs or INSERTs, is unacceptable.
*   **Action**:
    *   **Generics**: Use generics like `supabase.rpc<Args, Result>('func_name')` to fully guarantee return and argument types.

## 9. Domain Data Modeling Principles
*   **Structured Tagging Protocol (Search)**:
    *   **Tag-Based**: Manage feature data (e.g., **Wifi, Parking**) centrally in a `tags` table (`type: amenity | rule`) instead of adding columns.
    *   **Filtering**: Efficiently support AND searches like "Condition A" AND "Condition B" via server-side filtering.
*   **Business Hours Schema**:
    *   **JSONB Store**: Store business hours as structured data (JSONB) and standardize logic that prioritizes holiday settings (`holidays`).
    *   **Search Optimization**: For "Currently Open" searches, recommend using columns like `next_open_at` updated via Triggers instead of runtime calculation.
*   **Reputation & Trust System**:
    *   **Bayesian Average**: Use **Bayesian Average** for rankings to favor "High count ★4.8" over "Low count ★5".
    *   **Pre-Moderation**: Save new posts as `status: pending`, publishing immediately only for approved or trusted users. Block self-reviews systematically.
    *   **Immutable Deletion**: Since deletion is irreversible, require "Magic Word" input for admin operations.
*   **Interactive Content Schema**:
    *   **Unified Schema**: Manage quizzes, diagnoses, and surveys in a single `interactive_contents` table, branching logic via `config` JSONB.
    *   **Deterministic**: Ensure diagnosis logic is deterministic (e.g., score calculation), not random, to guarantee result reproducibility.
