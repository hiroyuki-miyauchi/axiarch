# 50. 管理運用と内部ツール (Admin Operations & Internal Tools)

## 1. 構築 vs 購入戦略 (Build vs Buy Strategy - "Retool First")
*   **Retoolファースト・ポリシー (Retool First Policy)**:
    *   **ルール**: 管理画面（Admin Panel）、CSツール、オペレーションダッシュボードは、**原則としてスクラッチ開発（React/Flutter）を禁止します**。
    *   **解決策**: **Retool** を第一選択肢とし、構築時間を「数週間」から「数時間」に短縮します。エンジニアのリソースは、ユーザー向けのコア機能（Core Product）に集中させます。
    *   **例外**: 顧客（エンドユーザー）が直接使用するダッシュボードや、Retoolでは実現不可能な高度なUI/UXが必要な場合のみ、スクラッチ開発を許可します。

## 2. 管理ダッシュボード要件 (Admin Dashboard Requirements)
*   **Mobile First Admin**:
    *   **緊急対応**: 管理者は、外出先や移動中に緊急対応（ユーザーBAN、返金承認）を行う可能性があります。管理画面であっても、**モバイルでの操作性**を担保します。
*   **KPIモニタリング (KPI Monitoring)**:
    *   常に重要なKPI（MRR, DAU, 解約率）をリアルタイムで表示するウィジェットを配置します。
    *   「ビジネスが今どうなっているか」が一目でわかる状態を維持します。
*   **UI/UX基準 (UI/UX Standards)**:
    *   **Modal Visibility**: メディアピッカー等の主要な作業用モーダルは、一覧性を確保するため画面幅の **80% (80vw)** 以上を使用します。狭いモーダルは作業効率を下げます。
    *   **Action Feedback**: 保存・更新アクションに対し、「Toast通知」に加え、「ボタンの緑色化 + テキスト変更（保存しました！）」による明確なフィードバックを実装します。ユーザーに「保存された？」と迷わせる時間をゼロにします。
    *   **Copyright Automation**: フッターのコピーライト年号は、`[開始年]–[現在年]` の形式でシステムが自動更新するように実装します。手動更新は廃止します。
    *   **The Code Input Standard (Mandatory Code Editor)**:
        *   **Law**: 生の `textarea` でのコード編集（HTML/CSS/JS/Config）は、シンタックスハイライトがなく、入力ミスを誘発し、サイトの品質を著しく損なうため厳禁です。
        *   **Action**: コード入力欄（HTML/CSS/Config等）には、必ず `react-simple-code-editor` (+ `prismjs`) などのエディタコンポーネントを使用してください。
    *   **The CSS Containment Protocol (Admin Layout Protection)**:
        *   **Law**: 管理画面のアコーディオン展開等によって、親コンテナのスクロール計算が狂い、膨大な余白が発生する「Whitespace Glitch」を防ぐ必要があります。
        *   **Action**: `overflow-y-auto` を持つ管理画面のメインコンテナには、原則として `style={{ contain: 'layout' }}` を付与し、子要素のレイアウト変動を隔離してください。
*   **Information Density Strategy**:
    *   **Card Strategy**: コンテンツエリアは白背景のカード (`bg-white shadow-sm`) とし、背景（`bg-muted/40`）から分離します。
    *   **Action Placement**: 「保存」等の主要アクションは、常にヘッダー右上に配置します。
    *   **Layman Accessibility**: "Slug", "UUID" 等の専門用語を避け、「URL識別子」等の平易な言葉を使用します。全ての入力項目にはツールチップ解説を義務付けます。
    *   **Visual Theme Editor**: 「サイト設定」内にテーマエディタを新設し、プレビューを見ながら直感的にデザイン変更可能にします。
    *   **The Audit Visibility Protocol (Freshness Indicator)**:
        *   **Law**: 管理画面のリストや詳細ヘッダーには、公開日（`published_at`）と最終更新日（`updated_at`）を並記してください。
        *   **Visual Cue**: `updated_at` が一定期間（例: 3ヶ月）以上前の場合、警告色（橙色等）で表示し、情報の陳腐化を視覚的に伝えてください。
*   **ユーザー管理 (User Management)**:
    *   **検索とフィルタ (Search & Filter)**: ユーザーをID、メールアドレス、ステータス、課金プランで即座に検索・フィルタリングできる機能を実装します。
    *   **監査ログ (Action Logs / Audit Trail)**: 管理者が行った操作（ユーザー削除、返金処理、ステータス変更等）は全てログとして記録し、誰がいつ何をしたかを追跡可能にします。
*   **代理ログイン (Impersonation / Masquerade)**:
    *   CS対応のために、管理者が特定のユーザーとしてログインし、ユーザーと同じ画面を確認できる機能（代理ログイン）を実装します。
    *   **セキュリティ**: この機能は最高レベルの権限を持つ管理者のみに制限し、使用時は必ずログに残します。


## 3. コンテンツ管理戦略 (Content Management Strategy)
*   **HTML直書きの完全撤廃**:
    *   **原則 (No Raw HTML)**: 管理画面において、管理者がHTMLタグを直接記述・編集する運用を**厳禁**とします。XSS脆弱性の原因となり、デザインの一貫性を破壊します。
    *   **ブロックエディタ**: 記事作成機能が必要な場合は、**Tiptap** 等のブロックベースエディタ、またはHeadless CMSを導入します。
    *   **構造化データ**: コンテンツはHTML文字列ではなく、JSON構造として保存し、プレゼンテーション層とデータを分離します。
*   **カスタムフィールドUI**:
    *   地図や動画の埋め込みは、iframeコードの貼り付けではなく、専用フィールド（住所入力、YouTube ID入力）を用意し、フロントエンド側で安全にタグを生成します。

## 4. 運用ワークフロー (Operational Workflow)
*   **承認フロー (Approval Flows)**:
    *   返金処理、データの物理削除、全ユーザーへの通知など、不可逆的またはリスクの高い操作には、必ず「承認フロー（ダブルチェック）」を組み込みます。Retoolの承認機能ワークフローを活用します。
*   **アラート連携 (Alerting Integration)**:
    *   システムエラー（5xxエラーの急増）や異常なKPI変動（解約の急増）が発生した場合、SlackやEmailに即座に通知する連携を構築します。
    *   管理画面を見に行かなくても、異常に気付ける仕組みを作ります。

## 5. データインポート・エクスポート (Data Import & Export)
*   **一括処理 (Bulk Operations)**:
    *   **非同期処理**: 数千件を超えるCSVエクスポートや一括更新は、タイムアウトを防ぐため必ず**バックグラウンドジョブ**（Cloud Tasks等）として実行し、完了後にメールまたは通知で知らせます。
    *   **バリデーション**: インポート機能には、必ず「プレビュー画面」を設け、エラー行（型不一致、必須欠け）を事前に表示して修正させるステップを挟みます。いきなりDBに書き込むことは禁止です。

### 5.1. The Data Import Validation Standard（データインポート検証基準）
*   **Law**: 一括インポート機能では、データをDBに書き込む前に以下の検証を**全て通過**させなければなりません。検証なしの一括投入は「データベース汚染」であり厳禁です。
*   **Action**:
    1.  **Type Check（型チェック）**: 各フィールドが期待されるデータ型（数値、日付、URL等）に準拠しているか検証してください。型不一致の行はインポート対象から除外します。
    2.  **Duplicate Detection（重複チェック）**: 一意制約カラム（メールアドレス、識別子等）について、インポートデータ内の重複およびDB上の既存データとの重複を検出してください。
    3.  **Character Encoding Normalization（文字コード正規化）**: 入力データの文字コード（UTF-8, Shift_JIS, EUC-JP等）を自動判定し、UTF-8に正規化してからインポートしてください。全角・半角の不統一も正規化対象とします。
    4.  **Error Report（エラーレポート）**: インポート失敗行には、**行番号**と**具体的なエラーメッセージ**（プロジェクト設定言語）を付与してレポートしてください。「3件失敗しました」のような曖昧な報告は禁止です。
    5.  **Preview Mandatory**: 検証結果のプレビュー（成功件数・失敗件数・エラー詳細）をユーザーに確認させ、明示的な承認を得てからDBへの書き込みを実行してください。
*   **Rationale**: 検証なしの一括インポートは、型不正データ・重複レコード・文字化けの三重汚染を引き起こし、修復コストが指数関数的に増大します。入口での厳格な検証により、データ品質を構造的に保証します。

## 6. サポートとFAQ (Support & FAQ)
*   **SLA (Service Level Agreement)**:
    *   **初回応答時間**: ユーザーからの問い合わせには、自動応答を含めて **即時** に反応し、有人対応が必要な場合も **24時間以内** の初回応答を目標とします。
*   **FAQ管理 (FAQ Management)**:
    *   ユーザーからの問い合わせが多い項目は、即座にFAQ（ヘルプセンター）に追加し、自己解決率を高めます。FAQはCMS（NotionやZendesk Guide）で管理し、エンジニア以外でも更新可能にします。
*   **チャットサポート (Chat Support)**:
    *   **Intercom / Zendesk**: アプリ内チャットサポートを導入し、ユーザーの課題をリアルタイムで解決します。AIボット（Finなど）を活用し、一次対応を自動化します。

## 7. セキュリティと権限 (Security & Access Control)
*   **RBAC (Role-Based Access Control)**:
    *   管理者にも権限レベル（Super Admin, Support, Analyst）を設け、必要な情報のみにアクセスできるようにします（最小権限の原則）。
*   **IP制限 (IP Restriction)**:
    *   管理画面へのアクセスは、VPN経由または特定のIPアドレスからのみ許可し、外部からの攻撃を防ぎます。

## 8. 管理運用プロトコル (Admin Operations Protocols)

### 8.1. The Admin Hygiene Protocol (No English in UI)
*   **Law**: 「管理画面だから英語でいい」という甘えは、非エンジニア（運用担当者）の生産性を著しく低下させます。
*   **Action**:
    *   **Full Localization**: ボタンラベル、エラーメッセージ、ログのアクション名に至るまで、画面上に表示される全てのテキストは日本語でなければなりません。
    *   **Mapping**: Enum型やシステム識別子は、必ず表示用の定数マップ（例: `AUDIT_ACTION_LABEL`）を通して表示してください。生の値（`create_post`）を表示することは禁止です。

### 8.2. The System Transparency Protocol (Tech Stack Sync)
*   **Context**: 技術構成がブラックボックス化すると、非エンジニア（経営陣・運用者）との共通認識がズレ、誤った意思決定を招きます。
*   **Law**: 技術スタックに変更があった場合（DB変更、新AIモデル導入等）は、必ずダッシュボード上の `src/components/admin/dashboard/tech-stack-card.tsx` を即座に更新し、実態と同期させてください。技術構成がブラックボックス化することは許されません。



### 8.3. The Explicit Explanation Protocol (Zero Jargon)
*   **Law**: 開発者にとっての「常識（UUID, CPM, MRR）」は、運用者にとっては「謎の記号」です。これらに説明がないことはツールとしての敗北です。
*   **Action**:
    *   **Zero Jargon**: 管理画面上の全ての専門用語・指標・計算値には、必ず `Info` アイコンと `Tooltip` を付与し、「それが何であり、どう計算され、ビジネスにどう影響するか」をド素人でも分かる言葉で解説してください。
    *   **No Assumptions**: 「見ればわかる」という推測を禁止します。全ての数値には定義が必要です。新しいダッシュボードや指標を追加する際は、Tooltipの実装完了をもって「完成」と見なします。

### 8.4. The Data Seeding & Caching Protocol
*   **Law**: キャッシュキーの陳腐化や、CLI結果（"Up to date"）への過信は、本番環境でのデータ不整合（マスターデータ欠損等）を招きます。
*   **Action**:
    *   **Versioning**: マスターデータのキャッシュキーには、データの量や質が変わる際にバージョンサフィックス（`_v2`等）を付与し、物理的にキャッシュを無効化してください。
    *   **Verification**: 本番同期後は、必ずアプリケーション経由でデータ件数が期待通りか確認する手順を必須とします。

### 8.5. The Label Mapping Protocol (ラベルマッピング義務)
*   **Law**: システム内部キー（Enum値、ステータスコード、アクション名等）をUIにそのまま表示することは、非エンジニアの運用者にとって「暗号」であり、ツールとしての敗北です。
*   **Action**:
    1.  **Display Map**: 全てのEnum値・システム識別子に対して、表示用の定数マップ（例: `STATUS_LABEL_MAP: Record<Status, string>`）を定義し、UIからはこのマップを経由してラベルを取得してください。
    2.  **Fallback**: マップに存在しない値（将来の拡張等）が渡された場合は、生のキーをフォールバックとして表示しつつ、Loggerに警告を出力してください。これにより「未翻訳」状態を即座に検知できます。
    3.  **Localization Ready**: ラベルマップはプロジェクト設定言語で記述し、将来の多言語対応時にi18nキーへの置き換えが容易な設計にしてください。
*   **Scope**: ボタンラベル、テーブルのカラムヘッダー、監査ログのアクション名、通知メッセージの全てに適用します。
    *   **Operator-Native Language**: 管理画面のUIは、運用者の母国語（プロジェクト設定言語）で表示してください。「管理画面だからシステム内部のキーでよい」という妥協は、非エンジニアの運用者にとって認知負荷を高め、操作ミスの温床となります。
    *   **Abbreviation Convention**: 略語や専門用語（IP, URL, API等）は、プロジェクト設定言語の文脈に馴染む表記を選択してください。全角・半角の混在や不統一を避け、一貫したスタイルを維持してください。

## 9. 管理者エスカレーションプロトコル (Admin Escalation Protocol)

### 9.1. The Automated Escalation Standard (自動エスカレーション基準)
*   **Law**: ユーザー通報が一定数を超えた場合の自動処理と、管理者の対応プロセスを明確に定義してください。
*   **Escalation Threshold**:

    | 通報数 | 自動アクション |
    |:------|:-------------|
    | **3件 / 同一対象** | 対象コンテンツを**自動非表示**にし、管理者に要審査通知を送信 |
    | **5件 / 同一投稿者** | 当該ユーザーの**全投稿を審査キュー**に移動 |
    | **10件 / 24時間以内** | アカウントを**一時停止**し、管理者の手動審査を必須とする |

*   **Admin Action Audit**: 管理者が実行した全ての制裁操作（削除・非表示・アカウント停止等）は、**操作者ID・対象・理由・タイムスタンプ**を監査ログに記録してください。
*   **Due Process**: 全ての制裁措置に対し、**異議申立ての機会**（例: 14日以内）を提供してください。
*   **Notification Template**: 制裁措置の通知文はターゲットロケールの自然な文面とし、理由の明示・問い合わせ導線・異議申立て期限を含めてください。
*   **Mandate**: 各エスカレーション閾値の具体的な数値はプロジェクト固有のBlueprint側で定義してください。本プロトコルはエスカレーション構造の「型」を提供します。
