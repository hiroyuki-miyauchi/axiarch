# 50. 管理運用と内部ツール (Admin Operations & Internal Tools)

## 1. 構築 vs 購入戦略 (Build vs Buy Strategy - "Retool First")
*   **Retoolファースト・ポリシー (Retool First Policy)**:
    *   **ルール**: 管理画面（Admin Panel）、CSツール、オペレーションダッシュボードは、**原則としてスクラッチ開発（React/Flutter）を禁止します**。
    *   **解決策**: **Retool** を第一選択肢とし、構築時間を「数週間」から「数時間」に短縮します。エンジニアのリソースは、ユーザー向けのコア機能（Core Product）に集中させます。
    *   **例外**: 顧客（エンドユーザー）が直接使用するダッシュボードや、Retoolでは実現不可能な高度なUI/UXが必要な場合のみ、スクラッチ開発を許可します。

## 2. 管理ダッシュボード要件 (Admin Dashboard Requirements)
*   **Mobile First Admin**:
    *   **緊急対応**: 管理者は、外出先や移動中に緊急対応（ユーザーBAN、返金承認）を行う可能性があります。管理画面であっても、**モバイルでの操作性**を担保します。
*   **KPIモニタリング (KPI Monitoring)**:
    *   常に重要なKPI（MRR, DAU, 解約率）をリアルタイムで表示するウィジェットを配置します。
    *   「ビジネスが今どうなっているか」が一目でわかる状態を維持します。
*   **UI/UX基準 (UI/UX Standards)**:
    *   **Modal Visibility**: メディアピッカー等の主要な作業用モーダルは、一覧性を確保するため画面幅の **80% (80vw)** 以上を使用します。狭いモーダルは作業効率を下げます。
    *   **Action Feedback**: 保存・更新アクションに対し、「Toast通知」に加え、「ボタンの緑色化 + テキスト変更（保存しました！）」による明確なフィードバックを実装します。ユーザーに「保存された？」と迷わせる時間をゼロにします。
    *   **Copyright Automation**: フッターのコピーライト年号は、`[開始年]–[現在年]` の形式でシステムが自動更新するように実装します。手動更新は廃止します。
*   **Information Density Strategy**:
    *   **Card Strategy**: コンテンツエリアは白背景のカード (`bg-white shadow-sm`) とし、背景（`bg-muted/40`）から分離します。
    *   **Action Placement**: 「保存」等の主要アクションは、常にヘッダー右上に配置します。
    *   **Layman Accessibility**: "Slug", "UUID" 等の専門用語を避け、「URL識別子」等の平易な言葉を使用します。全ての入力項目にはツールチップ解説を義務付けます。
    *   **Visual Theme Editor**: 「サイト設定」内にテーマエディタを新設し、プレビューを見ながら直感的にデザイン変更可能にします。
*   **ユーザー管理 (User Management)**:
    *   **検索とフィルタ (Search & Filter)**: ユーザーをID、メールアドレス、ステータス、課金プランで即座に検索・フィルタリングできる機能を実装します。
    *   **監査ログ (Action Logs / Audit Trail)**: 管理者が行った操作（ユーザー削除、返金処理、ステータス変更等）は全てログとして記録し、誰がいつ何をしたかを追跡可能にします。
*   **代理ログイン (Impersonation / Masquerade)**:
    *   CS対応のために、管理者が特定のユーザーとしてログインし、ユーザーと同じ画面を確認できる機能（代理ログイン）を実装します。
    *   **セキュリティ**: この機能は最高レベルの権限を持つ管理者のみに制限し、使用時は必ずログに残します。


## 3. コンテンツ管理戦略 (Content Management Strategy)
*   **HTML直書きの完全撤廃**:
    *   **原則 (No Raw HTML)**: 管理画面において、管理者がHTMLタグを直接記述・編集する運用を**厳禁**とします。XSS脆弱性の原因となり、デザインの一貫性を破壊します。
    *   **ブロックエディタ**: 記事作成機能が必要な場合は、**Tiptap** 等のブロックベースエディタ、またはHeadless CMSを導入します。
    *   **構造化データ**: コンテンツはHTML文字列ではなく、JSON構造として保存し、プレゼンテーション層とデータを分離します。
*   **カスタムフィールドUI**:
    *   地図や動画の埋め込みは、iframeコードの貼り付けではなく、専用フィールド（住所入力、YouTube ID入力）を用意し、フロントエンド側で安全にタグを生成します。

## 4. 運用ワークフロー (Operational Workflow)
*   **承認フロー (Approval Flows)**:
    *   返金処理、データの物理削除、全ユーザーへの通知など、不可逆的またはリスクの高い操作には、必ず「承認フロー（ダブルチェック）」を組み込みます。Retoolの承認機能ワークフローを活用します。
*   **アラート連携 (Alerting Integration)**:
    *   システムエラー（5xxエラーの急増）や異常なKPI変動（解約の急増）が発生した場合、SlackやEmailに即座に通知する連携を構築します。
    *   管理画面を見に行かなくても、異常に気付ける仕組みを作ります。

## 5. データインポート・エクスポート (Data Import & Export)
*   **一括処理 (Bulk Operations)**:
    *   **非同期処理**: 数千件を超えるCSVエクスポートや一括更新は、タイムアウトを防ぐため必ず**バックグラウンドジョブ**（Cloud Tasks等）として実行し、完了後にメールまたは通知で知らせます。
    *   **バリデーション**: インポート機能には、必ず「プレビュー画面」を設け、エラー行（型不一致、必須欠け）を事前に表示して修正させるステップを挟みます。いきなりDBに書き込むことは禁止です。

## 6. サポートとFAQ (Support & FAQ)
*   **SLA (Service Level Agreement)**:
    *   **初回応答時間**: ユーザーからの問い合わせには、自動応答を含めて **即時** に反応し、有人対応が必要な場合も **24時間以内** の初回応答を目標とします。
*   **FAQ管理 (FAQ Management)**:
    *   ユーザーからの問い合わせが多い項目は、即座にFAQ（ヘルプセンター）に追加し、自己解決率を高めます。FAQはCMS（NotionやZendesk Guide）で管理し、エンジニア以外でも更新可能にします。
*   **チャットサポート (Chat Support)**:
    *   **Intercom / Zendesk**: アプリ内チャットサポートを導入し、ユーザーの課題をリアルタイムで解決します。AIボット（Finなど）を活用し、一次対応を自動化します。

## 7. セキュリティと権限 (Security & Access Control)
*   **RBAC (Role-Based Access Control)**:
    *   管理者にも権限レベル（Super Admin, Support, Analyst）を設け、必要な情報のみにアクセスできるようにします（最小権限の原則）。
*   **IP制限 (IP Restriction)**:
    *   管理画面へのアクセスは、VPN経由または特定のIPアドレスからのみ許可し、外部からの攻撃を防ぎます。

## 8. 管理運用プロトコル (Admin Operations Protocols)

### 8.1. The Admin Hygiene Protocol (No English in UI)
*   **Law**: 「管理画面だから英語でいい」という甘えは、非エンジニア（運用担当者）の生産性を著しく低下させます。
*   **Action**:
    *   **Full Localization**: ボタンラベル、エラーメッセージ、ログのアクション名に至るまで、画面上に表示される全てのテキストは日本語でなければなりません。
    *   **Mapping**: Enum型やシステム識別子は、必ず表示用の定数マップを通して表示してください。

### 8.2. The System Transparency Protocol (Tech Stack Sync)
*   **Context**: 技術構成がブラックボックス化すると、経営陣との共通認識がズレます。
*   **Law**: 技術スタックに変更があった場合（DB変更、新AIモデル導入等）は、必ずダッシュボード上の「Tech Stack Card」を即座に更新し、実態と同期させてください。
