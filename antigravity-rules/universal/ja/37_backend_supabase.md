# 37. バックエンドエンジニアリング (Backend Engineering - Supabase & PostgreSQL)

## 1. ハイブリッドスタック責務 (Hybrid Stack Responsibility)
*   **The Hybrid Stack**:
    *   **Edge (Cloudflare)**: **Shield & Optimize**. DDoS防御、WAF、画像リサイズ、静的アセットキャッシュを担当します。ロジックは持たせません。
    *   **Frontend (Vercel)**: **Router & Renderer**. ステートレスなUIレンダリングとAPIルーティングを担当します。CPU高負荷処理（画像加工・AI待機）は禁止です。
    *   **Backend (Supabase)**: **Source of Truth**. DB、Auth、ストレージ（Origin）、非同期ジョブ（Edge Functions）を担当します。

## 2. データベース設計 (Database Design - PostgreSQL)
*   **整合性と所有権 (Data Integrity & Ownership)**:
    *   **RLSの絶対遵守**: 行レベルセキュリティ（Row Level Security - RLS）は絶対です。`service_role` キーの使用はバッチ処理などの例外を除き禁止し、原則として全てのクエリはRLSを経由させます。
    *   **明示的な所有権**: テーブル作成時には必ず `user_id` カラム（または適切な所有者カラム）を定義し、RLSポリシーで `auth.uid() = user_id` を検証します。「誰でも読込可能」なテーブルはマスターデータ以外認められません。
    *   **Shared Resource Pattern**: 家族やチームで共有するリソース（例: `projects`, `teams`）は、中間テーブル（例: `project_members`）を通じてアクセス権を管理し、「所有者のみ」の単純なRLSで完結しないように設計します。
*   **Type Safety**:
    *   **Database Types**: `supabase gen types` コマンドを使用して、DBスキーマからTypeScript型定義（`database.types.ts`）を自動生成し、これをアプリケーション全体で使用します。手動での型定義は禁止です。
*   **Schema Design Standards**:
    *   **Schema Separation**: `public` (App Data), `extensions` (PostGIS等), `admin` (監査ログ・管理ビュー) を厳格に分離し、`extensions` を `public` にインストールすることを禁止します。
    *   **Constraints**:
        *   **Identity**: 自動採番には `SERIAL` ではなく `GENERATED BY DEFAULT AS IDENTITY` (標準SQL) を使用します。
        *   **Money**: 金額計算には `numeric`/`decimal` または `integer` (最小単位) を使用し、`float` は完全禁止とします。
        *   **Boolean**: `boolean` カラムは `NOT NULL DEFAULT false` とし、3値論理 (`null`混入) を防ぎます。

## 3. インテグリティとCMSアーキテクチャ (Integrity & CMS Architecture)
*   **Soft Delete Strategy**:
    *   主要データの削除は `deleted_at` への日時記録（論理削除）とし、ユニーク制約には `WHERE deleted_at IS NULL` (Partial Index) を付与して再登録を許可します。
*   **CMS Data Strategy (Triple Write Protocol)**:
    *   **The Holy Trinity**: リッチテキストデータは以下の3カラムへの **Atomic Update** を義務付けます。
        1.  **`_json` (JSONB)**: 編集用構造化データ (Editor Source)。
        2.  **`content` (HTML)**: 表示用HTML (View Source)。パース負荷ゼロ。
        3.  **`search_text` (TEXT)**: 検索用プレーンテキスト (Search Source)。タグ除去済み。
    *   **JSON Maintenance**: JSON内の文字列置換をSQLの `REPLACE` で行うことは禁止です。必ずアプリ側でパースして安全に置換します。

## 3. パフォーマンスとスケーラビリティ (Performance & Scalability)
*   **無限のスケーラビリティ (Infinite Scalability)**:
    *   **Select All 禁止**: `select('*')` はプロトタイピング時以外禁止です。本番コードでは必要なカラムのみを明示的に指定します。
    *   **Pagination**: クライアントサイドで上限（Limit）のないクエリを実行することは**厳禁**です。データ量は常に増加することを前提とし、必ずページネーションまたは無限スクロールを実装します。
*   **インデックス戦略**:
    *   外部キー（Foreign Key）のカラム、および検索・フィルタリングに使用されるカラムには、必ずインデックスを貼ります。
    *   **Japanese Search**: 日本語全文検索には `pg_search` (tsvector) または `pgroonga` を使用し、単純な `LIKE` 検索のパフォーマンス劣化を回避します。
*   **Scalability Strategy**:
    *   **Partitioning**: 単一テーブルが **1,000万件 (10M)** を超える予測が立った時点で、`pg_partman` によるパーティショニング（時間/ID範囲）を導入します。
    *   **Read Replica**: ヘビーな分析クエリ（管理画面ダッシュボード等）は Read Replica へルーティングし、ユーザー向けトランザクションへの影響を防ぎます。
    *   **Archival**: 「1年以上アクセスがない」データは、アクティブDBから切り離し、**Object Storage (CSV/Parquet)** へ退避させます。

## 4. 認証とセキュリティ (Auth & Security)
*   **Gotrue**:
    *   認証フローはSupabase Auth (Gotrue) に完全に委任します。自前でのパスワードハッシュ化やセッション管理は禁止です。
*   **Triggerによる自動化**:
    *   更新日時（`updated_at`）の自動更新や、通知の作成は、アプリケーションコードではなくDBトリガーで実装し、データの整合性を保証します。
*   **通知システム (Notification Architecture)**:
    *   **集約ロジック**: 同一リソースへの重複アクション（例: 複数人のいいね）は、通知テーブルで「集約（Aggregation）」し、通知爆撃を防ぎます。
    *   **配信チャネル**: 即時通知はWebSocket (Realtime)、メールは非同期ジョブ (`pgmq`) を経由させ、UXと信頼性を両立させます。

## 5. ストレージと配信 (Storage & Delivery - "The Shield")
*   **一元管理 (Centralized Storage)**:
    *   全てのファイルはSupabase Storageを「正」とします。Vercelのローカルストレージや`public`フォルダへの動的保存は禁止です。
*   **Cloudflare Proxy Shield**:
    *   画像配信は必ずCloudflare（`<DOMAIN>/cdn-cgi/image/...`）を経由させ、Supabaseへの直接アクセス負荷を軽減します。
    *   **Cloudflare Standard (Hybrid Optimization Protocol)**:
        *   **Admin (Original Preservation)**: 運営者のアップロードは「マスターデータ」としての同一性を保証するため、アプリ側での圧縮・変換を禁止し、オリジナル形式（JP/PNG）を維持します。日本語ファイル名はSEOのためローマ字変換（`wanakana`）を行います。
        *   **User (UX & Capacity)**: ユーザー（UGC）のアップロードは、巨大ファイルの送信を防ぐためクライアント側でリサイズ（長辺1920px推奨）を行い、**EXIF情報（位置情報）を必ず削除**します。自動トリミングは禁止し、Crop UIを提供します。
        *   **Edge Resizing**: 最終的な配信フォーマット（WebP/AVIF）変換は、全てCDNエッジ（Cloudflare）に委任します。
    *   **Bucket Separation (広告分離)**:
        *   **General**: `bucket-main-app` (Public: 投稿・アバター / Private: 請求書)。
        *   **Ads**: `bucket-ads-assets` (広告バナー)。完全パブリック、CDNキャッシュ最大化。
        *   データのライフサイクルとアクセス制御レベルが異なるアセットを混在させません。
    *   **Sensitive Document Protocol (Medical/Financial)**:
        *   **Private Bucket Mandatory**: パスポート、財務報告書、本人確認書類などの機微なドキュメントは、**`private-secure-docs`** などの非公開バケットに保存することを義務付けます。公開バケット (`public`) は厳禁です。
        *   **Signed URL Access**: ファイルへのアクセスは、サーバーサイドでRLS権限を確認した上で発行する **Signed URL (署名付きURL)** を通じてのみ許可します。有効期限は短く（例: 15分）設定します。

## 6. 運用とマイグレーション (Operations & Migration)
*   **マイグレーション運用規定 (Migration Protocol)**:
    *   **Remote Execution**: リモートデータベース（本番/Staging）へのスキーマ変更は、CLI (`db push`) の不安定さを回避するため、原則として **Supabase管理画面のSQLエディタ** でSQLを実行することを推奨します。
    *   **Version Control**: 実行したSQLは必ず `supabase/migrations/` にファイルとして保存し、Git管理します。「誰がいつ何を実行したか」を追跡可能にします。
    *   **Code-First Migration**: Supabase管理画面（Table Editor）でのテーブル作成・変更は**厳禁**です。必ずマイグレーションファイル (`supabase/migrations/*.sql`) を作成します。
    *   **Preview Evacuation Protocol**: Preview環境でマイグレーション履歴の破壊的な整合性エラー（`duplicate key` 等）が発生した場合、**修復せず直ちに新しいブランチへ移行（疎開）** します。使い捨て環境の修復コストをゼロにします。
*   **Seed Data (決定論的データ)**:
    *   `seed.sql` で生成する開発用データは、乱数を使わず **固定ID・固定値** を使用し、常に同じデータ状態が再現される（Determinism）ようにします。デバッグ効率を高めます。

## 7. 運用保守とセキュリティ (Maintenance & Security)
*   **Hardening Protocols**:
    *   **Public Schema Guard**: `REVOKE CREATE ON SCHEMA public FROM PUBLIC;` を実行し、攻撃者による勝手なテーブル作成を防ぎます。
    *   **View Security**: View作成時は `security_invoker = true` を設定し、権限昇格（Definer View脆弱性）を防ぎます。
    *   **Cache Reload**: スキーマ変更後は、`NOTIFY pgrst, 'reload schema';` を実行し、APIキャッシュ不整合を防ぎます。

## 8. 高度データベース運用プロトコル (Advanced Database Protocols)

### 5.1. The Database Mutation Protocol (No Console Ops)
*   **Law**: DB管理画面（Supabase DashboardのTable Editor）での手動レコード追加・編集・削除は、履歴が残らないため**完全禁止**とします。
*   **Action**:
    *   **Local Migration**: DB変更（カラム追加等）は、必ずローカルでMigrationファイルを作成し、GitHub PRを経由してCI/CDで適用してください。
    *   **Seed Data**: 初期データ投入も `supabase/seed.sql` で管理してください。

### 5.2. The Strict Type Safety Protocol (No Any)
*   **Law**: `as any` は技術的負債への招待状です。特にRPCやINSERTで型定義をサボることは許されません。
*   **Action**:
    *   **Generics**: `supabase.rpc<Args, Result>('func_name')` のようにジェネリクスを使用し、戻り値と引数の型を完全に保証してください。

## 9. ドメインデータモデリング (Domain Data Modeling Principles)
*   **Structured Tagging Protocol (検索)**:
    *   **Tag-Based**: 特徴データ（Wifiあり、駐車場あり等）はカラム追加ではなく、`tags` テーブル (`type: amenity | rule`) で一元管理します。
    *   **Filtering**: 「条件A」かつ「条件B」のようなAND検索をサーバーサイドフィルタリングで効率的にサポートします。
*   **Business Hours Schema (構造化営業時間)**:
    *   **JSONB Store**: 営業時間は構造化データ（JSONB）として保存し、祝日設定 (`holidays`) を優先適用するロジックを標準とします。
    *   **Search Optimization**: 「現在営業中」検索には、現在時刻計算ではなく Trigger で更新される `next_open_at` カラム等の利用を推奨します。
*   **Reputation & Trust System (信頼スコアリング)**:
    *   **Bayesian Average**: ランキングには単純平均ではなく **ベイジアン平均 (Bayesian Average)** を採用し、「件数少ない★5」より「件数多い★4.8」を上位に表示します。
    *   **Pre-Moderation**: 新規投稿は `status: pending` で保存し、承認または信頼済みユーザーのみ即公開とします。自己レビュー（Self-Review）はシステムでブロックします。
    *   **Immutable Deletion**: 投稿削除は不可逆のため、管理操作では「Magic Word」入力を必須とします。
*   **Interactive Content Schema (診断・クイズ)**:
    *   **Unified Schema**: クイズ、診断、アンケートは単一の `interactive_contents` テーブルで管理し、`config` JSONBでロジックを分岐させます。
    *   **Deterministic**: 診断ロジックはランダムではなく決定論的アルゴリズム（点数計算等）で行い、結果の再現性を担保します。
