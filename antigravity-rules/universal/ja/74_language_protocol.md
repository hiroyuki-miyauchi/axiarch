# 74. 言語プロトコル (Language Protocol)

> [!CRITICAL]
> **Japanese First Protocol (日本語第一主義)**
> - **基本原則**: ユーザーは日本のプロフェッショナルであり、AIエージェントとのコミュニケーションは、文化的文脈を含めて「高度な日本語」で行われなければならない。
> - **適用範囲**:
>   - **思考と計画**: タスク管理 (`task.md`)、実装計画 (`implementation_plan.md`) のすべての記述。
>   - **報告と対話**: `notify_user` による報告、状況説明、質問。
>   - **成果物**: ユーザーが目にするUI、ドキュメント、図表、エラーメッセージ。
> - **英語の制限**: 英語の使用は、コード、変数名、技術用語、またはコミットメッセージのみに厳格に制限される。不自然な翻訳調や、文脈を無視した英語の使用は「バグ」と見なされる。

## 1. Zero Cognitive Load Protocol (認知負荷ゼロ義務)
*   **Law**: 日本人ユーザーおよび開発者とのコミュニケーションロスをゼロにするため、UI、ドキュメント、コミットメッセージ、変数名の意図（コメント）に至るまで、**「日本語」を第一言語**とし、認知負荷（翻訳の手間）を物理的に排除します。
*   **Localization Mandate**:
    *   **UI Compliance**: エラーメッセージやバリデーション通知は、システム生の英語（例: "Invalid input"）ではなく、ユーザーが直感的に理解できる**「自然で丁寧な日本語」**でなければなりません。
    *   **Validation Guard**: Zodなどのスキーマ検証ライブラリを使用する場合は、必ず日本語のカスタムメッセージを設定してください。英語のエラー文言をユーザーの目に触れさせることは「実装漏れバグ」と定義します。
*   **Quality Standard (The Professional Tone)**:
    *   **Anti-Translation Feel**: 直訳調の不自然な日本語（例:「それは成功しました」「無効な値です」）を禁止します。日本のSaaSとして自然な表現（例:「保存しました」「正しい形式で入力してください」）を追求してください。
    *   **Politeness**: ユーザーに対する敬意払い、"Please"の直訳ではない、日本的な「おもてなし（Hospitality）」の精神を感じさせるマイクロコピーを義務付けます。

## 2. Bilingual Strategy (バイリンガル戦略)
*   **Code in English, Docs in Japanese**:
    *   **Code**: ソースコード、変数名、JSDocは世界標準の **英語** で記述します。
    *   **Docs**: 設計書（Blueprint）、タスク定義、PRの説明、コミット要約は、チームの正確な意思疎通のため **日本語** で記述します。
    *   **Exception**: 管理画面UI等のローカライゼーションが必要な箇所は、プロジェクト設定言語（日本語）に従います。

## 3. Comprehensive UI Localization Protocol (包括的UIローカライゼーション義務)
*   **Law**: 公開・非公開を問わず、**全てのUIテキスト**（ボタン、プレースホルダー、ヘルプ文言、ステータス、スクリーンリーダー用テキスト、ドキュメント）をプロジェクト設定言語にローカライズしなければなりません。
*   **Action**:
    1.  **Hidden Text Coverage**: アイコンの `alt` 属性、`aria-label`、`title` 属性など、視覚的に隠れたテキストも**例外なく**ローカライズしてください。
    2.  **Error & Fallback**: データ欠落時やエラー時のフォールバックテキスト（「Unknown」「N/A」等）は、プロジェクト設定言語に置換してください（例:「不明」「未設定」）。英語のフォールバックをユーザーの目に触れさせることは「実装漏れバグ」です。
    3.  **Third-Party Extension**: 外部ライブラリ（エディタ、UIキット等）をラップする際、UIラベルやツールチップがプロジェクト設定言語と一致しているか必ず検証してください。
    4.  **Placeholder Integrity**: プレースホルダーは、そのフィールドが期待するデータ型とビジネス文脈に100%合致させてください。他コンポーネントからのコピー時は「意味的な整合性」の再検証が必須です。
    5.  **Context-Aware Nominalization**: ボタンやラベルには冗長な表現を避け、体言止め（名詞化）や簡潔な表現を用い、プロフェッショナルなトーンを維持してください。
*   **Exception**: APIパス、JSONキー、技術用語（HTML, CSS, JSON等）は開発者の利便性を考慮して英字を維持して良いですが、説明文は必ずプロジェクト設定言語にしてください。

## 4. Validation Message Localization Protocol (バリデーションメッセージ・ローカライゼーション義務)
*   **Law**: バリデーションライブラリ（Zod, Yup, React Hook Form等）のデフォルトエラーメッセージは英語です。これをユーザーの目に触れさせることは「バグ」です。
*   **Action**:
    1.  **Explicit Message**: 全てのバリデーションルール（特に `.url()`, `.email()`, `.datetime()`, `.enum()` 等）に対し、明示的な `message` プロパティをプロジェクト設定言語で設定してください。
    2.  **System Key Translation**: ステータスコードやEnum値（'active', 'pending'等）をUIに表示する際は、表示用ラベルマップ（Display Label Map）を経由し、プロジェクト設定言語に変換してください。
    3.  **Logic-Layer Coverage**: `throw new Error()` や Logger に渡すメッセージであっても、エンドユーザーに表示される可能性があるものは、プロジェクト設定言語で記述してください。
*   **Rationale**: 一見ローカライズされたUIの中に突如「Required」「Invalid input」と表示されることは、ユーザーの信頼を損ない、「未完成なシステム」という印象を与えます。

## 5. Translation Exclusion Policy (翻訳除外ポリシー)
*   **Law**: ローカライゼーション義務の一方で、「全ての英語文字列を翻訳するとプログラムロジックが破壊される」ケースが存在します。以下のパターンは**明示的に翻訳除外**としてください。
*   **Exclusion List**:
    1.  **プログラム定数**: フロントエンドの条件分岐で使用される文字列定数（例: `'QUOTA_EXCEEDED'`）。
    2.  **フロー制御文字列**: 認証フロー等の状態管理に使用される文字列（例: `'OTP Required'`）。
    3.  **外部API送信値**: サードパーティAPIの仕様に依存する文字列（例: Stripeの `status: 'active'`）。
    4.  **内部例外 (`throw new Error`)**: try-catch内で処理され、UIに到達しない開発者向けアサーション。
*   **Judgment Criteria**:
    *   ✅ **翻訳する**: `return { error: '...' }` → ユーザーに表示される可能性がある。
    *   ✅ **翻訳する**: `return { message: '...' }` → トーストやアラートに表示される。
    *   ❌ **翻訳しない**: `throw new Error('...')` で、catch内でユーザーに別メッセージを表示する場合。
    *   ❌ **翻訳しない**: 条件分岐の比較対象として使われる定数文字列。
