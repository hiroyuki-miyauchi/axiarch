# 190. 管理画面と運用 (Internal Tools & Operations)

## 1. Retool First 戦略 (Retool First Strategy)
*   **原則 (Principle)**: 管理画面（Admin Panel）や社内ツールは、エンジニアのリソースを顧客向けプロダクトに集中させるため、原則として **Retool** などのローコードツールを使用して構築します。
*   **例外 (Exception)**: 複雑な独自ロジックが必要な場合や、パフォーマンスが極めて重要な場合に限り、Next.js等でのスクラッチ開発を検討しますが、その場合もUIライブラリ（MUI, Ant Design）を活用し、デザインコストをかけないようにします。

## 2. 権限管理 (RBAC: Role-Based Access Control)
*   **厳格な権限分離**:
    *   **Super Admin**: 全権限を持つ。CTOおよび指名された責任者のみ。
    *   **Support**: ユーザー情報の閲覧のみ可。編集・削除は不可。
    *   **Content Manager**: コンテンツの入稿・編集のみ可。
*   **最小権限の原則**: 業務に必要な最小限の権限のみを付与します。

## 3. 監査ログ (Audit Logs)
*   **完全な追跡**: 管理画面での全ての操作（誰が、いつ、何を、変更前/変更後）をログとして記録します。これは不正防止およびトラブルシューティングのために必須です。
*   **不変性**: 監査ログは改ざんできない状態で保存します（例: Firestoreの書き込み専用コレクション、Cloud Logging）。

## 4. 運用フロー (Operational Workflow)
*   **CS対応**: ユーザーからの問い合わせ対応は、管理画面からスムーズに行えるように設計します（例: ユーザーID検索、ステータス確認、パスワードリセットメール送信）。
*   **マニュアル化**: オペレーション手順はNotion等で文書化し、属人化を防ぎます。AI（チャットボット）を活用して、マニュアル検索を効率化します。
