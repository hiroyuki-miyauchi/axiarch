# 110. データ基盤とドキュメント運用 (Data Ops & Doc Ops)

## 1. データ基盤 (Data Infrastructure - "Data Mesh")
*   **信頼できる唯一の情報源 (Single Source of Truth)**:
    *   **BigQuery**: 全ての分析データはBigQueryに集約します。スプレッドシートやローカルファイルでの管理は禁止です。
    *   **ELT**: データは加工せずにまずロード（Load）し、BigQuery内で変換（Transform）するELTパイプラインを採用します。
*   **データ品質 (Data Quality)**:
    *   **dbt**: データ変換には **dbt (data build tool)** を使用し、SQLをバージョン管理・テストします。
    *   **テスト**: データの欠損、重複、異常値を自動検知するテストをパイプラインに組み込みます。

## 2. ドキュメント運用 (Documentation Ops)
*   **Living Documentation (生きたドキュメント)**:
    *   **Mermaid.js**: アーキテクチャ図やフローチャートは、画像（png/jpg）ではなく、必ず **Mermaid.js** などのコードベースで記述します。これにより、バージョン管理と編集が可能になり、陳腐化を防ぎます。
    *   **ADR (Architecture Decision Records)**: 「なぜその技術を選んだか」「なぜその設計にしたか」という重要な意思決定は、`docs/adr` ディレクトリにMarkdown形式で記録し、歴史的経緯を保存します。
*   **Notion / GitHub Wiki**:
    *   **フロー情報**: 日々の議事録やアイデアはNotionで管理します。
    *   **ストック情報**: 仕様書、設計図、手順書はGitHub（Markdown）で管理し、コードと同期させます。
*   **検索性 (Searchability)**:
    *   ドキュメントは「書くこと」よりも「見つけられること」が重要です。適切なタグ付けと階層構造を維持します。
*   **更新責任 (Ownership)**:
    *   全てのドキュメントには「最終更新日」と「責任者（Owner）」を明記します。半年以上更新されていないドキュメントは、更新するかアーカイブ（削除）します。
*   **Docs as Code**:
    *   ドキュメントはコードと等価です。Gitで管理し、PRレビューの対象とします。
    *   コードの変更とドキュメントの更新は「アトミック（不可分）」です。ドキュメント更新なきコード変更はマージしてはなりません。
*   **鮮度とメンテナンス (Freshness & Maintenance)**:
    *   **Dead Link Check**: リンク切れを自動検知する仕組みを導入します。
    *   **Review Cycle**: 主要なルールファイルは四半期ごとに見直し（Review）を行い、陳腐化を防ぎます。
*   **AI Generation**:
    *   API仕様書やボイラープレートコードのドキュメントは、AIを活用して自動生成し、人間は「Why（なぜそうしたか）」の記述に集中します。
