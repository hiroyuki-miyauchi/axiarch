# 53. 危機管理とBCP (Crisis Management & BCP)

## 1. インシデント対応体制 (Incident Response - "War Room")
*   **重大度レベル (Severity Levels)**:
    *   **SEV-1 (Critical)**: サービスダウン、データ損失、セキュリティ侵害。**即時対応**（24/365）が必要です。全エンジニアを招集します。
    *   **SEV-2 (High)**: 主要機能の不具合（決済不可など）。当日中の修正が必要です。
    *   **SEV-3 (Medium)**: 軽微なバグ、UXの低下。次回のスプリントで対応します。
*   **War Roomの設置**:
    *   **基準**: SEV-1またはSEV-2が発生した場合、即座に「War Room（緊急対策本部）」を設置します（Slackチャンネル `#incident-yyyy-mm-dd` + Google Meet）。
*   **役割分担 (Roles)**:
    *   **Commander (指揮官)**: 全体の意思決定を行う唯一の人物。技術的な作業は行いません。
    *   **Ops Lead (作業責任者)**: 実際の復旧作業を指揮・実行します。
    *   **Comms Lead (広報責任者)**: ユーザーおよび社内への定期的（30分ごと）な状況報告を行います。
    *   **Scribe (書記)**: 全てのイベントとタイムスタンプを記録し、後のポストモーテムに使用します。
*   **The Incident Response Timeline (対応タイムライン基準)**:
    *   **Law**: インシデント発生時の「検知→初動→解決」の時間目標を、重大度ごとに明確に定義し、対応の遅延を防止しなければなりません。
    *   **Timeline Matrix**:

        | レベル | 検知 → 初動 | 初動 → 解決 | コミュニケーション | 例 |
        |:------|:----------|:----------|:---------------|:---|
        | **SEV-1** | **15分以内** | **1時間以内** | ステータスページ + 全社通知 | 全面停止、データ消失 |
        | **SEV-2** | **30分以内** | **4時間以内** | ステータスページ + チーム通知 | 主要機能停止、認証障害 |
        | **SEV-3** | **2時間以内** | **24時間以内** | チャット通知チャネル | UI崩れ、速度劣化 |

    *   **On-Call Escalation**: 主担当（Primary）が初動期限内に応答しない場合、副担当（Secondary）に自動エスカレーションしてください。
    *   **6-Step Incident Flow**:
        1.  **検知**: 監視アラート or ユーザー通報
        2.  **分類**: Severityレベル判定
        3.  **宣言**: インシデント通知チャネルで宣言（例: 「SEV-1: 決済機能障害 発生中」）
        4.  **対応**: ロールバック / ホットフィックス / 一時遮断
        5.  **解決**: ステータスページ更新、復旧通知
        6.  **振り返り**: 48時間以内にPost-Mortemを実施

## 2. コミュニケーション・テンプレート (Communication Templates)
*   **原則**: 「沈黙は不信を生む」。解決していなくても、状況を伝え続けます。
*   **ステータスページ (Status Page)**:
    *   **Investigating**: 「現在、アクセス障害を検知し、調査中です。」
    *   **Identified**: 「原因を特定しました（DB負荷）。現在復旧作業中です。」
    *   **Monitoring**: 「修正を適用し、監視中です。」
    *   **Resolved**: 「復旧しました。ご迷惑をおかけしました。」
*   **Risk**: プロンプトインジェクションによる「ヘイトスピーチ大量生成」や、APIループによる「クラウド破産」の発生。
*   **The Global Kill Switch Implementation (Zero Re-deploy)**:
    *   **Law**: 緊急停止スイッチは、再デプロイを必要としない **Edge Config** (Vercel) または **DB Flags** (`system_settings`) で実装されなければなりません。「コードを直してデプロイ」では遅すぎます。
    *   **Action**: `ai_enabled`, `billing_enabled` などのフラグを Middleware でチェックし、OFF時は即座に 503 を返す回路を組み込んでください。
*   **Solution**: コードの再デプロイを待たずに、一瞬でAI機能を全停止できる「緊急停止スイッチ (Global Kill Switch)」を実装してください。Edge ConfigやDBフラグ (`ai_enabled: false`) で制御します。
*   **ユーザーメール**:
    *   データ侵害等の重大インシデント時は、法務確認済みのテンプレートを使用し、迅速かつ透明性を持って通知します。
*   **The Status Page Component Health Matrix (コンポーネント別稼働監視)**:
    *   **Law**: ステータスページでは、サービスを構成するコンポーネントごとの稼働状況を個別にリアルタイム表示しなければなりません。「全体的に障害」ではなく「どのコンポーネントが影響を受けているか」をユーザーが一目で判別できる状態を維持してください。
    *   **Component Categories (最低限のカテゴリ)**:

        | カテゴリ | 監視対象（例） | ヘルスチェック方法（例） |
        |:--------|:-------------|:---------------------|
        | **Webアプリケーション** | フロントエンド / Edge | HTTP 200応答確認 |
        | **API** | バックエンドAPI | レスポンスタイム ≦ 閾値 |
        | **データベース** | プライマリDB | 接続プール使用率 |
        | **認証** | 認証基盤 | ログイン成功率 |
        | **メディア配信** | CDN / ストレージ | CDNヒット率 |
        | **外部連携** | 決済 / AI / メール等 | 応答成功率 |

    *   **Status Levels**: ステータスは以下の4段階で表示し、ユーザーが影響度を直感的に理解できるようにしてください。
        *   `Operational` — 正常稼働
        *   `Degraded Performance` — 性能劣化
        *   `Partial Outage` — 一部障害
        *   `Major Outage` — 全面障害
*   **The Incident Notification Template (障害通知テンプレート)**:
    *   **Law**: ユーザーへの障害通知は、慌てて自由文で書くのではなく、事前に用意されたテンプレートに基づいて発信しなければなりません。情報の欠落や不適切な表現を防ぎ、信頼を維持してください。
    *   **Initial Report Template (初報テンプレート)**:

        ```
        【障害情報】{コンポーネント名}に関するお知らせ

        いつも{サービス名}をご利用いただき、ありがとうございます。
        ただいま{コンポーネント名}において、{症状}が発生しております。

        ■ 発生時刻: {YYYY年MM月DD日 HH:MM}
        ■ 影響範囲: {影響を受ける機能}
        ■ 現在の状況: 原因調査中 / 復旧作業中

        ご不便をおかけし、誠に申し訳ございません。
        復旧次第、本ページにてお知らせいたします。
        ```

    *   **Required Fields (必須フィールド)**: 障害通知（初報・続報・復旧報）には、少なくとも以下のフィールドを含めてください。
        *   **発生時刻**: 障害が最初に検知された正確な日時
        *   **影響範囲**: 影響を受ける機能・ユーザー層の特定
        *   **現在の状況**: 「調査中 / 原因特定済み / 復旧作業中 / 復旧完了」の進捗
        *   **次回更新予定**: 続報の予定時刻（沈黙を防ぐ）

## 3. データ侵害プロトコル (Data Breach Protocol)
*   **初動対応 (Stop the Bleeding)**:
    *   データ侵害や不正アクセスが疑われる場合、迷わず「止血」を行います。APIキーの無効化、サーバーのシャットダウン、アクセスの遮断を最優先します。
    *   **Assess (評価)**: 止血後、影響範囲（漏洩したデータ、対象ユーザー数）を冷静に特定します。
    *   **Notify (通知)**: 法的要件（GDPRの72時間ルール等）に従い、ユーザーおよび規制当局への通知を行います。

## 4. 災害復旧 (Disaster Recovery)
*   **RPO / RTO**:
    *   **RPO (Recovery Point Objective)**: 目標復旧時点。どの時点までのデータ損失を許容するか（例: 最大1時間前まで）。
    *   **RTO (Recovery Time Objective)**: 目標復旧時間。ダウンしてから何時間以内に復旧させるか（例: 4時間以内）。
*   **クラウド冗長化**:
    *   AWS S3のクロスリージョンレプリケーションや、データベースのマルチAZ配置を利用し、特定のリージョンがダウンしてもサービスを継続（または早期復旧）できるようにします。

## 5. 事業継続計画 (BCP)
*   **キーマンリスクの排除**:
    *   「特定のエンジニアしか触れないコード」を排除します。ドキュメント化とコードの標準化により、誰でも復旧作業を行える状態を維持します。
    *   **緊急連絡網**: Slack以外の緊急連絡手段（電話、別チャットツール）を確保します。

### 5.1. 委託先障害連携プロトコル (The Vendor Incident Coordination Protocol)
*   **Law**: 外部委託先（クラウドプロバイダー、SaaS、決済サービス等）の障害が自社サービスに波及した場合の対応フローを事前に定義しなければなりません。「委託先の問題だから待つしかない」は事業継続の放棄です。
*   **Mandate**:
    *   **委託先障害時の対応フロー**:
        1.  **検知**: 委託先のステータスページ監視 + 自社モニタリングによる異常検知
        2.  **影響評価**: 自社サービスのどの機能・ユーザーセグメントに影響があるかを即時評価
        3.  **ユーザー通知**: 自社ステータスページに影響範囲と状況を掲載（委託先の復旧待ちでも沈黙禁止）
        4.  **代替策実行**: フォールバック手段（代替サービスへの切替、機能制限モード等）を実行
        5.  **復旧確認**: 委託先復旧後、自社サービスの正常動作を検証
    *   **委託先BCP確認義務**: 重要な委託先（High リスク等級：`60_security_privacy.md` §10.1参照）に対し、**年1回**以下を確認してください。
        *   事業継続計画（BCP）の整備状況
        *   RPO/RTOの設定値とSLAとの整合性
        *   直近1年間のインシデント履歴と復旧実績
        *   冗長化・バックアップ体制
    *   **代替委託先の事前準備**: 単一障害点（SPOF）となる委託先については、代替サービスの選定と最低限の接続検証を事前に完了しておいてください。「障害が起きてから代替を探す」のでは遅すぎます。
*   **Cross-Reference**: §7 (Incident Runbooks), `60_security_privacy.md` §10 (Vendor Security Management)

## 6. ポストモーテムと学習 (Post-Mortem & Learning)
*   **The Blameless Post-Mortem Protocol**:
    *   **Law (No Witch Hunts)**: インシデント発生時、個人の責任を追及することを**厳禁**とします。「なぜ彼が間違えたか」ではなく「なぜシステムが彼に間違いを許したか」を問います。これを「責任追及なし（Blameless）」の原則とします。
    *   **Action (5 Whys)**: 「なぜ？」を5回繰り返し、真の根本原因（Root Cause）に到達してください。「ヒューマンエラー」は原因ではなく、調査の出発点に過ぎません。
    *   **Output (Systemic Fix)**: ポストモーテムの完了条件は、再発を物理的に防ぐための **「新しい自動テスト」** または **「Lintルールの追加」** がPRとして提出されることです。「気をつける」「マニュアルを修正する」という対策は認めません。
*   **The Post-Mortem Template (事後検証テンプレート)**:
    *   **Law**: Post-Mortemレポートには、以下の必須セクションを全て含めなければなりません。
    *   **Required Sections**:

        | セクション | 内容 |
        |:---------|:-----|
        | **概要** | 何が起きたか（1-2文） |
        | **タイムライン** | 発生→検知→初動→解決の時系列 |
        | **影響範囲** | 影響ユーザー数、影響時間、データ損失の有無 |
        | **根本原因** | 技術的原因（コード変更、設定ミス、外部依存等） |
        | **改善アクション** | 再発防止策（具体的なタスク、担当者、期限） |
        | **教訓** | プロジェクト教訓ログに追記すべき内容 |

    *   **Blameless Culture**: Post-Mortemは**個人を責めず、システムとプロセスの改善**に集中してください。

## 7. インシデントランブック (Incident Runbooks)
*   **The Incident Runbook Framework**:
    *   **Law**: 主要な障害シナリオ（データベース障害、CDN障害、決済システム障害等）に対して、具体的な手順書（ランブック）を事前に作成しなければなりません。ランブックがないインシデント対応は「手探りの消火活動」であり、復旧時間を著しく長期化させます。
    *   **Runbook Template**: 各ランブックは以下の構造に準拠してください。

        | ステップ | 内容 |
        |:--------|:-----|
        | **1. 確認** | 障害発生の事実確認（ステータスページ・監視ツール） |
        | **2. 診断** | 原因の切り分け（メトリクス・ログの確認） |
        | **3. 緩和** | 影響範囲の縮小（設定変更・リソース拡張） |
        | **4. 代替** | フォールバック手段への切替（冗長化・代替サービス） |
        | **5. 通知** | ステータスページ・ユーザーへの通知 |
        | **6. 復旧** | 正常状態への復元・Post-Mortem作成 |

    *   **Coverage（最低限のシナリオ）**: 以下のシナリオについて、プロジェクト固有のランブックをBlueprintで定義してください。
        *   **Database障害**: 接続プール枯渇、レプリカ遅延、バックアップからのリストア
        *   **CDN/メディア配信障害**: キャッシュパージ、直接配信への切替
        *   **決済システム障害**: メンテナンスバナー表示、未処理トランザクションの整合
        *   **認証基盤障害**: セッション管理のフォールバック
    *   **Cross-Reference**: §1 (Incident Response), §4 (Disaster Recovery), §6 (Post-Mortem)
