# I. IP所有権・権利帰属（§1-§5）

---

## §1. IP所有権の基本原則

### Rule 63.001: 職務著作の明確化
- **必須要件**: 従業員が業務中に作成した成果物（ソースコード、設計書、アルゴリズム、学習済みモデル等）が**職務著作（Work for Hire）**であることを就業規則に明記すること
- **入社時手続き**: 入社日に以下を必ず締結する
  - **知的財産譲渡契約書（IP Assignment Agreement）**
  - **発明事前開示契約書（Prior Invention Disclosure）**: 入社前の既存発明を列挙し除外
  - **秘密保持契約書（NDA）**: 在職中・退職後の秘密保持義務

### Rule 63.002: IP譲渡の完全性（グローバル法域対応）
- 全ての著作物・発明・意匠・データベース権が法人に帰属することを明示
- **属地主義への対応**: 各法域で有効なIP譲渡条項を使用
  - **日本**: 著作権法第15条（職務著作）+ 特許法第35条（職務発明）の適用確認
  - **米国**: Work for Hire Doctrine + Assignment Agreement の併用（カリフォルニア州 Labor Code §2870 の除外規定に注意）
  - **EU**: 各加盟国の著作者人格権への配慮（ドイツ UrhG §7: 人格権放棄不可、フランス Code IP L.121-1: 同様）
  - **中国**: 著作権法第18条（職務著作）+ 専利法第6条（職務発明）— 法人帰属には明示的契約が推奨
  - **韓国**: 発明振興法第10条（職務発明）— 予約承継の有効性確認
  - **インド**: Patents Act §§ 19-21 — 雇用契約での明示的譲渡が必要

### Rule 63.003: 対価・報酬制度
- 日本の職務発明（特許法第35条）に基づく**相当の対価**の算定基準を社内規程で策定
- 発明報奨制度の設計: 出願時・登録時・実施時の段階的報奨
- 対価紛争リスク低減のため、算定プロセスの透明性を確保（従業者との協議手続きの文書化）
- **ドイツ ArbEG（従業者発明法）**: 報酬算定は法定ガイドラインに準拠（ライセンス類推法が標準）
- **中国**: 職務発明報酬の最低基準（専利法実施細則第77-78条）への準拠確認

## §2. コントリビューター権利管理

### Rule 63.010: CLA（Contributor License Agreement）の義務化
- 外部コントリビューター（フリーランス、業務委託、副業エンジニア）がプロジェクトに参加する場合、コード提出前に必ずCLAを締結
- **CLA種別の選定**:
  - **Individual CLA**: 個人コントリビューター向け
  - **Corporate CLA**: 法人に所属するコントリビューター向け（会社の承認が必要）
- CLAの管理ツール: CLA Assistant（GitHub App）等を導入し、プルリクエスト時にOIDC連携で自動検証

### Rule 63.011: 業務委託契約のIP条項
- **必須条項チェックリスト**:
  - [ ] 成果物のIP所有権が発注者に帰属する旨の明示
  - [ ] 著作者人格権の不行使特約（日本法域）
  - [ ] 第三者IPの不使用保証（Third-Party IP Warranty）
  - [ ] AI生成コードの使用有無・範囲の開示義務
  - [ ] 納品物のソースコード・設計資料の引渡し義務
  - [ ] 秘密保持義務（存続期間の明示: 最低3年、営業秘密は無期限）
  - [ ] 競業避止義務（合理的な範囲で — カリフォルニア州では原則無効に注意）

### Rule 63.012: OSSコントリビューションポリシー
- 自社エンジニアが業務時間中にOSSに貢献する場合のポリシーを策定
- **承認フロー**: マネージャー + 法務の事前承認を必須とする
- 自社の営業秘密やプロプライエタリコードがOSSに混入しないことを保証するレビュー手順
- **Inbound/Outbound分離**: OSSから受け入れるライセンス（Inbound）と自社が付与するライセンス（Outbound）の整合性を確認

## §3. 共同開発・JV時のIP取り決め

### Rule 63.020: バックグラウンドIP/フォアグラウンドIP分離
- **バックグラウンドIP**: 共同開発前から各当事者が保有する既存IP → 元の所有者に帰属
- **フォアグラウンドIP**: 共同開発により新たに創出されたIP → 契約で帰属を事前に決定
- **サイドグラウンドIP**: 共同開発に関連して一方当事者が独自に開発したIP → 開発者に帰属
- **改良IP（Improvement IP）**: バックグラウンドIPに対する改良 → 帰属・ライセンス条件を事前に合意

### Rule 63.021: 共同IP管理の契約要件
- **必須取り決め事項**:
  - 発明者認定の基準（各法域の発明者要件に準拠）
  - 特許出願の費用負担割合と出願・維持の意思決定プロセス
  - ライセンス供与の可否と条件（サブライセンス権を含む）
  - 共同開発終了後のIP利用制限とライセンスバック
  - 紛争解決条項（管轄・準拠法・ADR優先条項）
  - 上場・M&A時の共同IPの取扱い（優先交渉権・ROFR/ROFOの設定）

## §4. 従業員退職時のIP保護

### Rule 63.030: 退職時チェックリスト（IP観点）
- [ ] 退職日までに全ての発明開示を完了
- [ ] 会社所有のコード・資料・データへのアクセス権限を即日失効
- [ ] 個人デバイスからの会社データ削除を確認（MDM/DLP連携 + 削除証明書の取得）
- [ ] 退職後の秘密保持義務の再確認書面を取得
- [ ] 競業避止・勧誘禁止条項の範囲と期間の確認
- [ ] AI Copilot / LLMツールの業務アカウント解除（学習データへの社内情報混入リスク排除）
- [ ] Git履歴・コードレビュー記録の保全（退職者の貢献範囲の証拠保全）

### Rule 63.031: Garden Leave（自宅待機条項）
- 機密性の高い技術を扱うキーパーソンに対し、退職前の一定期間（3-6ヶ月）を自宅待機とする条項の検討
- **法域別の有効性確認**:
  - 米国カリフォルニア州: 競業避止義務が原則無効（Bus. & Prof. Code §16600）→ Garden Leaveで代替
  - 米国その他の州: 合理的範囲内で有効（期間・地理・業種の制限）
  - 日本: 合理的範囲内で有効（判例法理 — 対価の支払いが有効性の重要要素）
  - EU: 加盟国ごとに異なる（ドイツ: HGB §74以下、フランス: 対価必須）

## §5. IP権利帰属の自動監査

### Rule 63.040: 自動監査の仕組み（CI/CD連携）
- **コミット署名検証**: 全コミットに`Signed-off-by`を義務付け（DCO: Developer Certificate of Origin）
- **CLA検証ゲート**: PR作成時にCLA署名の有無を自動チェックし、未署名PRをマージブロック

```yaml
# GitHub Actions: CLA Check Example
name: CLA Check
on: [pull_request]
jobs:
  cla-check:
    runs-on: ubuntu-latest
    steps:
      - uses: contributor-assistant/github-action@v2
        with:
          path-to-signatures: 'signatures/cla.json'
          branch: 'main'
          remote-organization-name: 'your-org'
```

### Rule 63.041: 定期IP監査
- **四半期ごと**に以下を監査:
  - 新規入社者のIP譲渡契約の締結状況
  - 外部委託契約のIP条項の適切性
  - CLA未署名コントリビューションの検出
  - OSS利用状況とライセンス適合性（→ [602_oss_compliance.md](../security/200_oss_compliance.md) 参照）
  - AI生成コードの使用状況と権利帰属の記録確認

# II. 特許戦略（§6-§12）

## §6. 特許出願 vs 営業秘密の判断フレームワーク

### Rule 63.050: 意思決定マトリクス

| 判断基準 | 特許出願を選択 | 営業秘密を選択 |
|---|---|---|
| **リバースエンジニアリング** | 容易（ハードウェア/UI） | 困難（サーバーサイドアルゴリズム） |
| **技術の寿命** | 長期（20年以上有用） | 短期（急速に陳腐化） |
| **競合の模倣リスク** | 高い | 低い（自社のみ実現可能） |
| **保護期間** | 20年（特許権） | 無期限（秘密が維持される限り） |
| **公開の可否** | 公開可能 | 公開不可（競争優位のSecret Sauce） |
| **エンフォースメント** | 訴訟で権利行使可能 | 漏洩時の立証が困難 |
| **コスト** | 出願・維持費が高額 | 管理コストが中心 |
| **AI学習リスク** | 公開後にAIの学習データに組込まれるリスク | 非公開のため学習データ混入リスクが低い |

### Rule 63.051: 決定プロセス
1. 発明開示書（Invention Disclosure Form）の提出
2. 特許委員会（Patent Committee）によるレビュー
3. 上記マトリクスに基づく判定
4. 決定記録の保存（営業秘密選択時は特にアクセス制限記録を残す）
5. **定期再評価**: 技術環境の変化に応じて営業秘密 → 特許出願への戦略転換を検討

## §7. 防衛的特許戦略

### Rule 63.060: Defensive Publishing（防衛的公開）
- 全てを特許出願するのではなく、**他社に特許を取らせない**ために技術ブログ・学術論文・プレプリントサーバー等で先行技術として公開する戦略
- **公開媒体の推奨**:
  - 技術ブログ（自社エンジニアリングブログ）
  - arXiv / SSRN 等のプレプリントサーバー
  - Linux Foundation の Defensive Publication Program
  - IP.com Prior Art Database
- 公開日の証拠を確保（タイムスタンプ付きアーカイブ、Wayback Machine登録）

### Rule 63.061: Defensive Patent License（DPL）/ LOT Network
- **DPL**: 相互不攻撃協定 — DPL参加者間では特許訴訟を起こさない
- **LOT Network**: 特許がPAE（Patent Assertion Entity）に渡った場合にメンバー企業にライセンスが自動付与される仕組み
- **Open Invention Network (OIN)**: Linux関連特許の相互不攻撃（2,000+企業参加）
- スタートアップ・中小企業は加入を強く推奨

### Rule 63.062: 特許不侵害保険
- 特許訴訟（とくにPAE/NPE訴訟）への防衛策として**IP保険**の加入を検討
- **カバレッジ**: 訴訟費用、和解金、ライセンス料（→ §45 IP保険で詳細）

## §8. 特許ポートフォリオ管理

### Rule 63.070: 定期ポートフォリオ監査
- **年次監査項目**:
  - 各特許の事業との整合性（アクティブ製品/サービスとの対応マッピング）
  - 維持費対効果の評価 → 不要特許の放棄（Pruning）判断
  - 競合他社のポートフォリオとのGap分析
  - 地理的カバレッジの評価（主要市場での出願有無）
  - **AI活用のポートフォリオ分析**: Patsnap / Clarivate / LexisNexis PatentSight を活用した定量的評価

### Rule 63.071: ポートフォリオ可視化
- 特許マップ（Patent Landscape Map）を年次で作成
- 技術領域×事業領域のマトリクスで保有特許を俯瞰
- **ヒートマップ分析**: 技術クラスター別の特許密度・競合比較の可視化

### Rule 63.072: 継続的出願戦略（Continuation Strategy）
- 核となる発明に対し、分割出願（Divisional）・継続出願（Continuation）・CIP（Continuation-in-Part）を活用
- クレーム範囲の拡張・防衛的クレーム追加により保護範囲を最大化
- **ポートフォリオ・バランス**: 攻撃的特許（収益化/ライセンシング用）と防衛的特許（FTO確保用）のバランスを維持

## §9. 標準必須特許（SEP）と FRAND

### Rule 63.080: SEP宣言時の義務
- 標準化団体（ETSI, IEEE, IETF等）への技術提案時にSEP該当性を誠実に開示
- **FRAND条件**: Fair, Reasonable, And Non-Discriminatory な条件でのライセンス供与義務

### Rule 63.081: EU SEP規則の動向（2025年最新）
- **EU SEP規則の撤回**: 2025年10月、欧州委員会は「合意の見通しなし」として提案を正式撤回
- 撤回された提案の主要内容（今後再提案の可能性あり）:
  - SEPレジストリの設置（EUIPO管轄）
  - 必須性チェック（Essentiality Check）
  - 集計ロイヤリティに関する専門家意見
  - FRAND条件決定のための調停プロセス
- **UPC PMAC（Patent Mediation and Arbitration Centre）**: SEP紛争解決の専門フォーラムとして2025年末にWebサイト公開 — 今後の動向を注視
- **戦略的対応**: 規制環境の不確実性に備え、SEPライセンスプログラムの文書化・FRAND条件の根拠資料を整備

### Rule 63.082: 標準化活動参加時のIP方針
- 標準化団体参加前にIP方針を社内で策定
- エンジニアへの教育: 標準化会議での発言と特許権行使の関係
- **Anti-Hold-Up / Anti-Hold-Out**: ライセンサー・ライセンシー双方の義務を理解し、交渉の適正化を図る

## §10. ソフトウェア特許の特殊考慮

### Rule 63.090: 特許適格性の判断（米国）
- **Alice/Mayoテスト（2ステップ分析）**:
  - Step 1: クレームが抽象的アイデア・自然法則・自然現象に向けられているか
  - Step 2: 「inventive concept」（発明的概念）が抽象的アイデアを上回る追加的要素を含むか
- **戦略**: 技術的改善（Technical Improvement）を明確にクレームに記載
  - 計算リソースの効率改善（メモリ使用量削減、処理速度向上）
  - 特定のハードウェアとの連携による技術的効果
  - 従来技術の具体的な技術的課題に対する技術的解決策
- **2025年以降の司法動向**: 特許審判部（PTAB）と連邦巡回裁判所のSoftware Patent統計を四半期監視

### Rule 63.091: 日本・欧州でのソフトウェア特許
- **日本**: 自然法則の利用性を満たすこと（ソフトウェアによる情報処理がハードウェア資源を利用）
  - 特許庁「コンピュータソフトウェア関連発明審査基準（2023年改訂版）」に準拠
  - **AI関連発明**: 学習済みモデルを含む発明はハードウェア資源との結合で適格性を確保
  - **推奨戦略**: 「〇〇処理をコンピュータが実行する方法」ではなく、「〇〇サーバー」として装置クレームを主クレームに
- **欧州（EPO）**: 技術的効果（Further Technical Effect）の主張が重要 — COMVIK approach の適用
  - AI/MLアルゴリズムは「Further Technical Effect」の具体的記述（精度向上・処理時間短縮等の定量的効果）で適格性確保
- **統一特許裁判所（UPC）**: 2023年発足 → 欧州統一特許の活用を検討（→ §12で詳細）
- **中国**: 技術的解決策に焦点を当てたクレーム設計（CNIPA審査基準2023準拠）

### Rule 63.092: クロスライセンスと特許プール戦略
- **クロスライセンス（Cross-License）**:
  - 競合他社との特許相互ライセンスにより訴訟リスクを低減し、R\&D加速の原資を確保
  - **判断基準**: 自社ポートフォリオの「攻撃価値」vs 相手方ポートフォリオの「防衛価値」を定量比較してから交渉
  - **典型条件**: 非独占・無償・将来取得特許を含む双方向ライセンス + 改良発明の報告義務
  - **注意点**: 独占禁止法（反トラスト法）の観点でのリスク評価（水平的競合者間の範囲設計に注意）
- **特許プール（Patent Pool）**:
  - MPEG-LA / Via Licensing / Avanci等の特許プールへの参加を検討（SEP保有の場合は特に重要）
  - 参加前にプール内のライセンス条件・ランニングロイヤリティ料率を精査
- **パテント・トローリング（PAE）対策**:
  - LOT Network + OINの二重加入を推奨（→ §7 Rule 63.061参照）
  - PAEからの警告書受領時の72時間対応プロトコルを事前策定（→ §43参照）

## §11. 意匠権（デザインパテント）戦略

### Rule 63.095: 意匠権の戦略的活用
- **保護対象**: GUI/UIデザイン、アイコン、フォント、製品外観、パッケージデザイン
- **特許との併用**: 技術的発明は特許で、外観・UIは意匠権で二重保護を構築
- **出願戦略**:
  - **日本**: 意匠法（2020年改正で画像デザイン・建築物・内装の保護拡大）
  - **米国**: Design Patent（35 USC §171）— 15年保護、維持費不要
  - **EU**: 登録共同体意匠（RCD）— EUIPO経由、3年非登録意匠権（UCD）も活用
  - **ハーグ協定**: 国際出願で複数法域を一括カバー

### Rule 63.096: GUI/UIデザイン保護のベストプラクティス
- **記録習慣**: デザインの作成日・作成者・バージョン履歴を文書化
- **公開前出願**: UIデザインは公開前に出願すること（新規性要件 — 特に日本・EU）
- **部分意匠**: 製品全体ではなく特徴的なUI要素のみを部分意匠として出願し、権利範囲を最適化
- **アニメーション/トランジション**: 動的意匠（日本）やAnimated Design Patent（米国）の活用を検討

## §12. EU統一特許（UP）・統一特許裁判所（UPC）実務

### Rule 63.098: UPCの概要と戦略的活用
- **2023年6月発足**: 参加国18ヶ国、EU GDP の約80%をカバー
- **2025年2月時点**: 48,000件以上のUP登録、752件の訴訟提起
- **メリット**:
  - 単一出願で最大25ヶ国の特許保護を一括取得
  - 汎欧州的な差止命令の取得が可能
  - SEP紛争の専門フォーラム（PMAC）
- **リスク**:
  - 単一の無効判決で全参加国の保護を喪失（Central Revocation）
  - Long-arm jurisdiction（BSH v Electrolux, 2025年CJEU判決）による予期しない管轄リスク

### Rule 63.099: Opt-out戦略
- **移行期間（〜2030年頃）**: 既存の古典的欧州特許をUPC管轄から除外するOpt-out申請が可能
- **判断基準**:
  - **Opt-outが望ましい場合**: 重要な特許で中央無効判決のリスクを回避したい場合
  - **UPC管轄維持が望ましい場合**: 汎欧州的エンフォースメントを求める場合
- **Sunrise Period**: Opt-outは取消可能だが、一度当該特許で訴訟が提起された場合は変更不可
- **費用**: UPC裁判費用は2026年1月より平均33%値上げ — コスト見積もりに反映すること

# III. 営業秘密・機密情報（§13-§16）

---

## §13. 営業秘密の定義と保護体制

### Rule 63.100: 営業秘密の三要件
- 以下の3要件を全て満たすことが法的保護の前提:
  1. **秘密管理性**: アクセス制限・「Confidential」表示等の管理措置
  2. **有用性**: 事業活動に有用な技術上または営業上の情報
  3. **非公知性**: 公然と知られていない情報

### Rule 63.101: テクニカルコントロール
- **アクセス制御**:
  - Secret Sauceのソースコードは独立リポジトリで管理、Need-to-Know原則でアクセス権を付与
  - IAMロールによる最小権限アクセス
  - アクセスログの監査証跡を最低7年保存
- **暗号化**:
  - 保管時暗号化（AES-256以上）
  - 転送時暗号化（TLS 1.3）
  - 鍵管理はHSM/KMSを使用
- **DLP（Data Loss Prevention）**:
  - 機密コードのクリップボード・外部転送を検知・ブロック
  - 電子メール・クラウドストレージへの機密ファイルアップロード制御
  - AI Copilot/ChatGPT等のLLMへの機密コード貼付を検知・ブロック

### Rule 63.102: 法的保護の根拠
- **日本**: 不正競争防止法（第2条第6項）— 2024年改正で限定提供データの保護強化
- **米国**: Defend Trade Secrets Act（DTSA, 2016）+ 各州法（UTSA）
  - **DTSA Whistleblower Immunity**: 2016年5月11日以降の全契約にImmunity通知条項を含めること（18 USC §1833(b)）
- **EU**: Trade Secrets Directive（2016/943）— 各加盟国法への移行完了
- **中国**: 反不正当競爭法 — 営業秘密侵害の立証責任転換（2019年改正）
- 各法域の手続き要件（証拠保全、暫定的差止命令の要件）を事前に把握

---

## §14. 秘密保持契約（NDA）管理

### Rule 63.110: NDAの標準テンプレート
- **相互NDA（Mutual NDA）**: 両当事者が秘密情報を開示する場合
- **片務NDA（One-Way NDA）**: 一方のみが開示する場合（主に採用面接・投資家DD）
- **必須条項**:
  - [ ] 秘密情報の定義（包括型 vs 列挙型）
  - [ ] 除外事項（公知情報、独自開発、第三者から合法的に取得等）
  - [ ] 秘密保持期間（最低3年、営業秘密は無期限推奨）
  - [ ] 開示目的の限定
  - [ ] Return/Destroy条項（契約終了後の返却・破棄義務 + 破棄証明書）
  - [ ] エクイタブルリリーフ条項（差止請求の明示）
  - [ ] 準拠法と紛争解決（裁判管轄/仲裁）
  - [ ] **AIツールへの入力禁止条項**: 秘密情報をAI/LLMの入力データとして使用することを明示的に禁止

### Rule 63.111: NDAライフサイクル管理
- CLM（Contract Lifecycle Management）ツールで一元管理
- NDA期限のアラート通知（90日前/30日前）
- 年次棚卸し: 失効済み・不要NDAの整理

---

## §15. リモートワーク環境での営業秘密保護

### Rule 63.120: リモートワーク固有のリスク対策
- **物理的セキュリティ**:
  - スクリーンプライバシーフィルターの義務化
  - 公共Wi-Fi使用禁止（VPN必須）
  - 家庭内での画面共有・会話の盗聴リスクへの啓蒙
- **技術的制御**:
  - VDI（仮想デスクトップ）の導入（機密コードへのアクセスはVDI経由のみ）
  - BYOD端末にはMDM/MAMを必須適用
  - 機密情報のローカル保存禁止
  - 画面キャプチャ/録画の制限（DRM連携）
- **手続き的制御**:
  - リモートワーク開始時の誓約書取得
  - 年次の情報セキュリティ研修の受講義務
  - ホームオフィスのセキュリティ監査（自己申告チェックリスト）

## §16. GenAI環境での営業秘密リスク

### Rule 63.125: AI/LLMへの機密情報漏洩リスク
- **リスクシナリオ**:
  - エンジニアがAI Copilot/ChatGPTにプロプライエタリコードを貼付し、コード改善を依頼
  - 営業担当がLLMに顧客リスト・価格戦略を入力して分析を依頼
  - 経営陣がAIにM&A戦略・事業計画を入力してレビューを依頼
- **「合理的措置」の再定義**: AI時代においてDTSA/不正競争防止法の「合理的措置」にはLLMへの入力制御が含まれると解される（2025年判例動向）

### Rule 63.126: GenAI営業秘密保護ポリシー
- **Tier 1（禁止）**: 営業秘密・Secret Sauceは外部AIサービスへの入力を完全禁止
- **Tier 2（制限）**: 機密情報は Enterprise API（データ学習に使用しない契約のサービスのみ）で利用可
- **Tier 3（許可）**: 公開情報・一般的なコードパターンは制限なし
- **技術的制御**:
  - AIゲートウェイプロキシ（Nightfall AI / Harmonic Security等）によるリアルタイム検知・ブロック
  - エンドポイントDLPの拡張（AI Webアプリへのデータ送信監視）
  - 社内LLM/オンプレミスモデルの活用推進（機密データの外部送信回避）

```yaml
# DLP Policy: AI Service Upload Detection
rules:
  - name: "Block Trade Secret Upload to AI"
    condition:
      destination_domains:
        - "api.openai.com"
        - "generativelanguage.googleapis.com"
        - "api.anthropic.com"
      content_classification: ["TRADE_SECRET", "CONFIDENTIAL"]
    action: BLOCK
    notification: security-team@company.com
```

## §17. 商標戦略と出願管理

### Rule 63.130: 商標出願の基本方針
- **出願タイミング**: プロダクト名・サービス名の決定後、公開前に出願
- **出願前調査**:
  - 各国の商標データベース（J-PlatPat, USPTO TESS, EUIPO TMview, WIPO Global Brand Database等）で類似商標を調査
  - 共通法上の商標権（Common Law Rights）も確認（米国）
  - AI商標検索ツール（TrademarkVision / Corsearch AI等）による画像類似性チェック
- **国際出願**:
  - マドリッド議定書（マドプロ）を活用した一括出願
  - 事業展開予定国を優先し、少なくとも主要5市場をカバー
  - **ニース分類**: 主要な商品・サービス分類を網羅的に指定（特にソフトウェア: 第9類、SaaS: 第42類）

### Rule 63.131: 商標監視と使用管理
- **ブランド監視サービス**: 新規出願・ドメイン登録を自動監視（CompuMark / Corsearch / TrademarkNow等）
- **使用証拠の管理**:
  - 商標の使用実績（広告、パッケージ、Web画面）をスクリーンショット・タイムスタンプ付きで定期保存
  - 日本: 不使用取消審判（3年不使用）への備え
  - 米国: Section 8/9 Affidavit の期限管理（登録後5-6年目、9-10年目）

### Rule 63.132: 商標ガイドラインの社内展開
- ブランドガイドラインにIP保護の観点を含める
  - ™/®マークの正しい使用法
  - サードパーティによる商標使用時のライセンス条件
  - ジェネリサイド（商標の普通名称化）防止策（使用時は常に形容詞として使用し、動詞化を避ける）

- 退職者のアクセス権限の即時無効化手順
- **デジタルアセット台帳**: 全デジタルアセット（SNS、ストアアカウント、暗号鍵、証明書）を一元管理台帳に記録

# V. 著作権・データ権利（§20-§22）

---

## §20. 著作権登録戦略

### Rule 63.155: 著作権登録の戦略的活用
- **米国（USCO）**:
  - 著作権登録は訴訟提起の**前提条件**（17 USC §411(a)）— 未登録では連邦裁判所に提訴不可
  - 登録タイミングによる**法定損害賠償（Statutory Damages）**の可否:
    - 侵害発生前 or 公開後3ヶ月以内に登録 → 法定損害賠償（最大$150,000/著作物）+ 弁護士費用請求可
    - 侵害発生後に登録 → 実損害賠償のみ
  - ソフトウェアの著作権登録: ソースコード最初/最後の25ページの提出（Trade Secret部分の編集申請可）
- **日本**: 著作権の発生に登録不要（無方式主義）だが、以下で登録が有用:
  - プログラムの著作物の登録（文化庁）— 権利推定の効果
  - 第一公表年月日の登録 — 保護期間の起算点の証明
- **国際**: ベルヌ条約による無方式主義が基本だが、訴訟戦略上、主要法域での登録を推奨

### Rule 63.156: ソフトウェア著作権とコード保護
- **保護範囲**: ソースコード・オブジェクトコードは著作物として保護（API構造はOraclevsGoogle判決で争点）
- **著作権 vs 特許**: アルゴリズム・機能的側面は著作権で保護不可 → 特許による保護を検討
- **コード規約の文書化**: 自社コーディングスタイルの独自性・創作性を主張するための根拠を整備

---

## §21. データ権利・データベース権

### Rule 63.160: データに対する法的保護
- **EU suigenerisデータベース権**: 実質的投資を行ったデータベースの保護（Directive 96/9/EC）— 15年間
- **EU Data Act（2024年施行開始）**: IoTデータへのアクセス権・データポータビリティ義務
  - データホルダーのスイッチングコスト削減義務
  - 公的機関へのデータアクセス権（緊急時）
  - 不公正な契約条項に対する規制
- **日本**: 限定提供データ（不正競争防止法第2条第7項）— 2024年改正で保護強化
- **中国**: データ安全法（DSL）+ Privacy Laws（PIPL）のデータ管理権
- **データの財産権**: 現時点で多くの法域でデータ自体に所有権は認められないが、契約・営業秘密・データベース権による間接保護が可能

### Rule 63.161: データライセンスと収益化
- **データライセンス契約の必須条項**:
  - 利用目的の限定（AI学習目的を含むか否かを明示）
  - 再配布・二次利用の制限
  - データ品質の保証水準
  - 個人情報を含む場合のデータ保護条項（GDPR/APPI準拠）
  - 契約終了時のデータ返却・削除義務
- **データマーケットプレイス**: Snowflake Marketplace / AWS Data Exchange等での販売時のライセンス設計

## §22. IPトークン化・NFT・ブロックチェーン活用

### Rule 63.165: IPトークン化の概要と戦略
- **IPトークン化**: 特許・商標・著作権をブロックチェーン上のトークンとして表現し、取引・管理を効率化
- **主要ユースケース**:
  - **フラクショナルIP所有**: 特許権の持分をトークン化し、複数投資家による共同所有を実現
  - **ロイヤリティ自動分配**: スマートコントラクトによるライセンス収入の自動分配
  - **IP出自証明（Provenance）**: ブロックチェーンによるIP権利の不変的な所有権履歴追跡
  - **IP担保融資**: トークン化されたIPを担保とした融資（→ §41 IP担保融資参照）

### Rule 63.166: NFTとIP保護
- **NFTと著作権**: NFTの購入は著作権の譲渡を意味しない — 購入者が取得するのは通常「限定的ライセンス」
- **NFTライセンス設計**:
  - 商業利用権の範囲明示（CC0 / Creative CommonsベースまたはNFTライセンス2.0）
  - 二次販売時のロイヤリティ自動徴収（ERC-2981標準）
  - 派生著作物の作成可否
- **リスク管理**:
  - NFT化された著作物の原著作者の権利確認
  - ウォッシュトレーディング・マーケット操作の法的リスク
  - 各法域の暗号資産規制への準拠（日本: Financial Regulations、EU: MiCA規制）

### Rule 63.167: ブロックチェーンによるIP管理の将来展望
- **IPレジストリのオンチェーン化**: 特許庁・商標庁のレジストリをブロックチェーンに移行する構想（WIPO / EPO の研究プログラム）
- **C2PA（Coalition for Content Provenance and Authenticity）**: AI生成コンテンツの出自証明とブロックチェーンの融合
- **市場規模**: トークン化資産市場は2034年までに$30T規模と予測 — 早期のIP戦略策定が競争優位を構築

# VI. AI生成物のIP（§23-§26）

---

## §23. AI生成コンテンツの著作権

### Rule 63.170: ヒューマンオーサーシップ原則
- **米国（USCO 2025年1月 Part 2レポート）**:
  - AI単独で生成されたコンテンツには著作権は発生しない
  - 人間が表現要素を十分に決定した場合のみ著作権が認められる
  - プロンプトの入力のみでは著作権主張に**不十分** — 選択・配置・編集・修正の創作的判断が必要
  - AI生成素材を含む著作物の登録時はAI使用の**開示義務**あり
  - AI生成部分がde minimisを超える場合は明示的に除外が必要
  - 既存の著作権法の原則で対応可能とし、新規立法は不要との見解
- **日本（文化庁2025見解）**: AIを「道具」として利用し、人間が創作的寄与を行った場合に著作物性が認められる
- **EU（AI Act + Copyright Directive）**: AI生成物のラベリング義務、学習データの著作権尊重・Opt-out対応義務
- **判例**: Thaler v. Perlmutter（2023確定）— AI単独の著作者としての登録は不可

### Rule 63.171: AI生成コンテンツの社内ガイドライン
- **著作権取得が必要な成果物**（マーケティング素材、ブランドアセット等）には、人間が実質的な創作的判断を行い、その過程を記録
- **記録すべき事項**:
  - 人間が行った創作的選択（プロンプト設計、出力の選択、編集、配置、合成）
  - AI出力に対する人間の修正内容（変更箇所・変更量の定量記録）
  - 使用したAIツール・モデル名・バージョン
  - 最終成果物における人間創作部分とAI生成部分の区分
- AIプロンプトのみでは著作権主張に不十分であることを社内教育で徹底

### Rule 63.172: AI出力の第三者著作権侵害リスク
- AIが学習データに含まれる著作物を再現するリスクへの対策:
  - 類似性チェックツールの導入（盗作検知、画像類似度検索）
  - 高リスク用途（商用出版、広告）では著作権クリアランス調査を実施
  - AIツールのToS（利用規約）を確認し、出力物の権利帰属・免責条項を把握
- **2025年Fair Use判例動向**: 合法的に取得した著作物でのAI学習→Fair Use寄り、海賊版データでの学習→Fair Use否定の傾向

## §24. AI支援発明の特許出願

### Rule 63.175: ヒューマンインベンターシップ要件
- **USPTO 2025年11月改訂ガイダンス**:
  - 2024年2月ガイダンスを明示的に撤回
  - AIシステムは発明者として認められない（自然人のみ）
  - AIは実験器具・ソフトウェアツールと同等の「道具」として位置付け
  - 従来のPannu factorsはAI支援発明には適用しない新方針
  - 発明の「着想（Conception）」に対する人間の**実質的貢献**が必要
  - AI活用における誠実義務（Duty of Candor）の遵守 — AI支援の事実を適切に開示
- **日本**: 発明者は自然人に限定（特許法第36条）
- **欧州**: EPOはDabus判決（2025確定）でAI単独の発明者を拒否

### Rule 63.176: AI支援発明の出願戦略
- **文書化義務**: AIがどの段階で使用され、人間がどの段階で創作的判断を行ったかを発明ノートに詳細記録
- **クレーム設計**: AIモデル自体ではなく、AIを活用したプロセスや、AIの出力を人間が選択・改良した結果に焦点
- **開示義務**: 各国の出願時にAI使用を適切に開示（隠匿は後日の無効リスク）
- **AI発明ノートテンプレート**:
  - 技術課題の定義（人間が設定）
  - AIモデル/ツールの選定理由（人間が判断）
  - AIへの入力パラメータ・条件設定（人間が設計）
  - AI出力の評価・選択基準（人間が決定）
  - 選択された出力の改良・最適化（人間が実施）

### Rule 63.177: AI生成コードの特許適格性
- AI生成コードを含むソフトウェア発明の出願時:
  - コード自体ではなく、技術的課題に対する技術的解決策としてクレームを構成
  - AI生成部分と人間設計部分の切り分けを明確にする
  - ライセンスリスクの観点からも確認（→ [602_oss_compliance.md](../security/200_oss_compliance.md) §AI生成コード参照）

## §25. AI学習データの権利クリアランス

### Rule 63.180: 学習データのIP管理
- **著作権クリアランス**:
  - 学習データに第三者の著作物が含まれる場合のライセンス確認
  - EU AI Act: 汎用AIモデルプロバイダーは学習データの要約開示義務あり（2026年8月完全施行）
  - 日本著作権法第30条の4（情報解析用途の権利制限）の適用範囲確認
- **Opt-out対応**:
  - `robots.txt`の`ai`ディレクティブ、`ai.txt`（C2PA対応）でのクローリング制御
  - TDM（Text and Data Mining）のOpt-out通知への対応義務（EU DSM Directive第4条）
- **データ出自の追跡（Data Provenance）**:
  - 学習データセットのライセンス・出典・バージョンを記録するデータカタログの整備
  - C2PA/Coalition for Content Provenance and Authenticity への対応準備
  - **Model Card / Data Sheet**: モデルの学習データの構成・ライセンス状況を文書化

## §26. AI生成物のIPガバナンスフレームワーク

### Rule 63.185: 組織的AI-IPガバナンス
- **AI-IP委員会の設置**: 法務 + 技術 + 事業部門の横断チームによる意思決定
- **AI-IPポリシーの必須要素**:
  - AIツールの承認リスト（Approved AI Tool List）の管理
  - 用途別のリスクティアリング（著作権取得必要 vs 内部利用のみ vs 不要）
  - AI生成物の権利帰属ルール（デフォルトは法人帰属）
  - AI使用の記録・監査体制
  - インシデント対応（AI出力が第三者IPを侵害した場合のエスカレーションフロー）

### Rule 63.186: AI-IPリスクティアリングマトリクス

| リスクレベル | 用途例 | 著作権取得 | 人間創作性の要件 | 記録要件 |
|---|---|---|---|---|
| **High** | ブランドアセット、広告素材、商用出版物 | 必須 | 実質的な人間の創作的判断 | 詳細記録必須 |
| **Medium** | 内部プレゼン、ドキュメント、プロトタイプ | 推奨 | 編集・選択レベルの関与 | 要約記録 |
| **Low** | コード補完、テストデータ生成、内部分析 | 不要 | AI主導で可 | ツール名の記録のみ |

## §27. Exit戦略の類型と選定

### Rule 63.190: Exit類型の比較フレームワーク

| Exit類型 | 特徴 | IP観点の重要度 | 準備期間 |
|---|---|---|---|
| **M&A（買収）** | 最も一般的。事業シナジーが鍵 | ★★★★★ | 12-24ヶ月 |
| **IPO（株式公開）** | 最大の資金調達。規制準拠コスト高 | ★★★★☆ | 18-36ヶ月 |
| **SPAC** | IPOの迅速代替。SEC新規則で規制強化 | ★★★★☆ | 6-12ヶ月 |
| **Direct Listing** | 引受なし。既存株主の直接売出し | ★★★★☆ | 12-18ヶ月 |
| **セカンダリーセール** | 既存株主の部分的流動性確保 | ★★★☆☆ | 3-6ヶ月 |
| **IPライセンシング** | IP単体の収益化 | ★★★★★ | 要交渉 |
| **Acqui-hire** | チーム獲得が主目的 | ★★☆☆☆ | 1-3ヶ月 |

### Rule 63.191: Exit戦略とIP整備の関係
- **M&A**: 買収者はIPの完全性・排他性を最重視 → IP Due Diligenceの徹底が必須
- **IPO**: SEC/金融庁への知財リスク開示が必要 → リスクファクターの正確な記載
- **SPAC**: SEC 2024年最終規則でIPO同等の開示義務 → Target企業のco-registrant責任（→ §32詳細）
- **Direct Listing**: 引受審査がないため、IPリスクの自主的開示がより重要
- 事業の80%以上が無形資産で構成される技術企業では、IP評価がバリュエーションの核

## §28. IP資産の評価（Valuation）

### Rule 63.200: 評価手法の概要

| 手法 | 概要 | 適用場面 | 精度 |
|---|---|---|---|
| **コスト法** | IP開発にかかった実コスト（再構築コスト）を算定 | アーリーステージ、社内参考 | 低〜中 |
| **インカム法（DCF）** | IPから将来生み出される収益を割引現在価値で算定 | 収益実績のあるIP | 中〜高 |
| **マーケット法** | 類似IPの取引事例・ライセンス料率を参照 | 比較可能な事例がある場合 | 中 |
| **ハイブリッド法** | 上記の組み合わせ（2025トレンドで最推奨） | 複雑なIPポートフォリオ | 高 |
| **オプション法（Real Options）** | 将来の不確実性を考慮した柔軟な評価 | 研究段階のIP、AI/量子関連 | 中〜高 |

### Rule 63.201: 評価実施の要件
- **評価タイミング**: Exit準備開始時、資金調達ラウンド、年次IP棚卸し時
- **外部専門家の関与**: IP評価は第三者の知財専門家（公認会計士・弁理士・知財コンサルタント）が実施することを推奨
- **AIツールの活用**: 特許評価にAIアナリティクス（Patsnap / Clarivate / LexisNexis PatentSight / IPlytics等）を活用し、ポートフォリオの相対的価値を定量化

### Rule 63.202: ロイヤリティ料率ベンチマーク
- 業界標準のロイヤリティ料率を参考に算定:
  - ソフトウェア: 10-20%
  - 半導体: 3-5%
  - 医薬品: 15-25%
  - SaaS/クラウド: 15-30%
  - AI/MLモデル: 10-25%（新興分野、事例蓄積中）
- **25%ルール**: 利益の25%をロイヤリティとする経験則（現在は補助的参照に留め、多角的に算定）

## §30. M&A時のIP交渉

### Rule 63.220: IP関連の表明保証（Representations & Warranties）
- **売手が保証すべき事項**:
  - 全IPの合法的所有権を保有していること
  - 第三者のIPを侵害していないこと
  - 全てのIP譲渡契約が有効であること
  - 係争中のIP訴訟がないこと
  - OSSライセンスに完全に準拠していること
  - 重要な営業秘密への不正アクセスがないこと
  - AI生成コードの使用範囲と権利帰属が明確であること
  - データプライバシー規制（GDPR/APPI等）に準拠していること

### Rule 63.221: 補償条項（Indemnification）
- IP侵害が判明した場合の損害賠償条項
- **バスケット**: 少額クレームのフィルタリング（De Minimis / Basket）
- **キャップ**: 補償上限額の設定（通常は買収価格の10-20%、IP関連は特別キャップの検討）
- **エスクロー**: 買収金額の一部をエスクロー口座に一定期間留保（通常12-24ヶ月）
- **Earnout条項**: IPの収益化実績に基づく追加対価の設定

### Rule 63.222: チェンジオブコントロール条項
- 既存のライセンス契約にチェンジオブコントロール条項がある場合:
  - M&Aにより契約が終了・再交渉が必要になるリスクの事前評価
  - 重要なサードパーティライセンスの譲渡可能性確認
  - クラウドサービス契約（AWS/GCP/Azure）のチェンジオブコントロール条項確認

## §31. IPO時のIP開示要件

### Rule 63.230: SEC向けIP開示（米国上場）
- **Form S-1（Registration Statement）**:
  - 知的財産権の概要（特許・商標・著作権・営業秘密の数と重要性）
  - IP関連のリスクファクター（侵害リスク、特許満了、依存度）
  - 係争中・潜在的なIP訴訟の開示
  - OSSの使用状況と関連リスク
  - AI技術の使用とIP帰属に関するリスク
  - 海外事業におけるIP窃取・不正アクセスリスク（特に中国・ロシア等）
- **継続開示義務**: 10-K/10-Qでの知財関連リスクの定期更新
- **IPの実質的毀損**: IP侵害・窃取が発生した場合は仮定的リスク記述では不足 — 具体的事象・影響の開示が必要

### Rule 63.231: 金融庁向けIP開示（日本上場）
- **有価証券届出書**:
  - 知的財産の概要と事業上の重要性
  - 技術リスクの記載
  - 他社IPへの依存度
- **知財投資・活用戦略の開示ガイダンス**（内閣府2023策定）への対応
- **統合報告書**: 知財KPI（出願数、ライセンス収入、IP投資額）の開示推奨

## §32. SPAC/De-SPAC時のIP開示

### Rule 63.235: SEC 2024年最終規則（Subpart 1600）
- **2024年1月採択**: SPAC IPOおよびDe-SPAC取引に対しIPO同等の開示義務を課す包括的規則
- **Target企業のco-registrant責任**: De-SPAC取引でTarget企業は共同登録者となり、取締役・役員がRegistration Statementの開示に対し責任を負う
- **Forward-Looking Statement Safe Harbor の適用除外**: SPACにはPSLRA の safe harbor が適用されない — 将来予測（プロジェクション）に関する免責なし
- **希薄化開示**: スポンサー報酬・引受手数料・ワラント・転換証券・PIPE等による希薄化の詳細開示義務
- **最低配布期間**: De-SPAC取引のプロスペクタス/プロキシステートメントに20暦日の最低配布期間

### Rule 63.236: SPAC時のIP固有の開示事項
- 上記SEC新規則に加え、以下をIP観点で準備:
  - Target企業のIP資産の詳細記述（特許・商標・営業秘密のポートフォリオ）
  - IPに関連するリスクファクター（侵害リスク、OSSリスク、AIリスク）
  - IPの評価方法論とその根拠（プロジェクションのベースとなるIP価値）
  - 利益相反の開示（スポンサーとTarget間のIP関連取引）

## §33. Post-Merger IP統合（PMI）

### Rule 63.240: PMI・IP統合ロードマップ

| フェーズ | 期間 | アクション |
|---|---|---|
| **Day-1** | クロージング日 | IP所有権の法的移転確認、重要ライセンスの継続確認、IP台帳の統合開始 |
| **Day-30** | 〜1ヶ月目 | 統合IPポートフォリオの重複分析、不要IPのPruning計画、商標使用統一方針 |
| **Day-100** | 〜3ヶ月目 | 統合データルームの完成、IPコスト最適化、統合特許出願戦略の策定 |
| **Day-365** | 〜1年目 | 統合IPポートフォリオの完全移行、シナジー効果の測定、統合IP成熟度評価 |

### Rule 63.241: PMI固有のIPリスク
- **キーパーソンリテンション**: 重要な発明者・技術者の離職防止策（Retention Bonus / IP Earnout）
- **文化統合**: IP創出文化（発明開示の促進 vs 営業秘密重視）の統合
- **ブランド統合**: 共同ブランド → ブランド吸収 → 新ブランドのタイムライン設計
- **技術スタックの統合**: OSSライセンスの互換性確認、重複SaaSライセンスの整理

## §34. カーブアウト・スピンオフ時のIP分離

### Rule 63.245: IP分離フレームワーク
- **Shared IP**: 親会社・分離先が共に使用するIPの取扱い
  - **License-back**: 分離先に対するIPライセンスの供与（期間・範囲・料率を事前合意）
  - **IP共有契約（Shared IP Agreement）**: 共同所有IPの利用条件
- **Exclusive IP**: 一方にのみ帰属するIPの明確な区分
- **TSA（Transition Services Agreement）**: 分離移行期間中のIP関連サービスの提供条件

### Rule 63.246: カーブアウト時のIP Due Diligenceチェックリスト
- [ ] 分離対象IPの完全な特定と台帳化
- [ ] 混在IPの帰属決定（共有 vs 排他的ライセンス）
- [ ] 従業員のIP譲渡契約の移管手続き
- [ ] サードパーティライセンスの譲渡/再交渉
- [ ] OSSコンポーネントの分離先での利用可能性確認
- [ ] 営業秘密の分離先への移管セキュリティ手順
- [ ] データ（顧客データ含む）の帰属・移管・プライバシー準拠

## §37. 技術デューデリジェンス

### Rule 63.270: コード品質の評価基準

| 評価項目 | グリーン（良好） | イエロー（要注意） | レッド（リスク大） |
|---|---|---|---|
| テストカバレッジ | ≥80% | 50-79% | <50% |
| 技術的負債比率 | ≤5% | 5-15% | >15% |
| 依存パッケージの脆弱性 | 0 Critical/High | 1-3 High | 4+ High or Critical |
| ドキュメント充実度 | API/アーキテクチャ文書完備 | 主要部分のみ | ほぼ未整備 |
| CI/CDパイプライン | 完全自動化 | 部分自動化 | 手動デプロイ |
| デプロイ頻度 | 日次以上 | 週次 | 月次以下 |
| MTTR（平均復旧時間） | <1時間 | 1-4時間 | >4時間 |
| AI生成コード比率 | 記録あり・管理下 | 部分的記録 | 記録なし |

### Rule 63.271: アーキテクチャ評価
- **スケーラビリティ**: 10倍の負荷増加に対応できるアーキテクチャか
- **単一障害点（SPOF）**: 存在する場合は緩和策が講じられているか
- **クラウドネイティブ度**: コンテナ化/IaC/マネージドサービス活用の度合い
- **モジュラリティ**: マイクロサービス/Monolith比率、結合度の評価
- **AI/ML基盤**: モデルのバージョン管理・再現性・データパイプラインの成熟度

### Rule 63.272: 技術チームの評価
- 主要技術領域ごとのバスファクター（Bus Factor ≥ 2が目標）
- キーエンジニアのリテンションリスク評価
- 採用市場における技術スタック人材の採用難易度

## §38. OSS/ライセンスDD

### Rule 63.280: SCA連携による自動ライセンス監査
- **SCAツールの活用**: Snyk / FOSSA / Socket.dev / Black Duck 等を用い、DDプロセスの一環としてライセンスコンプライアンスを自動検証
- **詳細は** → [602_oss_compliance.md](../security/200_oss_compliance.md) を参照

### Rule 63.281: M&AにおけるOSSリスクの定量化
- **Black Duck 2025レポートの知見**:
  - M&A監査対象の**85%**でライセンスコンフリクトが検出
  - **96%**でパッチ未適用の脆弱性が存在
  - 1トランザクションあたり平均数千のOSSコンポーネントを使用
- **対策**: Exit準備の早期段階（18ヶ月前以上）からSCA/SBOM生成を継続実施

### Rule 63.282: コピーレフトへの対応
- GPL/AGPL等のコピーレフトライセンスがプロプライエタリコードに影響を与えていないか（コピーレフト汚染）の確認
- 汚染が検出された場合のリメディエーション計画の策定
- ライセンスの再交渉またはコンポーネント置換の実施

## §39. ベンダーロックイン評価とポータビリティ

### Rule 63.290: ベンダーロックイン評価マトリクス

| 評価項目 | 低リスク | 中リスク | 高リスク |
|---|---|---|---|
| **データポータビリティ** | 標準フォーマットでエクスポート可 | 部分的にカスタム形式 | ベンダー固有形式のみ |
| **API互換性** | 業界標準API使用 | 一部独自API | 完全独自API |
| **インフラ抽象化** | IaC + マルチクラウド対応 | 一部プロバイダー固有 | プロバイダー固有サービスに深く依存 |
| **データ移行コスト** | 数日〜数週間 | 数ヶ月 | 6ヶ月以上 |
| **契約上の制約** | 即時解約可 | 長期契約・解約金あり | 排他的契約 |

### Rule 63.291: ポータビリティ確保戦略
- **抽象化レイヤーの導入**: データベース/ストレージ/認証サービスへのアクセスを抽象化
- **マルチクラウド/ハイブリッド戦略**: 完全なマルチクラウドは不要だが、データのポータビリティは確保
- **エグレスコストの把握**: クラウドベンダーからのデータ転送コストを事前に見積もり
- **標準準拠**: OpenAPI, CloudEvents, OCI等の標準規格を優先使用

## §40. クロスボーダーIP紛争解決

### Rule 63.295: 国際IP紛争解決メカニズム

| メカニズム | 対象 | 特徴 | 期間 |
|---|---|---|---|
| **UPC** | 欧州特許紛争 | 汎欧州的差止命令、中央無効判決 | 12-18ヶ月 |
| **ITC Section 337** | 米国輸入品の特許侵害 | 排除命令（輸入禁止）、迅速手続 | 12-16ヶ月 |
| **WIPO仲裁調停センター** | 国際IP紛争全般 | 秘密性、専門性、執行可能性 | 6-12ヶ月 |
| **ICC国際仲裁** | 国際ライセンス紛争 | ニューヨーク条約による執行 | 12-24ヶ月 |
| **ISDS** | 投資協定に基づくIP関連紛争 | 投資家対国家紛争 | 24-48ヶ月 |

### Rule 63.296: 紛争解決条項の設計
- 国際契約のIP条項には以下を含めること:
  - **段階的紛争解決**: 協議 → 調停 → 仲裁/訴訟の段階的プロセス
  - **準拠法**: IP権利の存在・有効性は保護国法、契約関係は合意法
  - **仲裁条項**: WIPO / ICC / SIAC 規則のいずれかを指定
  - **暫定措置**: 証拠保全・差止命令の緊急仲裁人（Emergency Arbitrator）条項

## §41. IP担保融資（IP-backed Financing）

### Rule 63.300: IP担保権の設定
- **概要**: 特許・商標等のIPを担保として金融機関から融資を受ける仕組み
- **法的基盤**:
  - 米国: UCC Article 9（動産担保）— IP担保権のPerfection（登録）が必要
  - 日本: 特許法第95条以下（特許権の質権設定）— 特許庁への登録が対抗要件
  - EU: 各加盟国法に依存（統一的なIP担保権法制は未整備）
- **実務上の課題**:
  - IP資産の評価（→ §28 IP評価参照）
  - IPの陳腐化リスク
  - 倒産時のIPの取扱い

### Rule 63.301: IP担保融資の戦略的活用
- **エクイティ希薄化の回避**: 追加の株式発行なしで資金調達が可能
- **適用場面**: ブリッジファイナンス、R&D資金確保、Exit前のランウェイ確保
- **市場動向（2025）**: IP担保融資市場は成長傾向 — Aon / Intellectual Ventures / IPwe等の専門プレイヤーが台頭
- **注意点**: 担保IPの売却権制限、ライセンス供与への影響を事前に確認

## §43. IPインシデント対応

### Rule 63.320: 侵害通知受領時のフロー
1. **即日**: 法務チームに通知、証拠保全開始
2. **72時間以内**: 外部IP弁護士の起用判断
3. **2週間以内**: 非侵害/無効の反論可能性評価
4. **対応方針決定**: 訴訟・ライセンス交渉・設計回避（Design Around）の選択
5. **記録**: 全対応履歴をIPインシデントログに記録

### Rule 63.321: 自社IPの侵害発見時のフロー
1. 侵害の証拠収集（公証人による証拠保全を推奨 — Wayback Machine/スクリーンショット/ブロックチェーンタイムスタンプ）
2. 侵害の範囲・影響度の評価
3. 警告書（Cease & Desist Letter）の送付判断
4. 交渉/訴訟/ADR（仲裁・調停）の選択（→ §40 クロスボーダー紛争解決参照）
5. ライセンス供与の可能性検討

### Rule 63.322: インシデント対応チーム
- **構成**: 法務責任者 + CTO/VP Engineering + 外部IP弁護士 + 事業責任者
- **エスカレーション基準**: 潜在的損害額/差止リスク/事業影響度に応じた3段階基準を事前策定

## §44. IP成熟度モデル

### Rule 63.330: 5段階成熟度モデル

| レベル | 名称 | 特徴 | チェック項目 |
|---|---|---|---|
| **Level 1** | 初期（Ad Hoc） | IP管理が属人的、体系化されていない | IP台帳が存在しない |
| **Level 2** | 基礎（Basic） | 基本的なIP譲渡契約・NDAが整備 | 就業規則にIP条項がある |
| **Level 3** | 管理（Managed） | 特許/商標の出願戦略が存在、IP台帳を運用 | 定期的なIP監査を実施 |
| **Level 4** | 最適化（Optimized） | データルーム自動化、IP評価を定期実施、AI-IP方針策定済み | Exit Ready Scorecard 80%以上 |
| **Level 5** | 戦略的（Strategic） | IPが事業戦略の中核、ライセンシング・M&A活用、IP担保融資実績あり | IPが収益源の一つ |

### Rule 63.331: 成熟度向上のロードマップ
- **Level 1→2**: IP譲渡契約の全員締結、NDA標準テンプレートの策定
- **Level 2→3**: 特許委員会の設置、商標監視サービスの導入、IP台帳の整備
- **Level 3→4**: CI/CD連携のデータルーム自動化、IP評価の外部委託、OSSコンプライアンスの自動化、AI-IPポリシーの策定
- **Level 4→5**: IPライセンシング収益の創出、M&A時のIP活用戦略の確立、IPトークン化の検討

## §45. IP保険（IP Litigation Insurance）

### Rule 63.335: IP保険の類型

| 保険類型 | 保護対象 | カバレッジ | 適用場面 |
|---|---|---|---|
| **防御型（Defensive）** | 他社からの侵害訴訟の防御コスト | 訴訟費用・和解金・損害賠償金 | PAE/NPE訴訟対策 |
| **攻撃型（Offensive/Abatement）** | 自社IP侵害時のエンフォースメントコスト | 訴訟費用・調査費用 | 積極的なIP保護 |
| **表明保証保険（R&W Insurance）** | M&A時のIP関連表明保証違反 | 損害賠償金 | M&Aクロージング |
| **IPO保険（POSI）** | 上場後のIP関連訴訟 | 訴訟費用・損害賠償金 | IPO前の保護策 |

### Rule 63.336: IP保険導入の検討基準
- **加入判断の指標**:
  - 年間IP訴訟リスク評価スコアが中〜高の場合
  - M&A/IPOを計画している場合
  - PAE/NPEが活動的な技術分野の場合
  - 特許ポートフォリオの規模が中〜大（50件以上）の場合
- **主要プロバイダー**: Aon / Marsh / RPX / IPISC

## §46. ESG・サステナビリティとIP

### Rule 63.340: Green Patent・環境技術IP
- **WIPO GREEN**: 環境技術のIPマッチングプラットフォーム — 自社のグリーンIPの登録を検討
- **各国のGreen Patent Fast Track**:
  - 日本: スーパー早期審査制度（グリーン発明対象）
  - 米国: Patents for Humanity / Climate Change Mitigation Technologies Patent Pilot
  - 欧州: EPO PACE（早期審査プログラム）
- **SDGs関連IPポートフォリオ**: ESG投資家向けに環境・社会課題解決に資するIPの可視化

### Rule 63.341: ESG開示におけるIP
- **TCFD/ISSB/CSRD**: 気候関連リスクの開示にIP資産（環境技術特許）を含める
- **統合報告書**: IPが企業のサステナビリティ戦略にどう貢献しているかを定量的に開示
- **グリーンウォッシュ防止**: 環境関連IPの実際の使用状況・効果を正確に開示

## §47. オープンイノベーションIP方針

### Rule 63.345: Corporate Venture Capital（CVC）/ アクセラレーターのIP要件
- **CVC投資先のIP条項**:
  - 投資先のIP所有権は投資先に帰属（支配権の確保ではなくアクセス権の確保）
  - 優先的ライセンスの取得（ROFR for IP licensing）
  - M&A時のIPアクセス権の保証
- **アクセラレーター参加企業のIP保護**:
  - 参加企業のバックグラウンドIPの保護
  - メンターへの秘密保持義務
  - 共同開発成果物のIP帰属の明確化

### Rule 63.346: 産学連携のIP管理
- **大学との共同研究契約のIP条項**:
  - 研究成果のIP帰属（米国: Bayh-Dole Act — 大学に優先帰属権）
  - 企業への独占的/非独占的ライセンス権の確保
  - 学術公表権と特許出願のタイミング調整（Grace Period活用）
  - 学生・ポスドクの成果物のIP帰属確認

| **2028-2030年** | 量子コンピューティングの実用化予測窓: 既存RSA/ECC暗号の実用的脅威 | PQC移行計画とIP戦略の最終統合フェーズ（→ §50 Rule 63.362）|

- **年次**: グローバルIPコンプライアンスマトリクスの更新

## §50. 将来技術のIP戦略

### Rule 63.360: 新興技術のIP保護アプローチ

| 技術領域 | IP保護手法 | 注意点 |
|---|---|---|
| **量子コンピューティング** | 特許（量子アルゴリズム・量子ゲート設計）+ 営業秘密 | 量子耐性暗号への移行に伴うIP更新 |
| **Post-Quantum Cryptography (PQC)** | 特許（PQC実装最適化・HW加速）+ NIST標準化活動参加 | 標準アルゴリズム自体の特許困難 — 実装最適化が主戦場 |
| **Agentic AI / MCP / A2A** | 特許（エージェントオーケストレーション手法）+ 営業秘密（プロンプト設計） | エージェント間IP帰属はグレーゾーン — 人間の監督記録が必須 |
| **オープンソースAIモデル活用** | Fine-Tuning差分ウェイト（LoRA等）を営業秘密として保護 | Llama/Mistral等のライセンス条件を精査（商業利用制限に注意） |
| **Web3/分散型アプリ** | 特許 + 営業秘密（スマートコントラクトロジック） | OSS前提のエコシステム — ライセンス戦略の慎重な設計 |
| **メタバース/XR** | 意匠権（仮想空間デザイン）+ 商標（バーチャルブランド） | 仮想世界でのIP侵害の管轄問題 |

### Rule 63.361: 将来技術のIPポートフォリオ構築
- **早期出願戦略**: 技術が成熟する前に基幹特許を確保
- **防衛的公開**: 自社が追求しない技術方向性については防衛的公開で他社の特許化を阻止
- **標準化参加**: 新興技術の標準化団体（W3C / IEEE / IETF / NIST）への参加によるSEP獲得機会
- **技術ロードマップとの連動**: 3-5年の技術ロードマップとIP出願戦略を同期

### Rule 63.362: Post-Quantum Cryptography（PQC）IP戦略
- **NIST PQC標準化完了（2024年8月）**: CRYSTALS-Kyber（ML-KEM）、CRYSTALS-Dilithium（ML-DSA）、FALCON（FN-DSA）、SPHINCS+（SLH-DSA）
  - 標準アルゴリズム自体の特許取得は困難 → **高速実装・ハードウェアアクセラレーション・ハイブリッド移行ツール**に特許機会
  - TLS1.3 + PQCハイブリッド鍵交換の実装最適化は重要な特許領域
- **Harvest Now, Decrypt Later（HNDL）リスクへの対応**:
  - PQC移行計画を特許戦略と同期し、移行ツールおよび移行プロセス自体の特許を確保
  - 既存暗号化プロトコルとPQCのハイブリッド運用技術は早期出願を推奨

### Rule 63.363: Agentic AI / MCP / A2A プロトコルのIP問題
- **IP帰属のグレーゾーン対応**:
  - 複数のAIエージェントが協調して生成した成果物の著作権帰属は現時点で未確立
  - **原則**: 人間がオーケストレーション設計・目標設定・出力評価に実質的に関与した部分のみ著作権主張可能
  - **記録義務**: エージェント実行ログ（各エージェントの担当ステップ・判断過程）を完全保存（→ §23参照）
- **プロンプトとワークフロー設計のIP保護**:
  - 複雑なマルチエージェントオーケストレーション設計は**営業秘密**として保護優先（公開で競合が容易に複製可能）
  - 特許出願は「エージェント間の協調プロトコル」「タスク分解・割当アルゴリズム」等の技術的革新部分に焦点
- **MCP（Model Context Protocol）活用時の注意**:
  - MCPサーバーに組み込まれたビジネスロジック・プロンプトテンプレートは営業秘密として管理
  - サードパーティMCPサーバー利用時はToS確認（データ利用・IP帰属の条項）

### Rule 63.364: オープンソースAIモデルのFine-Tuning戦略
- **商業利用ライセンス確認（必須）**:

| ベースモデル | ライセンス | 商業利用 | Fine-Tuning上の制限 |
|---|---|---|---|
| Llama 3系 | Meta Llama 3 Community License | MAU 700M超は別途申請 | 可（ロゴ・名称の商業利用は制限） |
| Mistral / Mixtral | Apache 2.0 | 自由 | 可 |
| Gemma 2 | Gemma Terms of Use | 可 | Google競合製品への適用制限 |
| Phi-4 | MIT | 自由 | 可 |

- **Fine-Tuningモデルの保護方針**:
  - ベースモデルのライセンスを継承しつつ、**差分ウェイト（LoRA Adapter / QLoRA）を営業秘密として保護**
  - Fine-Tuning用の独自学習データセット・データパイプラインは特許または営業秘密で保護
  - モデルアーキテクチャの独自改良（新しいAttention機構・位置エンコーディング等）は特許出願を検討
  - **SaaS経由で提供する場合**: AGPLライセンスのモデルはネットワーク越しの提供でもOSSソース開示義務が生じる可能性に注意

# XI. 地政学的リスクと輸出管理（§51）

---

## §51. IP × Sovereign Risk（地政学的リスクと輸出管理）

### Rule 63.370: 技術覇権競争時代のIPリスク概観
- **背景**: 米中技術覇権競争・ロシア制裁・台湾海峡リスクなど地政学的緊張がIPの移転・保護・権利行使に直接影響
- **リスク分類**:
  - **輸出管理リスク**: 自社IP（特許技術・ソフトウェア・暗号技術）が輸出規制対象に該当するリスク
  - **IP強制移転リスク**: 市場参入条件として技術ライセンスを強制される国・契約リスク
  - **国家管轄外知財無効化リスク**: 地政学的対立下での相手国での権利行使不能リスク
  - **供給チェーン分断リスク**: 技術サプライヤーが制裁や輸出制限の対象になるリスク

### Rule 63.371: 輸出管理規制とIPの交差点
- **EAR（Export Administration Regulations / 米国）**:
  - EAR対象品目（ECCNリスト）に該当するソフトウェア・コード・アルゴリズムの特定（特に暗号技術: ECCN 5D002, 5E002）
  - **Fundamental Research Exclusion（FRE）**: 教育機関との共同研究で公開予定の成果はEAR除外が適用可能
  - **License Exception TSR**: 技術移転に関する一部例外の活用
  - 外国人従業員（非市民権者）への社内技術開示も「みなし輸出（Deemed Export）」として規制対象になる可能性
- **ITAR（International Traffic in Arms Regulations / 米国）**:
  - 防衛・宇宙・サイバーウェポン関連IPはITAR適用の可能性を弁護士に確認
  - ITARに抵触すると外国人エンジニアへの技術開示が制限される（採用・開発体制への影響）
- **EU輸出管理規制（EG No 428/2009 改正版）**:
  - デュアルユース技術のEU輸出管理規則への対応
  - サイバーセキュリティ技術のイントラジブル技術移転規制
- **日本: 外為法（外国為替及び外国貿易法）**:
  - 技術的優位性の高い技術を対内直接投資審査対象業種（コア業種）で保有する場合、外資の出資・買収に事前届出義務
  - 機微技術の外国への技術移転にも輸出許可が必要な場合あり

```yaml
# 輸出管理チェックリスト（IP観点）
ip_export_control_checklist:
  eccn_classification:
    - action: "全製品・ソフトウェア・暗号ライブラリのECCN分類実施"
    - action: "EAR99（規制対象外）とECCN分類品目を台帳で区分管理"
  deemed_export:
    - action: "外国籍従業員・インターンへの技術開示前にECCN確認"
    - action: "クラウド開発環境の所在地・アクセス権限の輸出管理リスク評価"
  license_enforcement:
    - action: "輸出先国が制裁リスト（OFAC SDN / BIS Entity List）に未掲載か確認"
    - action: "オープンソースコードへの組み込み前にEAR適用確認（OSS自体はEAR対象外だが組み合わせで規制対象になるケースあり）"
```

### Rule 63.372: 中国・ロシア等でのIPリスク管理
- **中国でのIP保護戦略**:
  - 中国での特許出願は利点と引き換えに技術内容が公開される → 営業秘密との使い分け戦略が重要
  - **秘密保持審査（Confidentiality Examination）**: 中国で開発・保有する発明を外国で先に出願する場合、CNIPAへの秘密保持審査申請が必要（違反時は中国特許権が無効）
  - Joint Venture強制ルールへの対応: バックグラウンドIPは別法人で管理し、JVにはライセンス供与のみ
  - **香港**: 現時点でIPの保護は本土と独立しているが、国家安全維持法（NSL）の影響を継続モニタリング
- **ロシア・ウクライナ情勢**:
  - 制裁対象企業・個人とのライセンス契約はOFACライセンスが必要
  - ロシア法改正（2022年）：特許権者への補償なしで制裁国への強制実施権設定を認める条項 → 露国内IP権利行使の実効性低下
- **インド**:
  - 国家安全保障上の理由による強制実施権制度（Patents Act §§100-103）への対応
  - Pharmaceutical/AI分野での強制実施権リスクの事前評価

### Rule 63.373: 技術デカップリングへの対応戦略
- **IPポートフォリオの地政学的分散**:
  - 地政学リスクの高い地域向けIPと、低リスク地域向けIPを分離して管理
  - クリティカルなIPのホスト法域を政治的安定国（シンガポール・オランダ・アイルランド等）に集約
- **Technology Partitioning（技術区分管理）**:
  - コア技術（核心的な競争優位）と汎用技術を分けて管理
  - コア技術: オンプレミス/プライベートクラウドで管理、外国籍者のアクセスを制限
  - 汎用技術: グローバル開発チームで共有可能
- **FDI（外国直接投資）審査対応**:
  - CFIUS（米国）/ FIRB（豪）/ NSI Act（英）/ 対内直接投資審査（日本）のトリガーとなるIP所有権変動を事前評価
  - M&A時のCFIUS審査が必要な場合の対応計画（FIRRMA以降の強化された審査基準）

### Rule 63.374: 安全保障輸出管理と研究開発ガバナンス
- **基礎研究と応用研究の分離管理**:
  - 公開前提の基礎研究（FRE対象）と秘密保持が必要な応用研究（EAR/ITAR対象）を組織・プロセス・ITシステムで物理的に分離
  - **Research Security Program**: NISTの推奨するResearch Security プログラム（NSPM-33準拠）の実装
- **国際共同研究のIP条項**:
  - 外国政府資金が含まれる共同研究のIPへの政府アクセス権条項（march-in rights等）に注意
  - 軍民両用技術の研究はDUAL USE EXPORTレビューを義務化

# XII. GenAI サプライチェーンのIP ガバナンス（§52）

---

## §52. GenAI サプライチェーン全体のIP責任連鎖

### Rule 63.380: AI Agents協調出力のIP帰属
- **問題の本質**: LLM → Tool Call → Subagent → 最終出力というGenAIサプライチェーンで「誰の知的財産か」が不明確になる
- **現行法的立場（2026年時点）**:
  - 複数AIエージェントが協調して生成した成果物への著作権は、人間が実質的な創作的寄与を行った部分にのみ発生（→ §23参照）
  - **エージェントオーケストレーター** が人間の指示の下で動作する場合、最終出力の著作権は人間（または法人）に帰属すると解釈される傾向
  - 完全自律型AIエージェント（human-in-the-loop なし）の出力は著作権保護を受けない可能性が高い

### Rule 63.381: GenAI サービスプロバイダーとのIP条項管理
- **モデルAPI利用時のToS精査（必須）**:

| プロバイダー | 出力物の権利帰属 | 学習利用 | 商用利用 | Enterprise契約での違い |
|---|---|---|---|---|
| **OpenAI API** | ユーザーに帰属 | Opt-out可（API Tier） | 可 | Data Processing Agreement締結でトレーニング不使用を明文化 |
| **Anthropic API** | ユーザーに帰属 | Opt-out可 | 可 | Enterprise向けデータ非学習保証あり |
| **Google Gemini API** | ユーザーに帰属 | 条件あり（Enterprise除く） | 可 | Vertex AIはデータ非学習 |
| **Azure OpenAI** | ユーザーに帰属 | データ非学習 | 可 | 企業向け標準でトレーニング不使用 |

- **Enterprise APIの義務化**: 機密プロジェクトでは必ずEnterprise契約（データ非学習保証）のAPIのみを使用
- **ToS変更モニタリング**: 主要AIプロバイダーのToSを四半期ごとにレビューし、IP条項の変更を追跡

### Rule 63.382: コード生成・補完ツールのIP リスク管理
- **GitHub Copilot / Cursor / Codeium等の利用時のIPリスク**:
  - **著作権侵害リスク**: 学習データに含まれる著作物を再現するリスク（フィルタリング機能の有効化を推奨）
  - **特許侵害リスク**: 特定のアルゴリズムパターンが特許技術の再現となる可能性
  - **ライセンスブリーチリスク**: OSSコードがライセンス条件なしに生成コードに混入するリスク
- **技術的対策**:

```yaml
# GitHub Copilot: IP保護設定
copilot_ip_protection:
  duplicate_detection:
    enabled: true  # 公開コードと一致するサジェストを非表示
  suggestions_matching_public_code: "blocked"  # 公開コードへの参照をブロック
  code_referencing:
    enabled: true  # コードの出典参照を有効化
  
# Cursor: プロプライエタリコードの漏洩対策
cursor_privacy:
  privacy_mode: true  # コードをAnthropicのサーバーに送信しない
  indexing_local_only: true  # インデックスをローカルのみに制限
```

- **AI生成コードの記録義務**: 生成したコードの`% of AI contribution`をコミットコメントまたはPRテンプレートに記録（→ §23, §24参照）

### Rule 63.383: マルチモーダルAI出力のIP管理
- **画像・映像・音声生成のIP管理**:
  - **画像**: DALL·E / Midjourney / Stable Diffusion 等の利用規約を確認し、生成画像の商業利用制限を把握
  - **音声**: 音源・声優の著作隣接権への留意（AI Voice Cloningは各法域で規制動向を注視）
  - **映像**: AI生成映像にはC2PA（Content Authenticity Initiative）メタデータを付与し、出自を証明
  - **AI生成コンテンツのラベリング義務**: EU AI Act §50（2026年8月完全施行）に基づきAI生成コンテンツへの技術的マーキングが義務化
- **Deep Fake・なりすましリスク**:
  - 自社ブランド・経営者の顔・声のDeep Fake利用を監視するブランドモニタリングの強化
  - Cease & Desist対象としての「Synthetic Media」利用への証拠保全戦略

### Rule 63.384: AI出力物の品質・セキュリティとIPの統合管理
- **AI生成コードのサプライチェーンリスク**:
  - LLMが脆弱性を持つコードパターンを生成するリスク（CWE Top 25への対照検証を自動化）
  - AI生成コードのSAST/DASTスキャンをCI/CDに統合（→ [000_security_privacy.md](../security/000_security_privacy.md) §AI-SPM参照）
- **AI生成コードのライセンス自動検出**: 

```python
# AI生成コードのライセンス・IP検出パイプライン例
# CI/CDに組み込み、マージ前に自動実行

steps:
  - name: "Detect AI-generated code fingerprints"
    run: |
      # 公開リポジトリとの類似度検出
      scancode-toolkit --output-format json --license --copyright ./src
      
  - name: "AI code attribution check"
    run: |
      # PR内のAI生成コードのアノテーション確認
      python scripts/check_ai_attribution.py --pr-diff ${{ github.event.pull_request.number }}
      
  - name: "License conflict detection"
    run: |
      # AI生成コード内のOSSライセンス混入検出
```

# XIII. IP収益化戦略（§53）

## §53. IP単体の収益源化（IP Monetization as a Revenue Stream）

### Rule 63.390: IPライセンシング収益化フレームワーク
- **収益化モデルの類型**:

| モデル | 概要 | 適合場面 | 収益規模 |
|---|---|---|---|
| **ライセンシング（非独占）** | 複数ライセンシーに同一IPをライセンス供与 | 広く普及した技術（SEP含む） | 中〜大 |
| **ライセンシング（独占）** | 単一ライセンシーへの排他的ライセンス | 特定市場での強力シナジー | 大 |
| **クロスライセンス** | 相互のIPをライセンス交換、実質無償 | 競合他社との係争回避・R&D加速 | キャッシュレス |
| **特許プール参加** | MPEG-LA / Via / Avanci等のプールへの特許拠出 | SEP保有、標準技術分野 | 中〜大（安定的） |
| **スピンオフ・ライセンシング会社** | IP保有専門会社（IPCo）を設立し運営 | 大規模ポートフォリオ、事業分離 | 大 |
| **IPのアウトライト売却（Assignment）** | IP所有権ごと第三者に移転 | 戦略的に不要なIP、資金調達 | 一時金 |
| **IPトークン化・フラクショナル** | ブロックチェーン上での持分販売 | 流動性確保、複数投資家参加 | 新興・不確実 |

### Rule 63.391: ライセンシングプログラムの設計
- **ライセンス料率設定の原則**:
  - **業界標準料率（§28 Rule 63.202参照）** をベンチマークとして使用
  - **Comparable License Agreements（比較可能ライセンス事例）**: 過去の類似ライセンス事例を収集・データベース化し、交渉の根拠として活用
  - **Nash Bargaining Solution**: 経済学的フレームワークを用いた料率設定（SSPPU: Smallest Saleable Patent Practicing Unit を基準とした積算）
  - **ロイヤリティスタッキング**: 複数IPが重複してロイヤリティが積み重なる場合の合計料率の合理性評価（FRANDの文脈で特に重要）

- **ライセンス契約の必須条項**:
  - [ ] ライセンス対象IPの明示（特許番号・バージョン管理されたIPリストの添付）
  - [ ] 使用許諾の範囲（地理・分野・使用目的・サブライセンス可否）
  - [ ] ロイヤリティの算定基準（SSPPU / エンドプロダクト価格 / 固定額）
  - [ ] 報告・監査権（Audit Rights: 年1回、費用負担、トリガー条件）
  - [ ] 最恵国待遇（Most Favored Licensee / MFN）条項の検討（SEP/FRANDでは慎重に）
  - [ ] 改良発明のグラントバック条項（Grantback: 独占的は独禁法リスク、非独占を推奨）
  - [ ] 終了条件（特許無効確定、支払停止、破産）
  - [ ] 仲裁・準拠法（ライセンシング紛争専門のICC / WIPO仲裁を推奨）

### Rule 63.392: IPCo（IP保有専門会社）戦略
- **IPCoの設計目的**:
  - 事業会社リスクからIPを分離（訴訟・倒産リスクのリングフェンス）
  - 税効率の最適化（IP集約法域: オランダ「イノベーションボックス」、アイルランド「Knowledge Development Box」、ルクセンブルク「IP Box」）
  - ライセンシングプログラムの一元管理

- **IPCo設立のための税務考慮**:
  - **BEPS（Base Erosion and Profit Shifting）**: OECDのBEPS行動計画13（移転価格・国別報告）への準拠
  - **Pillar Two（グローバル最低税率15%）**: IPCo法域の有効税率確認（エロージョン防止のための追加課税計算）
  - **移転価格文書化**: IP移転時のALP（独立当事者間原則）に基づく評価書の整備（国別報告義務）

### Rule 63.393: 特許プールとライセンシングコンソーシアム
- **参加判断基準**:
  - 自社特許がプール内の必須技術と認定されているか（Essentiality評価）
  - プールのロイヤリティ率が市場で通用するFRAND条件を満たしているか
  - プール参加による競合との関係性（共有ライセンス → 競合への利用許諾が生じる点に注意）

- **主要特許プール一覧（2025現在）**:

| プール | 技術分野 | 特記事項 |
|---|---|---|
| **MPEG-LA** | HEVC / VVC / H.264 / AVC | エンコーダー・デコーダーを必須特許対象 |
| **Via Licensing（Sisvel傘下）** | Wi-Fi / Bluetooth / 4G / 5G | 標準化済みSEPのプール |
| **Avanci** | IoT / 車載通信（4G/5G） | FRAND一括ライセンスモデル |
| **Access Advance** | VVC / H.266 / HEVC Plus | 映像圧縮の次世代標準 |
| **Open Patent Alliance（OPA）** | Wi-Fi / Bluetooth | 中小企業向け低コストアクセス |

### Rule 63.394: PAE（Patent Assertion Entity）vs. 実施企業の攻守戦略
- **自社がPAEから攻撃を受けた場合（防衛戦略）**:
  - IPR（Inter Partes Review: 特許無効審判）提起 — PTAB（Patent Trial and Appeal Board）での無効化
  - 優先的和解交渉（裁判コストとの比較衡量）
  - LOT Network / OINに未加入の場合は優先加入
  - 対PAEディフェンスサービス（RPX / Unified Patents）の活用

- **自社IPで積極的エンフォースメントを行う場合（攻撃戦略）**:
  - エンフォースメント前にFTO調査を必ず実施（報復特許訴訟リスクの事前評価）
  - ITC Section 337利用（輸入禁止命令 — 迅速かつ強力だが要件が厳格）
  - 訴訟費用保険とサクセスフィー契約（コンティンジェンシー）の法律事務所の活用

### Rule 63.395: IPライセンシングのKPIと財務管理
- **必須KPI**:
  - ライセンス収入（絶対額 + 事業収益に対する比率）
  - アクティブライセンシー数（地域別・分野別）
  - 報告・監査サイクルの遵守率（未提出ライセンシー数）
  - ロイヤリティ未払い・遅延の発生件数と回収率
  - 新規ライセンス取得平均交渉期間

- **IPライセンス収益の財務管理**:
  - ライセンス収益を**PL（損益計算書）の独立ラインアイテム**として可視化（IPOや投資家向け評価に直結）
  - ライセンス収益のうち、保証料（固定）とランニングロイヤリティ（変動）の分類管理（ASC 606 / IFRS 15準拠）
  - 無形資産のAmortization（償却）との対応付け（GAAP/IFRSの仕掛中の無形資産会計）

# XIV. 競争法とIP（§54）

---

## §54. GenAI時代のIP × 競争法（Antitrust & IP）

### Rule 63.400: 競争法とIPの交差点概論
- **基本緊張関係**: IPは一定期間の独占権を付与する（特許20年、著作権70年以上） — 競争法は市場独占を規制する
- **IP免除の原則（IP Exemption）**: IPの行使自体は通常競争法違反にならないが、以下の行為は規制対象：
  - 市場全体を独占する目的でのIPシェルタリング
  - FRAND条件を無視したSEPのライセンス拒否
  - パテントエバーグリーニング（特許の人工的延命による競合の市場参入阻害）
  - 反競争的なIPプール・クロスライセンスによる価格協調

### Rule 63.401: GenAI独占リスク（2025-2026最新動向）
- **世界規模のAI独占調査**:
  - **EU**: 欧州委員会がOpenAI / Microsoft / Google / AWS / NVIDIA のAI市場における競争上の懸念を調査中（2025年-）
  - **米国**: DOJ・FTCがFANG/Big Techのベースモデル囲い込み（APIアクセス制限・データ独占）を調査
  - **英国**: CMAが「AI Foundation Models: Initial Report」でAIサプライチェーンの競争上のリスクを特定（2024年）
  - **日本**: 公正取引委員会がAIシステム提供企業の優越的地位濫用リスクを調査（2024年報告書）

- **具体的な競争法リスク事例**:
  - 独占的なAPIアクセス制限 → 下流スタートアップの市場排除
  - 訓練データの独占（例: インターネット全体のスナップショット → 後発者の追随不可能な参入障壁）
  - ハードウェア（GPU）とAIソフトウェアのバンドル（tying arrangement）
  - AI能力を持った企業による競合スタートアップへのAcqui-hireを通じた特許・モデル独占

### Rule 63.402: FRAND原則のGenAI拡張論ＩP
- **学術・規制の動向（2025-2026）**:
  - 一部の規制当局・学者が、AI基盤モデルの学習データや推論APIへのFRAND類似のアクセス義務を主張
  - EU AI ACT Article 53: GPAI提供者への技術文書・トレーニングデータ開示義務 → FRAND類似の透明性要件
  - **今後の注視点**: 基盤モデルAPIへの強制アクセス（Compulsory Access）义务化が規制アジェンダに浮上している
- **戦略的示唆**:
  - 市場シェアが大きくなるとIPの独占的行使がより厳しく規制される
  - 自社のAI技術・データがFRAND的な開示要求を受けるリスクを事前に評価

### Rule 63.403: IP × 独禁法のグローバルコンプライアンス
- **米国（DOJ/FTC）**:
  - **IP Licensing Antitrust Guidelines（2017年版）**: DOJ/FTCのIPライセンスに関するガイドライン（現在改訂中 — AI・デジタル市場への対応）
  - **Single Entity Doctrine**: 関連企業間のクロスライセンスは通常適法、しかし非関連競合間での価格調整は違法
  - **Market Definition**: AI/IP市場の定義方法（特許クラス、技術市場、イノベーション市場）

- **EU（欧州委員会/競争総局 DG COMP）**:
  - **TTBER（Technology Transfer Block Exemption Regulation）2014/316/EU**: EUでの特許・ノウハウライセンスに関する一括適用除外規則（競争法適用除外の安全港）
  - TTBERの市場シェア閾値: 競合間は**20%**以下、非競合間は**30%**以下でセーフハーバー
  - **ハードコア制限**: 価格固定、市場分割、販売数量制限はTTBERの恩恵を受けられない

- **日本（公正取引委員会）**:
  - **知的財産の利用に関する独占禁止法上の指針（2023年改定）**: 特許ライセンス制限の判断基準
  - **優越的地位の濫用**: 大規模プラットフォームのスタートアップへのIPライセンス拒否

- **中国（市場監督管理総局 SAMR）**:
  - 知識産権の濫用による競争排除停止規定（2023年施行）への準拠
  - 標準必須特許（SEP）のFRAND違反に対する行政処分のリスク（Qualcomm Case以降の強化）

### Rule 63.404: Patent Evergreening vs. 合法的継続出願
- **不正なEvergreening（規制対象）**:
  - マイナーな製剤変更・製法変更のみを理由とする特許更新で市場独占を人為的に延命
  - 新規性・進歩性のない改良特許の大量出願によるブロッキング（特に製薬・AI分野）
  - Pay-for-Delay（Reverse Payment Settlement）: 後発薬企業への支払いによる市場参入延期

- **合法的継続出願戦略（→§8 Rule 63.072）との区別**:
  - 実質的な技術的改良を含む出願は正当
  - 改良の有無を判断する内部レビュー手順を文書化し、Evergreening認定リスクを低減

### Rule 63.405: デジタル市場法（DMA）とIPの交差
- **EU デジタル市場法（DMA）2023年施行**:
  - **ゲートキーパー指定**: Google / Apple / Meta / Amazon / Microsoft / ByteDance は「ゲートキーパー」— 追加義務を負う
  - **IPとの交差点**:
    - **相互運用性義務**: メッセージングサービスの相互運用義務 → プロトコル特許のFRAND的開示が事実上求められる可能性
    - **データポータビリティ義務**: ユーザーデータの転送 → データベース権・営業秘密との調整
    - **AppStore改革**: サードパーティアプリ配布義務 → プラットフォームIPへのアクセス義務化
  - **対応戦略**: ゲートキーパーに指定されていない場合でも、ゲートキーパー企業との取引にDMA準拠義務が間接的に波及する可能性を評価

### Rule 63.406: 競争法コンプライアンスプログラム
- **IPライセンスにおける競争法コンプライアンス必須要素**:
  - [ ] IPライセンス担当者への競争法研修（年1回）
  - [ ] 競合他社との情報交換・ライセンス交渉時の「Do's and Don'ts」ガイドの策定
  - [ ] クロスライセンス・特許プール参加前の競争法レビュー（外部弁護士確認）
  - [ ] TTBERの市場シェア閾値（EU）の定期確認 — 閾値超過時の対応フローの整備
  - [ ] IPに関連するM&A・JVの競争当局届出義務の事前評価

### Rule 63.413: VC条件交渉におけるIPの位置付け

- **ライセンス収益の先行開示**: 既存ライセンス収入は MRR に加算してバリュエーション算定根拠に含めること
- **IP Reps & Warranties（Preferred Stock Purchase Agreement）**:

```python
def attach_c2pa_manifest(
    input_file: str,
    output_file: str,

    ai_model_version: str,
    human_author: str,
    creation_timestamp: str

    AI生成コンテンツにC2PAマニフェストを付与し、
    EU AI Act Art.50 および著作権帰属を証明する。

            {
                "label": "c2pa.training-mining",
                "data": {"entries": {"c2pa.ai_generative_training": "notAllowed"}}
            },
            {
                "label": "stds.schema-org.CreativeWork",

                    "@type": "CreativeWork",

                    "description": (
                        f"Human-AI collaborative work. "
                        f"AI Model: {ai_model_name} v{ai_model_version}. "
                        f"Human creative judgment applied."
                    )
                }
            }
        ]
    }
    builder = Builder(manifest)
    # 署名鍵は HSM/KMS から取得（ハードコード禁止）
    signer = create_signer(
        sign_cert="certs/company_ip_cert.pem",
        private_key="certs/company_ip_key.pem",
        alg=SigningAlg.ES256,
        tsa_url="http://timestamp.digicert.com"
    )
    builder.sign_file(signer, input_file, output_file)
```

## §58. サードパーティ訴訟ファイナンス（TPLF）によるIP権利行使

### Rule 63.440: IP訴訟ファイナンスの概要と戦略的位置付け
- **市場規模（2025年）**: グローバルTPLF（Third-Party Litigation Finance）市場は約$175B超に成長し、IP訴訟が最大セグメント
- **基本構造**: 訴訟ファイナンサーが訴訟費用（法律事務所費用・証拠開示・専門家証人）を全額負担し、勝訴・和解時の回収金から一定割合（通常25-40%）またはMultiple（2-4x）で報酬を得る
- **戦略的意義**:
  - エクイティ希薄化・バランスシートへの影響なしにIP権利行使が可能
  - キャッシュフローが限定されるスタートアップが大企業の侵害に対抗する唯一の手段になり得る
  - VC投資家は訴訟ファイナンスをポートフォリオ企業のIP保護手段として積極評価

### Rule 63.441: 主要訴訟ファイナンサーと案件適合性

| ファイナンサー | 主要対象案件 | 最低案件評価額 | 特記事項 |
|---|---|---|---|
| **Burford Capital** | 特許・商標・ライセンス契約違反 | $5M以上 | 最大手。ポートフォリオファイナンスも提供 |
| **Harbour Litigation Funding** | 複合IPポートフォリオ | $3M以上 | 欧州・アジア案件に強み |
| **Longford Capital** | US特許訴訟（ITC含む） | $2M以上 | ITCセクション337専門チーム |
| **Woodsford** | UPC対応欧州特許 | €1M以上 | UPC発足後の欧州特許訴訟に注力 |
| **Parabellum Capital** | 営業秘密・不正競争 | $5M以上 | テク企業の営業秘密訴訟に特化 |


### Rule 63.442: 訴訟ファイナンス契約の必須条項（Funding Agreement）
- **開示義務（Disclosure）**: 米国連邦裁判所の多くで訴訟ファイナンス契約の開示義務が設けられ始めている（2024年NJおよびCa.地区裁判所ルール）— 開示義務の範囲を事前確認
- **ファンダーのコントロール**:
  - 和解の最終判断は被告側企業（クライアント）が保持すること（ファンダー主導の和解強制を禁止する条項）
  - 訴訟戦略の変更（法律事務所の交代・追加）に对するファンダーの拒否権範囲を明確に制限
- **リコース（求償権）の不存在**: ノンリコース（Non-Recourse）ファイナンスであることを明示（敗訴時に会社資産への請求権なし）
- **秘密保持**: ファンダーへの開示情報のACP（Attorney-Client Privilege）保護 — 共通利益特権（Common Interest Privilege）の適用確認
- **売却・譲渡禁止**: ファイナンス契約のセカンダリー市場への売却がある場合の通知・同意条件
- **EXIT条項**: M&A・IPO発生時のファイナンス契約の取扱い（早期返済・権利引継ぎの条件）

### Rule 63.443: VC/投資家へのIPファイナンス開示
- **デューデリジェンス上の位置付け**: 訴訟ファイナンス契約は「潜在的偶発負債（Contingent Liability）」ではなく「潜在的偶発資産（Contingent Asset）」として財務モデルに含める
- **表明保証への影響**: M&A時に訴訟ファイナンス契約の存在を適切に開示すること（Change of Control条項の影響を事前評価）
- **IPOへの影響**: SEC Form S-1では進行中のファイナンス付き訴訟を開示（訴訟の進展が事業・財務に重要な影響を持つ場合）

### Rule 63.444: 訴訟ファイナンスの倫理・規制上の注意点
- **弁護士倫理規則**: 訴訟ファイナンサーのコントロールがModel Rules of Professional Conduct 1.8(f)（第三者の費用負担）および5.4（弁護士業務の分割）に抵触しないことを確認
- **日本における位置付け**: 日本では弁護士法第72条（非弁護士の法律事務）の観点からTPLFの適法性グレーゾーンあり — 国内案件は弁護士費用の直接負担スキームとの区別を法務で確認
- **EU規制動向**: 欧州議会がTPLF規制指令を議論中（2024-2025年）— ファンダーの資本要件・利益相反開示義務化の可能性
- **Tax Treatment**: 訴訟ファイナンス収益（回収金のファンダー取り分）の税務処理は管轄によって異なる（米国: Ordinary Income扱いが一般的）

# XIX. 重層的IP保護戦略（§59）

## §59. IP Stacking：ハイブリッド・レイヤードIP保護フレームワーク

### Rule 63.450: 重層的IP保護の設計原則

> [!IMPORTANT]
> **IP保護は「単一権利」への依存を排除せよ。複数のIP権利を重層的に積み重ねることで、競合が一つの権利を迂回しても他の権利が作動する「多重防衛ライン」を構築する。**

- **IP Stackingの定義**: 1つのプロダクト・技術に対し、特許・商標・著作権・意匠権・営業秘密・データベース権・契約上の保護を戦略的に組み合わせる手法
- **なぜ重要か**:
  - 特許は有効期間が20年で終了するが、商標は更新で永続
  - 特許は無効審判で敗れても、営業秘密保護は残存
  - AI生成代替物による特許回避を商標・著作権・意匠が補完

### Rule 63.453: 競合の迂回（Design Around）への対抗スタッキング

- **単一特許への依存リスク**: 競合が意図的にDesign Aroundを行い、特許を回避する製品を開発するリスク
- **対抗策**:
  1. **クレーム囲い込み**: コア特許に加え、実装バリエーションを網羅する継続出願（→ §8 Rule 63.072）
  2. **意匠権によるUI囲い込み**: GUIの視覚的表現は意匠権で保護し、機能的Design Aroundが外観でも侵害を構成させる
  3. **商標による市場ロック**: 製品名・ロゴ・音・色彩等を多重商標登録し、競合が「同じに見える」製品を市場投入できないようにする
  4. **営業秘密のモート化**: 最もコアな「Secret Sauce」は公開せず営業秘密として維持（→ §6 Rule 63.050）

### Rule 63.454: 空間コンピューティング・ウェアラブルのIPスタッキング実務

- **Apple Vision ProライクなMR/XR製品のIP戦略**:
  - **深度センシング / 視線追跡**: センサーフュージョンアルゴリズムの特許
  - **ハンドジェスチャーUI**: GUI意匠権 + タッチフリーインタラクション特許
  - **空間音響**: 音声処理アルゴリズムの特許 + 音響ブランドの商標
  - **仮想空間内のブランド表示**: 仮想ブランドの3D商標登録（米国 USPTO 3D marks）
  - 仮想世界内でのIP侵害の管轄は**国際仲裁条項（WIPO / ICC）**を全ライセンス契約に必須化

- **ニューロテック / BCI ウェアラブルのIP優先事項**:
  - **神経データはPII最高機密ティア**: 神経信号データは個人識別可能性が極めて高く、GDPR/APPI の「特別カテゴリデータ」として扱うことを推奨
  - **データ収集同意とIPの交差**: 研究用に収集した神経データを商用AIモデルの学習に転用する場合は追加同意と著作権クリアランスが必要
  - **倫理審査との統合**: IRB（Institutional Review Board）承認取得プロセスにIP権利帰属の明確化を統合

## §60. Sovereign IP & National Security IP Framework

### Rule 63.460: CHIPS Act とIPガバナンス（米国・同盟国）

> [!CAUTION]
> **CHIPS Act補助金を受領した企業は、IPに対する政府アクセス権・強制ライセンス条件・利益超過回収（Clawback）条項への対応が義務化されている。補助金申請前にIP戦略と連動させよ。**

- **CHIPS and Science Act（2022年）の主要IP関連義務**:
  - **プロフィットシェアリング条項**: 補助金受領後に一定の利益閾値を超えた場合、超過分の一部を政府に返還義務
  - **強制ライセンスリスク**: 国家安全保障上の必要性がある場合、政府が補助金対象技術のライセンスを取得できる条文
  - **外国懸念企業（FEOC）制限**: 補助金受領後10年間、中国・ロシア・北朝鮮・イランとの重要技術共有・合弁禁止
  - **報告義務**: 技術使用状況・ライセンス供与状況の定期報告

```yaml
    - "補助金対象IPの特定と台帳化（特許・製造プロセス・ノウハウ）"
    - "プロフィットシェアリング閾値の財務モデルへの反映"
    - "FEOC制限の現在・将来のライセンス契約への影響評価"
  during_performance:
    - "IP使用状況の四半期報告体制の構築"

  post_completion:

    - "Clawback条項トリガーとなる利益レベルの年次評価"
```

### Rule 63.461: Dual-Use Research of Concern（DURC）とIPガバナンス

- **DURCの定義**: 正当な科学的目的のために行われる研究であっても、公衆衛生・安全・セキュリティ・経済的安全・国家安全保障への脅威となる非意図的リスクを有する研究
- **IP戦略への影響**:
  - DURC認定研究の成果は米国NSPM-33（National Security Presidential Memorandum）のResearch Securityルール適用対象
  - 外国人研究者との共同研究・成果発表・特許出願に**政府事前審査**が必要になる場合がある
  - 成果のOSSリリース・技術ブログ公開も「Deemed Export」として規制対象になる可能性

- **DURCリスク評価フレームワーク**:
  1. **技術分類**: NSPM-33の重要・新興技術リスト（AI・量子・バイオ・先端製造・宇宙）との照合
  2. **資金源の確認**: 外国政府・外国企業からの資金提供の有無とFRGC（Foreign Research Grant Compliance）確認
  3. **共著者・アクセス権の審査**: 外国籍研究者のアクセス制限（Need-to-Know）の適用
  4. **保護措置の文書化**: IP保護措置・アクセス制御の記録をDDの証拠として整備

### Rule 63.462: EU Critical Raw Materials Act とIPサプライチェーン

- **EU Critical Raw Materials Act（CRM Act、2024年施行）**:
  - 戦略的原材料（リチウム・コバルト・レアアース等）の採掘・加工技術のIP保護が欧州産業政策の重要課題に
  - **IP義務の概要**:
    - CRM Act対象プロジェクトへの公的補助金受領企業はIPポートフォリオの開示・技術移転条件の透明性が求められる
    - 中国系企業への技術移転・ライセンスに事前届出義務が生じる可能性
  - **戦略的示唆**: サプライチェーン上のCRM関連IP（製造工程特許・精製技術）のリスト化と地政学的リスク評価（→ §51）を統合

### Rule 63.463: FDI審査とIPの交差点高度化

- **FIRRMA以降のCFIUS審査強化（2022 TID Rulesets）**:
  - **TID US Business定義の拡大**: 重要技術（Critical Technology）・重要インフラ（Critical Infrastructure）・センシティブデータを扱う企業は強制申告義務
  - **Critical Technology**: EAR管理技術、ITAR管理技術、NSF指定新興技術 — IPポートフォリオとの照合が必須
  - IPのM&Aによる外国への移転がCFIUS審査の主要トリガーになることを事前評価（→ §51 Rule 63.373参照）

- **英国 National Security and Investment Act（NSI Act、2022年）**:
  - 17の重要セクター（AI・量子・先端素材・エンジニアリング生物学・防衛等）でのM&A・IPライセンス・合弁は**強制事前申告**
  - IPの単独譲渡（特許・営業秘密の移転）も審査対象になる可能性（2024年ガイダンス確認必須）

- **日本の外為法（FEFTA）の強化傾向**:
  - コア業種（半導体・AI・宇宙・防衛）への外国資本によるIP支配に対する審査が厳格化
  - 「技術的優位性の高い技術」の国外流出防止のため、IP移転を伴うM&Aには事前相談を推奨

### Rule 63.464: ソブリンIP戦略（国家レベルのIP保護への対応）

- **「ソブリンAI」政策との交差点**:
  - 多くの国家が自国のAIインフラ・学習データへの主権主張を強化。企業が国内で開発したAIモデルの海外への移転・ライセンシングに制限がかかるリスク
  - **対応策**: 国別法人による「ローカルIP保有構造」の設計（→ §61 IPディールアーキテクチャと連携）

- **データローカリゼーション法とIPの衝突**:
  - ロシア（FZ-242）、中国（PIPL / DSL）、インド（DPDP）等のデータローカリゼーション義務が、グローバルなIPデータの自由な移転を阻害
  - **学習データ・モデルウェイト**のローカリゼーション義務については各国法務専門家と事前評価が必須

## §61. IPを基軸としたM&A・ファンドレイジングのディールデザイン

### Rule 63.470: IP-Centric Deal Structuring の原則

> [!IMPORTANT]
> **「IPがどこに座すか」がディールの税効率・リスク配分・統合コストを決定する。IP戦略をディール設計の最初の議題に置け。**

- **IPの所在（IP Siting）によるディール設計の違い**:
  - **OpCo-IPCo構造**: 事業会社（OpCo）から知財保有会社（IPCo）を分離し、OpCoがIPCoからライセンスを受ける構造
    - M&A時: 買収者はIPCoのみを取得しOpCoは残す（事業リスクの分離）
    - 倒産保護: OpCo倒産時もIPCoは独立して事業継続可能
  - **シングルエンティティ構造**: IPを事業会社に統合保有 — 中小企業・スタートアップ段階では管理コスト面で現実的

### Rule 63.471: IP Earnout メカニズムの精緻設計

- **IP Earnoutとは**: M&Aクロージング後、IPが特定の収益・マイルストーンを達成した場合に追加対価を支払う仕組み
- **チェックリストと必須条項**:
  - [ ] **Earnout Base**: どのIPの、どの収益（ライセンス料/プロダクト売上/特許勝訴回収）をベースとするかを明示
  - [ ] **計算期間**: 通常12-36ヶ月。技術の成熟速度に応じて設定
  - [ ] **加速条項（Acceleration）**: 買収者がIPを積極的にライセンスせず、Earnoutを意図的に妨害した場合の加速支払い条件
  - [ ] **マイルストーン定義**: 「特許成立」「初回ライセンス収入$1M」「FDA/CEマーク取得」等を具体的に定義
  - [ ] **会計・報告権**: 売手の独立した会計士によるEarnout計算の検証権（Audit Right）
  - [ ] **MAC例外**: 市場全体の崩壊（パンデミック/戦争）によるEarnout未達を免責する条項

```yaml
# IP Earnout 計算モデル（概念例）
ip_earnout_model:
  base_consideration: "$50M（クロージング時）"
  earn_out_period: "36ヶ月"
  milestones:
    - trigger: "特許EP0000000の欧州登録完了"
      payment: "$2M"
      deadline: "クロージング後18ヶ月以内"
    - trigger: "ライセンス収入累計$5M達成"
      payment: "$5M（利益の20%を上限）"
      deadline: "クロージング後36ヶ月以内"
    - trigger: "FTO調査クリーン確認（コア製品領域）"
      payment: "$1M"
      deadline: "クロージング後6ヶ月以内"
  acceleration_trigger: "買収者が本契約後12ヶ月以内に対象IPのライセンス活動を停止した場合"
  acceleration_amount: "未達成Earnout残額の50%を即時支払い"
```

### Rule 63.472: IP Escrow の高度設計

- **標準的Escrow（→ §30 Rule 63.221）の高度化**:
  - **IP専用Escrow口座**: M&A対価の一部（通常10-20%）をIP関連表明保証違反のためにIP専用エスクロー口座へ隔離
  - **Basket / Deductible**: IP関連クレームの最低限度額（De Minimis: $50K-$200K）と総合ミニバスケット（1-2%）の設定
  - **Reverse Break-up Fee（逆解約金）**: 買収者がCFIUS審査・IP規制問題を理由にクロージングを拒否した場合の売手への補償を契約に明記

- **Source Code Escrow（ソースコード預託）**:
  - SaaSプロダクト会社のM&A/ファンドレイジングでは、買収者・投資家がソースコードを第三者のEscrowエージェントに預けることを要求するケースあり
  - **主要エージェント**: Iron Mountain / NCC Group / EscrowTech
  - **リリース条件**: 事業の停止・倒産・重大な契約違反時にライセンシーがソースコードアクセスを取得
  - **アップデート義務**: 四半期ごとのソースコード更新とエスクローへの再預託

### Rule 63.473: SPV（特別目的会社）を活用したIP保護と資金調達

- **IP-SPVの構造**:
  1. 親会社が特定のIPポートフォリオをSPVに移転
  2. SPVがIPを担保に証券化・債券発行または第三者ファイナンス調達
  3. 親会社はSPVからIPのライセンスバックを受ける（事業継続性維持）

- **IP-SPVの活用場面**:
  - バランスシートを傷めずにIPを担保としたデット調達
  - 破産隔離（Bankruptcy Remote）構造による訴訟・倒産リスクからのIPの保護
  - M&A時の特定IP資産のみの切り出し売却

- **税務・会計上の考慮事項**:
  - SPVへのIP移転は独立当事者間原則（ALP）に基づく移転価格評価が必要
  - IFRS/GAAPにおける無形資産の認識・償却ルールへの準拠を確認
  - BEPS Pillar Twoのグローバル最低税率15%がSPV法域に適用されるかを事前評価（→ §53 Rule 63.392参照）

### Rule 63.474: Secondary Transaction（セカンダリー取引）におけるIP考慮事項

- **Secondary市場でのIPリスク**:
  - LP持分の移転・GP引継ぎ時にポートフォリオ企業のIP状況がDD対象になる
  - **Continuation Vehicles（CVs）**: 特定のポートフォリオ企業のIPを長期保有するためのファンド別法人 — IP資産の評価が基準価格の核となる
  - IPの陳腐化リスクと差止命令リスクはCV運用期間中に変動するため、年次IP再評価を義務化

- **GP主導Secondary時のIP保護**:
  - 旧投資家（LP）から新投資家（LP）への持分移転にもChange of Control条項が適用される可能性を事前に確認
  - ターゲット企業のNDA・ライセンス契約における「スポンサーチェンジ条項」を精査
