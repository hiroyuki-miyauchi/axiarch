## §1. 最高指令・適用範囲 (Supreme Directive & Scope)

### 1.1. 基本原則
-   **Privacy by Default**: 全ての個人データ処理は、**利用目的の特定・最小化・保存期間の限定**を原則とする。
-   **Accountability**: データ管理者は、法的義務の遵守を**証明する責任**を負う（GDPR Art.5(2)）。
-   **Transparency**: データの収集・利用・共有について、ユーザーに**平易な言語**で説明する義務がある。
-   **Purpose Limitation**: 収集時に特定した目的以外でのデータ利用は、法的根拠なく**禁止**。

## §3. データ主権・越境移転 (Data Sovereignty & Cross-Border Transfer)

### 3.1. データレジデンシー
-   **Rule 3.1.1**: ユーザーデータは、そのユーザーが居住する地域のリージョンに保存する。
-   **Rule 3.1.2**: CDNキャッシュ・一時ファイルを含め、PIIが指定リージョン外に永続化されないことを保証。
-   **Cross-Reference**: `600_security_privacy.md` §8.2 (Data Residency)

### 3.2. データポータビリティ (Portability)
-   **Export権**: ユーザーは自身の全データを機械可読形式（JSON/CSV）で一括ダウンロードする権利を持つ。
-   **SLA**: 開示請求から**72時間以内**に提供可能な体制を維持。実装は非同期ジョブで行う。
-   **Export頻度制限**: エクスポート要求は**月2回**まで。超過時は次回利用可能日を通知。
-   **Import（受入）**: CSV/JSON形式での一括インポート機能を提供。
    -   **Validation Pipeline**: 型チェック、重複チェック（`ON CONFLICT`）、UTF-8正規化を**全て通過**してから保存。
    -   **Error Report**: 失敗行は行番号付きエラーメッセージで詳細レポート。
    -   **Dry Run Mode**: 本番書き込み前のバリデーション専用モードの実装を推奨。

### 3.3. 忘れられる権利 (Right to Erasure)
-   **Rule 3.3.1**: 退会処理は関連PIIを**物理削除（Hard Delete）**し、復元不可能にする。
-   **Rule 3.3.2**: 法的保管義務のあるデータのみCold Storageへ隔離。保存期間経過後に物理削除。
-   **Rule 3.3.3**: SAR（Subject Access Request）は受領から**30日以内**に対応完了。
-   **Rule 3.3.4**: 関連テーブルへの連鎖削除順序はService層で明示的に管理。
-   **Cross-Reference**: `600_security_privacy.md` §4.5 (Right to be Forgotten)

### 3.4. 越境移転フレームワーク

| メカニズム | 用途 | 要件 |
|:----------|:-----|:-----|
| **十分性認定** | 日EU間、日英間等 | 移転先国が適切な保護水準を有すると認定済み |
| **SCC** | 十分性認定のない国への移転 | EU標準契約条項の締結 + TIA実施 |
| **BCR** | グループ企業間移転 | 監督機関の承認 |
| **CBPR** | APEC圏 | CBPR認証取得 |
| **明示的同意** | 個別ケース | 最終手段。反復的移転には不適 |

#### 3.4.1. データローカライゼーション要件（国別）

| 国 | ローカライゼーション要件 | 越境移転条件 | 制裁リスク |
|:---|:----------------------|:-------------|:----------|
| **中国** | CIIOは個人情報・重要データを**国内保存必須** | 安全評価 / SCC / 認証 + 個別同意 | 売上5% or RMB 5,000万 |
| **ロシア** | 個人データの**収集・保存は国内DB必須** | 外国DBの使用禁止 | 高額罰金 |
| **ベトナム** | 「重要データ」「コアデータ」は国内保存 | 影響評価 + 当局の承認 | 売上の5%（2026年施行） |
| **インド** | SDFは特定カテゴリの個人データを国内保存。RBIは決済データ国内保存義務 | ホワイトリスト方式 | ₹250クロール |

-   **Transfer Impact Assessment (TIA)** — CNIL 2025.1最終ガイドライン準拠 6ステップ:
    1.  **移転の特定**: 移転対象データ、関係者（exporter/importer）、移転の文脈を明確化
    2.  **移転ツールの特定**: 使用するGDPR Art.46メカニズム（SCC/BCR等）を決定
    3.  **移転先国の法制・実務評価**: 移転先国の法制度（文書上の法律と実際の運用）を評価。政府がデータにアクセスするリスク（監視法制等）を重点的に評価
    4.  **補完措置の特定・採択**: 保護水準のギャップがある場合、技術的（暗号化・仮名化）・契約的・組織的補完措置を採択
    5.  **実施・文書化**: 評価結果・採択した措置・意思決定根拠を全て文書化。監査証跡として保全
    6.  **定期的再評価**: 法制度変更・新リスク・データ利用変更を踏まえ、**年1回以上**または重大変更時に再評価
-   **Anti-Pattern**: 「SCCを締結したからTIAは不要」は誤り。SCCはTIAと補完措置が前提。
-   **EU-US Data Privacy Framework (DPF)**: 2023.7十分性認定。2025.9に欧州一般裁判所がDPFへの挑戦を棄却し、一時的に安定。ただしnoybは再挑戦を表明。DPFに依拠する場合でもTIAは推奨。
-   **UK十分性認定延長**: EU-UK十分性認定は**2031年12月**まで延長（ドラフト決定2025.10）。DUAAの影響評価を経て欧州委員会が十分性を確認。

### 3.5. データ加工基準
-   **仮名加工情報**: 他情報と照合しない限り個人を識別不可な状態に加工。内部分析目的での利用緩和可能。第三者提供は原則禁止。
-   **匿名加工情報**: 再識別が不可能な状態に加工。本人同意なく第三者提供可能。加工基準遵守・公表義務・再識別禁止を厳守。
-   **使い分け**: 内部分析・プロダクト改善 → 仮名加工情報 / 外部提供・オープンデータ → 匿名加工情報。

### 3.6. DSAR自動化ワークフロー

| ステップ | 処理内容 | 自動化レベル |
|:--------|:---------|:-----------|
| **1. 受付** | Webフォーム / API / メール経由での一元受付 | 完全自動 |
| **2. 本人確認** | 段階的認証（ログイン済み → 即時 / 未認証 → メール+身分証照合） | 半自動 |
| **3. データ検索** | 全データストア横断の自動PII検索（DB、S3、ログ、バックアップ） | 完全自動 |
| **4. レビュー・墨消し** | 第三者PIIの自動マスキング + 人間レビュー | 半自動 |
| **5. 応答作成** | 準拠形式（JSON/CSV/PDF）での自動パッケージング | 完全自動 |
| **6. 配信** | 暗号化済みダウンロードリンクの自動送信 | 完全自動 |
| **7. 監査記録** | 全操作の不変ログ記録（改竄防止） | 完全自動 |

-   **SLA管理**: 要求種別ごとの法定期限を自動追跡し、期限5日前にアラートを発火:
    -   GDPR: **30日**（延長時は+60日、理由通知必須）
    -   CCPA/CPRA: **45日**（延長時は+45日）
    -   Global Privacy Laws: **30日**（速やかに対応する義務）
-   **Anti-Pattern**: DSARをCSチームの手動メール対応に依存する運用は規模拡大時に破綻する。

#### 3.6.1. DSAR API設計パターン

```typescript
// DSAR受付APIエンドポイント設計
interface DSARRequest {
  requestType: 'access' | 'deletion' | 'correction' | 'portability' | 'opt-out';
  subjectId: string;         // ユーザーIDまたはメール
  jurisdiction: 'GDPR' | 'CCPA' | 'Global Privacy Laws' | 'LGPD' | 'PIPA';
  verificationMethod: 'authenticated' | 'email' | 'id_document';
  requestedAt: string;       // ISO 8601
}

interface DSARResponse {
  requestId: string;         // UUID v4
  status: 'received' | 'verified' | 'processing' | 'completed' | 'rejected';
  deadlineAt: string;        // 法域別法定期限の自動計算
  estimatedCompletionAt: string;
}

// 法定期限の自動計算
const DSAR_DEADLINES: Record<string, number> = {
  'GDPR': 30,    // 30日（延長可能+60日）
  'CCPA': 45,    // 45日（延長可能+45日）
  'Global Privacy Laws': 30,    // 30日
  'LGPD': 15,    // 15日
  'PIPA': 10,    // 10日
};

function calculateDeadline(jurisdiction: string, requestedAt: Date): Date {
  const days = DSAR_DEADLINES[jurisdiction] ?? 30;
  const deadline = new Date(requestedAt);
  deadline.setDate(deadline.getDate() + days);
  return deadline;
}
```

-   **Mandate**: DSAR APIは全操作を不変ログとして記録し、監査証跡を保全すること。

### 3.7. 自動意思決定とプロファイリング (Automated Decision-Making)
-   **Law**: GDPR Art.22は、個人に法的効果またはこれに類する重大な影響を及ぼす**完全自動化された意思決定**に対し、異議申立権を保証する。CCPA/CPRAは2026.1よりADMT（Automated Decision-Making Technology）規制を施行。
-   **要件**:
    -   自動意思決定の存在、関与するロジック、想定される結果をユーザーに**事前開示**
    -   ユーザーが**人間によるレビューを要求する権利**を保証
    -   **オプトアウト権**: CCPA/CPRAはプロファイリング目的の自動意思決定に対するオプトアウト権を追加
-   **CCPA ADMT規則（2026.1施行）**:
    -   「人間の意思決定を実質的に代替する」技術が対象
    -   新規処理は開始**前**にリスク評価を完了
    -   2026.1以前の既存処理は**2027.12.31まで**にリスク評価を完了
-   **Colorado AI Act（2026.6施行）**:
    -   「consequential decisions」（金融、住宅、保険、医療、教育、雇用等に関する意思決定）を行うAIシステムが対象
    -   Deployer: NIST AI RMFまたはISO 42001に準拠したリスク管理プログラムの構築・維持義務
    -   Deployer: デプロイ前の影響評価実施 + 年次更新（または重大変更後90日以内に更新）
    -   消費者への開示義務: AIが意思決定に関与していること、異議申立方法の告知
-   **実装パターン**:

```typescript
// 自動意思決定の透明性フラグ
interface AutomatedDecision {
  decision_id: string;
  decision_type: 'credit_scoring' | 'content_recommendation' | 'pricing';
  is_fully_automated: boolean;
  logic_summary: string; // ユーザー向け平易な説明
  human_review_requested: boolean;
  opt_out_available: boolean;
}
```

#### 3.7.1. 説明権・アルゴリズム透明性要件
-   **EU AI Act Art.86**: 高リスクAIシステムによる意思決定の影響を受ける個人は、AIシステムの役割・主要な判断要素・意思決定の根拠について**明確で意味ある説明**を受ける権利を有する。
-   **GDPR Art.13-15**: 自動化された処理の「関与する論理、その重要性、想定される結果」についての意味ある情報を提供する義務。
-   **XAI（説明可能なAI）実装チェックリスト**:
    -   [ ] モデルの判断要因を抽出するXAI手法（SHAP / LIME等）を組み込んでいるか
    -   [ ] エンドユーザー向けに**平易な言葉**で説明を提供するUIがあるか
    -   [ ] 説明の内容が「違反を特定し苦情を申し立てる」に十分な具体性を有しているか
    -   [ ] 説明の生成・提供プロセスが監査ログとして記録されているか
    -   [ ] モデル更新時に説明の一貫性が検証されているか
-   **Anti-Pattern**: 「アルゴリズムが复雑なので説明不可能」は法的に許容されない。技術的複雑性は組織側の問題であり、データ主体の権利を制限する理由にならない。

## §4. 同意管理・CMP (Consent Management)

### 4.1. CMP導入義務
-   **Mandate**: Cookieやトラッキングの同意は、OneTrust等の信頼できるCMPを使用して管理し、各国の法改正に自動追従する。
-   **Cross-Reference**: `600_security_privacy.md` §4.12 (Cookie Consent技術実装)

### 4.2. Strict Opt-In
-   **Rule 4.2.1**: 必須Cookie以外のトラッキングは、ユーザーの**明示的な同意**なしにブロック。ナッジ禁止。
-   **Rule 4.2.2**: 同意バナーでは「すべて拒否」を「すべて許可」と**同等の視認性**で配置。

#### 4.2.1. Consent Management API実装パターン

```typescript
// --- Consent Management API エンドポイント設計 ---
// POST   /api/v1/consents          - 同意記録の作成
// GET    /api/v1/consents/:userId   - ユーザーの同意状態取得
// PATCH  /api/v1/consents/:userId   - 同意状態の更新（部分撤回等）
// DELETE /api/v1/consents/:userId   - 全同意の撤回

interface ConsentRecord {
  userId: string;
  consents: {
    category: 'necessary' | 'analytics' | 'marketing' | 'personalization';
    granted: boolean;
    grantedAt: string | null;     // ISO 8601
    revokedAt: string | null;
    legalBasis: 'consent' | 'legitimate_interest' | 'contract';
    policyVersion: string;       // 同意時のポリシーバージョン
  }[];
  metadata: {
    ipAddress: string;           // ハッシュ化して保存
    userAgent: string;
    jurisdiction: string;        // GeoIPから自動判定
    consentMethod: 'banner' | 'settings' | 'api';
  };
}
```

```typescript
// --- Server-Side Consent Enforcement ミドルウェア ---
import type { NextRequest, NextResponse } from 'next/server';

export function consentMiddleware(req: NextRequest): NextResponse {
  const consentCookie = req.cookies.get('consent_state');
  const consent = consentCookie
    ? JSON.parse(consentCookie.value)
    : { analytics: false, marketing: false };

  // 同意無しのトラッキングスクリプトをブロック
  const response = NextResponse.next();
  response.headers.set('X-Consent-Analytics', String(consent.analytics));
  response.headers.set('X-Consent-Marketing', String(consent.marketing));

  // CSPで同意なしのサードパーティスクリプトをブロック
  if (!consent.marketing) {
    const csp = response.headers.get('Content-Security-Policy') || '';
    response.headers.set(
      'Content-Security-Policy',
      `${csp}; script-src 'self'` // マーケティングスクリプトのドメインを除外
    );
  }
  return response;
}
```

-   **Mandate**: 同意状態はDBで管理し、Cookieはクライアント側のキャッシュとしてのみ使用。サーバーサイドでの強制を必ず実装。
-   **Cross-Reference**: `600_security_privacy.md` (CSP設定)

### 4.3. Cookie分類と同意要件

| 種別 | 例 | 同意 |
|:-----|:---|:-----|
| **必須Cookie** | セッションID, CSRFトークン | 不要（サービス運営に必須） |
| **機能Cookie** | 言語設定, テーマ設定 | 不要（ユーザー利便性） |
| **分析Cookie** | Analytics, ヒートマップ | **要同意** |
| **広告Cookie** | 広告配信, リターゲティング | **要同意** |

### 4.4. CMP実装基準
-   **初期状態**: 分析・広告Cookieは初回訪問時**無効**。明示的同意後に有効化。
-   **バナーUI**: 「すべて同意」「カスタマイズ」「拒否」の3選択肢を提供。
-   **永続化**: 同意状態はDBまたはLocalStorageに保存。再訪時にバナーを再表示しない。
-   **同意撤回**: ユーザーはいつでもフッター等から同意を撤回できる導線を確保。
-   **サーバーサイド記録**: 同意状態をサーバーサイドに記録し、監査証跡を保持。
-   **Mandate**: デフォルトは「全拒否」。ユーザーが能動的に同意したカテゴリのみ有効化。

### 4.7. GDPR Digital Omnibus改革（2025.11提案）

> [!NOTE]
> Digital Omnibus PackageはGDPRの「置換」ではなく「調和・明確化・簡素化」を目的とする。欧州議会・理事会で審議中。施行まで現行ルールを遵守。

#### 4.7.1. Cookie同意の簡素化
-   **ワンクリック拒否**: Cookieバナーにおける「すべて拒否」のワンクリック実装が義務化される見込み。「すべて同意」と同等の目立ちやすさが必須。
-   **同意免除の拡大**: **オーディエンス計測Cookie**および**セキュリティ維持Cookie**が同意不要に。ターゲティング・広告Cookieは引き続き同意必須。
-   **再要求制限**: 同意を拒否したユーザーへの再同意要求は**6ヶ月間禁止**。
-   **Cookie違反罰金**: Cookie関連違反に対する制裁金上限をGDPR水準に統一（最大€20M / 全世界売上4%）。
-   **集中設定構想**: ユーザーがブラウザ設定でCookie設定を一括管理し、個別サイトでのバナー表示を簡素化する構想が提案中。

#### 4.7.2. 「個人データ」定義の明確化
-   **エンティティ固有識別可能性テスト**: 「合理的に使用されうる手段」に基づきエンティティごとに識別可能性を判断。CJEU SRB判決の法典化。
-   **仮名化データの明確化**: コントローラーが特定エンティティについて仮名化データがGDPR適用範囲外となる条件を評価可能に。分析・AIパイプラインに影響。
-   **エンジニアリング影響**: 仮名化処理のフローを文書化し、再識別リスクの評価プロセスを確立すること。

#### 4.7.3. DSAR（データ主体アクセス要求）改革
-   **濫用拒否の新根拠**: データ主体がデータ保護とは無関係な目的でDSARを濫用する場合、拒否または合理的手数料の請求が可能に。
-   **Action**: DSAR処理フローに濫用判定プロセスを追加。ただし判定基準は慎重に運用し、正当な要求を誤って拒否しないこと。

#### 4.7.4. AIデータ処理の正当利益明確化
-   **AI開発への正当利益**: AI開発・運用における個人データ処理に正当利益を法的根拠として適用可能に（バランシングテスト充足が条件）。
-   **特別カテゴリデータ**: AI開発・運用における特別カテゴリデータ（SCD）の処理に新条件を導入。
-   **Anti-Pattern**: 「Digital OmnibusでAI学習が自由になった」は誤り。バランシングテストと透明性義務は引き続き必須。

#### 4.7.5. データ漏洩通知の変更
-   **通知閾値引き上げ**: DPAへの通知閾値を「リスクあり」から「**高リスク**」に引き上げ。
-   **通知期限延長**: 72時間→**96時間**に延長提案。
-   **ワンストップ通知**: 複数規制レジームにまたがる場合の**単一通知窓口**を提案。

#### 4.7.6. DPIA統一化
-   **EU全体のガイダンス統一**: DPIA要件をEU全体で調和。DPIAが必要な処理活動の統一リストと**標準テンプレート**策定を提案。

-   **Mandate**: 提案段階のため施行まで現行ルールを遵守。ただし「すべて拒否」ボタンの同等配置は現行Rule 4.2.2で既に義務付け済み。Digital Omnibusの動向を**四半期ごと**にモニタリングし、施行に備えた実装準備を開始すること。

### 4.5. Cookieレス追跡技術の法的地位
-   **ブラウザフィンガープリンティング**: ePrivacy指令下ではCookieと**同等の同意義務**が適用。同意なきフィンガープリンティングは違法。
-   **Server-Side Tagging**: サーバーサイドタグも個人データを処理する場合は同意必要。「サーバー側だから同意不要」は**誤り**。
-   **Privacy Sandbox API廃止（2025.10）**: GoogleはTopics API、Protected Audience、Attribution Reporting、IP Protectionを**利用終了**とした。低採用率とエコシステムのフィードバックが要因。
    -   **継続API**: CHIPS（パーティションCookie）、FedCM（連合認証管理）、Private State Tokens（不正防止）は維持。
    -   **エンジニアリング影響**: Privacy Sandbox依存の広告/アトリビューション実装は**即座にファーストパーティデータ戦略へ移行**が必須。
-   **Google Cookie廃止撤回（2025.4）**: Googleは第三者Cookie完全廃止計画を公式撤回。Chrome設定でのユーザー選択方式に移行。
    -   **現実**: Safari/Firefoxは既にデフォルトブロック。全ユーザーの約30-40%は第三者Cookie非対応環境。
    -   **Mandate**: Cookie依存を前提としないデータ収集・広告設計をデフォルトとし、ファーストパーティデータ・コンテキスト広告を優先。
    -   **Anti-Pattern**: 「Googleが撤回したからCookieに依存して良い」は誤り。規制とブラウザ仕様の両面でCookieレス化は不可逆。
-   **UK DUAA 2025**: 英国は低リスクな分析Cookieの同意要件を緩和（通知+オプトアウト方式）。マーケティングCookieは引き続き同意必須。Cookie違反罰金をGDPR水準に引き上げ（最大£17.5M / 全世界売上4%）。
-   **GDPR Digital Omnibus（2025Q4提案）**: Cookie同意の簡素化、SME向け免除拡大を提案中。施行まで現行ルールを遵守。

### 4.6. サードパーティスクリプトガバナンス
-   **原則**: サードパーティのJS/SDKは、ユーザーが該当カテゴリに同意するまで**一切ロードしてはならない**。
-   **Tag Managerルール**: GTM等のタグマネージャーを使用する場合、CMPと連動して同意ステータスに応じたタグ発火制御を実装。
-   **ベンダー契約**: サードパーティSDKの導入時に、データ収集範囲・目的・保持期間をDPA（§13）で明記。
-   **監査**: 実際にロードされるスクリプトを四半期ごとに監査し、未承認のスクリプト混入を検出。

### 4.8. Universal Opt-Out Mechanism（GPC）実装

> [!IMPORTANT]
> 米国複数州でGlobal Privacy Control（GPC）シグナルの認識が**法的義務**となっている。CaliforniaではCCPA/CPRAの下で2023年より義務化済み。2026年1月よりConnecticut、Oregonも義務化。

-   **定義**: GPCはユーザーがブラウザ設定で「販売・共有しない」意思を一括表明する技術仕様（`Sec-GPC: 1`ヘッダー）。
-   **義務化州（2026年時点）**: California, Colorado, Connecticut, Montana, Oregon, Texas
-   **実装要件**:

```typescript
// --- GPC Signal Detection Middleware ---
import type { NextRequest, NextResponse } from 'next/server';

export function gpcMiddleware(req: NextRequest): NextResponse {
  const gpcSignal = req.headers.get('Sec-GPC');
  const response = NextResponse.next();

  if (gpcSignal === '1') {
    // GPCシグナル検出: オプトアウトとして処理
    response.headers.set('X-GPC-Honored', 'true');

    // 1. ターゲティング広告Cookieを無効化
    // 2. サードパーティデータ共有を停止
    // 3. 「販売・共有しない」状態をDB記録
    // 4. 分析Cookieの同意状態を保守的に設定
  }

  return response;
}
```

-   **Mandate**: GPCシグナルを受信した場合、**追加のユーザーアクションなし**にオプトアウトとして処理すること。「GPCを無視してバナーで再確認を求める」は違法。
-   **Cross-Reference**: §6.3.2 (米国州法), §4.2 (Strict Opt-In)

## §5. 利用規約・プライバシーポリシー (ToS & Privacy Policy)

### 5.1. 動的同意管理
-   **Rule 5.1.1**: ユーザーが「いつ、どのバージョンの規約に同意したか」をDBに記録し、法的証跡として保持。
-   **Rule 5.1.2**: 規約改定時は、次回ログイン時に同意画面を強制表示。
-   **同意記録の必須フィールド**: ユーザーID、同意種別、同意日時、IPアドレス、User-Agent、規約バージョン

### 5.2. 法的アーカイブ
-   **Rule 5.2.1**: 過去の規約は削除せず、Permalinkとして永続的にアクセス可能にする。
-   **Rule 5.2.2**: 規約ページ更新時は、更新前データをリビジョンテーブルに自動保存（削除禁止）。

### 5.3. レイヤード・ノーティス
-   プライバシーポリシーは、法的詳細条文の前に**平易な言葉（Plain Language）**での要約を必ず表示。

### 5.4. 法的文書のSSoT
-   **Law**: 法的文書をソースコード内にハードコードすることは禁止。
-   **Action**: 法的文書は**Headless CMS**または**データベース**を唯一の正とする。
-   **No Code Deployment for Terms**: 法的文言の修正にアプリの再デプロイを要する設計は設計上の欠陥。
-   **Cross-Reference**: `600_security_privacy.md` §3.3 (Legal Content Consistency Protocol)

### 5.5. 同意の法的有効性基準

| 同意形式 | 法的有効性 | 推奨場面 |
|:---------|:----------|:---------|
| **Clickwrap** | ✅ 強い | 利用規約同意、課金同意 |
| **Sign-in-wrap** | ⚠️ 中程度 | 低リスクの規約更新 |
| **Browsewrap** | ❌ 弱い | 使用禁止 |

### 5.6. 紛争解決・仲裁条項
-   **仲裁条項**: B2B契約では仲裁条項の採用を推奨。準拠法・仲裁機関・言語を明記。
-   **クラスアクション免除**: B2C規約でのクラスアクション免除条項はEUでは無効。米国も州法により制限あり。
-   **ADR**: EU向けサービスではODR（オンライン紛争解決）プラットフォームへのリンクを提供義務。

## §6. 地域別コンプライアンス (Regional Compliance)

### 6.1. 日本法

#### 6.1.1. 特定商取引法
-   有料課金を行う場合、特定商取引法に基づく表記をアプリ内に必ず設置。
-   **開示項目**: 事業者名、住所、電話番号、代表者名、販売価格、支払時期・方法、商品引渡時期、キャンセルポリシー。
-   **最終確認画面**: 決済直前画面に**契約期間**、**課金タイミング**、**解約条件**を明確表示。

#### 6.1.2. Financial Regulations
-   **供託金回避**: 前払式支払手段の発行は供託金義務のリスクあり。原則Apple/GoogleのIAP利用。

#### 6.1.3. 広告表示義務（ステルスマーケティング規制）
-   **Law**: 広告主からの金銭授受があるコンテンツには「PR」「広告」等の表記を必ず表示。
-   **System Guard**: `is_sponsored`フラグを持つ投稿にシステムレベルで自動的に広告表記を付与。

#### 6.1.4. メール配信コンプライアンス
-   **Opt-In**: 広告メール送信には事前の明示的同意を必須とし、同意日時を記録。
-   **Sender Disclosure**: 全メールに送信者名称、連絡先、配信停止方法を表示。
-   **Unsubscribe**: ワンクリック配信停止リンク必須。
-   **Audit Trail**: 全システムメール送信をログに記録。メールアドレスはマスキングして保存。

#### 6.1.5. サブスクリプション自動更新法
-   自動更新の事実と条件を契約**前**に明確に開示
-   無料トライアルから有料への自動移行時は、移行**前**に明示的な確認ステップを必須
-   解約方法を登録方法と**同等以下の難易度**で提供
-   更新**7日前**までにリマインダー通知を送信
-   **FTC Click-to-Cancel Rule**: 解約を登録と同じチャネルで完了可能にする義務。

#### 6.1.6. APPI改正法（2027年施行見込み）
-   **立法進捗**: PPCが2023年11月に見直しを開始。2024年6月中間報告。**2026年1月9日に「改正方針」を正式公表**。2026年通常国会（1月23日開会）に改正法案を提出予定。2027年施行見込み。
-   **確定方針（2026.1改正方針）**:
    -   **行政課徴金制度の導入**: 悪質な個人データ大量売買等に対する**行政罰金**の新設。現行の指導・勧告・命令中心の執行体制を大幅強化
    -   **AI開発におけるデータ利用緩和**: 犯歴・病歴・人種等の**要配慮個人情報**について、AI学習目的での**同意なし利用を許可**する方針。統計処理・モデル学習に限定
    -   **認定団体のインシデント報告緩和**: 認定団体については不正アクセスに対する報告期限を**30日または60日**に延長（現行の即時報告義務から緩和）
-   **引き続き検討中の項目**:
    -   **差止請求権**: 適格消費者団体による差止請求・損害回復制度の導入検討
    -   **生体認証データ規制**: 現行APPIでは生体認証データが「要配慮個人情報」に含まれていない。規制強化を検討中
    -   **子供データ**: 児童の個人情報に関する明確なガイダンス（同意手続き、安全管理措置）の策定
    -   **不正利用規制の明確化**: 個人情報の不正利用・不正取得の解釈範囲を特定・類型化
    -   **オプトアウト制度の強化**: 第三者提供におけるオプトアウト制度の義務強化
-   **Action**: 2026年国会での法案審議を監視。行政課徴金制度は**罰金額・適用基準の確定次第**でコンプライアンス予算を確保。AI開発緩和は要配慮個人情報の利用方針に直接影響するため、AIガバナンス委員会（§11.9）で対応方針を策定。
-   **Anti-Pattern**: 「AI開発緩和が確定したから要配慮個人情報を自由に学習データに使える」は誤り。緩和は統計処理・モデル学習に限定され、**目的外利用・再識別・第三者提供は引き続き禁止**。

### 6.2. EU法

#### 6.2.1. GDPR固有要件
-   **DPO設置**: 大規模な個人データ処理を行う場合、DPOの選任が義務。
-   **DPIA**: 高リスクのデータ処理を開始する前にDPIAを実施。
-   **ROPA**: データ処理活動の記録を維持。
-   **EDPB 2026協調執行**: 2026年はGDPR Art.12-14（透明性義務）が協調執行の優先テーマ。
-   **Cross-Reference**: `600_security_privacy.md` §4.3 (PIA)

#### 6.2.2. DPIA実施メソドロジー (7ステップ)
-   **Trigger**: 大規模プロファイリング / センシティブデータの大規模処理 / AI・新技術の導入 / 自動意思決定 / 脆弱者データ処理 / データセットの結合
-   **7ステップ**:
    1.  **必要性判定**: トリガーに該当するか評価
    2.  **処理の記述**: 性質・範囲・文脈・目的・データフロー・保存期間の詳細記述
    3.  **必要性・比例性評価**: 目的達成に必要最小限か検証
    4.  **リスク特定・評価**: 「発生可能性 × 深刻度」のマトリクスで評価
    5.  **軽減措置の策定**: 匿名化、暗号化、アクセス制御等の具体策を文書化
    6.  **DPO・ステークホルダー協議**: DPOの助言取得
    7.  **承認・記録・定期見直し**: 残留リスクレベルの承認、処理変更時の再評価
-   **Living Document**: DPIAは処理内容に重大な変更がある度に再評価する**生きた文書**。

#### 6.2.3. DSA/DMA
-   **透明性報告**: コンテンツモデレーション活動の透明性レポートを公開する義務。
-   **広告透明性**: ターゲティング広告の配信理由をユーザーに開示する義務。

#### 6.2.4. European Accessibility Act (EAA)
-   **Law**: 2025年6月施行。WCAG 2.1 AA以上の準拠を法的義務として位置付け。
-   **Cross-Reference**: `200_design_ux.md` (アクセシビリティ詳細基準)

#### 6.2.5. UK Data (Use and Access) Act 2025 (DUAA)
-   **Law**: 2025年6月Royal Assent。UK GDPRの対象を絞った改正。段階施行（2025.6-2026.6）。
-   **主要変更点**:
    -   **SAR対応緩和**: 「合理的かつ比例的」な検索基準。追加情報が必要な場合の「時計停止」ルール導入
    -   **ADM（自動化意思決定）緩和**: 特別カテゴリデータ以外のADMを許可（異議申立・人的介入の権利は維持）
    -   **認定正当利益（Recognised Legitimate Interests）**: 国家安全保障、犯罪防止、脆弱者保護等の特定目的ではバランステスト不要
    -   **国際データ移転**: 「本質的に同等」→「重大な低下がない」基準に緩和
    -   **Cookie違反罰金**: GDPR水準に引き上げ（最大**£17.5M / 全世界売上4%**）
    -   **苦情対応義務**: 30日以内の受領確認、不当な遅延のない調査
-   **Smart Dataスキーム**: Open Banking拡張型のデータポータビリティ制度。金融・エネルギー・通信・交通セクターへ展開予定。
-   **DVS（Digital Verification Services）**: デジタル身分証明サービスの法的基盤。UK CertifIDトラストマークによる認証済みDVS登録制度（2025.12施行）。
-   **Action**: 英国ユーザー向けサービスの場合、SAR処理フロー・ADMポリシー・Cookie管理をDUAAに対応更新。

### 6.3. 米国法

#### 6.3.1. CCPA/CPRA固有要件
-   **Do Not Sell / Share**: 「個人情報を販売・共有しない」リンクをWebサイトのフッターに設置。
-   **Sensitive Personal Information**: センシティブ個人情報の処理制限。
-   **Right to Correct**: 個人情報訂正権への対応。
-   **Right to Know（強化）**: 2026.1より過去12ヶ月を超えて2022.1.1以降に収集された情報へのアクセス権を付与。
-   **サイバーセキュリティ監査**: 売上$100M超の企業は**2028.4.1まで**に初回認証を完了。

#### 6.3.2. 米国州法の急増（2025-2026）
-   **Mandate**: 米国は連邦プライバシー法が不在のため、州法の乱立が加速中。サービスが複数州のユーザーを対象とする場合、**最も厳格な州法**を基準としてプライバシー体制を設計せよ。
-   **2025年施行州**: Delaware, Iowa, Nebraska, New Hampshire, New Jersey, Tennessee
-   **2026年施行州**: Kentucky, Rhode Island, Indiana, Maryland, Minnesota
-   **Neural Data追加**: Connecticut（2026.7）がNeural Dataをセンシティブデータカテゴリに追加。
-   **州間協調執行（2026年動向）**: Colorado, Connecticut, Maryland, Minnesota, Oregon, New Jersey等の複数州のAGが**共同執行アクション**を活発化。単一の違反行為に対して複数州から同時に執行される事例が増加。
-   **執行重点分野（2026年）**:
    -   ユーザー選択の整合性（ダークパターン対策）
    -   Universal Opt-Out Signal（GPC等）の認識義務
    -   データブローカー登録義務
    -   子供・ティーンのプライバシー保護
    -   プライバシーポリシーの適法性
-   **連邦動向**:
    -   **2025.12大統領行政命令**: 州AI規制が国家競争力を阻害する場合の**連邦プリエンプション（優先適用）**方針を確立。AI Litigation Task Forceを設置し、州法への法的挑戦を推進。
    -   **連邦資金の条件付与**: 特定の連邦資金は、州がAI規制で矛盾する法律を制定しないことを条件化。
    -   **商務長官への評価指示**: 既存州AI法の国家競争力への影響評価を指示。
    -   **影響**: 州法と連邦方針の間の緊張が高まっており、**州法の有効性が法廷で争われる**可能性がある。当面は既存州法を遵守しつつ、連邦動向を四半期ごとに監視。
-   **Oregon精密位置情報禁止（2026）**: Oregonのプライバシー法改正により、精密位置情報（半径1,750フィート以内）を**センシティブデータ**として分類し、収集にはオプトイン同意を義務付け。

#### 6.3.3. California Delete Act / DROP
-   **Law**: データブローカーはDROP（Delete Request and Opt-Out Platform）経由の削除要求を**2026.8.1**より処理義務。
-   **エンジニアリング影響**: データブローカーに該当する場合、DROP APIとの統合を設計時に考慮。
-   **Cross-Reference**: §32 (データブローカー規制)

#### 6.3.4. Maryland Online Data Protection Act (MODPA)
-   **Law**: 2025年10月施行。データ収集を「合理的に必要かつ比例的」な範囲に制限する強力なデータ最小化義務。
-   **特徴**: センシティブデータへのターゲティング広告禁止、18歳未満へのターゲティング広告禁止。

## §7. 子供・脆弱者データ保護 (Children & Vulnerable Persons Data Protection)

### 7.1. 年齢確認 (Age Gate)
-   **Rule 7.1.1**: 課金機能を持つサービスでは年齢確認（生年月日入力）を実施。
-   **Rule 7.1.2**: 年齢確認を回避する手段を技術的に防止。

### 7.2. 親権者同意 (Parental Consent)
-   **Law**: 法定年齢未満のユーザー（日本: 18歳未満、EU: 16歳未満※加盟国により13-16歳、米国COPPA: 13歳未満）の個人データ処理には親権者の同意が必要。
-   **Action**: 同意チェックボックス必須、タイムスタンプ・IPのログ保存、13歳未満にはVPC取得。
-   **India DPDP Act（2026.11）**: Verifiable Parental Consentと同意マネージャー登録が義務化。

### 7.3. Age-Appropriate Design
-   **Rule 7.3.1**: 子供がアクセスする可能性のあるサービスでは**Privacy by Default**を最高レベルで適用。
-   **Rule 7.3.2**: 子供向けコンテンツでのプロファイリング・ターゲティング広告を**禁止**。
-   **Rule 7.3.3**: 位置情報の追跡はデフォルトで**無効**。
-   **Rule 7.3.4**: プッシュ通知・エンゲージメント促進機能のデフォルト**無効化**。

### 7.4. 課金上限
-   **Mandate**: 未成年者の月額課金上限を設定し、超過時は親権者への通知を実施。
-   **Anti-Pattern**: Loot Box（ガチャ）の確率非開示は各国で規制強化中。確率を明示的に開示すること。

### 7.5. 生体認証データ (Biometric Data)
-   **Law**: 生体認証データは一度漏洩すると変更不可能なため、最高レベルの保護が必要。
-   **主要規制**:
    -   **BIPA (Illinois)**: 収集前の書面同意、目的の明示、保持ポリシー策定が義務。違反は**個人単位**で損害責任。
    -   **EU AI Act**: 公共空間でのリアルタイム遠隔生体認証は原則**禁止**（2025.2施行済）。
    -   **GDPR Art.9**: 生体認証データは「特別カテゴリ」として原則処理禁止。
-   **実装要件**:
    -   収集前に書面でのインフォームドコンセント（電子署名可）
    -   保持期間を明示（目的達成後または最終操作から**3年以内**に削除）
    -   保存時は**AES-256以上**で暗号化、アクセスログ必須
    -   Deepfake構成に使用されるリスクを評価し対策を文書化
-   **Cross-Reference**: `600_security_privacy.md` (Zero Trust認証)

### 7.6. 年齢確認義務のグローバル拡大
-   **背景**: 2025-2026年、未成年者のSNS利用規制が世界的に急速に拡大。自己申告式の年齢確認は不十分とみなされる傾向。

| 法域 | 法律 | 対象年齢 | 施行日 | 要件 |
|:-----|:-----|:---------|:-------|:-----|
| **UK** | Online Safety Act 2023 | 全年齢 | 2025.7 | 「高度に効果的な年齢保証」義務。Ofcom罰金: 全世界売上10% |
| **Australia** | Social Media Ban | <16 | 2025.12 | 16歳未満のSNS利用禁止。プラットフォーム側に年齢確認義務 |
| **Malaysia** | SNS Ban + eKYC | <16 | 2026.1 | 16歳未満禁止。eKYCによる年齢確認必須 |
| **Louisiana** | SB 162 | <16 | 2025.7 | 全ユーザーの年齢確認。16歳未満は親権者同意必須 |
| **California** | SB 976 | <18 | 2026.12 | 年齢確認後、18歳未満のアディクティブフィード除外 |
| **Nebraska** | LB 383 | <18 | 2026.7 | 年齢確認+親権者同意必須 |
| **Tennessee** | HB 1891 | <18 | 2025.1 | 年齢確認+親権者同意必須 |

-   **Mandate**: グローバル展開サービスは**最も厳格な法域の年齢確認義務**を基準として設計。
-   **Anti-Pattern**: 自己申告（チェックボックス「18歳以上です」）のみでの年齢確認は、UK/AU/MY法下では**不十分**。

### 7.7. COPPA 2.0（KIDS Act）
-   **法案**: 2026年3月上院全会一致通過。保護対象を13歳未満→**17歳未満**に拡大。
-   **主要要件**:
    -   未成年者へのターゲティング広告を**禁止**
    -   アディクティブ機能（無限スクロール、自動再生等）のデフォルト**無効化**
    -   最も強力なプライバシー設定をデフォルト適用
    -   ペアレンタルコントロール機能の提供義務
-   **FTC COPPA規則改正（2025.6施行）**: ターゲティング広告へのopt-in親権者同意、データ保持期間制限を強化。FTCは年齢確認技術の利用を推奨するポリシーステートメントを発行（2026.2）。
-   **Action**: 現行§7.1-7.4の要件をCOPPA 2.0の17歳基準に拡張する準備を開始。施行確定後に即時対応できるよう設計段階で組み込む。

### 7.8. 年齢確認技術要件
-   **技術オプション比較**:

| 手法 | 精度 | プライバシーリスク | コスト | 採用例 |
|:-----|:-----|:-----------------|:-------|:-------|
| **自己申告（生年月日入力）** | 低 | 低 | 低 | 現行多数のサービス |
| **顔推定AI** | 中-高 | 中（生体データ処理） | 中 | UK Online Safety Act推奨 |
| **デジタルID（DVS/eKYC）** | 高 | 中（ID連携） | 中 | UK DUAA DVS、Malaysia eKYC |
| **政府発行ID照合** | 最高 | 高（PII処理） | 高 | Australia ban対応 |
| **クレジットカード確認** | 中 | 低 | 低 | COPPA既存方式 |

-   **Mandate**: 年齢確認実装時は、**プライバシー影響評価（DPIA）**を必ず実施（§6.2.2準拠）。
-   **原則**: 年齢確認に収集したデータは年齢確認目的以外に使用禁止。確認完了後は速やかに削除。
-   **Cross-Reference**: §6.2.5 (UK DUAA DVS)

## §8. ダークパターン禁止 (Dark Pattern Prohibition)

### 8.1. 定義と禁止行為
-   **Law**: ダークパターンはEU DSA・CCPA/CPRAで明文的に禁止されている。規制当局はCMP同意UIの操作性にも監視を強化中。

### 8.2. 禁止パターン一覧

| パターン | 定義 | 禁止例 |
|:---------|:-----|:-------|
| **Confirm Shaming** | 拒否選択肢に罪悪感を与える表現 | 「いいえ、割引は要りません」 |
| **Trick Questions** | 二重否定等で混乱させる | 「通知を受け取らないことを希望しない」 |
| **Hidden Costs** | 最終段階で追加費用を提示 | カート画面での突然の手数料追加 |
| **Forced Continuity** | 解約を意図的に困難にする | 解約に電話必須、解約ボタンの隠蔽 |
| **Roach Motel** | 登録は容易、退会は困難 | 1クリック登録 → 10ステップ退会 |
| **Privacy Zuckering** | デフォルトで過剰な情報共有 | 全情報を「公開」にデフォルト設定 |
| **Nagging** | 繰り返しの許可要求 | 拒否後すぐに再表示されるポップアップ |
| **Preselection** | 不利な選択肢を事前選択 | 広告メール受信のチェックボックスがデフォルトON |

### 8.3. 設計原則
-   **Rule 8.3.1**: 同意の取得と撤回は**同等の容易さ**で提供する（Symmetry of Choice）。
-   **Rule 8.3.2**: 解約・退会プロセスは登録プロセスと**同等以下のステップ数**で完了可能にする。
-   **Rule 8.3.3**: デフォルト設定は常に**ユーザーのプライバシーを最大限保護**する方向に設定。
-   **Rule 8.3.4**: 拒否選択肢を視覚的に劣位に配置する行為を禁止（フォントサイズ、色、位置）。

## §9. 知的財産・ライセンスガバナンス (IP & License Governance)

### 9.1. ライセンス汚染防止
-   **GPL禁止**: コピーレフトライセンス（GPL/AGPL）のコード混入は厳禁。
-   **権利クリアランス**: 使用する全アセットの商用利用可否を確認し、ライセンス証跡を一元管理。
-   **Cross-Reference**: `602_oss_compliance.md`

### 9.2. UGCライセンス付与
-   **Law**: 利用規約に、UGCに対する「プロモーション目的での無償かつ非独占的利用権」許諾条項を含める。
-   **Safeguard**: 著作者人格権の不行使特約を含め、ユーザー権利の不当侵害を防止。

### 9.3. 業務委託知財 (Work-for-Hire Protocol)
-   **Mandate**: 業務委託契約に以下を必須とする:
    -   成果物の著作権（27条・28条含む）が検収時に発注者に移転
    -   著作者人格権の不行使条項
-   **Cross-Reference**: `603_ip_due_diligence.md`

## §10. 反社会的勢力排除 (Anti-Social Forces Exclusion)

### 10.1. 契約条項（必須）
-   **Mandate**: 全ての契約書・規約に以下を含める:
    -   反社会的勢力でないことの表明保証
    -   反社会的勢力を利用しないことの誓約
    -   該当が判明した場合の即時契約解除権（催告不要）
    -   損害賠償請求権の留保

### 10.2. KYB/KYC Due Diligence
-   **法人取引**: KYB審査を実施。**個人取引**: KYCスクリーニングを実施。
-   **スクリーニング**: 反社データベースとの照合。
-   **定期チェック**: 初回だけでなく**年次での再スクリーニング**を義務付け。

### 10.3. 発覚時の即時対応
1.  法務部門に即時報告
2.  契約解除の内容証明送付
3.  関連サービスアカウントの即時停止
4.  所轄警察・暴力団追放センターへの相談・通報

## §11. AI法規制・EU AI Act対応 (AI Regulation & EU AI Act)

### 11.1. EU AI Act リスク分類

| リスクレベル | 定義 | 例 | 義務 |
|:-----------|:-----|:---|:-----|
| **禁止** | 基本的権利を脅かすAI | ソーシャルスコアリング、職場での感情認識、リアルタイム遠隔生体認証 | **使用禁止**（2025.2施行済） |
| **高リスク** | 安全性・権利に重大な影響 | 採用選考AI、信用スコアリング、重要インフラ管理 | リスク管理、適合性評価、人間の監視、技術文書、ログ（**2026.8施行**） |
| **限定リスク** | 透明性義務のみ | チャットボット、ディープフェイク生成 | AI生成である旨の**明示的ラベリング**（**2026.8施行**） |
| **最小リスク** | 追加義務なし | スパムフィルター、ゲームAI | 任意の行動規範遵守を推奨 |

### 11.2. EU AI Act 段階施行タイムライン

| 施行日 | 適用内容 |
|:------|:---------|
| **2025.2.2** | 禁止規定（Unacceptable Risk）、AIリテラシー義務（Art.4） |
| **2025.8.2** | GPAI義務、ガバナンス組織（AI Board/Scientific Panel）、加盟国の所管当局指定・罰則法制化 |
| **2026.8.2** | 高リスクAI義務（Annex III）、透明性義務（Art.50）、AIサンドボックス設置義務 ※**施行延期提案あり**（下記参照） |
| **2027.8.2** | 高リスクAI（製品安全法令対象: 医療機器、航空等） |

> [!WARNING]
> **EU AI Act施行延期提案（2026.3）**: 欧州委員会はAnnex III高リスクAI義務の施行を**2027年12月**に延期する提案を公表。加盟国・欧州議会の合意が必要。**確定するまで2026.8基準で準備を継続**すること。延期が確定した場合でも、AI Officeによる非公式ガイダンス・期待値は変わらない可能性が高い。

### 11.3. 高リスクAIシステムの義務
-   **リスク管理システム**: AIライフサイクル全体を通じたリスク管理体制の構築・文書化。
-   **データ品質**: 学習データの代表性・正確性・バイアス排除を保証。
-   **技術文書**: 市場投入前に包括的な技術文書を作成・維持。
-   **ログ記録**: 自動生成イベントのログを**最低6ヶ月間**保持。
-   **人間の監視**: AIシステムの運用中に人間が効果的に監視可能な設計。
-   **適合性評価**: Conformity Assessmentの実施。

### 11.4. GPAI（汎用AI）モデル義務
-   **全GPAIモデル**: 技術文書提供、著作権指令遵守、学習データの要約公開。
-   **システミックリスクGPAI（追加）**: モデル評価（敵対的テスト含む）、リスク軽減措置、重大インシデント報告、サイバーセキュリティ保護。
-   **施行タイムライン**: 2025.8.2以降にリリースされる新GPAIモデルは即時準拠。AI Officeによる正式執行は**2026.8.2**より開始。2025.8.2以前のモデルは**2027.8.2**までに準拠。

#### 11.4.1. GPAI Code of Practice（2025.7公表）
-   **概要**: 独立専門家により開発された自主的行動規範。GPAIモデルプロバイダーの透明性・著作権・安全性義務に関する**実務ガイダンス**を提供。
-   **法的位置付け**: 法的拘束力はないが、AI Act準拠を証明する**重要な手段**として機能。遵守しない場合はAI Officeによる個別審査のリスク。
-   **主要コンプライアンス領域**:

| 領域 | 要件 |
|:-----|:-----|
| **透明性** | 技術文書の作成・公開、学習データの十分な要約（データソース種別・規模・キュレーション方法）、モデルのCapability・Limitation記述 |
| **著作権コンプライアンス** | 学習データにおける著作権コンプライアンス方針の策定・公開。EU著作権指令のテキスト&データマイニング（TDM）除外規定の遵守。権利者のオプトアウト（Art.4 TDM予約権）を尊重する技術的措置 |
| **安全性・セキュリティ** | システミックリスクGPAIは敵対的テスト（レッドチーミング）の実施、リスクアセスメント、サイバーセキュリティ保護措置の記録・公開 |

-   **著作権コンプライアンス方針の実装要件**:
    -   robots.txt / ai.txt のクローラ制御ルールの遵守（§11.11参照）
    -   権利者からのオプトアウト要求処理フローの文書化
    -   学習データセットのデータソース・キュレーションプロセスの記録
    -   TDM除外規定の適用可否の法的評価
-   **Mandate**: GPAIモデルを利用・提供する場合、Code of Practiceの主要要件への対応状況を文書化し、AIガバナンス委員会（§11.9）でレビューすること。

### 11.5. Agentic AI賠償責任フレームワーク
-   **Risk**: Agentic AI（自律的に意思決定・行動するAIシステム）は、従来の予測可能なツールとは異なり、既存の賠償責任モデルに「責任ギャップ」を生む。
-   **法的課題**:
    -   従来の不法行為法では、適応・進化するAIエージェントの行為に対する責任帰属が困難
    -   多くの技術契約はAgentic AIが引き起こす種類の損害をカバーしていない
    -   顧客側がリスクを負担する契約構造が一般的
-   **対策**:
    -   Agentic AIの意思決定ログを**全件記録**し、責任追跡可能性を確保
    -   ベンダー契約にAgentic AIの行為に対する**責任分界条項**を明記
    -   Human-in-the-Loop（HITL）を重要な意思決定に必須とし、完全自律動作の範囲を制限
    -   Agentic AIの行動範囲を**明示的なガードレール**で制約
-   **Cross-Reference**: `400_ai_engineering.md` (ガードレール), `600_security_privacy.md` (AI/LLMセキュリティ)

### 11.6. AI生成コンテンツ著作権
-   **No Direct Publish**: AI生成テキストをそのまま公開記事として掲載することを**禁止**。
-   **AI Image Risk Assessment**: AI生成画像の商用利用時は学習データの著作権リスクを事前評価。
-   **DB Flag**: `is_ai_assisted: boolean`（デフォルト: `false`）。

### 11.7. ユーザーデータのAI学習利用制御
-   **Opt-Out by Default**: ユーザーデータはAI学習に**利用しない**ことをデフォルト。
-   **Explicit Opt-In**: 明示的同意を取得した場合のみ、匿名化の上で利用可能。
-   **DB Flag**: `ai_training_opt_in: boolean DEFAULT false`。
-   **Right to Withdraw**: いつでもOpt-Inを撤回可能。撤回後は次回学習バッチから除外。

### 11.8. データ商業利用のOpt-Out
-   **Opt-Out Switch**: ユーザー設定画面に「データ活用への協力」ON/OFFスイッチを設置。
-   **DB Flag**: `data_commercialization_opt_out: boolean DEFAULT false`。
-   **Anonymization Mandate**: 外部提供データには**k-匿名化**または**差分プライバシー**技術を適用。

### 11.9. AIガバナンス委員会
-   事業規模が一定以上に到達した段階で**AIガバナンス委員会**を設置:
    -   技術責任者、法務・コンプライアンス担当、ユーザー代表
-   **Scope**: モデル選定、プロンプト方針変更、新規AI機能のリスク評価、インシデント対応。
-   **Cadence**: 四半期ごとのレビュー + 緊急時臨時招集。
-   **AI Literacy Mandate**: AI関連業務に従事する全従業員にAIリテラシー教育を実施（EU AI Act Art.4、**2025.2施行済**）。

### 11.10. GenAIハルシネーション賠償責任
-   **Risk**: ハルシネーションが法的責任・名誉毀損・規制違反のリスクを生む。
-   **対策**:
    -   GenAI出力を**公式情報**として直接使用禁止
    -   ユーザー向けAI機能に「回答は不正確な場合があります」の**免責表示を必須**
    -   GenAIプロバイダーとの契約で**補償条項 (Indemnification)**・責任分界・学習データオプトアウト権を確保
-   **GDPRリスク**: AIが個人について不正確な情報を生成する場合、GDPR **Art.5(1)(d)**（データ正確性原則）に違反しうる。

### 11.11. GenAI学習データコンプライアンス

#### 11.11.1. 著作権・スクレイピング法制

| 法域 | 規制 | 要件 |
|:------|:-----|:-----|
| **EU** | EU AI Act + DSM指令 Art.4 | GPAIモデル提供者は学習データの構造化要約を公開。テキストーデータマイニングの**オプトアウトを尊重する義務** |
| **米国** | CA AB 2013（2026.1施行） | 学習データのソース・種類・著作権保護コンテンツ含有の有無を開示義務 |
| **米国** | AI Accountability for Publishers Act（2026提案） | robots.txtの法的強制力化、Fair Use抗弁の制限、版元への対価支払い義務 |
| **日本** | 著作権法30の4 | 情報解析目的の著作物利用は原則適法。ただし「利益を不当に害する」場合は除外 |

-   **Anti-Pattern**: Anthropicは海賊版データを学習に使用し、**$1.5B**の和解金を支払い（2025.8）。学習データの適法性検証は**モデル開発前**に完了すること。

#### 11.11.2. クローラー制御とオプトアウト
-   **robots.txt**: AIクローラーのブロック設定は必須。ただし任意遵守のため、法的強制力は限定的。
-   **ai.txt**（2025提案）: ドメイン単位でAIによる利用許可を細粒度で指定（例: 要約は許可、学習は禁止）。採用は低いが動向を監視。
-   **llms.txt**: AIエージェント向けのコンテンツ参照用ファイル。法的強制力なし。
-   **Mandate**: 自社コンテンツを保護するため、robots.txtでAIクローラー（GPTBot, Google-Extended, CCBot等）をブロックし、利用規約でAI学習目的のスクレイピングを明示的に禁止すること。
-   **GenAIプロバイダー契約**: 学習データオプトアウト権、出力の著作権帰属、補償条項（Indemnification）を契約で確保。

## §12. データライフサイクル管理 (Data Lifecycle Management)

### 12.1. エンゲージメントデータ保持基準

| データ種別 | 保持期間 | 期限後の処理 |
|:----------|:---------|:------------|
| **お気に入り / ブックマーク** | 無期限 | 退会時に物理削除 |
| **閲覧履歴** | 直近**90日** | 月次バッチで物理削除 |
| **操作ログ** | 直近**1年** | 法的保管義務があるものは別途保管 |
| **監査ログ** | **1年** | アーカイブ後削除 |
| **決済データ** | **法定7年** | 法定期限後削除 |
| **キャンペーンデータ** | 終了後**90日以内** | 物理削除 |

-   **Mandate**: 各データの具体的保持期間はBlueprint側で定義。本プロトコルは「型」を提供。

### 12.2. コンテンツアーカイブ基準
-   **サービス終了エンティティ**: ステータスを`closed`/`archived`に変更し検索結果から除外。関連UGCは永久保持。
-   **退会ユーザーのコンテンツ**: `owner_id: null`に変更。公益性のあるコンテンツは保持。
-   **休眠データ**: 2年以上更新のない非公開コンテンツは管理者にアーカイブ提案を自動通知。

### 12.3. 自動クリーンアップ
-   **Mandate**: `pg_cron` / Cloud Scheduler等で定期クリーンアップジョブを自動実行。

```sql
-- pg_cron による自動クリーンアップジョブ例
SELECT cron.schedule(
  'cleanup-browsing-history',
  '0 3 * * 0', -- 毎週日曜 03:00 UTC
  $$DELETE FROM browsing_history WHERE created_at < NOW() - INTERVAL '90 days'$$
);
```

-   **匿名化の義務**: 削除が法的に困難な場合、**不可逆な匿名化**を実施。ハッシュ化だけでは不十分。

### 12.4. 法域別データ保持期間テンプレート

| データカテゴリ | GDPR（EU） | CCPA/CPRA（CA） | Global Privacy Laws | 共通推奨 |
|:-------------|:----------|:--------------|:-----------|:--------|
| **顧客アカウント情報** | 目的達成まで + 退会後3年 | 開示義務: 12ヶ月分 | 利用目的達成まで | 退会後**最長3年** |
| **取引・決済データ** | 法定義務期間（税: 7年） | 法定義務期間 | 法定義務期間（税: 7年） | **法定期間に準拠** |
| **マーケティング同意** | 同意撤回まで + 証跡3年保存 | opt-out後即時停止 | 同意撤回まで | 証跡は**5年**保存 |
| **アクセスログ** | 正当利益期間（推奨: 6ヶ月） | 規定なし | 推奨: 1年 | **6ヶ月〜1年** |
| **CCTV・監視映像** | 72時間〜30日（国による） | 規定なし | 規定なし | **最長30日** |
| **従業員データ** | 雇用終了後3〜10年（国による） | 雇用終了後の合理的期間 | 退職後3年（労基法） | **法定期間に準拠** |
| **健康データ** | HIPAA: 6年 / GDPR: 目的達成まで | 規定なし | 個情法 + 医師法 | **法定期間に準拠** |
| **子供のデータ** | COPPA/GDPR: 目的達成後速やかに削除 | COPPA: 合理的期間 | Global Privacy Laws: 同左 | **最短期間で削除** |

-   **Mandate**: 上記はテンプレート。Blueprint側で具体的値を定義すること。
-   **CCPA/CPRA開示義務**: プライバシーポリシーで各カテゴリの具体的保持期間を**公開する義務**あり。

#### 12.4.1. 法的ホールド（Litigation Hold）
-   **定義**: 訴訟・調査・規制対応が合理的に予見される場合、関連データの通常削除を**即時停止**する法的義務。
-   **トリガー**: 訴訟予告（保全通知）、規制当局からの調査開始通知、内部告発、監査通知。
-   **手順**:
    1.  法務チームが法的ホールド通知を発行
    2.  該当データの自動削除ジョブを**即座に一時停止**（`pg_cron`ジョブの無効化等）
    3.  対象データの範囲・保管場所を特定・文書化
    4.  関係者に保全義務を通知（定期リマインダー送付）
    5.  ホールド解除は法務チームの書面承認のみ
-   **Anti-Pattern**: 法的ホールド中にデータを削除した場合、**証拠隠滅（Spoliation）**として制裁・推定不利益が発生。

## §13. 委託先データガバナンス (Vendor Data Governance)

### 13.1. データ処理委託契約 (DPA)
-   **契約条項（必須）**:
    -   取扱いデータの範囲・種類・保管先の明示
    -   目的外利用の禁止
    -   再委託の原則禁止（必要時は書面での事前承認）
    -   契約終了時のデータ返却・削除義務
    -   インシデント発覚時の報告義務（発覚から**24時間以内**の初報）
-   **DPAテンプレートチェックリスト**:
    -   [ ] 処理対象データのカテゴリと量の明示
    -   [ ] 処理の目的と法的根拠の記載
    -   [ ] データ処理者の義務（暗号化、アクセス制御、ログ管理）
    -   [ ] Sub-Processor事前承認条項
    -   [ ] 監査権の明記
    -   [ ] DSAR協力義務
    -   [ ] 契約終了後のデータ削除/返却期限（**30日以内**）と証跡提出
    -   [ ] 準拠法と紛争解決手段
    -   [ ] 損害賠償の上限と免責条件

### 13.2. データ分類と共有制限

| データ分類 | 共有可否 | 条件 |
|:---------|:--------|:-----|
| **Sensitive PII** | 原則禁止 | やむを得ない場合のみ暗号化+アクセスログ付きで共有 |
| **Personal Info** | 条件付き許可 | DPA締結+暗号化転送+アクセス制限 |
| **Quasi-Personal Info** | 許可 | DPA締結のみ |
| **匿名化済みデータ** | 自由共有 | 再識別不可であることの検証済みが前提 |

### 13.3. 委託先監査義務
-   **年次監査**: 委託先のセキュリティ管理体制を**年1回以上**評価。
-   **評価項目**: セキュリティポリシー、アクセス制御、暗号化、インシデント対応、教育、第三者認証（ISO 27001、SOC 2等）
-   **不適合時**: 改善期限（最長**90日**）。期限内に改善されない場合は契約見直しまたは解除。

### 13.4. ベンダーリスクティアリング

| ティア | 基準 | 審査頻度 | DPIA | 監査義務 |
|:------|:-----|:--------|:-----|:--------|
| **Tier 1（高リスク）** | Sensitive PII処理 / 大量データ処理 / コアインフラ運用 | **四半期** | **必須** | 年次オンサイト監査 |
| **Tier 2（中リスク）** | Personal Info処理 / 非コアシステム | **半年** | 推奨 | 年次リモート監査 |
| **Tier 3（低リスク）** | 匿名化データのみ / PII非接触 | **年1回** | 不要 | SOC 2 / ISO 27001証明書確認 |

-   **Mandate**: 新規ベンダー契約前に必ずティア分類を実施。Tier 1ベンダーは**CISO承認必須**。
-   **ティア変更トリガー**: データ処理範囲の変更、セキュリティインシデント発生、規制環境の変化。
-   **継続モニタリング**: Tier 1ベンダーはセキュリティスコアリングサービス（SecurityScorecard等）による**継続的リスクモニタリング**を実施。

### 13.5. 第四者リスク管理（Fourth-Party Risk）
-   **定義**: 委託先（Processor）がさらに再委託する先（Sub-Processor）のリスク管理。
-   **フロースルー義務**: DPA契約条項で以下を義務化:
    -   Sub-Processorへのデータ保護義務の**全量フロースルー**（同等以上の義務を課す）
    -   Sub-Processor一覧の**事前開示**と変更時の**30日前通知**
    -   データ管理者（Controller）の**異議申立権**（Sub-Processor変更への拒否権）
-   **チェーン監視**:
    -   Tier 1ベンダーのSub-Processor一覧を**四半期で取得・更新**
    -   Sub-Processorが外国敵対国（§33.4）管轄下でないことを検証
    -   Sub-Processorの重大インシデント発生時の**即時エスカレーション**義務
-   **Anti-Pattern**: 「委託先が管理するから自社は関与不要」は誤り。GDPR Art.28に基づきControllerが最終責任を負う。

## §14. データ漏洩法的通知義務 (Data Breach Legal Notification)

### 14.1. 法的根拠
-   **Law**: GDPR Art.33/34、APPI改正第26条等は、個人データ漏洩時の監督機関への報告およびユーザー通知を義務付ける。
-   **Cross-Reference**: `600_security_privacy.md` §10.7, `503_incident_response.md` §3

### 14.2. 5段階フロー

| フェーズ | 期限 | 担当 | アクション |
|:--------|:-----|:-----|:----------|
| **1. 検知** | 即時 | 監視システム | 異常検知 → 即時通知 |
| **2. 初動** | 検知後**1時間以内** | 技術責任者 | 漏洩経路特定・遮断、影響範囲調査、証拠保全 |
| **3. 速報** | 検知後**72時間以内** | 法務 | 監督機関への速報提出 |
| **4. ユーザー通知** | 速報と**同時** | CS/運営 | 影響ユーザーへの通知 |
| **5. 事後** | 検知後**30日以内** | 全チーム | 確報提出、Post-Mortem、再発防止策 |

### 14.3. 報告閾値
-   個人データの漏洩が**1,000件以上**
-   要配慮個人情報が含まれる場合
-   財産的被害が生じるおそれがある場合
-   不正アクセスによる漏洩の場合

### 14.4. 法域別通知期限差分表

| 法域 | 監督機関への通知期限 | ユーザー通知 | 備考 |
|:-----|:-------------------|:-----------|:-----|
| **GDPR** | **72時間以内** | 高リスク時は「不当な遅延なく」 | 改正案で**96時間**に緩和検討中 |
| **Global Privacy Laws** | 速報: **3-5日以内** / 確報: **30日以内** | 本人への通知義務 | 2027年改正で変更の可能性 |
| **CCPA/CPRA** | 法定期限の明示なし | 「最も迅速な手段」で通知 | 州司法長官への報告は500件以上 |
| **CA SB 446** | AG通知: **15日以内**（500件超） | **30日以内** | **2026.1施行**。従来の「最も迅速な手段」を30日に明確化 |
| **New York** | NYDFS通知義務 | **30日以内** | **2024.12施行**。2025.3より「医療情報」「健康保険情報」もPI定義に追加 |
| **COPPA** | FTCへ速やかに通知 | 保護者へ通知 | 児童データは即時対応 |
| **NIS2** | 初報: **24時間以内** / 詳報: **72時間以内** | サービス利用者への通知 | 重要インフラ事業者向け |
| **CIRCIA** | **72時間以内**（ランサム支払: 24時間以内） | — | 重要インフラ対象。**2026.5施行**（当初2025.10から延期） |

-   **Mandate**: 複数法域にまたがる場合、**最も厳格な期限**を基準としてIRフローを設計。

#### 14.6.4. IRランブックコードテンプレート

```typescript
// --- Incident Response Runbook: Notification Deadline Calculator ---
import { differenceInHours, addHours, format } from 'date-fns';

interface JurisdictionDeadline {
  jurisdiction: string;
  regulation: string;
  deadlineHours: number;
  authority: string;
  notificationUrl: string;
}

const NOTIFICATION_DEADLINES: JurisdictionDeadline[] = [
  { jurisdiction: 'EU/EEA', regulation: 'GDPR Art.33', deadlineHours: 72, authority: 'Lead DPA', notificationUrl: '' },
  { jurisdiction: 'Japan', regulation: 'Global Privacy Laws', deadlineHours: 72, authority: 'PPC', notificationUrl: 'https://www.ppc.go.jp/personalinfo/legal/leakAction/' },
  { jurisdiction: 'US-CA', regulation: 'CCPA/SB446', deadlineHours: 720, authority: 'CA AG', notificationUrl: '' },
  { jurisdiction: 'Australia', regulation: 'Privacy Act (NDB)', deadlineHours: 720, authority: 'OAIC', notificationUrl: '' },
  { jurisdiction: 'US-Federal', regulation: 'CIRCIA', deadlineHours: 72, authority: 'CISA', notificationUrl: '' },
];

function calculateDeadlines(discoveryTime: Date) {
  return NOTIFICATION_DEADLINES.map(d => ({
    ...d,
    deadline: addHours(discoveryTime, d.deadlineHours),
    formattedDeadline: format(addHours(discoveryTime, d.deadlineHours), 'yyyy-MM-dd HH:mm (zzz)'),
    remainingHours: d.deadlineHours - differenceInHours(new Date(), discoveryTime),
  })).sort((a, b) => a.deadlineHours - b.deadlineHours);
}
```

-   **Mandate**: 上記デッドライン計算ロジックをIRツールに組み込み、インシデント発見時に**自動的に全法域の通知期限を算出**し、最短期限の法域をハイライトすること。
-   **Cross-Reference**: §14.4 (法域別通知期限), `503_incident_response.md` (インシデントランブック)
-   [ ] 越境データ移転の有無と移転先の監視法制リスクの評価
-   [ ] IRタイムラインの全記録（証跡としての監査ログ保全）

-   **Mandate**: テーブルトップ演習の実施後は必ず**振り返り（After Action Review）**を実施し、改善点をIR計画に反映すること。

## §16. データガバナンス組織体制 (Data Governance Organization)

### 16.1. 必須役割

| 役割 | 責務 | 設置要件 |
|:-----|:-----|:---------|
| **DPO** | データ保護法遵守の独立的監視、DPIA助言、監督機関との窓口 | GDPR対象時は必須 |
| **CPO** | プライバシー戦略の策定・実行 | 事業規模に応じて推奨 |
| **Data Steward** | データ品質管理、メタデータ管理 | ドメインごとに配置推奨 |

### 16.2. ROPA
-   **Mandate**: GDPR Art.30準拠のデータ処理活動記録を維持。
-   **記録項目**: 処理目的、データカテゴリ、受領者、越境移転、保存期間、安全措置。
-   **更新頻度**: 新規処理活動追加時に即時更新 + **年1回**の全体レビュー。

### 16.3. データ分類体系

| 分類レベル | 定義 | 取扱い基準 |
|:----------|:-----|:----------|
| **Restricted** | 最高機密（医療、金融、認証情報） | 暗号化必須、アクセスログ、最小権限 |
| **Confidential** | 業務上機密（個人情報、契約情報） | アクセス制限、暗号化転送 |
| **Internal** | 社内向け | 社外共有禁止 |
| **Public** | 公開情報 | 制限なし |

### 16.4. Privacy Engineering技術パターン

-   **Data Minimization at Code Level**: `SELECT *`の使用を禁止し、明示的フィールド選択を義務付け。

```typescript
// ✅ Correct: 必要カラムのみ選択（Data Minimization）
const user = await supabase.from('profiles')
  .select('id, display_name, avatar_url')
  .eq('id', userId);
```

-   **PII Tagging in Schema**:

```sql
COMMENT ON COLUMN profiles.email IS '@pii:email';
COMMENT ON COLUMN profiles.phone IS '@pii:phone';
COMMENT ON COLUMN profiles.full_name IS '@pii:name';
```

-   **Automated PII Redaction in Logs**:

```typescript
function redactPII(data: Record<string, unknown>): Record<string, unknown> {
  const PII_KEYS = ['email', 'phone', 'full_name', 'address', 'ip_address'];
  return Object.fromEntries(
    Object.entries(data).map(([key, value]) =>
      PII_KEYS.includes(key) ? [key, '[REDACTED]'] : [key, value]
    )
  );
}
```

-   **Privacy-Preserving Computation**:

| 技術 | 概要 | 適用ケース |
|:-----|:-----|:----------|
| **差分プライバシー (DP)** | 統計ノイズを加えて個人の特定を防止 | 分析ダッシュボード、A/Bテスト |
| **連合学習 (FL)** | データを移動せず分散学習 | 越境データ制約下のAI学習 |
| **準同型暗号 (HE)** | 暗号化したまま計算 | 委託先でのデータ処理 |
| **セキュアマルチパーティ計算 (MPC)** | 複数者のデータを非公開のまま共同計算 | ベンチマーク、業界横断分析 |

### 16.5. Policy-as-Code
-   **Concept**: ガバナンスポリシーを機械が実行可能なコードとして定義し、CI/CDで自動検証。
-   **適用領域**:
    -   スキーマコメント（`@pii:*`）に基づくPIIの自動検出
    -   マイグレーション時に`retention_days`設定がないテーブルの作成をブロック
    -   RLSポリシーとROPA記録の自動照合
    -   E2Eテストで同意前のトラッキングスクリプト読み込みを検出

#### 16.5.1. CI/CDプライバシーチェックパイプライン例

```yaml
# .github/workflows/privacy-checks.yml
name: Privacy Compliance Checks
on: [pull_request]
jobs:
  privacy-audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: PII Schema Audit
        run: |
          # @pii:タグのない個人データカラムを検出
          grep -rn 'email\|phone\|full_name\|address\|ip_address' \
            supabase/migrations/ | grep -v '@pii:' && exit 1 || true

      - name: Retention Policy Check
        run: |
          # retention_daysのない新規テーブルCREATEをブロック
          grep -l 'CREATE TABLE' supabase/migrations/*.sql | while read f; do
            grep -q 'retention_days' "$f" || \
              { echo "FAIL: $f missing retention_days"; exit 1; }
          done

      - name: Consent-Before-Tracking Test
        run: |
          # Playwrightで同意前のトラッキングスクリプト検出
          npx playwright test tests/privacy/consent-enforcement.spec.ts
```

-   **Mandate**: PRマージ前に上記チェックを全てパスさせること。失敗時はマージをブロック。

#### 16.5.2. ガバナンス自動化パイプライン
-   **目的**: Policy-as-Codeを拡張し、データガバナンスの検証を全自動化。
-   **実装レイヤー**:

| レイヤー | 自動化対象 | ツール例 |
|:--------|:-----------|:--------|
| **スキーマレイヤー** | PIIタグの網羅性検証、保持期限設定の必須化 | CI/CDスクリプト、Open Policy Agent |
| **アクセス制御レイヤー** | RLSポリシーとROPAの自動照合 | カスタムリンター、Supabase CLI |
| **ランタイムレイヤー** | 同意前のトラッキング・スクリプト検出 | Playwright E2Eテスト |
| **監査レイヤー** | DSAR応答期限の自動アラート、保持期限超過データの検出 | Cronジョブ、監視ダッシュボード |

-   **Mandate**: 少なくともスキーマレイヤーとランタイムレイヤーの自動化をCI/CDに組み込むこと。監査レイヤーは事業規模に応じて導入。

### 16.6. データマッピング・データフロー図
-   **Mandate**: 全データ処理活動について以下を含むデータフロー図を作成・維持:
    -   データの収集ポイント、保存先、共有先、保持期間、削除トリガー
-   **更新頻度**: 新機能リリース時に更新 + **半年に1回**の全体棚卸し。

### 16.7. データ倫理フレームワーク
-   **核心原則**: 透明性、公平性、目的限定、説明責任、人間中心性
-   **適用**: 新機能のデータ活用設計時に「Newspaper Test」を実施。

## §18. 法執行機関データ要求対応 (Law Enforcement Data Requests)

### 18.1. 基本原則
-   全ての政府・法執行機関からのデータ要求は法務が法的有効性を検証してから対応。
-   法律で禁止されていない限り、データ開示前に影響を受けるユーザーに通知。
-   要求に必要な最小限のデータのみを開示。
-   **CLOUD Act**: 米国拠点のサービスプロバイダーは海外保存データも開示対象となりうる。

### 18.2. 要求分類と対応

| 要求種別 | 開示範囲 | 法務確認 | ユーザー通知 |
|:---------|:---------|:---------|:----------|
| **任意の協力要請** | 任意。拒否可 | 推奨 | 可 |
| **召喚状** | 非コンテンツデータのみ | 必須 | 可 |
| **捜索令状** | コンテンツ含む | 必須 | 通常禁止 |
| **NSL** | 制限あり | 必須 | 禁止 |
| **緊急開示要求** | 生命の危険時のみ | 事後確認 | 事後通知 |

### 18.3. 透明性レポート
-   事業規模が一定以上の場合、政府からのデータ要求に関する**透明性レポート**を年次で公開。

### 18.4. 内部通報者保護 (Whistleblower Protection)
-   **Law**: EU内部通報者指令は50人以上の組織に内部通報チャネル設置を義務付け。
-   **実装要件**: 匿名報告チャネル、受領確認**7日以内**、フィードバック**3ヶ月以内**、報復からの保護。
-   **AI Act連動**: 2026年8月より、EU AI Act違反の報告も内部通報者保護の対象。

## §19. リーガルホールド・eDiscovery (Legal Hold & eDiscovery)

### 19.1. リーガルホールド
-   **Law**: 訴訟が合理的に予見される時点で、関連データの保全義務が発生。自動削除ポリシー（§12）を**即座に停止**。
-   **フロー**:
    1.  法務がリーガルホールドの必要性を判断
    2.  ホールド通知を関連Custodianに発行
    3.  自動削除ジョブを対象データについて停止
    4.  Custodianの承認を追跡・リマインダー自動送信
    5.  ホールド解除時は正式な解除通知 + 通常の保持ポリシー再開
-   **Anti-Pattern**: ホールド通知の口頭伝達。書面で発行し承認記録を保全せよ。

### 19.2. eDiscovery対応
-   **情報ガバナンス**: 平時からデータ所在マップ（§16.6）を整備し、eDiscovery要求時に迅速に対象データを特定可能に。
-   **能力**:
    -   ESI収集・保全手順書の整備
    -   機密区分・特権データのフィルタリング手順
    -   AI支援による大量データのレビュー・関連性判定
    -   PIIのマスキング・レダクション手順
    -   全工程の文書化（防御可能性の確保）

### 19.3. eDiscoveryにおけるAI活用とAI生成証拠保全

> [!IMPORTANT]
> 2025年時点で法律事務所の**79%**がAIをeDiscoveryに活用（2023年の19%から急増）。AI生成コンテンツの証拠保全は新たな法的課題。

-   **AI支援レビュー（TAR/CAL）**:
    -   Technology Assisted Review（TAR）またはContinuous Active Learning（CAL）の導入を推奨
    -   大量のESI（電子保存情報）の関連性判定・秘密特権フィルタリングを自動化
    -   AIモデルの訓練データ・精度指標（適合率/再現率）を文書化し**防御可能性**を確保
-   **AI生成コンテンツの証拠保全**:
    -   AIが生成したドキュメント・メール・コードも**発見可能なeDiscovery対象**
    -   AIチャットログ、プロンプト履歴、生成物の保全義務をリーガルホールドポリシーに組み込む
    -   **Shadow AI**（未承認のAIツール使用）による証拠散逸リスクを管理
-   **Action**: eDiscovery対応計画にAI生成コンテンツの保全範囲を明示的に含める。AIチャットログの保持期間を§12に基づき定義。
-   **Cross-Reference**: §12 (データライフサイクル), `400_ai_engineering.md` (AIツールログ管理)

## §20. 合成データガバナンス (Synthetic Data Governance)

### 20.1. 定義と法的位置付け
-   **合成データ**: 実データの統計的特性を再現しつつ、個人を直接参照しない人工的に生成されたデータ。
-   **法的注意**: 合成データであっても、元データと組み合わせて再識別が可能な場合は個人データとして扱われる（EDPB見解）。「合成データだからGDPR対象外」は**誤り**。

### 20.2. ガバナンスフレームワーク

| 要素 | 要件 |
|:-----|:-----|
| **生成目的の文書化** | 合成データの意図された用途を生成前に明確に定義・記録 |
| **バイアス管理** | 人口統計的代表性のバランシング努力を文書化。合成データが元データのバイアスを増幅しないことを検証 |
| **品質検証** | Train-Synthetic-Test-Real (TSTR) 検証: 合成データで訓練したモデルを実データの小サンプルでテスト |
| **メタデータ管理** | 合成データに生成手法、元データの目的、既知の制約を記録するメタデータタグを付与 |
| **監査可能性** | 生成プロセスの完全な再現可能性を保証。生成パラメータ・シード値の記録 |
| **アクセス制御** | 合成データであっても、生成元の実データの分類レベルに準じたアクセス制御を適用 |

### 20.3. 利用制限
-   **禁止**: 合成データを「実データの代替」として規制当局に提出することは禁止。
-   **許可用途**: AI/MLモデルのトレーニング、テスト環境でのデータ利用、データアクセスが制限される越境シナリオ。
-   **Cross-Reference**: `400_ai_engineering.md` (RAG・学習データ), `700_qa_testing.md` (テストデータ)

## §21. Data Clean Room規制 (Data Clean Room Regulation)

### 21.1. 定義と法的位置付け
-   **Data Clean Room**: 複数の当事者がデータを安全に結合・分析するための、行レベルのデータが外部に露出しない**隔離された分析環境**。
-   **FTC警告**: Data Clean Roomは不正なデータ開示や利用を自動的に防止するものではない。法的義務を迂回する手段として使用してはならない（FTC 2024ガイダンス）。

### 21.2. 運用要件

| 要件 | 内容 |
|:-----|:-----|
| **データ最小化** | 分析目的に必要最小限のデータのみをClean Roomに投入 |
| **匿名化閾値** | 出力は事前定義された閾値（例: k≥50）を満たす集計データのみ。行レベルデータの流出を禁止 |
| **差分プライバシー** | 出力にノイズ注入を適用し、個人の特定可能性を数学的に制限 |
| **契約的制約** | 参加者間でデータ利用目的・範囲・期間を規定するDCR契約を締結 |
| **第三者監査** | Clean Roomの運用がプライバシー規制に準拠していることを年次で第三者が監査 |
| **ログ記録** | 全クエリ・出力の完全な監査ログを保持 |

### 21.3. 適用ケース
-   広告効果測定（Cookie代替としてのポストCookieソリューション）
-   業界横断ベンチマーク分析
-   委託先とのプライバシー準拠データ協業

## §22. 神経データ・新興バイオメトリクス (Neural Data & Emerging Biometrics)

### 22.1. 定義と法的リスク
-   **Neural Data（神経データ）**: 脳・神経系から直接取得される情報。精神状態、感情、意図に関する極めてセンシティブな洞察を含む。
-   **リスク**: 一度漏洩すると変更不可能であり（生体認証と同等）、精神的プライバシーの侵害、差別、操作に悪用される可能性がある。

### 22.2. 規制マップ

| 法域 | 規制 | 施行 | 要件 |
|:-----|:-----|:-----|:-----|
| **カリフォルニア** | SB 1223 (CCPA改正) | 2025.1 | 神経データをセンシティブ個人情報に分類。アクセス・削除・共有制限権を付与 |
| **Colorado / Montana** | 神経データ保護法 | 2025 | 収集時の明示的同意、第三者開示への別途同意、削除権 |
| **Connecticut** | プライバシー法改正 | 2026.7 | 神経データをセンシティブデータカテゴリに追加 |
| **EU** | GDPR Art.9 | 適用中 | 神経データを特別カテゴリ情報として分類 |
| **チリ** | 憲法改正 | 2021 | 世界初の「神経権（Neurorights）」の憲法的保護 |

### 22.3. 実装要件
-   神経データを処理する場合は**明示的な書面同意**を取得（電子署名可）
-   神経データの**商業利用は原則禁止**。例外は医療・ヘルスケア目的のみ
-   保存時は**AES-256以上**で暗号化し、他の個人データから**分離保管**
-   **DPIA必須**: 神経データの処理を開始する前にDPIA（§6.2.2）を実施
-   神経データの第三者提供には**別途の明示的同意**を取得
-   **保持期間**: 目的達成後**1年以内**に削除

## §23. ESG・サステナビリティ報告義務 (ESG & Sustainability Reporting)

### 23.1. CSRD（企業サステナビリティ報告指令）
-   **Law**: EUのCSRDは段階的に適用範囲を拡大:
    -   **2025年**（FY2024報告）: 旧NFRD対象企業
    -   **2026年**（FY2025報告）: EU大企業（従業員250人超 or 売上€50M超 or 総資産€25M超のうち2基準充足）
    -   **2027年**（FY2026報告）: 上場SME（オプトアウト延期可）
    -   **2029年**（FY2028報告）: EU売上€150M超の非EU企業
-   **Omnibus簡素化（2026.2）**: 閾値引上げ・タイムライン延長・データポイント70%削減が提案。施行まで現行ルールを遵守。

### 23.2. デジタルプロダクトパスポート (DPP)
-   **Law**: EU ESPR（エコデザイン規則）に基づく製品ライフサイクル情報のデジタル記録。
-   **施行タイムライン**: バッテリー（2026.7）、鉄鋼（2027）、繊維・アルミ（2028）。
-   **エンジニアリング対応**: DPP対応製品を扱うサービスは、DPPデータの取得・表示・APIアクセス機能を設計時に考慮。

### 23.3. デジタルサービスへの影響
-   **データセンターの環境報告**: 大規模デジタルサービスはデータセンターのエネルギー効率・炭素排出量の開示を求められる可能性。
-   **グリーンウォッシング禁止**: 環境に関する虚偽・誤解を招く主張は法的リスク。AI生成の環境主張にも適用。
-   **Cross-Reference**: `101_revenue_monetization.md` (FinOps)

## §24. 独占禁止法・プラットフォーム規制 (Antitrust & Platform Regulation)

### 24.1. EU DMA（デジタル市場法）
-   **Law**: ゲートキーパー（Alphabet、Amazon、Apple、ByteDance、Meta、Microsoft）に対し、以下を禁止:
    -   **セルフプリファレンシング**: 自社サービスを検索結果やランキングで優遇する行為
    -   **クロスプラットフォームデータ結合**: ユーザーの明示的同意なくプラットフォーム間でデータを結合する行為
    -   **サイドローディング制限**: アプリストア以外からのアプリインストールを不当に制限する行為
-   **罰則**: 全世界売上の**10%**（反復違反時は20%）。

### 24.2. アルゴリズム共謀の規制
-   **Risk**: 価格設定AI等のアルゴリズムが意図せず共謀的な結果を生む場合、独占禁止法違反となりうる。
-   **事例**: 米国DOJがRealPage社との和解（2025.11）— アルゴリズムを用いた情報共有が集中市場で競争を阻害。
-   **対策**: 価格設定・市場分析にAIを使用する場合、競合他社のデータと直接共有されない設計を保証。

### 24.3. エンジニアリングへの影響
-   プラットフォーム上でアプリ・サービスを提供する場合、ゲートキーパーのDMA対応による仕様変更に備えたアーキテクチャの柔軟性を確保。
-   **Cross-Reference**: `301_api_integration.md` (APIゲートウェイ), `103_appstore_compliance.md` (ストアガイドライン)

## §25. AI管理システム標準（ISO 42001 / NIST AI RMF）

### 25.1. ISO/IEC 42001 (AI Management System)
-   **概要**: 2023年12月発行。AIの責任ある開発・展開・運用のためのマネジメントシステム規格。
-   **主要要件**:
    -   AIリスク管理プロセスの確立
    -   AIシステム影響評価の実施
    -   ライフサイクル管理
    -   サードパーティサプライヤーの監督
    -   継続的改善プロセス
-   **適用**: EU AI Actの規制ガイダンスでも参照。認証取得はコンプライアンスの証明に有効。

### 25.2. NIST AI RMF（AI Risk Management Framework）
-   **概要**: 2023年1月発行。AI信頼性向上のための自主的ガイドライン。
-   **4つのコア機能**:

| 機能 | 目的 |
|:-----|:-----|
| **Govern** | AIリスク管理のガバナンス体制構築 |
| **Map** | AIシステムのリスクをコンテキスト内で特定 |
| **Measure** | 特定されたリスクの分析・評価 |
| **Manage** | リスク軽減措置の実施・モニタリング |

-   **GenAIプロファイル**: 2024年7月にNISTが公開した生成AI固有のリスク管理プロファイル。ハルシネーション、データプライバシー、知的財産リスクに特化。
-   **Cross-Reference**: `400_ai_engineering.md` (AI実装), §11 (AI法規制)

### 25.3. 統合アプローチ
-   EU AI ActコンプライアンスにISO 42001の枠組みを利用し、NIST AI RMFの実務ガイダンスで補完する**統合的アプローチ**を推奨。
-   **Mandate**: 高リスクAIシステムを展開する場合、ISO 42001またはNIST AI RMFに基づくリスク管理文書を作成・維持。

## §26. デジタル主権・技術的独立性 (Digital Sovereignty & Technological Independence)

### 26.1. 概念
-   **デジタル主権**: 国家・組織・個人が、自らのデータ・技術・デジタルインフラを外部からの依存なく管理する能力。
-   **2つの次元**: データレジデンシー（物理的保存場所）+ 法的管轄権（データに対する法的支配）。

### 26.2. EUの取り組み
-   **EuroStack / SECA**: 欧州主導のデジタルサプライチェーン構築。主権クラウド、AI、量子コンピューティング、サイバーセキュリティへの投資。
-   **DMA/DSA/AI Act**: デジタル経済と新興技術の規制的枠組み。

### 26.3. エンジニアリングへの影響
-   **クラウド戦略**: EUユーザーを対象とするサービスは、EU域内のクラウドリージョンを優先。主権クラウド（Sovereign Cloud）の利用を検討。
-   **OSS優先**: デジタル主権の観点からオープンソースソフトウェアの採用を推奨。透明性と技術的独立性を確保。
-   **ベンダーロックイン回避**: 特定の非EU拠点クラウドプロバイダーへの過度な依存を避け、マルチクラウド・ポータビリティを設計段階で考慮。
-   **Cross-Reference**: `361_aws_cloud.md` (クラウドアーキテクチャ), §3.4.1 (データローカライゼーション)

## §27. ヘルスデータ保護 (Health Data Protection)

### 27.1. HIPAA改正（2026年施行見込み）
-   **Law**: HIPAA Security Ruleの大幅改正が提案中（2025.1提案、2026年H2施行見込み）。数十年ぶりの包括的見直し。
-   **主要変更点**:
    -   「required」と「addressable」の区別を廃止 — ほぼ全てのセキュリティ対策が**必須化**
    -   ePHIアクセスに**MFA（多要素認証）必須**
    -   全ePHIの**暗号化義務**拡大
    -   **年次コンプライアンス監査**、半年次脆弱性スキャン、年次ペネトレーションテスト
    -   施行後**180日以内**に大半の規定への準拠が必要
-   **Cross-Reference**: `600_security_privacy.md` (認証・暗号化)

### 27.2. HIPAA非対象ヘルスデータの保護
-   **背景**: スマートウォッチ、ヘルスアプリ等が収集する健康データの多くはHIPAA対象外。
-   **Health Information Privacy Reform Act**: HIPAA類似の保護を消費者ヘルスTechに拡張する法案（2025.11上院提出）。
-   **Washington MHMDA**: 消費者ヘルスデータの収集・共有に対する同意義務と削除権を規定。

### 27.3. エンジニアリング対応
-   ヘルスデータを処理するサービスは、HIPAA対象の有無に関わらず**最高レベルのデータ保護**を設計に組み込む
-   ヘルスデータのDB分類は `@pii:health` タグで管理し、§16.4のPrivacy Engineeringパターンを適用
-   **Cross-Reference**: §16.4 (Privacy Engineering)

## §28. リプロダクティブヘルスデータ保護 (Reproductive Health Data Protection)

### 28.1. 法的背景
-   **Dobbs判決後の影響**: HIPAA Reproductive Health Care Privacy Rule（リプロダクティブヘルスデータへのHIPAA保護拡大ルール）は、2025年6月テキサス連邦判事により**全国的に無効化**。
-   **州法での保護**: 一部の州（California、Virginia等）が独自のリプロダクティブヘルスデータ保護を制定。連邦レベルの保護がない現在、対応は**州法ベース**。

### 28.2. エンジニアリング要件
-   リプロダクティブヘルスに関連するデータ（位置情報、検索履歴含む）は**センシティブデータ**として分類
-   法執行機関からの要求に対し、§18の手続きを厳格に適用
-   位置情報データの保持期間を最小化し、リプロダクティブ関連施設への訪問推定が困難な設計を採用
-   **Anti-Pattern**: 位置情報データを無期限に保持し、法執行機関の要求時にリプロダクティブヘルス情報が推定可能な状態を放置すること

## §29. 従業員データ保護・職場監視 (Employee Data Protection & Workplace Monitoring)

### 29.1. 法的枠組み

| 法域 | 規制 | 要件 |
|:-----|:-----|:-----|
| **米国（連邦）** | ECPA | 正当なビジネス目的での監視を許可。ただし通知推奨 |
| **カリフォルニア** | CCPA/CPRA | 従業員に対するデータ収集実務の開示義務。アクセス・削除・訂正権 |
| **EU** | GDPR + AI Act | 正当な利益の厳格な評価。AI監視ツールは高リスクカテゴリ（2026.8施行） |
| **N/A** | NLRB判例 | 過度に侵襲的な電子監視・アルゴリズムベースの管理を精査 |

### 29.2. AI支援監視ツールの法的リスク
-   **EU AI Act**: 雇用・労務管理におけるAIシステムは**高リスク**に分類（2026.8施行）。適合性評価・技術文書・人間の監視が必須。
-   **職場での感情認識AI**: EU AI Act下で**禁止**（2025.2施行済）。
-   **アルゴリズムベースの人事評価**: 従業員の昇進、解雇等の意思決定にAIを使用する場合、§3.7の自動意思決定規制が適用。

### 29.3. 実装要件
-   従業員監視を行う場合、**書面でのプライバシー通知**を提供
-   収集データの範囲を**業務上必要な最小限**に限定
-   リモートワーク環境での画面監視・キーストロークログは**目的の正当性**を文書化
-   監視データのアクセスを**人事・セキュリティ部門**に限定し、アクセスログを保持

## §30. セクター別データ保護規制 (Sector-Specific Data Protection)

### 30.1. 金融データ

| 規制 | 適用 | 主要要件 |
|:-----|:-----|:---------|
| **GLBA** | 米国金融機関 | プライバシー通知義務、情報共有オプトアウト、データセキュリティ |
| **FCRA** | 信用情報 | 消費者信用報告の正確性担保、異議申立権、利用目的の制限 |
| **PCI DSS v4.0** | カード決済 | カード保有者データの保護（2025.3完全施行） |

### 30.2. 教育データ
-   **FERPA**: 学生の教育記録の保護。親・成人学生のアクセス権と同意権。
-   **エンジニアリング影響**: 教育関連サービスでは、学生データの第三者開示に対するオプトアウト機能を実装。

### 30.3. 通信データ
-   **TCPA**: テレマーケティングと自動化された通信に対する同意義務。違反は**$500-$1,500/件**の損害賠償。
-   **TCPA AI影響**: AI生成の音声通話・テキストはTCPA準拠の事前同意が必要。FCC規制でAI生成音声は「人工的な声」として分類（2024）。

### 30.4. FTC Section 5
-   **執行強化**: FTCは不公正・欺瞞的なデータ慣行に対する執行を強化。「AI Washing」（AI能力の誇張）への取締りも活発化。
-   **Cross-Reference**: §8 (ダークパターン禁止)

### 30.5. 防衛産業（CMMC）
-   **Law**: Cybersecurity Maturity Model Certification（CMMC）は、米国国防総省（DoD）の調達先に対するサイバーセキュリティ認証要件。**2025年10月**に最終規則が発効。
-   **3階層構造**:

| レベル | 対象 | 要件 |
|:------|:-----|:-----|
| **Level 1** | FCI（連邦契約情報）の取扱 | 自己評価、基本的サイバーハイジーン（17プラクティス） |
| **Level 2** | CUI（管理対象非機密情報）の取扱 | NIST SP 800-171r2（110コントロール）、第三者認証 |
| **Level 3** | 高優先CUIの取扱 | NIST SP 800-172（追加要件）、政府主導評価 |

-   **エンジニアリング影響**: 米国政府機関を顧客とするサービス、または防衛サプライチェーンに参加する場合はCMMCレベルに応じたセキュリティ対策を実装。
-   **Cross-Reference**: `600_security_privacy.md` (セキュリティ実装)

## §33. 政府監視・国家安全保障とデータプライバシー (Government Surveillance & National Security)

### 33.1. FISA Section 702
-   **Law**: 米国外の非米国人に対する令状なしの監視を許可。2026年4月に**失効予定**（再認可審議中）。
-   **リスク**: 米国人の通信が「付随的」に収集される問題。EU-US Data Privacy Frameworkへの影響。
-   **エンジニアリング影響**: Schrems II判決に基づき、EU→米国へのデータ移転ではTIA（§3.4）で米国の監視法制を評価する義務。

### 33.2. Schrems II影響と越境移転
-   **Mandate**: EU→米国データ移転では、EU-US Data Privacy Frameworkに依拠する場合でも、FISA 702の動向を継続的に監視。
-   **補完措置**: 暗号化（保存時・転送時）、仮名化、分割処理など技術的補完措置をTIAに文書化。
-   **Cross-Reference**: §3.4 (越境移転フレームワーク)

### 33.3. 政府データアクセスへの技術的対抗策
-   **エンドツーエンド暗号化**: 可能な場合はE2E暗号化を実装し、サービスプロバイダー自身もデータにアクセス不能な設計を検討
-   **データ最小化**: 保持するデータ量を最小化することで、政府アクセス要求の影響範囲を限定
-   **透明性レポート**: §18.3のレポートに政府監視要求の統計を含める

### 33.4. FTC外国敵対者データ法（Protecting Americans’ Data From Foreign Adversaries Act）
-   **Law**: 2024年成立。データブローカーが米国個人の機密個人データ（位置情報・健康・金融・生体データ等）を**外国敵対国**（中国・ロシア・イラン・北朝鮮等）またはその支配下団体に共有することを**禁止**。
-   **FTC執行権**: FTCが執行権限を保有。
-   **Action**: ㍲4件スクリーニングの一環として、データ共有先が外国敵対国の支配下にないかを検証。
-   **Cross-Reference**: §32 (データブローカー規制)

### 33.5. DOJ Bulkデータルール（2025.4施行）
-   **Law**: 米国人のバルク個人データまたは政府関連データを外国および対象者（covered persons）との特定取引で処理する新たな規制フレームワーク。
-   **施行**: 2025年4月施行（2025年10月に追加要件発効）。
-   **Action**: 点検対象取引のスクリーニングを実施。特に外国ベンダーへのデータ処理委託において、バルク個人データ取引に該当しないかを検証。

## §34. EU Data Act（EUデータ法）

### 34.1. 概要と施行タイムライン
-   **Law**: EU Data Act（Regulation (EU) 2023/2854）は、コネクテッドデバイスが生成するデータへのアクセス・共有・利活用を規定する新たな規制。
-   **発効**: 2024年1月11日発効。
-   **段階施行**:

| 段階 | 施行日 | 内容 |
|:-----|:------|:-----|
| **Phase 1** | **2025.9.12** | ユーザーのデータアクセス権・共有権、不公正契約条項規制、データ共有義務 |
| **Phase 2** | **2026.9.12** | 新製品の**Access-by-Design**（設計段階でのアクセス義務）、相互運用性要件強化 |
| **Phase 3** | **2027.1** | クラウドサービスの**スイッチング料金・データエグレス料金の廃止** |
| **Phase 4** | **2027.9.12** | データポータビリティ標準の完全実装 |

-   **適用対象**: コネクテッド製品のメーカー、関連サービスプロバイダー、クラウド事業者。**EU外企業**であってもEU顧客にサービスを提供する場合は対象。

### 34.2. コネクテッドデバイスのデータアクセス権
-   **ユーザーアクセス権**: ユーザー（企業・消費者を問わず）は、自身のコネクテッドデバイスが生成するデータへのアクセス権を有する。
-   **対象データ**: デバイス使用中に生成される全データ（個人データ・非個人データ含む）。生データ出力、前処理情報、メタデータを包含。
-   **第三者共有**: ユーザーは自身のデータをユーザーが指定する第三者と共有する権利を有する。
-   **コネクテッド製品の定義**: データを収集・生成し、通信可能な物理的アイテム（スマートホーム製品、コネクテッドカー、産業機器、医療機器等）。
-   **データ形式**: 構造化された、一般的に使用される、機械可読な形式でデータを提供。技術的に可能な場合は**継続的かつリアルタイム**で提供。
-   **無償提供**: ユーザーへのデータ提供は**無償**。

### 34.3. Access-by-Design要件（2026.9.12以降）
-   **設計段階義務**: 2026.9.12以降にEU市場に投入される新たなコネクテッド製品・関連サービスは、**設計段階からデータアクセス性**を組み込む義務。
-   **実装要件**:
    -   ユーザーが容易・安全・無償でデータを取得可能なインターフェースの設計
    -   データアクセスAPIの標準実装
    -   製品説明書にデータの種類・量・形式・アクセス方法を明記
-   **Anti-Pattern**: 「後付けでAPIを追加すればよい」は不可。設計段階からの組込みが義務。

### 34.4. クラウドサービスポータビリティ
-   **スイッチング義務**: クラウドサービスプロバイダーは、クラウドサービス間の切替えを促進する義務。
-   **障壁除去**: データ移行の契約的・技術的障壁を除去し、相互運用性を確保。
-   **料金廃止**: **2027年1月**以降、スイッチング料金・データエグレス料金は**廃止**。
-   **Cross-Reference**: `361_aws_cloud.md` (クラウドアーキテクチャ)

### 34.5. エンジニアリングへの影響
-   **IoTサービス**: コネクテッドデバイスを扱うサービスは、データアクセスAPI・データエクスポート機能を**設計段階**で組み込む。
-   **データフォーマット**: データ提供のフォーマット標準化（JSON/CSV等の機械可読形式）を設計時に考慮。
-   **GDPRとの棲み分け**: Data Actは**GDPRに完全準拠**。個人データが関与する場合はGDPRが優先。個人データと非個人データの分離管理が必要。
-   **営業秘密保護**: 営業秘密保護を理由にデータアクセスを不当に拒否することは禁止。ただし明確かつ深刻な経済的損害を証明した場合は例外的にデータ開示を制限可能。
-   **事前開示義務**: データホルダーは、生成されるデータの種類・量・形式、アクセス方法、製造者・第三者のアクセス有無をユーザーに**事前に開示**する義務。

### 34.6. 契約上の保護
-   **不公正条項規制**: データ共有契約における不公正条項を規制（特にSME保護）。
-   **Take-it-or-leave-it禁止**: ユーザーに過度な負担を課す「受け入れか拒否か」条項は禁止。
-   **Model Contractual Terms (MCT)**: 欧州委員会がデータアクセス・利用に関するモデル契約条項を策定中（2025年9月）。合理的報酬・営業秘密保護に関する条項を含む。
-   **Cross-Reference**: §13 (委託先データガバナンス・DPA)

## §35. AI Companion・フロンティアモデル安全規制

### 35.1. California SB 53（フロンティアモデル安全義務）
-   **Law**: 2026年1月施行。売上$500M超のフロンティアAIモデル開発者に**リスク管理フレームワーク**の公開を義務付け。
-   **主要要件**:
    -   リスク管理フレームワークの策定・公開
    -   安全性テスト結果の文書化
    -   重大リスクの軽減措置の実装
-   **適用範囲**: フロンティアモデルの開発者に限定。利用者・デプロイヤーは直接の対象外。

### 35.2. AI Companion Chatbot安全規制
-   **Law**: California法（2026年1月施行）。AIコンパニオンチャットボットに対する安全基準を導入。
-   **背景**: AIコンパニオンとの感情的依存やメンタルヘルスへの影響（特に未成年者）に対する懸念が立法化。
-   **要件**:
    -   AIコンパニオンであることの**明示的開示**
    -   未成年者に対する**追加保護措置**（利用時間制限、感情的操作の禁止等）
    -   自傷・自殺に関する会話への**緊急リソース案内**機能
    -   AIコンパニオンとのインタラクションデータの保護

### 35.3. AI生成コンテンツラベリング Code of Practice（EU Art.50）
-   **Law**: EU AI Act Art.50に基づく透明性義務。AIシステムとの対話や、AI生成コンテンツ（ディープフェイク含む）のラベリングを義務付け。
-   **Code of Practice第2ドラフト（2026.3）**: AI生成コンテンツのマーキング・ラベリングに関する自主的行動規範。
-   **主要要件**:
    -   チャットボットとの対話時に**AI対話であることの通知**義務（2026.8施行）
    -   ディープフェイク・AI生成テキスト（公共情報目的）への**明確なラベリング**義務
    -   **EU共通アイコン標準**の策定: AI生成コンテンツを示す標準化されたEUアイコンの使用を推奨
    -   オープン技術標準の採用を奨励
-   **エンジニアリング実装**:
    -   AI生成コンテンツに`ai_generated: true`フラグとメタデータ（生成モデル、生成日時）を付与
    -   ユーザーインターフェースにAI生成であることを示す**視覚的インジケーター**を配置
    -   EU共通アイコン標準が確定次第、採用の準備

### 35.4. エンジニアリング要件
-   **Cross-Reference**: §11 (AI法規制), §31 (グローバルAI規制マッピング), §25 (ISO 42001 / NIST AI RMF)
-   **Mandate**: AI機能をユーザーに提供する場合、上記規制への準拠状況をAIガバナンス委員会（§11.9）で**四半期ごと**にレビュー。

## §36. プライバシー・バイ・デザイン実装フレームワーク (Privacy by Design Implementation Framework)

### 36.1. GDPR Art.25 7原則の実装マッピング

| # | PbD原則 | 実装パターン | 検証方法 |
|:--|:--------|:-----------|:--------|
| **1** | **予防的（Proactive）** | 脅威モデリング（STRIDE/LINDDUN）を設計段階で実施 | DPIA実施記録の存在確認 |
| **2** | **デフォルト設定（Default）** | 最小限のデータ収集をデフォルト。`SELECT *`禁止、Cookie同意はデフォルトOFF | CI/CDでSELECT *検出、同意バナーのE2Eテスト |
| **3** | **設計に組込み（Embedded）** | PIIタグ付きスキーマ（§16.4）、RLSポリシー、暗号化をアーキテクチャに統合 | スキーマ監査、RLS検証スクリプト |
| **4** | **全機能性（Full Functionality）** | プライバシーとUXのトレードオフではなく両立を設計 | ユーザビリティテストとプライバシーレビューの並行実施 |
| **5** | **ライフサイクル保護（End-to-End）** | データ保持期間の自動適用（§12）、退会時の物理削除フロー | 保持期間ジョブの動作検証、退会フローのE2Eテスト |

| **7** | **ユーザー中心（User-Centric）** | 粒度の高い同意管理（§4）、プライバシーダッシュボード | 同意管理UIのアクセシビリティテスト |

### 36.2. PbDアセスメントチェックリスト

> 新規機能・サービスのリリース前に以下を検証する。

-   [ ] **データ最小化**: 収集するデータは目的達成に必要最小限か
-   [ ] **目的限定**: データの利用目的が明確に定義・文書化されているか
-   [ ] **デフォルト設定**: プライバシー設定がデフォルトで最も保護的な状態か
-   [ ] **同意取得**: 適切な法的根拠（同意、契約履行、正当利益等）が特定されているか
-   [ ] **アクセス制御**: RLS/RBAC等で最小権限アクセスが実装されているか
-   [ ] **暗号化**: 保存時・転送時の暗号化が実装されているか
-   [ ] **保持期間**: データ保持期間が定義され、自動削除が実装されているか
-   [ ] **PIIタグ付け**: PIIカラムに`@pii:*`タグが付与されているか
-   [ ] **DPIA**: 高リスクな処理についてDPIA（§6.2.2）が実施済みか
-   [ ] **ログのPIIレダクション**: ログにPIIが不要に含まれていないか（§16.4）
-   [ ] **第三者データ共有**: サードパーティとのデータ共有にDPA（§13）が締結されているか

-   **Mandate**: 上記チェックリストの全項目がパスしない限り、新規機能のリリースを**ブロック**する。

### 36.3. 設計フェーズ別プライバシーレビューゲート

### 36.4. PET（Privacy Enhancing Technologies）選択ガイド

| 技術 | 適用ケース | プライバシー保証 | パフォーマンス影響 |
|:-----|:----------|:---------------|:----------------|
| **差分プライバシー** | 分析・レポーティング | 数学的プライバシー保証（ε-DP） | 低〜中 |
| **k-匿名化** | データ公開・共有 | k人以上のグループ化 | 低 |
| **連合学習** | 越境データ制約下のAI学習 | データが移動しない | 高（通信コスト） |
| **準同型暗号** | 委託先でのデータ処理 | 暗号化状態での演算 | 非常に高 |
| **合成データ** | テスト・開発環境 | 元データとの非連結性 | 低（生成時のみ） |
| **トークナイゼーション** | 決済・認証データ | 不可逆トークンによる保護 | 低 |

-   **選択基準**: データの機密度、処理目的、パフォーマンス要件、規制要件に基づきPETを選択。複数のPETを組み合わせた**多層防御**を推奨。
-   **Cross-Reference**: §16.4 (Privacy-Preserving Computation), §20 (合成データガバナンス)

## §37. データガバナンス成熟度モデル (Data Governance Maturity Model)

### 37.1. 5段階成熟度モデル

| 段階 | レベル名 | 特徴 | 代表的な状態 |
|:-----|:--------|:-----|:-----------|
| **Level 1** | **Ad-Hoc（場当たり的）** | ガバナンスが未定義。個人の裁量に依存 | PIIの所在不明、保持期間未設定、DSAR対応が手作業 |
| **Level 2** | **Managed（管理開始）** | 基本的なポリシーが存在するが一貫性なし | プライバシーポリシーは存在するが更新不定期、DPAは一部のみ締結 |
| **Level 3** | **Defined（定義・標準化）** | 組織横断的なフレームワークが確立 | ROPA運用中、DPIAプロセス標準化、CI/CDにプライバシーチェック導入 |
| **Level 4** | **Measured（測定・最適化）** | KPIに基づくガバナンスの定量的管理 | §17.5のKPIダッシュボード稼働中、インシデント対応時間の継続的改善 |
| **Level 5** | **Optimized（自動化・継続改善）** | ガバナンスが全自動化され継続的に最適化 | Policy-as-Code完全稼働、AI駆動のリスク検出、規制変更の自動影響分析 |

### 37.2. 段階別KPI・評価基準

| KPI | L1目標 | L3目標 | L5目標 |
|:----|:------|:------|:------|
| **DSAR応答期限内率** | 手作業対応 | ≥90% | ≥99%（自動化） |
| **PIIカラムのタグ付け率** | 未実施 | ≥80% | **100%** |
| **DPA締結率** | 主要ベンダーのみ | ≥90% | **100%** |
| **インシデント平均対応時間** | 測定なし | <4時間 | **<1時間** |
| **CI/CDプライバシーチェック通過率** | N/A | ≥95% | **100%** |
| **データ保持期限遵守率** | 未管理 | ≥80% | **100%**（自動削除） |
| **プライバシー研修完了率** | 任意 | ≥80% | **≥95%** |

### 37.3. 段階移行のためのアクションプラン

-   **L1→L2（基盤構築）**:
    1.  プライバシーポリシーの策定と公開（§5）
    2.  PIIの棚卸しとデータマッピング（§16.6）
    3.  DSAR対応手順書の作成（§3.6）
    4.  主要ベンダーとのDPA締結（§13）

-   **L2→L3（標準化・体制構築）**:
    1.  DPO/CPOの任命またはプライバシー責任者の指定（§16.1）
    2.  DPIAプロセスの標準化と運用開始（§6.2.2）
    3.  ROPAの構築と維持（§16.2）
    4.  CI/CDへのプライバシーチェック導入（§16.5）
    5.  インシデント対応計画の策定と訓練（§14）

-   **L3→L4（定量管理）**:
    1.  KPIダッシュボードの構築と運用（§17.5）
    2.  取締役会/経営会議への定期報告体制の確立（§17.5.1）
    3.  ベンダーリスクティアリングの導入（§13.4）
    4.  テーブルトップ演習の四半期実施（§14.6）

-   **L4→L5（自動化・AI活用）**:
    1.  Policy-as-Codeの全レイヤー導入（§16.5.2）
    2.  AI駆動のリスク検出・分類の導入
    3.  規制変更の自動監視・影響分析パイプラインの構築
    4.  継続的コンプライアンス・モニタリングの実装

### 37.4. フレームワーク参照

## §38. 国際仲裁・ADR紛争解決 (International Arbitration & ADR)

### 38.1. 紛争解決策の設計
-   **原則**: データプライバシー紛争の解決手段をDPA・ToS内に明示的に規定すること。
-   **ADR（裁判外紛争解決）優先**: データプライバシー関連紛争は訴訟よりも仲裁・調停を優先。機密保持と専門性の観点から推奨。

### 38.2. 紛争解決条項チェックリスト

-   [ ] **準拠法**: DPA・ToSの準拠法を明記（データ処理地の法律を推奨）
-   [ ] **仲裁機関**: ICC、LCIA、SIAC、JCAA等から選択
-   [ ] **仲裁地**: 中立的な仲裁地を指定（シンガポール、ロンドン、パリ等）
-   [ ] **言語**: 仲裁言語の指定
-   [ ] **機密保持**: 仲裁手続き・裁定の機密保持義務
-   [ ] **多段階条項**: 調停→仲裁の段階的解決フロー
-   [ ] **仲裁人の専門性**: データプライバシー・テクノロジー法の専門知識を有する仲裁人の選定基準

### 38.3. AIと仲裁の新動向（2025-2026）
-   **AI仲裁人**: AAA/ICDRが2025年9月に書類専用のAI仲裁人を開始。2026年に他機関への拡大を計画。
-   **AI利用ガイドライン**: 仲裁機関がAIの責任ある使用に関するガイドラインを発行。デュープロセス、機密保持、透明性、強制執行可能性を優先。
-   **手続命令でのAI規定**: 仲裁廷が手続命令でAIの許容用途、AI支援の開示、データセキュリティ、人間による監督を明示的に規定する動き。

### 38.4. 法改正動向
-   **英国仲裁法（Arbitration Act 2025）**: 2025年8月施行。仲裁人免責と準拠法に関する重要な変更。
-   **中国新仲裁法**: 2026年3月施行。国際基準への近代化、臨時仲裁制度の導入。
-   **Cross-Reference**: §13 (委託先データガバナンス - DPAの紛争解決条項)

## §39. デジタル契約法・スマートコントラクト (Digital Contract Law & Smart Contracts)

### 39.1. デジタル契約の法的有効性
-   **原則**: デジタル署名・電子契約はほとんどの法域で合法。ただし、法的有効性は**明確な開示と同意**に依存。
-   **日本**: 電子署名法（電子署名及び認証業務に関する法律）に基づき、電子署名は書面署名と同等の法的効力。
-   **EU**: eIDAS 2.0によるEU Digital Identity Wallet義務化（2026-2027）
-   **US**: ESIGN Act / UETAによる電子署名の広範な法的承認

### 39.2. スマートコントラクトの法的フレームワーク

> [!WARNING]
> スマートコントラクトの法的フレームワークはまだ発展途上。強制執行可能性、責任、管轄に関する解釈の問題が残る。

-   **定義**: ブロックチェーン上で自動執行されるプログラム。条件充足時に契約条項を自動的に実行。
-   **法的課題**:
    -   **強制執行可能性**: 裁判所がコードを契約として認めるかは法域により異なる
    -   **責任**: コードのバグ・脆弱性による損害の責任所在
    -   **紛争解決**: 分散型システムでの紛争の管轄地決定
    -   **消費者保護**: 複雑な契約条件の可読性・透明性の確保
-   **規制動向**:
    -   **EU MiCA**: 暗号資産・ブロックチェーン金融サービスの規制（スマートコントラクト含む）
    -   **US Deploying American Blockchains Act (2025)**: ブロックチェーン技術の推進とスマートコントラクトの活用支援
-   **Mandate**: スマートコントラクトを使用する場合、必ず**人間が読める法的拘束力のある契約書**で補完すること。コードのみで契約とすることは禁止。

### 39.3. Australia ADM透明性義務（2026.12施行）
-   **Law**: Privacy Act改正（POLA Act 2024）により、APPエンティティは**自動意思決定（ADM）**の使用についてプライバシーポリシーで開示義務。
-   **対象**: 個人の権利・利益に**重大な影響**を与えると合理的に予想されるADMプロセス。AI/MLだけでなく、ルールベースのツールやスプレッドシートも対象になり得る。
-   **開示要件**:
    -   ADMに使用される個人情報の**種類**
    -   ADMによって下される意思決定の**種類**
-   **罰則**: 最大$330,000（コンプライアンスノーティス）
-   **Action**: 2026年12月までにプライバシーポリシーを更新し、ADMの使用についての開示を追加。
-   **Cross-Reference**: §3.7 (自動意思決定), §11 (AI法規制)

## §40. データ倫理ボード・倫理レビュー (Data Ethics Board & Ethics Review)

### 40.1. 倫理レビューボードの設置と構成
-   **Mandate**: AI/データ活用が事業の中核となる段階で、**データ倫理ボード（Ethics Board）**を設置する。
-   **構成（推奨）**:

| 役割 | 人数 | 選定基準 |
|:-----|:-----|:---------|
| **技術リード** | 1名 | AI/MLエンジニアリングの実務経験 |
| **法務・コンプライアンス** | 1名 | データ保護法・AI規制の専門知識 |
| **プロダクトマネージャー** | 1名 | ユーザー影響の評価能力 |
| **外部有識者（倫理）** | 1名 | テクノロジー倫理・社会影響の学術的知見 |
| **ユーザー代表 / D&Iオフィサー** | 1名 | 多様性・包摂性の観点 |

-   **独立性**: 倫理ボードは経営からの独立性を確保。利益相反がある場合は該当メンバーを議決から除外。
-   **Cross-Reference**: §11.9 (AIガバナンス委員会) — AIガバナンス委員会と倫理ボードの役割分担を明確化すること。

### 40.2. 倫理レビュートリガーと意思決定基準
-   **トリガー条件（以下のいずれかに該当する場合、倫理レビューを必須とする）**:
    -   [ ] ユーザーの行動・嗜好・感情を**プロファイリング**する新機能
    -   [ ] **脆弱者**（子供、高齢者、障害者）に影響を与える可能性のあるデータ処理
    -   [ ] **差別的結果**を生む可能性のあるアルゴリズム（採用、融資、保険、住宅等）
    -   [ ] **大規模な個人データ収集**を伴う新サービス
    -   [ ] ユーザーの**デジタルウェルビーイング**に影響を与えるデザイン（アディクティブUI等）
    -   [ ] **生体認証・神経データ**の新規利用
    -   [ ] **合成データ**の生成・利用（バイアス増幅リスク）
    -   [ ] **法的グレーゾーン**にある技術の導入
-   **意思決定基準（FAST原則）**:
    -   **F**airness（公平性）: 特定グループへの不当な不利益がないか
    -   **A**ccountability（説明責任）: 意思決定の責任所在が明確か
    -   **S**ustainability（持続可能性）: 長期的な社会的影響を考慮しているか
    -   **T**ransparency（透明性）: ユーザーに対する説明が十分か

### 40.3. 倫理的リスクアセスメントフレームワーク

| リスクレベル | 定義 | 対応 | 承認権限 |
|:-----------|:-----|:-----|:---------|
| **Critical** | 基本的人権の侵害リスク | **開発中止**または抜本的再設計 | 倫理ボード全会一致 + 経営承認 |
| **High** | 特定グループへの重大な不利益リスク | 軽減措置の実装 + 定期的監視 | 倫理ボード過半数 |
| **Medium** | 潜在的なバイアス・不公平のリスク | 軽減措置の文書化 + モニタリング | 技術リード + 法務承認 |
| **Low** | 最小限の倫理的懸念 | 標準的な開発プロセスで対応 | PM承認 |

### 40.4. Newspaper Test
-   **定義**: 「この機能/データ利用が新聞の一面記事になった場合、企業の評判にどのような影響を与えるか」を検証する思考実験。
-   **実施タイミング**: 倫理レビュートリガー該当時、および新機能のデータ活用設計時。
-   **文書化**: テスト結果と判断理由を倫理レビュー記録に含める。
-   **Anti-Pattern**: 「法的には問題ない」を理由に倫理レビューをスキップすること。法的合法性と倫理的正当性は別の概念。

### 40.5. 倫理レビュー記録と監査
-   **記録項目**: レビュー日時、参加者、対象機能/サービス、リスク評価結果、決定事項、条件付き承認の場合の条件
-   **保存期間**: **最低5年間**
-   **年次報告**: 倫理ボードの活動サマリーを年次で経営層に報告
-   **Cross-Reference**: §17 (コンプライアンスレビュー)

## §41. 生成AI著作権・訓練データガバナンス強化 (GenAI Copyright & Training Data Governance Enhancement)

### 41.1. AI著作権訴訟の判例マッピング（2024-2026）

> [!WARNING]
> AI著作権法は急速に発展中。以下は主要な訴訟・和解事例であり、確定判例として依拠しないこと。四半期ごとに法務チームが最新状況をレビュー。

| 事例 | 法域 | 状況 | 主要争点 | 影響 |
|:-----|:-----|:-----|:---------|:-----|
| **NYT v. OpenAI/Microsoft** | 米国 | 2024提訴 → 2026審理中 | Fair Use, 出力の実質的類似性 | 大規模言語モデルの学習データ適法性の先例 |
| **Anthropic和解** | 米国 | 2025.8和解（**$1.5B**） | 海賊版データの学習利用 | 学習データの適法性検証の重要性を実証 |
| **Getty v. Stability AI** | 米国/英国 | 審理中 | 画像の無断学習利用 | 画像生成AIの著作権リスク |
| **Thomson Reuters v. Ross Intelligence** | 米国 | 2025判決 | AI学習目的の著作物利用はFair Use非該当 | 「学習＝Fair Use」の安易な前提への警告 |
| **Concord Music v. Anthropic** | 米国 | 審理中 | 歌詞の無断学習・再生成 | 音楽著作権とAI出力の関係 |

-   **Mandate**: AI機能の開発・導入時に、使用するモデルプロバイダーの訴訟リスクを**デューデリジェンス**で評価すること。

### 41.2. 訓練データデューデリジェンスチェックリスト

> 自社でAIモデルを訓練する場合、または外部モデルを利用する場合に実施。

#### 41.2.1. 自社訓練時
-   [ ] 訓練データセットの**全ソース一覧**を文書化しているか
-   [ ] 各ソースの**ライセンス条件**を確認し、AI訓練目的での利用が許可されているか
-   [ ] **著作権保護コンテンツ**の含有割合を評価しているか
-   [ ] EU DSM指令Art.4の**TDMオプトアウト**を尊重しているか（robots.txt / ai.txt確認）
-   [ ] 日本著作権法30の4の**「利益を不当に害する」例外**に該当しないか法的評価を実施したか
-   [ ] 訓練データに**個人データ**が含まれる場合、§1-§4のプライバシー要件を充足しているか
-   [ ] **バイアス監査**: 訓練データの人口統計的代表性を評価したか
-   [ ] 訓練データの**出所証明（Data Provenance）**を記録・保全しているか

#### 41.2.2. 外部モデル利用時
-   [ ] モデルプロバイダーが訓練データの**適法性を保証**しているか（契約条項確認）
-   [ ] **補償条項（Indemnification）**がプロバイダー契約に含まれているか
-   [ ] プロバイダーの**訓練データオプトアウトポリシー**を確認したか（ユーザーデータの学習利用を拒否可能か）
-   [ ] プロバイダーが公表している**著作権訴訟リスク**を評価したか（§41.1参照）
-   [ ] **出力の著作権帰属**が契約で明確に定義されているか

### 41.3. AI生成物の著作物性判断基準（法域別）

| 法域 | AI生成物の著作物性 | 条件 |
|:-----|:-----------------|:-----|
| **米国** | ❌ 原則否定 | 人間の創造的関与が必要。純粋なAI生成物はパブリックドメイン（USCO 2023, Thaler v. Perlmutter） |
| **EU** | ❌ 原則否定 | 「著者の知的創作」要件。人間の創造的選択・配置が必要 |
| **日本** | ⚠️ 未確定 | 文化庁2024年素案: 「AI活用著作物」は人間の創作的寄与の程度により個別判断 |
| **中国** | ✅ 条件付き肯定 | 北京インターネット法院（2023）: 十分な「知的投入」があればAI生成物に著作権を認定 |
| **英国** | ✅ 条件付き肯定 | CDPA s.9(3): 「コンピュータ生成著作物」の著者は必要な手配をした者 |

-   **Mandate**: AI生成コンテンツを公開する際は、対象法域の著作物性判断基準に基づき著作権保護の可否を事前評価。保護が得られない法域では**営業秘密・契約的保護**を検討。

### 41.4. Input/Output著作権リスクマトリクス

| リスク分類 | Input（学習データ） | Output（生成物） | 対策 |
|:----------|:-------------------|:-----------------|:-----|
| **高リスク** | 著作権保護コンテンツを無許諾で学習 | 既存著作物と実質的に類似する出力 | 使用禁止。フィルタリング・ガードレール実装 |
| **中リスク** | オープンライセンスデータを学習 | 既存スタイルを模倣するが独自性あり | ライセンス条件の遵守確認。出力の類似性チェック |
| **低リスク** | ライセンスクリア済みデータを学習 | 完全に独自の出力 | 標準的なプロセスで対応 |

-   **Output Filteringの実装**:

```typescript
// AI出力の著作権リスクフィルタリング
interface ContentRiskCheck {
  outputId: string;
  similarityScore: number;     // 0-1: 既知著作物との類似度
  riskLevel: 'high' | 'medium' | 'low';
  matchedSources: string[];    // 類似が検出された著作物
  action: 'block' | 'review' | 'pass';
}

const SIMILARITY_THRESHOLDS = {
  block: 0.85,    // 85%以上: 自動ブロック
  review: 0.60,   // 60%以上: 人間レビュー
  pass: 0.0,      // 60%未満: 通過
};
```

-   **Mandate**: AI生成コンテンツを外部公開する場合、類似性チェックを実施すること。
-   **Cross-Reference**: §11.6 (AI生成コンテンツ著作権), §11.11 (GenAI学習データコンプライアンス), `602_oss_compliance.md`

## §42. プライバシー影響評価（PIA/DPIA）統合フレームワーク (Integrated Privacy Impact Assessment Framework)

### 42.1. 影響評価マトリクス

> [!IMPORTANT]
> 複数の規制が異なる種類の影響評価を要求する。本セクションは各評価の関係性を整理し、統合的な実施を可能にする。

| 評価種別 | 法的根拠 | トリガー | 主要な評価対象 | 更新頻度 |
|:--------|:---------|:---------|:-------------|:---------|
| **DPIA** | GDPR Art.35 | 高リスクデータ処理 | プライバシーリスク・データ主体への影響 | 処理変更時 + 年次レビュー |
| **TIA** | GDPR Art.44-49 + CNIL | 越境データ移転 | 移転先国の法的保護水準 | 年1回以上 + 重大変更時 |
| **AI Impact Assessment** | EU AI Act / Colorado AI Act | 高リスクAIシステム | AIの安全性・権利への影響・バイアス | デプロイ前 + 年次 (or 重大変更後90日) |
| **CCPA Risk Assessment** | CCPA/CPRA (2026.1施行) | ADMT利用、プロファイリング | 消費者への利益と権利侵害リスクのバランス | 新規処理前 + 年次 |
| **Ethics Impact Assessment** | §40 (本ファイル) | 倫理レビュートリガー該当 | 公平性・透明性・社会的影響 | 機能リリース前 |

### 42.2. CCPA Risk Assessment要件（2026.1施行）
-   **対象**: 重大なプロファイリング、ADMT利用、センシティブPI処理を行うビジネス
-   **要件**:
    -   処理の**目的と必要性**の記述
    -   消費者への**利益**とプライバシー侵害**リスク**の比較衡量
    -   リスク軽減措置の文書化
    -   **2026.1以前の既存処理**: 2027.12.31までに評価完了
-   **Action**: CCPA Risk Assessmentテンプレートを作成し、既存のDPIAプロセスと統合。

### 42.3. 統合影響評価テンプレート

```typescript
// 統合影響評価の管理フレームワーク
interface IntegratedImpactAssessment {
  assessmentId: string;          // UUID
  projectName: string;
  assessmentTypes: ('DPIA' | 'TIA' | 'AI_IA' | 'CCPA_RA' | 'Ethics_IA')[];
  status: 'draft' | 'in_review' | 'approved' | 'expired';
  riskLevel: 'critical' | 'high' | 'medium' | 'low';
  dataCategories: string[];      // 処理対象データカテゴリ
  jurisdictions: string[];       // 影響法域
  approvedBy: string;            // 承認者
  approvedAt: string;            // ISO 8601
  nextReviewDate: string;        // 次回レビュー日
  mitigationMeasures: {
    measure: string;
    status: 'planned' | 'implemented' | 'verified';
    verifiedAt: string | null;
  }[];
}

// 評価種別の自動判定
function determineRequiredAssessments(
  processing: { hasHighRiskAI: boolean; hasCrossBorderTransfer: boolean;
    hasADMT: boolean; hasEthicsTrigger: boolean; jurisdictions: string[] }
): string[] {
  const required: string[] = ['DPIA']; // DPIAは常に推奨
  if (processing.hasCrossBorderTransfer) required.push('TIA');
  if (processing.hasHighRiskAI) required.push('AI_IA');
  if (processing.hasADMT && processing.jurisdictions.includes('US-CA'))
    required.push('CCPA_RA');
  if (processing.hasEthicsTrigger) required.push('Ethics_IA');
  return required;
}
```

-   **Mandate**: 新規プロジェクト開始時に`determineRequiredAssessments`に相当する判定を実施し、必要な評価を**漏れなく特定**すること。

## §44. データ信託・データ共有協定 (Data Trusts & Data Sharing Agreements)

### 44.1. EU Data Governance Actのデータ利他主義
-   **Law**: EU Data Governance Act（DGA, 2023.9施行）は、公益目的での自発的なデータ共有（**データ利他主義 / Data Altruism**）の法的フレームワークを構築。
-   **主要概念**:

| 概念 | 定義 | 法的位置付け |
|:-----|:-----|:-----------|
| **データ利他主義** | 公益目的（科学研究、公衆衛生等）でのデータの自発的提供 | DGA Chapter IV |
| **データ利他主義組織（DAO）** | データ利他主義活動を促進する認定団体 | EU加盟国の登録・監督下 |
| **データ仲介サービス** | データ保有者とデータ利用者のマッチング | DGA Chapter III, 通知制度 |
| **欧州共通データ空間** | セクター別のデータ共有エコシステム（ヘルス、農業、製造等） | EUデータ戦略の中核 |

-   **エンジニアリング影響**: EUデータ空間に参加する場合、データ共有のAPIインターフェース・アクセス制御・利用目的のトレーサビリティを設計時に組み込む。

### 44.2. Data Trust構造と法的フレームワーク
-   **定義**: Data Trustは、信託法に基づき独立した受託者（Trustee）がデータ提供者の利益のためにデータを管理する構造。
-   **主要モデル**:

| モデル | 特徴 | 適用ケース |
|:------|:-----|:----------|
| **Civic Data Trust** | 市民データを公益のために管理 | スマートシティ、公衆衛生 |
| **Commercial Data Trust** | 企業間のデータ共有を仲介 | 業界横断ベンチマーク |
| **Personal Data Trust** | 個人が自らのデータを信託し管理を委ねる | 個人データの集合的管理 |

-   **法的考慮**: 信託法は法域によって大きく異なる。英国のData Trust Pioneerプロジェクトが先行事例。日本では2022年の「データに関する取引についての検討」（経済産業省）が参考。

### 44.3. データ共有契約テンプレート要件
-   **必須条項**:
    -   [ ] 共有データの範囲・カテゴリの明確な定義
    -   [ ] 利用目的の限定（Purpose Limitation）
    -   [ ] 再共有・再利用の禁止条項
    -   [ ] データの品質・正確性に関する免責
    -   [ ] 知的財産権の帰属
    -   [ ] データ漏洩時の通知義務
    -   [ ] 契約終了時のデータ削除・返却義務
    -   [ ] 準拠法と紛争解決手段
    -   [ ] 個人データが含まれる場合のDPA条項（§13準拠）
-   **Cross-Reference**: §13 (委託先データガバナンス), §21 (Data Clean Room)

### 44.4. 公益データ活用と倫理的考慮
-   **原則**: 公益目的のデータ共有であっても、個人のプライバシー権は保護される。
-   **要件**:
    -   データ利他主義に基づく提供には**明示的かつ自発的な同意**が必要
    -   同意はいつでも**撤回可能**
    -   公益の定義は**狭く厳格に**解釈（「事業のマーケティング改善」は公益に非該当）
    -   匿名化・仮名化を最大限適用し、個人の再識別リスクを最小化
-   **Cross-Reference**: §20 (合成データガバナンス), §3.5 (データ加工基準)

## §45. 量子コンピューティング脅威・暗号アジリティ (Quantum Computing Threats & Cryptographic Agility)

### 45.1. 脅威概要

> [!CAUTION]
> 暗号学的に関連する量子コンピュータ（CRQC）は10-20年以内に現行の公開鍵暗号を解読可能にすると予測される。**Harvest-Now-Decrypt-Later（HNDL）**攻撃は既に進行中であり、長期保護が必要な機密データへの対策は**即座に**開始すべき。

-   **Harvest-Now-Decrypt-Later (HNDL)**: 現在暗号化されたデータを収集・保存し、将来的に量子コンピュータで解読する攻撃戦略。政府レベルで既に実行されている可能性が高い。
-   **影響を受ける暗号**:

| 暗号方式 | 影響 | 現行用途 |
|:--------|:-----|:---------|
| **RSA** | ❌ **破壊** | TLS/SSL、デジタル署名、鍵交換 |
| **ECDSA / ECDH** | ❌ **破壊** | TLS 1.3、暗号通貨、コード署名 |
| **AES-256** | ⚠️ **安全（鍵長倍増相当）** | 対称暗号。AES-256はGroverのアルゴリズムの影響下でもAES-128相当のセキュリティを維持 |
| **SHA-256** | ⚠️ **安全（衝突耐性低下）** | ハッシュ。衝突探索が加速されるがpreimage攻撃には十分な耐性 |

### 45.2. NIST Post-Quantum Cryptography標準

| 標準 | アルゴリズム | 種別 | NIST公開 | 用途 |
|:-----|:-----------|:-----|:---------|:-----|
| **FIPS 203** | ML-KEM (CRYSTALS-Kyber) | 鍵カプセル化 | 2024.8 | TLS鍵交換（RSA/ECDH代替） |
| **FIPS 204** | ML-DSA (CRYSTALS-Dilithium) | デジタル署名 | 2024.8 | コード署名、認証（ECDSA/RSA代替） |
| **FIPS 205** | SLH-DSA (SPHINCS+) | ステートレス署名 | 2024.8 | 長期文書署名（ステートフル署名の代替） |
| **FIPS 206** | FN-DSA (FALCON) | デジタル署名 | 2025策定中 | コンパクトな署名（帯域幅制約環境） |

### 45.3. 法域別PQC移行義務・ガイダンス

| 法域 | 規制/ガイダンス | 要件 | 期限 |
|:-----|:--------------|:-----|:-----|
| **米国** | NSM-10 (2022) + OMB M-23-02 | 連邦政府システムのPQC移行計画策定義務 | 2035年（段階的） |
| **米国** | CISA PQCガイダンス | 重要インフラのCRQCリスク評価とPQC移行ロードマップ策定を推奨 | 推奨（義務化検討中） |
| **EU** | ENISA PQCレポート (2025) | EU機関のPQC移行ロードマップのガイダンス提供 | 推奨 |
| **フランス** | ANSSI (2024) | ハイブリッド暗号（PQC + 従来暗号の併用）を推奨 | 推奨 |
| **ドイツ** | BSI (2024) | ML-KEM / ML-DSAの政府利用を推奨 | 推奨 |

### 45.4. 暗号アジリティガバナンスフレームワーク
-   **定義**: **暗号アジリティ（Cryptographic Agility）**とは、暗号アルゴリズムの変更がシステム全体の大規模な改修なしに実行可能な設計特性。
-   **実装原則**:
    -   暗号アルゴリズムを**構成パラメータ**として管理し、コードにハードコードしない
    -   TLS設定で**複数のciphersuiteをサポート**し、PQCアルゴリズムの追加を容易にする
    -   暗号化されたデータに**アルゴリズムメタデータ**（使用した暗号方式・鍵長・バージョン）を付与
    -   **CBOM（Cryptography Bill of Materials）**を作成・維持し、使用中の全暗号資産を可視化

```typescript
// 暗号アジリティ設計パターン
interface CryptoConfig {
  keyEncapsulation: 'ML-KEM-768' | 'ML-KEM-1024' | 'X25519';
  digitalSignature: 'ML-DSA-65' | 'ML-DSA-87' | 'Ed25519';
  symmetricEncryption: 'AES-256-GCM';
  hashFunction: 'SHA-256' | 'SHA-3-256';
}

// 環境変数でアルゴリズムを切り替え可能にする
const cryptoConfig: CryptoConfig = {
  keyEncapsulation: (process.env.CRYPTO_KEM as CryptoConfig['keyEncapsulation']) || 'X25519',
  digitalSignature: (process.env.CRYPTO_SIG as CryptoConfig['digitalSignature']) || 'Ed25519',
  symmetricEncryption: 'AES-256-GCM',
  hashFunction: 'SHA-256',
};
```

### 45.5. PQC移行ロードマップ

| フェーズ | 期間 | アクション |
|:--------|:-----|:----------|
| **Phase 1: 棚卸し** | 即時開始 | CBOM作成。使用中の全暗号アルゴリズム・プロトコル・鍵の棚卸し |
| **Phase 2: リスク評価** | 3ヶ月以内 | HNDLリスクの評価。長期保護が必要なデータの特定（ヘルスデータ、政府データ等） |
| **Phase 3: ハイブリッド移行** | 6-12ヶ月 | 高リスク領域からハイブリッド暗号（PQC + 従来暗号の併用）を導入 |
| **Phase 4: PQC完全移行** | 2030年目標 | 全システムをPQCアルゴリズムに完全移行 |

-   **Mandate**: Phase 1（CBOM作成）は**即座に**開始すること。暗号資産の可視化なくしてPQC移行計画は策定不可。
-   **Anti-Pattern**: 「量子コンピュータはまだ先の話」として対策を先送りすること。HNDLリスクは**現在進行中**。
-   **Cross-Reference**: `600_security_privacy.md` (暗号化・鍵管理), §30.5 (CMMC — NIST SP 800-171のPQC対応)
