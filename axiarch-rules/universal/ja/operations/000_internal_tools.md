## Part I. 基本原則・Admin Excellence Culture

### 1.1. Admin Excellence Culture（管理ツール品質文化）

- **Rule 50.001**: 管理ツール・内部ツールは「プロダクトの一部」であり、エンドユーザー向け機能と同等の品質基準を適用すること
- **Rule 50.002**: 運用者（非エンジニア）の認知負荷をゼロに近づけることが最優先目標である
- **Rule 50.003**: 管理画面での操作ミスは顧客データの毀損に直結する。**構造的に誤操作を防止する設計**を義務とする

### 1.2. 優先順位の階層

```
セキュリティ（データ保護）> 操作安全性（誤操作防止）> 運用効率 > 開発速度
```

- 管理ツールの全設計判断はこの順序に従う
- 開発速度のために操作安全性を犠牲にすることは禁止

### 1.3. 管理ツール3原則

| 原則 | 説明 |
|------|------|
| **Operator-First** | 運用者の視点で設計する。エンジニアの都合を押し付けない |
| **Audit Everything** | 全操作を記録する。追跡不能な操作を許容しない |
| **Fail Safe** | 不明な状態ではデフォルトで安全側に倒す。破壊的操作には二重確認を必須とする |

### 1.4. Admin as a Product（管理ツールもプロダクトである）

- **Rule 50.004**: 管理ツールにもプロダクトロードマップ・ユーザーストーリー・スプリント計画を適用する
- 運用者NPS（Net Promoter Score）を四半期ごとに計測し、改善サイクルを回す
- 管理ツールの機能要望はバックログで管理し、優先度付けを行う

### 1.5. 運用者の権利章典（Operator Bill of Rights）

| # | 権利 |
|---|------|
| 1 | 操作結果を即座に確認できる権利 |
| 2 | 誤操作を取り消せる権利（Undo / Cooling-off） |
| 3 | 技術用語なしで操作を理解できる権利 |
| 4 | 操作が監査ログに正しく記録されていることを確認できる権利 |
| 5 | 安全な環境で操作を練習できる権利（サンドボックス） |
| 6 | システム状態の変化をリアルタイムに把握できる権利 |
| 7 | 操作の影響範囲を事前に確認できる権利（Blast Radius Preview） |

### 1.6. Platform Thinking（プラットフォーム思考）

- **Rule 50.005**: 管理ツールを単なる画面集ではなく「運用プラットフォーム」として設計する
- 拡張性（Plugin）、セルフサービス、API-First を基盤方針とする
- 運用者を「プラットフォームの顧客」と位置づけ、フィードバックループを構築する

## Part II. Build vs Buy 戦略

### 2.1. 低コードプラットフォーム・ファースト

- **Rule 50.010**: 管理ツールの開発は、まず低コード/ノーコードプラットフォーム（Retool, ToolJet, Appsmith等）の採用を検討する
- スクラッチ開発は例外条件を満たす場合のみ許可

### 2.2. スクラッチ開発の例外条件

以下の**いずれか**に該当する場合のみスクラッチ開発を許可:
- セキュリティ要件が低コードプラットフォームで充足不可能
- パフォーマンス要件（100ms以下のレスポンスタイム等）が低コードプラットフォームで達成不可能
- 既存システムとの深い統合が必要で、APIだけでは不十分
- 規制要件（データ所在地等）により外部プラットフォーム利用が禁止

### 2.3. プラットフォーム選定基準

| 評価軸 | 重み | 基準 |
|--------|------|------|
| セキュリティ・コンプライアンス | 30% | SOC2 Type II、RBAC/ABAC対応、監査ログ |
| カスタマイズ性 | 20% | カスタムコンポーネント、Plugin対応 |
| データソース接続 | 20% | REST/GraphQL/gRPC/DB直接接続対応 |
| コスト効率 | 15% | TCO（開発工数+ライセンス+運用）で評価 |
| AI統合 | 15% | AI Copilot、MCP Server対応、自然言語クエリ |

### 2.4. ベンダーロックイン対策

- **Rule 50.011**: プラットフォーム固有のロジックは全体の30%以下に抑える
- データアクセス層はAPI経由に統一し、プラットフォーム直結のDB接続を制限する
- 移行計画をドキュメント化し、3ヶ月以内に代替プラットフォームへ移行可能な状態を維持する

### 2.5. AI-Native Admin プラットフォーム

- **Rule 50.012**: 2025年以降の新規ツール選定では、AI-Native機能（自然言語クエリ、自動フォーム生成、インテリジェントサジェスト）の有無を必須評価項目とする
- AIによるクエリ生成を使用する場合、生成されたクエリの**人間によるレビュー承認**を必須とする

### 2.6. TCO（Total Cost of Ownership）評価

```
TCO_annual = License_cost + (Dev_hours × Hourly_rate) + Infra_cost + Training_cost + Migration_risk_cost
```

- **Rule 50.013**: Build vs Buy の意思決定は最低3年間のTCOで比較する
- 隠れコスト（カスタマイズ工数、アップグレード対応、セキュリティパッチ適用）を見積もりに含める

### 2.7. Agentic Admin Platforms（2025-2026動向）

- AI Agentが管理操作を自律的に実行するプラットフォームが台頭
- Agent実行も監査ログ対象とし、人間の操作と同等に追跡する
- **Rule 50.014**: Agentic Platformを採用する場合、Agent権限はRBACで管理し、破壊的操作には人間承認（HITL）を必須とする

### 2.8. プラットフォーム移行戦略

- 既存プラットフォームからの移行はStrangler Figパターンで段階的に実行
- データ移行は並行稼働期間（最低2週間）を設け、差分検証を自動化する
- 移行完了後、旧プラットフォームのアクセスを30日間読み取り専用で維持する

## Part III. 管理ツールアーキテクチャ

### 3.1. Admin Shell パターン

- **Rule 50.020**: 管理ツールのUIフレームワークは「Shell + Module」パターンで構築する
- Shell: 認証、ナビゲーション、通知、権限制御を一元管理
- Module: 各機能領域（ユーザー管理、コンテンツ管理等）を独立したモジュールとして実装

```typescript
// Admin Shell構成例
interface AdminShell {
  auth: AuthModule;          // 認証・認可
  navigation: NavModule;     // ナビ・サイドバー
  notification: NotifModule; // グローバル通知
  featureModules: Map<string, FeatureModule>; // 機能モジュール群
}
```

### 3.2. Micro-Frontend Admin

- **Rule 50.021**: 大規模管理ツールではMicro-Frontendアーキテクチャを採用する
- Module Federationによる動的ロードで各チームの独立デプロイを実現
- 共有依存関係（React, デザインシステム）はShellから提供

### 3.3. Plugin アーキテクチャ

- **Rule 50.022**: 管理ツールはPlugin拡張ポイントを持ち、カスタム機能の追加を容易にする
- Plugin Registry、Plugin Lifecycle（install/enable/disable/uninstall）を標準実装する
- Plugin間の依存関係を宣言的に管理し、循環依存を禁止する

### 3.4. Admin API Gateway

- **Rule 50.023**: 管理用APIは専用のAPI Gatewayを経由し、エンドユーザー向けAPIと分離する
- Rate Limiting、認証強化、監査ログ出力をGateway層で統一的に実施
- 管理APIのOpenAPI仕様書を自動生成し、常に最新状態を維持する

```yaml
# Admin API Gateway設定例
admin-gateway:
  rate_limit:
    default: 100/min
    bulk_operations: 10/min
    export: 5/min
  auth:
    required: true
    mfa_required_for:
      - DELETE
      - PATCH (bulk)
  audit:
    log_request_body: true
    log_response_status: true
    redact_fields: ["password", "ssn", "credit_card"]
```

### 3.5. Server-Driven UI（SDUI）

- **Rule 50.024**: 動的なフォーム・リスト・ダッシュボードにはServer-Driven UIパターンを検討する
- サーバー側でUI構造をJSON Schemaとして定義し、クライアントがレンダリング
- メリット: デプロイなしでUI変更可能、A/Bテスト容易、マルチプラットフォーム対応

## Part IV. 管理ダッシュボード設計

### 4.1. KPIモニタリング

- **Rule 50.030**: ダッシュボードのKPIは「ビジネスメトリクス」と「オペレーショナルメトリクス」を分離して表示する
- 各KPIにはトレンド（前日比・前週比）と閾値アラートを設定する
- ダッシュボードのデータ鮮度を明示する（例: 「最終更新: 5分前」）

### 4.2. リアルタイムストリーミングダッシュボード

- **Rule 50.031**: クリティカルなメトリクス（エラー率、キュー深度、アクティブユーザー数）はWebSocket/SSEによるリアルタイム更新を提供する
- ポーリング間隔は最大30秒。リアルタイム表示が不可能な場合、その旨をUI上に明示する

### 4.3. コラボレーティブダッシュボード

- **Rule 50.032**: 複数の運用者が同時にダッシュボードを閲覧・操作する場合のプレゼンス表示を提供する
- 同一リソースへの同時編集はConflict Resolution（Last Write Wins or Merge）を実装する

### 4.4. ダッシュボードパーソナライゼーション

- **Rule 50.033**: 運用者がウィジェットの配置・表示切替をカスタマイズ可能にする
- カスタマイズ設定はユーザー単位で永続化し、デバイス間で同期する
- デフォルトレイアウトはロール別に最適化して提供する

### 4.5. テクスタック透明性（Tech Stack Sync）

- **Rule 50.034**: ダッシュボードに表示されるデータのソース（DB, API, キャッシュ）を開発者モードで確認可能にする

## Part V. UI/UX 基準・デザインシステム

### 5.1. モーダル設計

- **Rule 50.040**: 削除等の破壊的操作はモーダルダイアログで確認を求める。対象名称のタイプ入力（例: `DELETE user-123`）を必須とする
- モーダル内でのショートカットキー無効化を徹底する（誤操作防止）

### 5.2. アクションフィードバック

- **Rule 50.041**: 全ての操作にリアルタイムフィードバック（Toast通知、進捗バー、ステータス変更）を提供する
- 非同期操作の場合、ジョブIDと進捗状況をポーリングまたはWebSocketで通知する

### 5.3. Command Palette（⌘K パターン）

- **Rule 50.042**: 管理ツールにはCommand Palette（⌘K / Ctrl+K）を実装し、以下を提供する:
  - ページ遷移（例: `Go to User #12345`）
  - アクション実行（例: `Disable user`）
  - 検索（例: `Search orders by email`）
  - 最近のアクション履歴

```typescript
// Command Palette設計例
interface CommandPaletteItem {
  id: string;
  label: string;
  category: 'navigation' | 'action' | 'search' | 'recent';
  shortcut?: string;
  requiresPermission?: string;
  dangerLevel?: 'safe' | 'warning' | 'destructive';
}
```

### 5.4. Natural Language Input（自然言語入力）

- **Rule 50.043**: AI搭載管理ツールでは自然言語によるクエリ・操作入力を提供する
- 自然言語入力から生成されたクエリ/操作は、実行前に**必ず人間がプレビュー・承認**する

### 5.5. コードエディタ義務（The Code Input Standard）

- **Rule 50.044**: JSON / SQL / 正規表現の入力フィールドにはシンタックスハイライト付きエディタ（Monaco Editor等）を組み込む
- バリデーションエラーをインラインで即座に表示する

### 5.6. レイアウト保護（CSS Containment Protocol）

- **Rule 50.045**: ユーザー入力コンテンツ（UGC）をプレビュー表示する際は`contain: layout style`で親要素のレイアウトへの影響を防止する

### 5.7. 情報鮮度表示（Audit Visibility Protocol）

- **Rule 50.046**: データ表示画面には「最終更新日時」「更新者」を常に表示する
- 5分以上古いデータにはStale Indicatorを表示し、手動リフレッシュを促す

### 5.8. Zero Jargon（専門用語の排除）

- **Rule 50.047**: 管理画面のラベル・メッセージに技術用語を使用しない。DB列名やAPI名をそのまま表示することは禁止
- 例: ❌ `is_active` → ✅ `アカウント状態: 有効`

### 5.9. Admin デザインシステム

- **Rule 50.048**: 管理ツール専用のデザインシステム（コンポーネントライブラリ + デザイントークン）を構築・維持する
- Figmaとコードの同期を自動化（Design Token連携）
- データテーブル、フォーム、フィルター、ステータスバッジの標準コンポーネントを提供する
- **デザイントークンの階層**: Global → Semantic → Component-specific

### 5.10. ダークモード対応

- **Rule 50.049**: 管理画面はダークモードを標準サポートする（長時間作業の目の疲労軽減）
- システム設定に自動追従 + 手動切替を提供
- ダークモードでのコントラスト比はWCAG AA基準を満たす

## Part VI. アクセシビリティ（WCAG 2.2 / EAA 2025）

### 6.1. 管理画面a11y基本方針

- **Rule 50.050**: 管理画面はWCAG 2.2 AA準拠を必須とする
- EAA（European Accessibility Act）2025年6月施行対象の場合、AA+追加要件を充足する
- a11y対応は「後付け」ではなく設計フェーズから組み込む

### 6.2. キーボードナビゲーション

- **Rule 50.051**: 全操作がキーボードのみで完了可能であること
- Tab順序は論理的で予測可能であること
- Focus Trapをモーダル・ドロワーに実装（フォーカスがモーダル外に移動しない）
- Skip Linkを提供（「メインコンテンツへ移動」）

### 6.3. スクリーンリーダー対応

- **Rule 50.052**: ARIA属性を適切に使用し、スクリーンリーダーで操作可能にする
- ダイナミックコンテンツ更新にはARIA Live Regionを使用する
- データテーブルには`role="grid"`と適切な`aria-label`を設定する

```html
<!-- ARIA Live Region例: 操作結果通知 -->
<div aria-live="polite" aria-atomic="true" class="sr-only">
  ユーザー「田中太郎」のアカウントを無効化しました
</div>
```

### 6.4. カラーコントラスト

- **Rule 50.053**: テキストと背景のコントラスト比は4.5:1以上（AA基準）を厳守
- ステータス表示は色だけでなく、アイコン+テキストを併用する（色覚多様性対応）

### 6.5. EAA 2025対応

- **Rule 50.054**: EU市場向けプロダクトの管理画面もEAA対象となる場合がある。以下を確認:
  - ウェブアクセシビリティ指令対応（公共セクター）
  - EAA対応（民間セクター、2025年6月28日施行）
  - 定期的なアクセシビリティ監査の実施と報告

### 6.6. 認知アクセシビリティ

- **Rule 50.055**: 認知障害を持つユーザーへの配慮を含める:
  - 明確で一貫したナビゲーション構造
  - 段階的な情報開示（Progressive Disclosure）
  - エラーメッセージに具体的な修正手順を含める
  - タイムアウトの延長オプション

## Part VII. 認証・認可アーキテクチャ

### 7.1. RBAC（ロールベースアクセス制御）

- **Rule 50.060**: 管理画面のアクセス制御はRBACを基本とし、最低限以下のロールを定義する:

| ロール | 権限概要 |
|--------|----------|
| Super Admin | 全操作権限。システム設定変更可能 |
| Admin | ユーザー管理、コンテンツ管理。システム設定は読み取り専用 |
| Operator | 日常運用操作（コンテンツ承認等）。ユーザー削除不可 |
| Viewer | 読み取り専用。操作実行不可 |
| Auditor | 監査ログ閲覧専用。操作実行不可 |

- **Rule 50.061**: ロール割り当ての変更は監査ログに記録し、上位ロール保持者のみが実行可能とする
- ロールの自己昇格は技術的に不可能な設計にする

### 7.2. ABAC（属性ベースアクセス制御）

- **Rule 50.062**: RBACで不十分な場合（テナント横断、時間帯制限等）はABACを追加レイヤーとして実装する
- 判断属性: ユーザー属性、リソース属性、環境属性（時刻・IP・デバイス）、操作属性

```typescript
// ABACポリシー例
const policy: ABACPolicy = {
  effect: 'allow',
  subject: { role: 'operator', department: 'support' },
  resource: { type: 'user_profile', sensitivity: 'PII' },
  action: 'read',
  condition: {
    timeRange: { start: '09:00', end: '18:00', timezone: 'Asia/Tokyo' },
    ipRange: ['10.0.0.0/8'],
    deviceTrust: 'verified'
  }
};
```

### 7.3. MFA（多要素認証）

- **Rule 50.063**: 管理画面へのアクセスにはMFA（多要素認証）を**必須**とする
- **Phishing-Resistant MFA**（FIDO2/WebAuthn/Passkeys）を優先的に導入する
- SMS認証はフォールバック専用とし、新規設定を非推奨化する

### 7.4. セッション管理

- **Rule 50.064**: 管理画面のセッションタイムアウトは最大8時間（アイドル30分）
- セッション延長時は再認証（少なくとも1要素）を求める
- 並行セッション数は同一ユーザー3セッション以下に制限する

### 7.5. JIT（Just-In-Time）アクセス

- **Rule 50.065**: 高権限操作（Super Admin相当）はJITアクセスで運用する:
  1. 運用者が権限昇格をリクエスト
  2. 承認者がリクエストを審査・承認
  3. 時間制限付き（最大4時間）で権限付与
  4. 有効期限切れで自動的に権限失効
  5. 全操作を監査ログに記録

### 7.6. CAEP（Continuous Access Evaluation Protocol）

- **Rule 50.066**: セッション中もリスクシグナル（IP変更、デバイス変更、異常行動）を継続的に評価し、リスク検知時はセッション無効化またはステップアップ認証を実行する
- IETF CAEP標準に準拠した実装を推奨

### 7.7. Passkeys完全移行ロードマップ

| Phase | 期間 | 内容 |
|-------|------|------|
| 1 | 〜2025 Q4 | Passkey登録UIの提供開始。既存MFAと並行運用 |
| 2 | 2026 Q1-Q2 | 新規管理者はPasskey必須。既存管理者への移行促進 |
| 3 | 2026 Q3 | SMS認証の新規登録を停止 |
| 4 | 2027 Q1 | Passkey完全移行。レガシーMFA廃止 |

### 7.8. SPIFFE/SPIRE 連携（サービス間認証）

- **Rule 50.067**: 管理ツールからバックエンドサービスへのアクセスはSPIFFE IDベースの相互認証を使用する
- サービスメッシュ環境ではmTLSを標準とし、APIキーによるサービス間認証を廃止する

## Part VIII. セキュリティ強化

### 8.1. Zero Trust アーキテクチャ

- **Rule 50.070**: 管理ツールはZero Trustモデルで設計する: 「Never Trust, Always Verify」
- ネットワーク境界に依存しない。VPN接続中でも全リクエストを検証する
- 各APIコールでIdentity + Device Trust + Context を検証する

### 8.2. ネットワーク制限

- **Rule 50.071**: 管理画面へのアクセスはIP制限 + VPN/ZTNAを併用する
- 管理用APIエンドポイントはパブリックインターネットに公開しない
- CDN経由の場合、Origin直接アクセスをブロックする

### 8.3. WAF・レート制限

- **Rule 50.072**: 管理画面用WAFルールを設定する:
  - ログイン試行: 5回/分（失敗時ロックアウト15分）
  - API呼び出し: 100回/分（通常操作）、10回/分（一括操作）
  - エクスポート: 5回/分

### 8.4. CSP（Content Security Policy）

- **Rule 50.073**: 管理画面のCSPは厳格に設定する:
  - `default-src 'self'`
  - `script-src 'self'`（インラインスクリプト禁止）
  - `style-src 'self' 'unsafe-inline'`（CSS-in-JSの場合のみnonce使用）
  - `frame-ancestors 'none'`（Clickjacking防止）

### 8.5. セキュリティテスト

- **Rule 50.074**: 管理画面には以下のセキュリティテストを実施する:
  - SAST（静的解析）: CI/CDパイプラインで毎コミット実行
  - DAST（動的解析）: 週次スケジュール実行
  - ペネトレーションテスト: 年次（外部ベンダー）
  - 権限バイパステスト: リリース前に必須実行

### 8.6. Device Trust（デバイス信頼性）

- **Rule 50.075**: 管理画面アクセスにはDevice Trustを要求する:
  - OSパッチ適用状態の確認
  - ディスク暗号化の確認
  - MDM/UEM管理下であることの確認
  - 未認証デバイスからのアクセス拒否

### 8.7. Supply Chain Security（管理ツール用）

- **Rule 50.076**: 管理ツールの依存関係はSBOM（Software Bill of Materials）で管理する
- 依存パッケージの脆弱性を週次でスキャンし、Critical/Highは72時間以内に対応する
- → 詳細は [602_oss_compliance.md](../security/200_oss_compliance.md) を参照

### 8.8. AI Red Teaming for Admin

- **Rule 50.077**: AI統合管理ツールには定期的なAI Red Teamingを実施する:
  - プロンプトインジェクション攻撃テスト
  - 権限エスカレーション攻撃テスト
  - データ漏洩テスト（PII抽出試行）

### 8.9. RASP（Runtime Application Self-Protection）

- **Rule 50.078**: 管理画面にはRASPを導入し、実行時の攻撃検知・防御を実施する:
  - SQLインジェクション検知
  - パストラバーサル検知
  - コマンドインジェクション検知
  - 検知時はリクエストをブロックし、セキュリティチームにアラートを送信

### 8.10. Admin-Specific Threat Model

- **Rule 50.079**: 管理ツール固有の脅威モデルを作成し、年次で更新する:
  - 内部不正（Insider Threat）
  - 認証情報の窃取（Credential Stuffing）
  - セッションハイジャック
  - 権限昇格（Privilege Escalation）
  - データ一括エクスポートによる情報漏洩

## Part IX. データプライバシー・GDPR管理画面対応

### 9.1. PII表示制御

- **Rule 50.080**: PII（個人識別情報）はデフォルトでマスキング表示する
- マスキング解除には追加認証（ステップアップ認証）+ 理由入力を必須とする
- マスキング解除操作は監査ログに記録し、解除回数を運用者別に追跡する

### 9.2. データマスキング

```typescript
// データマスキング戦略例
const maskingRules: Record<string, MaskingStrategy> = {
  email: (v) => v.replace(/(.{2}).*@/, '$1***@'),      // hi***@example.com
  phone: (v) => v.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2'), // 090****1234
  name: (v) => v[0] + '***',                            // 田***
  address: () => '***非表示***',
  creditCard: (v) => '**** **** **** ' + v.slice(-4),    // **** **** **** 1234
};
```

### 9.3. 忘れられる権利（Right to Erasure）対応

- **Rule 50.081**: GDPR第17条に基づくデータ削除リクエストを管理画面から実行可能にする
- 削除は論理削除 → 保持期間経過後の物理削除（30日ルール）
- 削除対象項目のチェックリストを提供し、漏れを防止する
- バックアップからの削除義務を運用手順に含める

### 9.4. 同意管理ダッシュボード

- **Rule 50.082**: ユーザーの同意状態を一覧表示するダッシュボードを提供する
- 同意バージョン管理と、同意撤回の即時反映を実装する

### 9.5. データ分類タグ

- **Rule 50.083**: 管理画面で扱うデータにはデータ分類タグを自動付与する:

| 分類 | 例 | 表示制御 |
|------|-----|----------|
| **Public** | 公開プロフィール | 制限なし |
| **Internal** | 注文履歴 | ロール制限 |
| **Confidential** | メールアドレス、電話番号 | マスキング + ステップアップ認証 |
| **Restricted** | クレジットカード番号、SSN | アクセス禁止（対象システムでのみ閲覧可能） |

### 9.6. データ分類の自動化

- **Rule 50.084**: MLを使用してデータフィールドを自動分類し、新規フィールド追加時のタグ付け漏れを防止する
- 自動分類結果は人間がレビュー・承認してから適用する

### 9.7. DPO（Data Protection Officer）ダッシュボード

- **Rule 50.085**: DPO/プライバシー責任者向けのダッシュボードを提供する:
  - データアクセス頻度レポート
  - PII閲覧ログの可視化
  - データ削除リクエストのステータス追跡
  - 同意取得率のモニタリング

## Part X. 監査ログ・コンプライアンス

### 10.1. 不変監査ログ（Immutable Audit Log）

- **Rule 50.090**: 管理画面の全操作は不変（Append-Only）監査ログに記録する
- 必須記録フィールド:

```typescript
interface AuditLogEntry {
  id: string;                    // UUID v7（時系列ソート可能）
  timestamp: string;             // ISO 8601（タイムゾーン付き）
  actor: {
    userId: string;
    role: string;
    ipAddress: string;
    userAgent: string;
    sessionId: string;
    deviceId?: string;
  };
  action: string;                // CRUD + カスタムアクション
  resource: {
    type: string;                // 'user' | 'order' | 'config'
    id: string;
    tenant?: string;
  };
  changes?: {
    before: Record<string, unknown>;
    after: Record<string, unknown>;
  };
  metadata: {
    requestId: string;           // 分散トレーシング用
    source: 'admin_ui' | 'api' | 'automation' | 'ai_agent';
    justification?: string;      // 操作理由（破壊的操作の場合必須）
    approvedBy?: string;         // 承認者（承認フローの場合）
  };
}
```

- **Rule 50.091**: 監査ログの削除・改変は技術的に不可能な設計にする（Write-Once Storage）

### 10.2. ログ保持・検索

- **Rule 50.092**: 監査ログの保持期間は最低7年間（SOC2/GDPR要件）
- 全文検索（Elasticsearch等）を提供し、任意の条件でフィルタリング可能にする
- ログのエクスポート（CSV/JSON）を提供する（エクスポート自体も監査ログに記録）

### 10.3. Tamper-Evident Logging（改竄検知ログ）

- **Rule 50.093**: Merkle Tree ハッシュチェーンによるログ改竄検知を実装する
- 一定間隔（例: 1時間ごと）でハッシュチェーンの整合性を自動検証する
- 改竄検知時はセキュリティチームにアラートを即時送信する

### 10.4. SOC2対応チェックリスト

- **Rule 50.094**: 管理ツールはSOC2 Trust Services Criteria に対応する:

| TSC | 対応項目 |
|-----|----------|
| CC6.1 | 論理アクセス制御（RBAC/ABAC） |
| CC6.2 | 認証メカニズム（MFA必須） |
| CC6.3 | 権限予防・検知（最小権限） |
| CC7.2 | システムコンポーネント監視（可観測性） |
| CC8.1 | 変更管理 |

### 10.5. コンプライアンスダッシュボード

- **Rule 50.095**: コンプライアンス状態をリアルタイムで可視化するダッシュボードを提供する
- 準拠率、違反件数、未対応項目を一覧表示する

### 10.6. SIEM連携

- **Rule 50.096**: 監査ログはSIEM（Security Information and Event Management）に連携する
- 標準フォーマット（CEF/OCSF）で出力し、SIEMでの相関分析を可能にする
- → セキュリティ詳細は [600_security_privacy.md](../security/000_security_privacy.md) を参照

### 10.7. WORM Storage（Write Once Read Many）

- **Rule 50.097**: 規制対応が必要な監査ログはWORMストレージに保管する
- S3 Object Lock / Azure Immutable Blob Storage 等を使用する
- 保持期間中の削除を技術的に不可能にする

### 10.8. Chain of Custody（証拠の連鎖）

- **Rule 50.098**: フォレンジック調査に備え、監査ログのChain of Custodyを確立する:
  - ログ収集 → 暗号化 → 署名 → WORM保管 → アクセスログ
  - 法的手続きで証拠として使用可能な状態を維持する

## Part XI. ユーザー管理

### 11.1. 検索・フィルタ

- **Rule 50.100**: ユーザー検索は以下の条件で高速（500ms以内）に実行可能にする:
  - メールアドレス（部分一致）
  - 表示名、ユーザーID
  - 登録日、最終ログイン日
  - ステータス（有効/無効/停止/削除済み）
  - カスタム属性

### 11.2. 代理ログイン（Impersonation / Masquerade）

- **Rule 50.101**: エンドユーザーの代理ログイン機能を提供する（サポート用途）
- 代理ログイン中はUIにバナーを常時表示し、代理ログイン中であることを明示する
- 代理ログインの開始・終了・全操作は監査ログに記録する
- 代理ログイン中の破壊的操作（データ削除等）は禁止する

```typescript
// 代理ログインの監査ログ例
const impersonationLog: AuditLogEntry = {
  action: 'impersonation.start',
  actor: { userId: 'admin-001', role: 'support' },
  resource: { type: 'user', id: 'user-12345' },
  metadata: {
    source: 'admin_ui',
    justification: 'チケット #SUPPORT-789 調査のため'
  }
};
```

### 11.3. アカウントライフサイクル管理

- **Rule 50.102**: ユーザーアカウントのライフサイクルを管理画面で完結させる:
  - 作成 → 有効化 → 停止 → 削除（論理） → 削除（物理）
  - 各ステート遷移にはバリデーションルールと承認フローを設定可能にする

### 11.4. SCIM 2.0 統合

- **Rule 50.103**: エンタープライズ顧客向けにSCIM 2.0（System for Cross-domain Identity Management）エンドポイントを提供する
- IdP（Okta, Azure AD等）からのプロビジョニング/デプロビジョニングを自動化する

### 11.5. Ghost Account 検知

- **Rule 50.104**: 以下の条件のアカウントをGhost Accountとして自動検知し、定期レポートを生成する:
  - 90日以上ログインなし
  - メールバウンス発生
  - 作成後一度もログインしていない
  - 退職者アカウント（HR連携により検知）

## Part XII. コンテンツ管理戦略

### 12.1. No Raw HTML原則

- **Rule 50.110**: 管理画面でのコンテンツ入力にRaw HTMLを許可しない
- リッチテキストエディタ（TipTap, Lexical等）を使用し、XSS安全な出力を保証する
- 許可するHTML要素のホワイトリストを設定し、サーバーサイドでサニタイズする

### 12.2. バージョニング・変更履歴

- **Rule 50.111**: コンテンツの全変更をバージョン管理し、任意のバージョンにロールバック可能にする
- 変更のdiff表示を提供する
- 公開バージョンと下書き（Draft）バージョンを分離する

### 12.3. AI-Assisted Content Moderation

- **Rule 50.112**: UGCの自動モデレーションにAIを活用する:
  - テキスト: 有害コンテンツ検知（Hate Speech, Spam, PII露出）
  - 画像: NSFW検知、著作権侵害検知
  - AI判定結果には信頼度スコアを付与し、低信頼度の場合は人間レビューにエスカレーションする

### 12.4. コンテンツ承認ワークフロー

- **Rule 50.113**: 公開コンテンツは承認ワークフローを経由する:
  - Draft → Review → Approve → Publish
  - 承認者はコンテンツ作成者と別のユーザーでなければならない（四眼原則）

## Part XIII. データインポート・エクスポート

### 13.1. 一括インポート

- **Rule 50.120**: CSVインポート時に以下を実施する:
  - スキーマバリデーション（列名・型チェック）
  - 重複検出（既存データとの照合）
  - ドライラン（Dry Run）モード（実データを変更せずに結果をプレビュー）
  - インポート結果レポート（成功/失敗/スキップ件数）

### 13.2. ストリーミングインポート

- **Rule 50.121**: 100万行以上のデータインポートはストリーミング方式で処理する
- 進捗バー + 予想残り時間を表示する
- 中断・再開機能を提供する（冪等性の確保）

### 13.3. エクスポート制御

- **Rule 50.122**: データエクスポートは以下の制御を実施する:
  - エクスポート件数上限の設定
  - PII含有データのマスキングオプション
  - エクスポート操作の監査ログ記録
  - エクスポートファイルの有効期限設定（7日後に自動削除）

### 13.4. Data Quality Gate

- **Rule 50.123**: インポートデータは品質ゲートを通過しなければ最終コミットしない:
  - Null値許容率チェック
  - フォーマット整合性（日付、電話番号等）
  - 参照整合性（外部キーの存在確認）
  - ビジネスルールバリデーション

## Part XIV. マスターデータ管理

### 14.1. Golden Record（ゴールデンレコード）

- **Rule 50.130**: マスターデータは「Golden Record」（単一の正規化されたレコード）を維持する
- 複数データソースからの情報をマージする際のルール（優先順位）を明確に定義する

### 14.2. MDM Hub

- **Rule 50.131**: マスターデータの変更は全てMDM Hubを経由する
- 直接DBアクセスによるマスターデータ変更は禁止する
- 変更履歴を完全に追跡可能にする

### 14.3. データ品質スコアリング

- **Rule 50.132**: マスターデータの品質をスコアリングし、低品質レコードを自動検出する:
  - 完全性（必須フィールドの充填率）
  - 正確性（フォーマット準拠率）
  - 一貫性（重複レコード数）
  - 鮮度（最終更新日からの経過日数）

## Part XV. 構成管理・フィーチャーフラグ

### 15.1. 構成変更プロトコル

- **Rule 50.140**: 本番環境の構成変更は以下のプロトコルに従う:
  - 変更申請 → 承認 → ステージング検証 → 本番適用 → 監視
  - 即座ロールバック手順を事前に準備する

### 15.2. フィーチャーフラグ管理

- **Rule 50.141**: フィーチャーフラグの管理UIを提供し、以下を可能にする:
  - フラグの有効/無効切替
  - ターゲティング設定（ユーザーセグメント、テナント、環境）
  - A/Bテスト割り当て管理
  - フラグのライフサイクル管理（作成日、有効期限、オーナー）

### 15.3. OpenFeature SDK 統合

- **Rule 50.142**: フィーチャーフラグのSDKはOpenFeature（ベンダー非依存標準）に準拠する
- バックエンド: LaunchDarkly / Unleash / Flagsmith 等から選択
- フラグ評価のレイテンシは10ms以下を目標とする

### 15.4. ドリフト検知

- **Rule 50.143**: 構成値のドリフト（意図しない変更）を自動検知する:
  - Git管理された構成値との差分を定期チェック
  - ドリフト検知時はアラートを送信し、自動修復オプションを提供
  - 環境間（Staging/Production）の構成差分を可視化する

### 15.5. Feature Flag SLO

- **Rule 50.144**: フィーチャーフラグの運用にSLOを設定する:
  - フラグ評価レイテンシ: p99 < 10ms
  - フラグ更新伝播時間: < 30秒
  - フラグシステム可用性: 99.99%
  - 古いフラグ（90日以上更新なし）の定期クリーンアップ義務

## Part XVI. 運用ワークフロー・承認フロー

### 16.1. 承認フロー設計

- **Rule 50.150**: 以下の操作には承認フローを必須とする:
  - ユーザーの一括操作（一括停止、一括削除）
  - 本番環境の構成変更
  - 金銭に関わる操作（返金、クレジット付与）
  - 権限変更

### 16.2. Multi-Party Approval（複数承認者）

- **Rule 50.151**: 高リスク操作にはMulti-Party Approvalを実装する:
  - リスクレベルに応じた必要承認者数（Low: 1名、Medium: 2名、High: 3名）
  - Quorum-Based Decision: 全承認者の過半数の承認で実行可能

### 16.3. Approval SLA

- **Rule 50.152**: 承認リクエストにはSLAを設定する:
  - 通常: 4時間以内に承認/却下
  - 緊急: 30分以内
  - SLA超過時は自動エスカレーション

### 16.4. 四眼原則（Dual Control）

- **Rule 50.153**: 破壊的操作は「実行者 ≠ 承認者」を技術的に強制する
- 自身が承認した操作を自身で実行することは不可能な設計にする

## Part XVII. Runbook・運用自動化

### 17.1. Runbook as Code

- **Rule 50.160**: Runbookはバージョン管理されたコード（Markdown + 実行可能スクリプト）として管理する
- 各ステップは原子的（Atomic）で検証可能（Verifiable）であること
- 安全チェック（Pre-condition / Post-condition）を各ステップに含める

### 17.2. Event-Driven Runbook

- **Rule 50.161**: アラートイベントに対応するRunbookを自動的に表示・実行提案する:
  - アラート → 対応Runbookの自動紐付け
  - 自動実行可能なステップは自動実行（人間確認後）
  - 手動ステップは担当者にアサインし、完了報告を追跡

### 17.3. ChatOps 統合

- **Rule 50.162**: Runbookの実行はChatOps（Slack Bot等）からもトリガー可能にする
- チャットでの操作も全て監査ログに記録する
- 破壊的操作のChatOps実行には追加確認ステップを挿入する

### 17.4. Toil 計測・削減

- **Rule 50.163**: 反復的な手作業（Toil）を定量的に計測する:
  - 作業カテゴリ別の時間追跡
  - Toil率の目標: チーム作業時間の50%以下（SRE標準）
  - Toil上位5項目の自動化ロードマップを四半期ごとに策定

## Part XVIII. ワークフロー自動化エンジン

### 18.1. 自動化プラットフォーム選定

- **Rule 50.170**: ワークフロー自動化には以下の基準で技術を選定する:

| 技術 | 適用場面 | 特徴 |
|------|----------|------|
| **Temporal** | ミッションクリティカル、長時間実行、レガシー統合 | 状態保持型、リトライ自動化、企業レベル信頼性 |
| **Inngest** | イベント駆動、サーバーレス、高速開発 | HTTP呼び出し型、インフラ不要、開発者体験優先 |
| **Step Functions** | AWS環境、Simple State Machine | AWSネイティブ統合 |

### 18.2. No-Code Workflow Builder

- **Rule 50.171**: 非エンジニア運用者向けにNo-Code Workflow Builderを提供する:

  - 条件分岐、ループ、並列処理のサポート
  - テスト実行（Dry Run）機能
  - 本番適用前のレビュー承認フロー

### 18.3. Saga Pattern

- **Rule 50.172**: 複数のサービスにまたがるワークフローにはSagaパターンを適用する:
  - 各ステップに対応する補償トランザクション（Compensating Transaction）を定義
  - 部分的な失敗時は自動的に補償トランザクションを逆順実行
  - Sagaの状態を永続化し、システム再起動後も再開可能にする

## Part XIX. アラート・通知統合

### 19.1. アラート設計

  - 動的閾値: ベースラインからの逸脱で判定（固定閾値ではなく）
  - アラートノイズ率の計測: 目標 < 5%

|--------|----------|-------------|
| Critical | PagerDuty + 電話 | 15分以内 |
| High | Slack + メール | 1時間以内 |
| Medium | Slack | 4時間以内 |
| Low | メール + アプリ内 | 次営業日 |

### 19.4. Notification SLO

- **Rule 50.182**: 通知配信にSLOを設定する:
  - 配信レイテンシ: Critical通知は30秒以内
  - 配信成功率: 99.9%以上
  - False Positive率: 5%以下

## Part XXII. AI統合・インテリジェント管理

### 22.1. Agentic AI 成熟度モデル（L1-L5）

| Level | 名称 | 説明 | 人間関与 |
|-------|------|------|----------|
| L1 | 提案型 | AIが操作を提案。人間が全て承認・実行 | 100% |
| L2 | 半自動型 | 低リスク操作はAIが自動実行。高リスクは人間承認 | 70% |
| L3 | 協調型 | AIがワークフローを自律実行。例外時に人間介入 | 40% |
| L4 | 自律型 | AIが大部分を自律運用。人間は監視・ポリシー設定 | 10% |
| L5 | 完全自律型 | AIが全運用を自律実行。人間はガバナンスのみ | <5% |

- **Rule 50.210**: 現在の推奨レベルはL2-L3。L4以上は高度なガードレール実装後にのみ許可する

### 22.2. MCP Server 設計（管理ツール用）

- **Rule 50.211**: 管理ツールの機能をMCP（Model Context Protocol）Serverとして公開する:
  - 一つのMCP Serverは単一のビジネスドメインに対応
  - 各ToolはJSON Schemaで入出力を厳密定義
  - 破壊的操作のToolには`confirmation_required: true`を設定

```typescript
// MCP Server Tool定義例
const userManagementTools = [
  {
    name: 'search_users',
    description: 'Search users by email, name, or ID',
    inputSchema: {
      type: 'object',
      properties: {
        query: { type: 'string', description: 'Search query' },
        limit: { type: 'number', default: 20, maximum: 100 }
      },
      required: ['query']
    },
    confirmation_required: false
  },
  {
    name: 'disable_user',
    description: 'Disable a user account',
    inputSchema: {
      type: 'object',
      properties: {
        userId: { type: 'string' },
        reason: { type: 'string' }
      },
      required: ['userId', 'reason']
    },
    confirmation_required: true  // 破壊的操作: 人間確認必須
  }
];
```

### 22.3. AI Copilot UX Pattern

- **Rule 50.212**: 管理画面のAI Copilotは以下のUXパターンに従う:
  - **Inline Suggestion**: フォーム入力中の値サジェスト
  - **Context Panel**: 選択中のリソースに関するインサイト表示
  - **Action Proposal**: 現在の状況に基づく次のアクション提案
  - **Natural Language Query**: 自然言語での検索・集計

### 22.4. AI ガードレール

- **Rule 50.213**: AI統合管理ツールのガードレール:
  - AIが生成したクエリ/操作は必ず人間がプレビューしてから実行
  - AIの操作範囲を明示的に定義（ホワイトリスト方式）
  - AIの操作は全て監査ログに`source: 'ai_agent'`として記録
  - AI操作の異常（通常と異なるパターン）を検知してアラート
  - → AI安全性詳細は [400_ai_engineering.md](../ai/000_ai_engineering.md) を参照

- **Rule 50.273**: 管理ツールにOpenTelemetryを導入し、Traces/Metrics/Logsを統合的に収集する:

```typescript
// OpenTelemetry管理画面計装例
import { trace } from '@opentelemetry/api';

const tracer = trace.getTracer('admin-panel');

async function handleUserAction(action: string, userId: string) {
  const span = tracer.startSpan(`admin.${action}`, {
    attributes: {
      'admin.action': action,
      'admin.target_user': userId,
      'admin.operator': getCurrentOperator().id,
    },
  });
  try {
    const result = await executeAction(action, userId);
    span.setStatus({ code: SpanStatusCode.OK });
    return result;
  } catch (error) {
    span.setStatus({ code: SpanStatusCode.ERROR, message: error.message });
    throw error;
  } finally {
    span.end();
  }
}
```

## Part XXXIII. 管理ツールテスト戦略

### 33.1. E2Eテスト

- **Rule 50.320**: 管理画面のクリティカルパスにE2Eテスト（Playwright推奨）を実装する:
  - ログイン → ユーザー検索 → 詳細表示 → 操作実行 → 監査ログ確認

### 33.2. 権限テスト

- **Rule 50.321**: 全ロール × 全操作の権限マトリクスを自動テストで検証する:
  - 許可されている操作が実行可能であること
  - 禁止されている操作が確実にブロックされること
  - 権限バイパスの回帰テスト

```typescript
// 権限マトリクステスト例
describe('Permission Matrix', () => {
  const permissionMatrix = [
    { role: 'viewer', action: 'user.delete', expected: 'denied' },
    { role: 'operator', action: 'user.delete', expected: 'denied' },
    { role: 'admin', action: 'user.delete', expected: 'allowed' },
    { role: 'super_admin', action: 'user.delete', expected: 'allowed' },
  ];

  permissionMatrix.forEach(({ role, action, expected }) => {
    test(`${role} should be ${expected} for ${action}`, async () => {
      const result = await checkPermission(role, action);
      expect(result).toBe(expected);
    });
  });
});
```

### 33.3. 監査ログテスト

- **Rule 50.322**: 全ての操作が監査ログに正しく記録されることを自動テストで検証する

### 33.4. アクセシビリティテスト

### 40.1. Break Glass プロトコル

- **Rule 50.400**: 緊急時に通常の認証・承認フローをバイパスするBreak Glass手順を事前定義する:
  1. 緊急事態の宣言（定義されたトリガー条件に基づく）
  2. Break Glassアカウントの有効化（2名以上の承認 or 物理的セキュリティキー）
  3. 時間制限付きアクセス（最大2時間）
  4. 全操作の強化監査ログ記録
  5. アクセス終了後の即時レビュー・報告

### 40.2. Break Glass アカウント管理

- **Rule 50.401**: Break Glassアカウントは以下の要件を満たす:
  - 通常運用アカウントとは完全に分離
  - 認証情報は物理的に安全な場所に保管（金庫等）
  - 使用されていないことを定期的（月次）に監査
  - アクセス時はセキュリティチームに即座にアラート送信

### 40.3. JIT昇格との連携

- **Rule 50.402**: 通常のJIT昇格プロセスが機能しない場合のみBreak Glassを使用する
- Break Glass使用後は必ずポストモーテムを実施し、JITプロセスの改善点を特定する

### 40.4. 事後監査

- **Rule 50.403**: Break Glassアクセスの事後監査は24時間以内に完了する:
  - 全操作ログのレビュー
  - 操作の正当性の確認
  - 不正使用の検出
  - 改善提案の策定

## Part XLI. 管理ツールサプライチェーンセキュリティ

### 41.1. SBOM管理

- **Rule 50.410**: 管理ツールの全依存関係をSBOM（CycloneDX 1.6 / SPDX 3.0）で管理する
- SBOMは各リリース時に自動生成し、アーカイブする

### 41.2. 脆弱性スキャン

- **Rule 50.411**: 依存関係の脆弱性スキャンをCI/CDに統合する:
  - Critical: ビルド即時ブロック
  - High: 72時間以内に修正
  - Medium: 次回スプリントで対応

### 41.3. Admin用CI/CDセキュリティ

- **Rule 50.412**: 管理ツールのCI/CDパイプラインは以下のセキュリティゲートを含む:
  - SAST（静的解析）
  - SCA（依存関係チェック）
  - シークレットスキャン
  - コンテナイメージスキャン（使用する場合）
  - SBOM生成
  - → 詳細は [602_oss_compliance.md](../security/200_oss_compliance.md) を参照

## Part XLII. Sustainability & Green IT

### 42.1. カーボンフットプリント計測

- **Rule 50.420**: 管理ツールのインフラが消費するカーボンフットプリントを計測する:
  - Cloud Carbon Footprint等のツールを活用
  - 月次レポートでCO2排出量を可視化

### 42.2. エネルギー効率最適化

- **Rule 50.421**: 管理ツールのエネルギー効率を最適化する:
  - 不要なバックグラウンド処理の削減
  - アイドル時のリソーススケールダウン
  - 効率的なクエリ設計（不要なDB呼び出しの削減）
  - ダークモードによる消費電力削減（OLED画面）

### 42.3. Green SRE連携

- **Rule 50.422**: 管理ツールの運用もGreen SRE原則に従う:
  - カーボンアウェアなスケジューリング（低炭素時間帯での処理実行）
  - リージョン選択時のカーボン係数の考慮
  - → 詳細は [502_site_reliability.md](../operations/400_site_reliability.md) を参照

## Part XLIII. データ主権・越境管理

### 43.1. Sovereign Cloud対応

- **Rule 50.430**: データ主権要件がある市場向けに、管理ツールのデータ所在地を制御する:
  - EU: EU域内のデータ保管義務
  - 中国: 中国国内のデータ保管義務
  - ロシア: ロシア国内のデータ保管義務
  - テナント別のデータ所在地設定を管理画面から管理可能にする

### 43.2. 越境データ転送

- **Rule 50.431**: 管理画面からの越境データ転送を制御する:
  - データエクスポート時に越境転送チェックを実施
  - GDPR SCC（Standard Contractual Clauses）準拠の確認
  - 越境転送のログ記録
  - → 詳細は [601_data_governance.md](../security/100_data_governance.md) を参照

### 43.3. データローカライゼーション

- **Rule 50.432**: データローカライゼーション要件に応じた管理画面の構成:
  - リージョン別データアクセス制限
  - 管理者のアクセス可能なリージョンの制御
  - 越境アクセスの監査ログ

## Part XLIV. 管理者教育・継続的トレーニング

### 44.1. セキュリティ意識向上

- **Rule 50.440**: 管理者向けセキュリティトレーニングを年次で必須実施する:
  - フィッシング攻撃の識別
  - ソーシャルエンジニアリング対策
  - セキュリティインシデント報告手順
  - 新たな脅威動向のアップデート

### 44.2. ロールベース研修

- **Rule 50.441**: ロール別のトレーニングプログラムを提供する:
  - Operator: 日常操作手順、エスカレーション手順
  - Admin: ユーザー管理、構成変更手順、インシデント対応
  - Super Admin: セキュリティ設定、DR手順、コンプライアンス

### 44.3. 認定プログラム

- **Rule 50.442**: 管理ツールの認定プログラムを設け、操作権限の前提条件とする:
  - 基本操作認定（全運用者必須）
  - 高度操作認定（Admin/Super Admin向け）
  - セキュリティ認定（セキュリティ関連操作に必須）
  - 認定は年次で更新を義務化する

## Part XLV. 内部ツール廃止・移行管理

### 45.1. EOL（End of Life）計画

- **Rule 50.450**: 内部ツールの廃止は以下のプロセスに従う:
  1. 廃止決定の通知（6ヶ月前）
  2. 代替ツール/プロセスの準備と周知
  3. 移行期間（最低3ヶ月の並行稼働）
  4. 移行完了の確認
  5. 旧ツールの停止（データは保持期間ポリシーに従い保管）

### 45.2. データ移行

- **Rule 50.451**: ツール廃止時のデータ移行:
  - 移行計画の策定とレビュー
  - テスト環境での移行リハーサル
  - diff検証による完全性確認
  - ロールバック計画の準備

### 45.3. 並行稼働期間

- **Rule 50.452**: 新旧ツールの並行稼働中は以下を実施する:
  - データの双方向同期（可能な場合）
  - 両ツールでの操作結果の一致確認
  - ユーザーフィードバックの収集
  - 問題発生時の旧ツールへのロールバック手順の維持

## Part XLVI. 管理画面の人間工学

### 46.1. Cognitive Load最適化

- **Rule 50.460**: 管理画面の情報密度を運用者の認知負荷に合わせて最適化する:
  - Progressive Disclosure: 詳細情報は段階的に表示
  - Chunking: 関連情報をグルーピング
  - Visual Hierarchy: 重要度に応じた視覚的階層

### 46.2. 情報密度設計

- **Rule 50.461**: ダッシュボードの情報密度設計ガイドライン:
  - 1画面に表示するKPIは最大7±2項目（ミラーの法則）
  - データテーブルの列数は10列以下（スクロールなし表示の場合）
  - フィルターは最初5個表示、追加は「詳細フィルター」で展開

### 46.3. Fatigue Prevention（疲労防止）

- **Rule 50.462**: 長時間の管理作業による疲労を防止する:
  - ダークモードの提供
  - フォントサイズの調整機能
  - 定期的な休憩リマインダー（オプション）
  - 画面の明暗バランスの最適化

## Part XLVII. 管理ツール固有のインシデント対応

### 47.1. Admin Panel障害対応

- **Rule 50.470**: 管理画面自体の障害に対する対応手順:
  1. 障害検知（モニタリング/ユーザー報告）
  2. 影響範囲の評価
  3. 代替手段の通知（CLI/API直接アクセス）
  4. 修復対応
  5. ポストモーテム

### 47.2. 権限漏洩対応

- **Rule 50.471**: 権限の不正昇格や漏洩が検知された場合:
  1. 影響アカウントの即時無効化
  2. 影響範囲の監査ログ分析
  3. 不正操作のロールバック（可能な場合）
  4. 原因分析と再発防止策の実施

### 47.3. データ漏洩即時対応

- **Rule 50.472**: 管理画面経由のデータ漏洩が検知された場合:
  1. 漏洩元アカウントの即時無効化
  2. 漏洩データの範囲特定
  3. 法的義務に基づく通知（GDPR: 72時間以内）
  4. フォレンジック調査の開始
  5. → 詳細は [503_incident_response.md](../operations/500_incident_response.md) を参照

## Part XLVIII. Agentic Workflow Orchestration

### 48.1. Multi-Agent System設計

- **Rule 50.480**: 管理ツールにおけるMulti-Agent System（MAS）の設計原則:
  - 各Agentは単一責任（ユーザー管理Agent、分析Agent等）
  - Agent間通信はA2A（Agent-to-Agent）プロトコルに準拠
  - Agentの行動はすべて監査ログに記録
  - Human-in-the-Loop（HITL）のブレークポイントを定義する

### 48.2. Agent権限管理

- **Rule 50.481**: AI Agentの権限はRBACで管理し、人間の運用者と同等のアクセス制御を適用する:
  - Agentごとに専用のサービスアカウントを割り当て
  - Agent権限は最小権限の原則に厳格に従う
  - 破壊的操作の実行にはHITL承認を必須とする

### 48.3. A2A Protocol統合

- **Rule 50.482**: Agent間通信にはA2A（Agent-to-Agent）/ ACP（Agent Communication Protocol）を採用する:
  - エージェントカード（Agent Card）による能力宣言
  - タスク通信プロトコル（Task Protocol）の標準化
  - エラーハンドリングと補償処理の定義

### 48.4. Agent Observability

- **Rule 50.483**: AI Agentの動作を以下の指標で監視する:
  - Agent実行頻度とスループット
  - 判定精度（True Positive / False Positive）
  - レイテンシ（Agent応答時間）
  - HITL呼び出し率（人間介入の頻度）
  - コスト（Agent実行あたりのトークン消費量）

  - 空間UIでのコラボレーション

  - 破壊的操作は音声のみでは実行不可（視覚的確認を必須とする）

### 49.3. Digital Twin管理

  - リアルタイムなシステム状態の3D可視化
  - What-If分析（構成変更の影響シミュレーション）
  - 障害の自動検知と視覚的なアラート

  - PQC（Post-Quantum Cryptography）への移行計画
