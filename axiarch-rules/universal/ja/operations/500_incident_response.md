> [!IMPORTANT]
> **Supreme Directive（最高指令）**
> - 危機発生時は、**事業継続と人命安全**が他の全ての優先事項（コスト、スケジュール、機能開発）に絶対的に優先する。
> - 「報告の遅延」は「対応の遅延」と同義である。インシデントを検知した時点で、その影響の大小に関わらず、**即座にエスカレーション**せよ。
> - 本ドキュメントに定義されたプロトコルを無視・省略した対応は、いかなる理由があっても**重大な憲法違反**として扱う。

## Part I: 基盤・哲学

### §1. 危機管理哲学と基本原則

-   **Law（レジリエンスファースト原則）**: 危機管理の目的は「障害をゼロにすること」ではなく、「障害が発生しても迅速に復旧し、事業を継続できるレジリエンス（回復力）を構築すること」です。完璧な予防は不可能であるという前提に立ち、**検知・対応・復旧・学習**の4能力を組織に実装してください。
-   **Proactive vs Reactive**:

    | アプローチ | 内容 | 例 |
    |:---------|:-----|:---|
    | **Proactive（事前対策）** | 障害が起きる前に備える | BIA、演習、ランブック作成、カオスエンジニアリング |
    | **Reactive（事後対応）** | 障害発生後に対処する | インシデントレスポンス、ポストモーテム |

    **Mandate**: 組織のリソース配分は **Proactive 70% : Reactive 30%** を目標としてください。事後対応に追われる組織は成熟度が低い状態です。

-   **5つの基本原則**:
    1.  **Assume Breach（侵害前提）**: 「いつか侵害される」ではなく「既に侵害されている可能性がある」と想定する
    2.  **Blast Radius Minimization（爆発半径の最小化）**: 障害の影響範囲を最小化する設計を常に優先する
    3.  **Graceful Degradation（優雅な劣化）**: 全停止ではなく機能制限モードでサービスを維持する
    4.  **Transparency（透明性）**: 「沈黙は不信を生む」— 進捗がなくても状況を伝え続ける
    5.  **Blameless Culture（非難なき文化）**: 個人の責任追及ではなくシステム改善に集中する

-   **Cross-Reference**: `000_core_mindset.md` (優先順位の階層), `502_site_reliability.md` §4 (SLI/SLO)

### §2. フレームワーク準拠

-   **Law**: 危機管理・BCPは、国際標準フレームワークに準拠した体系的なアプローチで構築しなければなりません。独自フレームワークのみに依存することは禁止です。
-   **フレームワークマッピング**:

    | フレームワーク | 適用領域 | 本ルールとの対応 |
    |:------------|:---------|:--------------:|
    | **ISO 22301:2019** | 事業継続マネジメントシステム（BCMS） | §3 BIA, §11 BCP, §13 DR戦略 |
    | **NIST CSF 2.0** | サイバーセキュリティフレームワーク（6機能モデル） | §6 インシデントフロー, §7 サイバーインシデント |
    | **NIST SP 800-61 Rev.3** | インシデントレスポンスガイド（2025年4月公開・**CSF 2.0統合モデル**に全面改訂。従来の4フェーズモデルを廃止し、CSF 2.0の6機能にIR活動をマッピング） | §6 インシデントフロー（CSF 2.0統合）, §5-§8 インシデント対応全般 |
    | **ITIL 4** | ITサービスマネジメント | §5 インシデント管理, §18 ポストモーテム |
    | **EU DORA** | デジタルオペレーショナルレジリエンス（2025年1月施行） | §21 DORA規制統合 |
    | **EU NIS2** | ネットワーク・情報セキュリティ指令（2024年10月国内法化期限） | §32 NIS2統合 |
    | **EU CRA** | サイバーレジリエンス法（2024年12月発効、2026年9月報告義務開始） | §33 CRA統合 |
    | **APRA CPS 230** | オペレーショナルリスク管理基準（豪州、2025年7月施行） | §34 インシデント報告国際調和 |

    - **Identify（識別）**: §3 BIA, §4 リスクアセスメント

## Part II: リスク分析と準備

### §3. BIA（事業影響度分析）

-   **Law**: 危機管理・BCPの設計に先立ち、**BIA（Business Impact Analysis）**を実施し、業務プロセスの優先度とRTO/RPOを定義しなければなりません。BIAなきBCPは「何を守るべきか」が不明確な計画であり無効です。
-   **業務優先度分類**:

    | ティア | 分類 | RTO目標 | RPO目標 | 例 |
    |:------|:-----|:--------|:--------|:---|
    | **Tier 1（ミッションクリティカル）** | 即時復旧必須 | ≦ 1時間 | ≦ 15分 | 認証、決済、コアAPI |
    | **Tier 2（ビジネスクリティカル）** | 当日復旧 | ≦ 4時間 | ≦ 1時間 | 検索、ユーザーダッシュボード |
    | **Tier 3（重要）** | 翌営業日復旧 | ≦ 24時間 | ≦ 4時間 | 管理画面、レポート生成 |
    | **Tier 4（一般）** | 計画復旧 | ≦ 72時間 | ≦ 24時間 | 社内ツール、バッチ処理 |

-   **BIA拡張メトリクス**:
    - **MTPD（Maximum Tolerable Period of Disruption）**: 事業が耐えられる最大停止期間
    - **WRT（Work Recovery Time）**: RTO達成後、通常業務レベルに戻るまでの追加作業時間
    - **MBCO（Minimum Business Continuity Objective）**: 最低限維持すべき業務レベル（例: 処理能力の50%）
    - **関係式**: MTPD ≧ RTO + WRT であることを必ず検証

-   **BIA実施義務**:
    - **初回**: プロジェクト立ち上げ時にBIAを実施し、Blueprintに記録
    - **更新**: 年1回、または重大な機能追加・アーキテクチャ変更時に更新
    - **記録項目**: 業務プロセス名、依存するシステム/サービス、停止時の影響（収益・法的・レピュテーション）、RTO/RPO/WRT/MTPD/MBCO
    - **依存関係マッピング**: 各業務プロセスの上流・下流依存関係を可視化し、カスケード障害リスクを特定
-   **Cross-Reference**: `502_site_reliability.md` §7 (RPO/RTO), §4 (SLI/SLO)

### §4. リスクアセスメントと脅威モデリング

-   **Law**: 定期的なリスクアセスメントを実施し、脅威の特定・評価・対策を体系的に管理しなければなりません。
-   **脅威カタログ（最低限のカバレッジ）**:

    | カテゴリ | 脅威例 | 影響度 |
    |:--------|:------|:------:|
    | **サイバー攻撃** | ランサムウェア、DDoS、データ侵害、サプライチェーン攻撃 | 極高 |
    | **AI/ML脅威** | プロンプトインジェクション、Model Poisoning、Deepfake | 極高 |
    | **インフラ障害** | クラウドリージョン障害、DB障害、CDN障害、DNS障害 | 高 |
    | **人的リスク** | キーマン離脱、内部不正、ソーシャルエンジニアリング | 高 |
    | **外部依存** | 決済サービス障害、AI APIベンダー障害、認証基盤障害 | 高 |
    | **自然災害** | 地震、台風、洪水（データセンター所在地） | 中〜高 |
    | **気候変動** | 極端気象の頻発、ヒートウェーブによるDC冷却障害 | 中〜高 |
    | **地政学リスク** | 制裁、データ主権規制変更、サプライチェーン分断 | 中〜高 |
    | **パンデミック** | 感染症流行、オフィス閉鎖、人員不足 | 中〜高 |
    | **法規制** | データローカライゼーション要件、緊急の規制変更 | 中 |
    | **量子コンピューティング** | HNDL攻撃、暗号解読、長期機密データ漏洩 | 中（将来的に極高） |
    | **OT/ICS** | 産業制御システム攻撃、SCADAマルウェア | 高（該当業種） |

-   **リスクマトリクス（評価基準）**:

    | | 影響度: 低 | 影響度: 中 | 影響度: 高 | 影響度: 極高 |
    |:------------|:---------|:---------|:---------|:----------|
    | **発生確率: 高** | 中リスク | 高リスク | 極高リスク | 極高リスク |
    | **発生確率: 中** | 低リスク | 中リスク | 高リスク | 極高リスク |
    | **発生確率: 低** | 低リスク | 低リスク | 中リスク | 高リスク |

    **Action**: 高リスク以上の脅威には、必ず対応策（ランブック、フェイルオーバー計画）を事前に準備してください。
-   **脅威インテリジェンス統合**:
    - ISAC（Information Sharing and Analysis Center）等の業界情報共有組織への参加を推奨
    - MITRE ATT&CK フレームワークに基づく脅威モデリング
    - CTI（Cyber Threat Intelligence）フィードの監視システムへの統合
-   **レビューサイクル**: 四半期ごとにリスクアセスメントをレビューし、新たな脅威の追加・リスクレベルの再評価を行ってください。

## Part III: インシデント対応

### §5. インシデント対応体制（War Room）

-   **重大度レベル（Severity Levels）**:

    | レベル | 定義 | 初動期限 | 解決目標 | コミュニケーション |
    |:------|:-----|:---------|:---------|:----------------|
    | **SEV-1 (Critical)** | サービス全停止、データ損失、セキュリティ侵害 | **15分以内** | **1時間以内** | ステータスページ + 全社通知 + 経営報告 |
    | **SEV-2 (High)** | 主要機能停止（決済不可、認証障害等） | **30分以内** | **4時間以内** | ステータスページ + チーム通知 |
    | **SEV-3 (Medium)** | 非主要機能の不具合、パフォーマンス劣化 | **2時間以内** | **24時間以内** | チャット通知チャネル |
    | **SEV-4 (Low)** | 軽微なバグ、UI崩れ、ログ警告増加 | **次営業日** | **次スプリント** | Issue Tracker |

-   **War Room設置基準**: SEV-1またはSEV-2が発生した場合、**即座に**War Room（緊急対策本部）を設置します。
    - **物理**: Slackチャンネル `#incident-yyyy-mm-dd-{summary}` + ビデオ会議
    - **自動化**: SEV-1検知時はPagerDuty等でon-call担当を自動招集

-   **ロール（RACI）**:

    | ロール | 責務 | 禁止事項 |
    |:------|:-----|:---------|
    | **Incident Commander（指揮官）** | 意思決定、優先順位付け、エスカレーション判断 | 技術的作業への直接従事 |
    | **Ops Lead（作業責任者）** | 復旧作業の指揮・実行、技術的判断 | Commander不在での独断対応 |
    | **Comms Lead（広報責任者）** | 30分ごとのステータス更新、ユーザー/社内通知 | 未確認情報の発信 |
    | **Scribe（書記）** | 全イベント・判断のタイムスタンプ付き記録 | 記録を後回しにすること |
    | **Subject Matter Expert（SME）** | 専門領域（DB、インフラ、セキュリティ等）の技術支援 | 指揮系統を無視した独断行動 |

-   **On-Call エスカレーション**:
    - Primary → **5分無応答** → Secondary自動エスカレーション
    - Secondary → **10分無応答** → Engineering Manager自動エスカレーション
    - 全員無応答 → **15分** → CTO/VPへ緊急通知

### §6. インシデントレスポンスフロー（NIST SP 800-61 Rev.3 / CSF 2.0統合モデル）

-   **Law**: インシデント対応は、**NIST SP 800-61 Rev.3（2025年4月公開）**に準拠してください。Rev.3は従来の4フェーズモデルを廃止し、**NIST CSF 2.0の6機能にIR活動をマッピング**する統合モデルに全面改訂されました。インシデントレスポンスを「単発イベント」ではなく「継続的なサイバーセキュリティリスク管理の一部」として位置づけます。

-   **CSF 2.0統合IRモデル（6機能マッピング）**:

    | CSF機能 | IR活動 | 本ルール対応 |
    |:--------|:------|:-----------|
    | **Govern（統治）** | IR方針の策定、リスクガバナンス、規制要件統合、経営層の責任範囲定義 | §19 ガバナンス, §48 CISO責任 |
    | **Identify（識別）** | 資産・リスクの把握、BIA、脅威インテリジェンス統合、CSF 2.0コミュニティプロファイル活用 | §3 BIA, §4 リスクアセスメント |
    | **Protect（防御）** | 準備活動（ランブック整備、連絡網維持、フォレンジックツール事前準備、教育訓練） | §17 ランブック, §43 教育, `600_security_privacy.md` |
    | **Detect（検知）** | 監視アラート、異常検知、トリアージ判定、規制報告トリガー判定 | §5 インシデント対応体制, §22 AIOps |
    | **Respond（対応）** | 封じ込め、根絶、証拠保全、コミュニケーション、規制報告 | §5-§10 IR全般, §34 規制報告 |
    | **Recover（回復）** | サービス復旧、ポストモーテム、改善アクション、レピュテーション回復 | §13-§15 DR, §18 ポストモーテム, §28 レピュテーション |

-   **従来モデルとの互換マッピング**:

    > Rev.3以前のNIST SP 800-61 Rev.2の4フェーズモデルとの対応関係（移行参考用）:

    | Rev.2（旧4フェーズ） | Rev.3（CSF 2.0統合） |
    |:-------------------|:-------------------|
    | **Phase 1: 準備（Preparation）** | → **Govern** + **Identify** + **Protect** |
    | **Phase 2: 検知と分析（Detection & Analysis）** | → **Detect** + **Respond**（分析部分） |
    | **Phase 3: 封じ込め・根絶・復旧** | → **Respond** + **Recover** |
    | **Phase 4: 事後活動（Post-Incident）** | → **Govern**（改善フィードバック） + **Identify**（リスク再評価） |

-   **Govern + Identify + Protect（事前準備）**:
    - ランブックの事前作成と定期レビュー（§17参照）
    - 緊急連絡網の維持（Slack以外の代替手段を確保: 電話、SMS、別チャットツール）
    - フォレンジックツール・ログ保全手順の事前準備
    - **規制報告テンプレート**: DORA/NIS2/CRA/GDPR/Global Privacy Laws/APRA CPS 230各規制の報告書テンプレートを事前準備（§34参照）
    - **CSF 2.0コミュニティプロファイル**: NIST SP 800-61 Rev.3が定義するインシデントリスク管理コミュニティプロファイルを基にIR優先事項を設定
-   **Detect（検知と分析）**:
    - 監視アラートまたはユーザー報告による検知
    - **トリアージ判定**: 重大度レベルの即座の判定（§5マトリクス参照）
    - 影響範囲の初期評価（影響ユーザー数、影響機能、データ損失有無）
    - **Evidence Preservation（証拠保全）**: ログ、スクリーンショット、メモリダンプの即時保全（§37 Chain of Custody準拠）
    - **規制報告トリガー判定**: インシデントが規制報告義務に該当するかの即時判定（§34参照）
-   **Respond（対応）**:
    - **短期封じ込め**: 影響範囲の拡大を即座に防止（ネットワーク隔離、API遮断、Kill Switch）
    - **根絶**: 根本原因の除去（マルウェア除去、脆弱性パッチ適用、不正アクセス経路の遮断）
    - **コミュニケーション**: §9-§10に基づくステークホルダー通知の実施
    - **規制報告**: §34に基づく法定報告の実施（最短期限に準拠）
-   **Recover（回復と事後活動）**:
    - **復旧**: サービスの段階的復旧（カナリアリリースでの正常性確認を推奨）
    - **48時間以内**にポストモーテムを実施（§18参照）
    - 改善アクションの起票と追跡 → **Govern（統治）**へのフィードバックループ
    - 教訓のプロジェクト教訓ログへの追記
    - 規制当局への最終報告の提出（該当する場合）

-   **6ステップ・インシデントフロー（簡易版）**:
    1.  **検知**: 監視アラート or ユーザー通報
    2.  **分類**: Severityレベル判定 + War Room設置判断 + 規制報告トリガー判定
    3.  **宣言**: インシデント通知チャネルで宣言（例: `🔴 SEV-1: 決済機能障害 発生中`）
    4.  **対応**: ロールバック / ホットフィックス / Kill Switch / 一時遮断
    5.  **解決**: ステータスページ更新、復旧通知、正常性監視開始
    6.  **振り返り**: 48時間以内にPost-Mortemを実施 → Governフィードバック

### §7. サイバーインシデント固有対応

-   **Law**: サイバーインシデントは、一般的なシステム障害とは異なる対応プロトコルが必要です。脅威タイプごとの固有プレイブックを事前に準備してください。

#### §7.1. ランサムウェア対応プレイブック

-   **脅威ランドスケープ（2025-2026）**:
    - **RaaS（Ransomware-as-a-Service）**: ランサムウェア攻撃の参入障壁が低下。技術力の低い攻撃者でも高度な攻撃を実行可能に
    - **二重恐喝（Double Extortion）**: データ暗号化＋データ公開脅迫の二段構え攻撃が主流化
    - **純粋恐喝（Pure Extortion）**: 暗号化なしでデータを窃取し、公開を脅して身代金を要求する手法が増加
    - **AI連携フィッシング**: AI生成による高精度なスピアフィッシング・ビッシング（音声フィッシング）が初期侵入経路として増加
    - **低ボリューム・高インパクト攻撃**: 大量データの窃取を暗号化なしで行い、検知を回避する戦術
    - **主要ターゲット**: 製造業、医療、教育、エネルギー、金融セクター

-   **即時対応（Golden Hour — 最初の1時間）**:
    1.  **隔離**: 感染が疑われるシステムをネットワークから即座に隔離（物理/論理的に切断）
    2.  **支払い禁止**: 身代金の支払いは**原則禁止**（法的助言を受けた上での例外判断のみ許可）
    3.  **証拠保全**: メモリダンプ、ログ、感染ファイルのフォレンジック用保全（§37参照）
    4.  **バックアップ確認**: バックアップの汚染有無を確認（イミュータブルバックアップの存在確認）
    5.  **法執行機関への通報**: 所轄警察、JPCERT/CC（日本）、FBI IC3（米国）等に通報
    6.  **規制報告**: DORA/NIS2該当時は規制当局への初期通知を開始（§34参照）
    7.  **IRリテーナー連絡**: 事前契約済みのインシデント対応リテーナー企業に即時連絡（§47参照）
    8.  **サイバー保険通知**: 保険会社への初期通知（保険請求に必要な証拠保全手順の確認）

-   **復旧手順**:
    - クリーンなバックアップからのリストア（§15参照）
    - 侵入経路の完全な特定と遮断
    - 全クレデンシャルの強制ローテーション
    - 段階的なサービス復旧（隔離環境での検証 → 制限付き公開 → 全面復旧）
    - データ窃取が確認された場合のダークウェブモニタリング開始

-   **防御策**:
    - **3-2-1-1バックアップルール**の厳守（§15参照）
    - **イミュータブルバックアップ**の必須実装
    - **ネットワークセグメンテーション**による横展開防止
    - **EDR/XDR**の全エンドポイントへの導入
    - **AI対応フィッシング訓練**: AI生成フィッシングを想定した従業員教育の強化
    - **MFA必須化**: 全特権アカウントおよびVPNアクセスへの多要素認証の義務化

-   **Cross-Reference**: §27 (サイバー保険), §47 (Ransomware Negotiation・IR Retainer)

#### §7.2. データ侵害対応プロトコル

-   **Stop the Bleeding（止血）**:
    - データ侵害または不正アクセスが疑われる場合、**躊躇なく止血**を行う
    - APIキーの無効化、影響サーバーの隔離、アクセストークンの失効を**最優先**で実行
-   **影響評価**:
    - 漏洩データの種類と件数の特定（PII、決済情報、認証情報）
    - 影響ユーザーの特定
    - 侵害の時間軸の特定（いつからいつまで）
-   **法的通知義務**:

    | 法規制 | 通知期限 | 通知先 |
    |:------|:---------|:------|
    | **GDPR（EU）** | **72時間以内** | 監督当局 + 影響を受けるデータ主体 |
    | **Global Privacy Laws** | **速報: 速やかに / 確報: 30日以内** | 個人情報保護委員会 + 本人 |
    | **CCPA/CPRA（米CA）** | **遅滞なく** | カリフォルニア州司法長官 + 消費者 |
    | **DORA（EU金融）** | **重大ICTインシデント: 4時間以内（初報）** | 管轄金融監督当局 |
    | **NIS2（EU）** | **早期警告: 24時間以内 / 通知: 72時間以内** | 管轄CSIRT + 監督当局 |

-   **Cross-Reference**: `600_security_privacy.md`, `601_data_governance.md`, §34 (インシデント報告国際調和)

#### §7.3. DDoS攻撃対応

-   **多層防御**:
    - **L3/L4**: CDN/WAFでのトラフィックフィルタリング（Cloudflare、AWS Shield等）
    - **L7**: レート制限の動的強化、CAPTCHAの一時的適用
    - **自動スケーリング**: トラフィック急増時のインフラ自動拡張
    - **Anycast**: DNS Anycastによるトラフィック分散
-   **判定基準**: 通常時の**10倍以上**のリクエスト急増を検知した場合、DDoS対応プロトコルを自動発動
-   **コスト保護**: DDoS攻撃によるオートスケーリングコスト爆発を防ぐ上限設定（§31 FinOps参照）

#### §7.4. サプライチェーン攻撃対応

-   **即時対応**:
    - 侵害された依存パッケージのバージョンを特定・アンインストール
    - 影響を受けた全環境のクレデンシャルをローテーション
    - SBOMに基づく影響範囲の迅速な特定
-   **Cross-Reference**: `602_oss_compliance.md` (SBOM/SCA/サプライチェーンセキュリティ)

#### §7.5. ゼロデイ脆弱性対応

-   **Law**: ゼロデイ脆弱性が公開された場合、パッチ適用までの間に攻撃を防ぐための緊急対応プロトコルを事前に準備してください。
-   **即時対応（パッチ未提供時）**:
    1.  **仮想パッチ適用**: WAF/IPS/IPSルールの緊急追加による攻撃ベクトルの遮断
    2.  **攻撃面の縮小**: 脆弱な機能の一時的無効化（Kill Switch、Feature Flag）
    3.  **監視強化**: 脆弱性を悪用する攻撃パターンの検知ルールを即時追加
    4.  **Threat Intelligence確認**: 既知のExploit / PoC / IoC（Indicators of Compromise）を確認
-   **パッチ適用時**:
    - 緊急パッチのテスト → ステージング検証 → カナリアデプロイ → 全面展開
    - **SLA**: Critical CVSSスコア9.0+のゼロデイは**24時間以内**にパッチ適用完了
-   **脆弱性開示調整（CVD: Coordinated Vulnerability Disclosure）**:
    - 自社製品の脆弱性発見時: 責任ある開示ポリシーに従い、修正後に公開
    - EU CRA下では脆弱性の積極的な悪用を**24時間以内**にENISAに報告（§33参照）
-   **Cross-Reference**: `600_security_privacy.md` (脆弱性管理), §40 (ゼロデイ詳細)

#### §7.6. アイデンティティベース攻撃対応

-   **Law**: 2025-2026年においてアイデンティティは最も攻撃される面（Attack Surface）です。「侵入する」のではなく「ログインする」攻撃が主流化しています。アイデンティティベースの攻撃に対する固有の対応プロトコルを準備してください。

-   **主要攻撃ベクトルと対策**:

    | 攻撃手法 | 概要 | 対策 |
    |:--------|:-----|:-----|
    | **MFA Fatigue Attack** | 大量のMFAプッシュ通知を送り、ユーザーに誤って承認させる | Number Matching MFA、FIDO2/WebAuthn移行、異常なMFA要求の検知 |
    | **SIM Swapping** | キャリアを欺いてSIMを乗っ取り、SMS-MFAを突破 | SMS-MFA廃止→認証アプリ/FIDO2、テレコムPINの設定 |
    | **Adversary-in-the-Middle（AiTM）** | プロキシ経由でセッションクッキーを窃取し、MFAを迂回 | トークンバインディング、条件付きアクセス（準拠デバイス必須） |
    | **Session Hijacking** | 窃取されたセッショントークンによる不正アクセス | セッショントークンの短寿命化、継続的認証、デバイスバインディング |
    | **Credential Stuffing** | 漏洩クレデンシャルDB（Dark Web）を利用した自動認証試行 | パスワードレス認証、Have I Been Pwned統合、レート制限強化 |
    | **ヘルプデスク攻撃** | IT部門になりすましてパスワードリセットを要求 | 本人確認の多段階プロセス、帯域外確認、リセット操作の監査ログ |

-   **即時対応（アイデンティティ侵害検知時）**:
    1.  **影響アカウントのセッション無効化**: 全セッション・全デバイスからの強制ログアウト
    2.  **クレデンシャルローテーション**: パスワード・APIキー・トークンの即時リセット
    3.  **横展開調査**: 侵害されたアカウントが持つアクセス権限に基づき、影響範囲を調査
    4.  **条件付きアクセスの一時的厳格化**: 影響組織全体でのMFA再認証強制
    5.  **IDPログの保全**: Azure AD/Okta/Google Workspace等のサインインログ、監査ログの即時保全

-   **予防戦略**:
    - **パスワードレス認証**: FIDO2/WebAuthn/Passkeyの段階的導入（最終的にはパスワード完全廃止を目標）
    - **継続的認証（Continuous Authentication）**: セッション中のリスクベース再認証（異常な振る舞いの検知時）
    - **特権アクセス管理（PAM）**: Just-in-Time（JIT）アクセス、最小権限原則の厳格化
    - **ID脅威検知（ITDR: Identity Threat Detection and Response）**: 専用ITDR製品またはXDRのID機能の導入

-   **Cross-Reference**: `600_security_privacy.md` (認証・アクセス制御), §8 (Deepfake/ソーシャルエンジニアリング)

#### §7.7. 非人間アイデンティティ（NHI）セキュリティとインシデント対応

-   **Law**: APIキー、サービスアカウント、OAuthクライアント、マシン間通信のトークン等の**非人間アイデンティティ（Non-Human Identity: NHI）**は、人間アイデンティティの圧倒的多数を上回り（典型的に10:1〜50:1の比率）、しかし管理が行き届いていないケースが多く、重大なセキュリティリスクとなっています。NHI固有の危機管理プロトコルを確立してください。

-   **NHIリスクカテゴリ**:

    | リスク | 概要 | 影響 |
    |:------|:-----|:-----|
    | **権限の肥大化（Privilege Sprawl）** | 過剰な権限を持つサービスアカウントの蓄積 | 侵害時の爆発半径が甚大 |
    | **ライフサイクル管理の欠如** | 退役・ローテーションされないAPIキー・トークン | 長期間の不正アクセスリスク |
    | **シークレットのハードコード** | ソースコード・CI/CD設定にAPIキーを直書き | GitHub等での公開→即時侵害 |
    | **ゾンビアカウント** | 使用されなくなったが残存するサービスアカウント | 検知されにくい攻撃経路 |
    | **OAuth過剰許可** | 必要以上の広範なスコープを持つOAuthアプリ | サードパーティ経由の横展開 |

-   **インシデント対応（NHI侵害時）**:
    1.  **影響NHIの即時無効化**: APIキー失効、サービスアカウント停止、OAuthトークン取り消し
    2.  **シークレットスキャン**: 全リポジトリ、CI/CDパイプライン、環境変数のスキャン（GitGuardian、Trufflehog等）
    3.  **横展開の調査**: 侵害されたNHIが持つ権限に基づく影響範囲の特定
    4.  **全関連クレデンシャルのローテーション**: 侵害されたNHIに接触した全サービスのクレデンシャル更新
    5.  **監査ログの確認**: NHIによる異常なAPI呼び出しパターンの過去ログ分析

-   **予防策**:
    - **シークレット管理**: HashiCorp Vault、AWS Secrets Manager等の集中管理。ハードコード禁止
    - **自動ローテーション**: APIキー・トークンの定期自動ローテーション（最長90日）
    - **最小権限**: サービスアカウントの権限をジョブ遂行に必要な最小限に制限
    - **NHIインベントリ**: 全NHIの棚卸しと定期的な棚卸し更新（四半期）
    - **有効期限の義務化**: 全APIキー・トークンに有効期限を設定。無期限トークン禁止
    - **CI/CDパイプラインセキュリティ**: パイプライン内でのシークレット使用はOIDCフェデレーション/Workload Identityを優先

    ```yaml
    # GitHub Actions — OIDC Workload Identity例（ハードコードシークレット回避）
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: 'projects/123/locations/global/workloadIdentityPools/pool/providers/provider'
          service_account: 'deploy@project.iam.gserviceaccount.com'
    ```

-   **サイバー保険への影響**: NHI侵害はサイバー保険の補償対象外となるケースが増加（§27参照）
-   **Cross-Reference**: `600_security_privacy.md` (アクセス制御), §27 (サイバー保険), §37 (フォレンジクス)

## Part IV: AI脅威対応

### §8. AI駆動脅威対応

-   **Law**: AI技術を悪用した新種の脅威（Deepfake、プロンプトインジェクション、AI生成フィッシング等）に対する固有の対応手順を準備してください。
-   **脅威カタログ**:

    | 脅威 | 影響 | 対応 |
    |:-----|:-----|:-----|
    | **プロンプトインジェクション** | AI機能の悪用、有害コンテンツ生成 | Global Kill Switch（§9.2参照）、ガードレール強化 |
    | **AI APIコスト攻撃** | 大量リクエストによるクラウド破産 | レート制限、コストサーキットブレーカー |
    | **Deepfake詐欺** | なりすましによる不正承認 | 多要素認証の厳格化、帯域外確認 |
    | **AI生成フィッシング** | 高精度なフィッシングメール/サイト | セキュリティ啓発強化、メール認証（DMARC） |
    | **Model Poisoning** | 学習データ汚染によるAI出力の改ざん | データパイプラインの整合性検証、モデル署名 |
    | **LLM Jailbreak** | セーフティガードレールの迂回 | 多層ガードレール、出力フィルタリング強化 |
    | **AI生成マルウェア** | 多態性マルウェアの自動量産 | EDR/XDR強化、ビヘイビア分析 |
    | **Shadow AI** | 未承認AIツールの業務利用 | AIガバナンスポリシー、DLP統合 |
    | **Adversarial ML** | 入力操作によるAIモデルの誤判定 | 敵対的テスト、入力バリデーション強化 |

-   **Deepfake対策プロトコル**:
    - **検出技術**: Deepfake検出ツール（Microsoft Video Authenticator、Sensity AI等）の導入検討
    - **認証強化**: 経営層のなりすまし（CEO詐欺）対策として、高額送金・重大意思決定には**帯域外認証**（別チャネルでの直接確認）を必須化
    - **Voice Cloning対策**: 音声ベースの承認（電話での送金指示等）には**コードワード方式**を導入（事前に合意した秘密の合言葉で本人確認）
    - **動画認証**: ビデオ会議でのなりすまし対策として、リアルタイムの動作指示（特定の動作を要求）による検証プロトコル
    - **全従業員教育**: Deepfake攻撃事例の定期的な共有と訓練（§43参照）

-   **Shadow AI管理**:
    - 未承認AIツールの業務利用を検出するDLP（Data Loss Prevention）統合
    - 承認済みAIツールの一覧と利用ガイドラインの維持
    - Shadow AIリスクの四半期評価と従業員啓発

-   **Global Kill Switch 実装義務**:
    - AI機能の緊急停止スイッチは、**再デプロイ不要**な **Edge Config** または **DBフラグ**（`system_settings`）で実装
    - Middlewareで `ai_enabled`, `billing_enabled` 等のフラグをチェックし、OFF時は即座に503を返却
    - **「コードを直してデプロイ」では遅すぎる** — 秒単位での停止を可能にすること

    ```typescript
    // Global Kill Switch — Middleware実装例
    import { NextResponse } from 'next/server';

    export async function middleware(request: NextRequest) {
      // Edge Config or DB flags check (no re-deploy required)
      const flags = await getSystemFlags(); // Edge Config / KV Store

      if (!flags.ai_enabled && request.nextUrl.pathname.startsWith('/api/ai')) {
        return NextResponse.json(
          { error: 'AI_SERVICE_TEMPORARILY_DISABLED', message: 'AI機能は一時停止中です' },
          { status: 503 }
        );
      }

      if (!flags.billing_enabled && request.nextUrl.pathname.startsWith('/api/billing')) {
        return NextResponse.json(
          { error: 'BILLING_SERVICE_TEMPORARILY_DISABLED', message: '決済機能は一時停止中です' },
          { status: 503 }
        );
      }
    }
    ```

-   **Cross-Reference**: `400_ai_engineering.md` (ガードレール), `502_site_reliability.md` §9.6 (AIコストガバナンス)

## Part V: コミュニケーション

### §9. 危機コミュニケーション管理

-   **Law**: 「沈黙は不信を生む」。インシデント発生時は、解決していなくても状況を伝え続けなければなりません。

#### §9.1. ステータスページ管理

-   **コンポーネント別稼働監視**:
    - ステータスページでは、サービスを構成するコンポーネントごとの稼働状況を**個別にリアルタイム表示**してください。
    - 「全体的に障害」ではなく「どのコンポーネントが影響を受けているか」をユーザーが一目で判別できる状態を維持

-   **Component Categories（最低限のカテゴリ）**:

    | カテゴリ | 監視対象（例） | ヘルスチェック方法（例） |
    |:--------|:-------------|:---------------------:|
    | **Webアプリケーション** | フロントエンド / Edge | HTTP 200応答確認 |
    | **API** | バックエンドAPI | レスポンスタイム ≦ 閾値 |
    | **データベース** | プライマリDB | 接続プール使用率 |
    | **認証** | 認証基盤 | ログイン成功率 |
    | **メディア配信** | CDN / ストレージ | CDNヒット率 |
    | **外部連携** | 決済 / AI / メール等 | 応答成功率 |

-   **Status Levels（4段階）**:
    - `Operational` — 正常稼働
    - `Degraded Performance` — 性能劣化
    - `Partial Outage` — 一部障害
    - `Major Outage` — 全面障害

-   **ステータス遷移フロー**:
    - `Investigating` → `Identified` → `Monitoring` → `Resolved`

#### §9.2. 障害通知テンプレート

-   **Law**: 障害通知は、**事前に用意されたテンプレート**に基づいて発信してください。慌てて自由文で書くことは情報欠落と不適切な表現のリスクを高めます。

-   **初報テンプレート**:
    ```
    【障害情報】{コンポーネント名}に関するお知らせ

    いつも{サービス名}をご利用いただき、ありがとうございます。
    ただいま{コンポーネント名}において、{症状}が発生しております。

    ■ 発生時刻: {YYYY年MM月DD日 HH:MM JST}
    ■ 影響範囲: {影響を受ける機能}
    ■ 現在の状況: 原因調査中 / 復旧作業中
    ■ 次回更新予定: {HH:MM JST}

    ご不便をおかけし、誠に申し訳ございません。
    復旧次第、本ページにてお知らせいたします。
    ```

-   **必須フィールド（初報・続報・復旧報共通）**:
    - **発生時刻**: 障害が最初に検知された正確な日時（タイムゾーン明記）
    - **影響範囲**: 影響を受ける機能・ユーザー層の特定
    - **現在の状況**: 「調査中 / 原因特定済み / 復旧作業中 / 復旧完了」
    - **次回更新予定**: 続報の予定時刻（沈黙を防ぐ）

-   **更新頻度**:

    | レベル | ステータスページ更新 | 社内更新 |
    |:------|:-----------------:|:-------:|
    | **SEV-1** | 30分ごと | 15分ごと |
    | **SEV-2** | 1時間ごと | 30分ごと |
    | **SEV-3** | 4時間ごとまたは変化時 | 必要時 |

#### §9.3. メディア・SNS対応

-   **原則**: SEV-1で外部メディアの関心を集める可能性がある場合、以下を遵守
    - **スポークスパーソンの一本化**: 外部メディアへの発言権限は指定された1名のみ
    - **ファクトチェック**: SNSでの誤情報の監視と、公式チャネルでの迅速な訂正
    - **法務レビュー**: データ侵害等の重大インシデントに関するプレスリリースは法務確認を必須とする
    - **ダークサイト準備**: 重大インシデント用の事前準備済み告知ページ（即座に公開可能な状態）

### §10. ステークホルダー通知とレピュテーション管理

-   **ステークホルダー別通知マトリクス**:

    | ステークホルダー | SEV-1 | SEV-2 | SEV-3 | 通知手段 |
    |:---------------|:-----:|:-----:|:-----:|:--------|
    | **経営陣** | ✅ 即時 | ✅ 1時間以内 | ❌ | 電話 + メール |
    | **全社/チーム** | ✅ 即時 | ✅ 即時 | ✅ | Slack + メール |
    | **ユーザー** | ✅ 即時 | ✅ 1時間以内 | ❌ | ステータスページ |
    | **規制当局** | ✅ 法定期限内 | △ 必要時 | ❌ | 公式書簡/ポータル |
    | **取引先/パートナー** | ✅ 1時間以内 | △ 必要時 | ❌ | メール + 電話 |
    | **サイバー保険会社** | ✅ 契約条件に準拠 | △ 必要時 | ❌ | 指定連絡先 |

-   **データ侵害時のユーザーメール**:
    - 法務確認済みのテンプレートを使用し、迅速かつ透明に通知
    - 含めるべき情報: 何が起きたか、影響範囲、ユーザーが取るべき行動（パスワード変更等）、問い合わせ先
-   **Cross-Reference**: `601_data_governance.md` (GDPR/APPI通知義務)

## Part VI: 事業継続

### §11. BCP（事業継続計画）

-   **Law**: BCPはIT部門だけのものではなく、**組織全体の事業継続能力**を定義する計画です。ISO 22301に準拠し、経営陣の承認を得た上で維持してください。
-   **BCP必須要素**:

    | 要素 | 内容 |
    |:-----|:-----|
    | **スコープ** | 対象業務・システム・拠点の明確な定義 |
    | **BIA結果** | §3で実施した事業影響度分析の反映 |
    | **復旧戦略** | 各ティアの復旧手順とリソース |
    | **役割と責任** | BCP発動時の指揮系統とRACIマトリクス |
    | **コミュニケーション計画** | 内外への通知手順と連絡網 |
    | **代替手段** | リモートワーク、代替インフラ、手動運用手順 |
    | **テストと訓練** | §16 テーブルトップ演習参照 |
    | **規制要件マッピング** | 適用される規制（DORA/NIS2/CRA）とBCPの対応関係 |

-   **キーマンリスクの排除**:
    - 「特定のエンジニアしか触れないコード」を**排除**する
    - ドキュメント化とコードの標準化により、**誰でも**復旧作業を行える状態を維持
    - **Bus Factor ≧ 2**: 全ての重要システムについて、最低2名以上が運用・復旧可能な状態を維持

-   **緊急連絡網**:
    - Slack以外の緊急連絡手段（電話、SMS、別チャットツール）を**確保**
    - 連絡網は**四半期ごとに更新**し、陳腐化を防止
    - **到達性テスト**: 半年に1回、実際に緊急連絡網を使用したテスト通知を実施
    - **帯域外通信**: 主要な通信インフラ自体が障害の場合の代替通信手段を確保

-   **リモートワークBCP**:
    - VPN/ゼロトラスト接続の容量が全従業員同時接続に耐えうることを確認
    - 主要ツール（ソースコード、CI/CD、監視）へのリモートアクセスが保証されていること
    - オフィス不可時の代替作業場所の事前特定

### §12. 委託先・サプライチェーンBCP

-   **Law**: 外部委託先（クラウドプロバイダー、SaaS、決済サービス等）の障害が自社サービスに波及した場合の対応フローを事前に定義してください。「委託先の問題だから待つしかない」は**事業継続の放棄**です。
-   **委託先障害時の対応フロー**:
    1.  **検知**: 委託先ステータスページ監視 + 自社モニタリングによる異常検知
    2.  **影響評価**: 自社サービスのどの機能・ユーザーセグメントに影響があるかを即時評価
    3.  **ユーザー通知**: 自社ステータスページに影響範囲と状況を掲載（委託先復旧待ちでも**沈黙禁止**）
    4.  **代替策実行**: フォールバック手段（代替サービス切替、機能制限モード等）を実行
    5.  **復旧確認**: 委託先復旧後、自社サービスの正常動作を検証

-   **委託先BCP確認義務**: 重要な委託先（Highリスク等級）に対し、**年1回**以下を確認:
    - BCP整備状況
    - RPO/RTOの設定値とSLAとの整合性
    - 直近1年間のインシデント履歴と復旧実績
    - 冗長化・バックアップ体制
    - **契約上の出口戦略**: ベンダーロックインを回避するための移行計画

-   **代替委託先の事前準備**: SPOF（単一障害点）となる委託先については:
    - 代替サービスの選定を事前に完了
    - 最低限の接続検証（PoC）を実施
    - 切替手順をランブック化（§17参照）
    - **「障害が起きてから代替を探す」のでは遅すぎる**

-   **委託先リスクティア分類**:

    | ティア | 基準 | 代替要件 | 例 |
    |:------|:-----|:---------|:---|
    | **Critical** | 停止 = 即サービス全停止 | 代替サービス事前検証済み | クラウドプロバイダ、認証基盤 |
    | **High** | 停止 = 主要機能停止 | 代替候補選定済み | 決済、CDN、メール配信 |
    | **Medium** | 停止 = UX劣化 | Graceful Degradation設計 | AI API、分析ツール |
    | **Low** | 停止 = 影響軽微 | 手動代替可能 | CMS、内部ツール |

-   **NIS2サプライチェーン要件**:
    - NIS2 Article 21に基づくサプライチェーンセキュリティ評価の実施
    - 委託先のセキュリティ慣行の定期評価
    - 契約へのサイバーセキュリティ義務条項の組み込み
    - サプライチェーン全体にまたがるインシデント管理計画の統合

-   **Cross-Reference**: §17 (ランブック), `600_security_privacy.md` §10 (ベンダーセキュリティ管理)

## Part VII: 災害復旧

### §13. 災害復旧戦略

-   **Law**: DR（Disaster Recovery）戦略は、BIA（§3）で定義したRTO/RPOに基づき、ティアごとに適切なDR手法を選択してください。
-   **DRティア分類**:

    | DRティア | 戦略 | RTO | RPO | コスト | 用途 |
    |:--------|:-----|:----|:----|:------|:-----|
    | **Tier 0: Active-Active** | 複数リージョン同時稼働 | ≈ 0 | ≈ 0 | 最高 | ミッションクリティカル |
    | **Tier 1: Hot Standby** | DR環境を常時起動・同期 | ≦ 15分 | ≦ 5分 | 高 | ビジネスクリティカル |
    | **Tier 2: Warm Standby** | DR環境を縮小起動・定期同期 | ≦ 1時間 | ≦ 15分 | 中 | 重要業務 |
    | **Tier 3: Pilot Light** | 最小構成のみDR環境に準備 | ≦ 4時間 | ≦ 1時間 | 低 | 一般業務 |
    | **Tier 4: Backup & Restore** | バックアップからの復元のみ | ≦ 24時間 | ≦ 24時間 | 最低 | 非クリティカル |

-   **RPO/RTO設計の必須確認項目**:
    - **RPO**: どの時点までのデータ損失を許容するか
    - **RTO**: ダウンしてから何時間以内に復旧させるか
    - **MTPD（Maximum Tolerable Period of Disruption）**: 事業が耐えられる最大停止期間
    - これらの値はBIAの結果と整合していること

### §14. クラウドネイティブDR

-   **Law**: クラウド環境でのDRは、従来のオンプレミスDRとは異なるアプローチが必要です。IaC（Infrastructure as Code）とクラウドネイティブサービスを最大限活用してください。

-   **マルチリージョンフェイルオーバー**:
    - クロスリージョンレプリケーションの活用（DB、オブジェクトストレージ）
    - DNSベースのフェイルオーバー（Route 53 Health Check、Cloudflare Load Balancing等）
    - フェイルオーバー時のデータ整合性の保証（結果整合性の許容範囲を事前定義）

-   **IaC DR（Infrastructure as Code による DR）**:
    - 全インフラをコードで定義し、**別リージョンに即時再構築**可能な状態を維持
    - Terraform/Pulumi/OpenTofu等のステートファイルのバックアップと復元手順を確立
    - **Recovery Playbook**: IaCからの環境再構築手順をランブック化

-   **Serverless / Edge Computing DR**:
    - サーバーレス関数の複数リージョン自動デプロイ
    - Edge Functions のグローバル分散による自然な冗長化
    - コールドスタート遅延を考慮したフェイルオーバー設計

-   **DRaaS（Disaster Recovery as a Service）**:
    - Pilot Light戦略で最小限のリソースのみDR環境に維持
    - フェイルオーバー時に自動スケールアップ
    - 月次でフェイルオーバーテストを実施

-   **コンテナ/Kubernetes DR**:
    - Kubernetes ManifestのGitOps管理による別クラスタへの即時デプロイ
    - Persistent Volumeのクロスリージョンスナップショット
    - Velero等のバックアップツールの活用

-   **マルチクラウドDR**（§39で詳述）:
    - 単一クラウドプロバイダへの依存リスクの評価
    - クラウド間フェイルオーバーの設計と検証

-   **Cross-Reference**: `361_aws_cloud.md` (マルチリージョン), `502_site_reliability.md` §7 (RPO/RTO), §39 (マルチクラウドDR)

### §15. データ保護とバックアップ戦略

-   **Law**: バックアップは「取得」だけでは不十分です。「復元可能であること」を定期的に検証しなければなりません。未検証のバックアップは**存在しない**と見なします。
-   **3-2-1-1 バックアップルール**:
    - **3**: データのコピーを最低**3つ**保持
    - **2**: **2種類**の異なるストレージメディアに保存
    - **1**: **1つ**はオフサイト（別リージョン/別プロバイダ）に保管
    - **1**: **1つ**はイミュータブル（改変不能）バックアップとして保持

-   **イミュータブルバックアップ義務**:
    - ランサムウェアがバックアップを暗号化するリスクに対抗
    - Object Lock（S3）、WORM（Write Once Read Many）ストレージの活用
    - バックアップの保持期間中は**削除・変更が物理的に不可能**な状態を維持

-   **バックアップ検証**:
    - **月次**: 自動リストアテストの実施（`502_site_reliability.md` §10.3参照）
    - **四半期**: 手動でのフルリストア訓練（Fire Drill Protocol）
    - **検証項目**: リストア完了、テーブル数一致、行数整合性、アプリ起動確認

-   **バックアップ暗号化**:
    - 全バックアップはAES-256以上で暗号化
    - 暗号鍵はバックアップとは**別の場所**に保管（KMS推奨）
    - **暗号アジリティ**: PQC移行に備え、暗号アルゴリズムの切替可能な設計（§35参照）

-   **Cross-Reference**: `502_site_reliability.md` §7 (Fire Drill Protocol, Off-Site Backup), §10.3 (自動バックアップ検証)

## Part VIII: 演習と検証

### §16. テーブルトップ演習とGameDay

-   **Law**: 計画は演習で検証しなければ信頼できません。定期的なテーブルトップ演習とGameDayを実施し、対応能力を継続的に向上させてください。

-   **テーブルトップ演習（Tabletop Exercise）**:

    | 項目 | 基準 |
    |:-----|:-----|
    | **頻度** | 四半期に1回以上 |
    | **参加者** | インシデントレスポンスチーム + 経営陣 |
    | **シナリオ** | §4リスクアセスメントの高リスク項目から選定 |
    | **成果物** | 発見された課題、改善アクションリスト |
    | **記録** | 演習の全過程を記録し、次回の改善に活用 |

-   **推奨シナリオ**:
    - ランサムウェアによる全システム暗号化
    - クラウドプロバイダのリージョン障害
    - 内部不正によるデータ漏洩
    - サプライチェーン攻撃（依存パッケージの侵害）
    - DDoS攻撃 + ランサムウェアの同時発生（マルチベクター攻撃）
    - AI APIの暴走/コスト爆発
    - パンデミックによる全社リモート移行
    - ゼロデイ脆弱性の公開（自社利用OSSの重大脆弱性）
    - 規制当局への報告訓練（DORA/NIS2報告タイムライン遵守）

-   **GameDay（カオスエンジニアリング統合）**:
    - **ステージング環境**での意図的な障害注入
    - ChaosMesh/Gremlin/Litmus等のツール活用
    - 予告あり → 予告なし → 本番環境（成熟度に応じて段階的に拡大）
    - **成功基準**: 自動復旧が機能すること、RTO以内に復旧できること

-   **演習後の改善サイクル**:
    1.  演習結果のレビュー（発見された課題の一覧化）
    2.  改善アクションの起票（担当者、期限を明記）
    3.  次回演習前に改善の完了を確認
    4.  改善内容のランブック・BCPへの反映

-   **Cross-Reference**: `502_site_reliability.md` §1 (カオスエンジニアリング), `700_qa_testing.md` Part VIII (レジリエンステスト), §38 (高度演習プログラム)

### §17. ランブックフレームワーク

-   **Law**: 主要な障害シナリオに対する具体的な手順書（ランブック）を**事前に**作成してください。ランブックがないインシデント対応は「手探りの消火活動」であり、復旧時間を著しく長期化させます。
-   **ランブックテンプレート**:

    | ステップ | 内容 | 所要時間目安 |
    |:--------|:-----|:-----------:|
    | **1. 確認** | 障害発生の事実確認（ステータスページ・監視ツール） | 5分 |
    | **2. 診断** | 原因の切り分け（メトリクス・ログの確認） | 15分 |
    | **3. 宣言** | インシデント宣言・War Room設置判断 | 5分 |
    | **4. 緩和** | 影響範囲の縮小（設定変更・リソース拡張・Kill Switch） | 15分 |
    | **5. 代替** | フォールバック手段への切替（冗長化・代替サービス） | 30分 |
    | **6. 通知** | ステータスページ・ユーザーへの通知 | 10分 |
    | **7. 復旧** | 正常状態への復元 | 可変 |
    | **8. 検証** | 正常性の確認テスト | 15分 |
    | **9. 記録** | Post-Mortem作成の開始 | — |

-   **必須ランブック（最低限のカバレッジ）**:

    | シナリオ | 主要対応 |
    |:--------|:--------|
    | **Database障害** | 接続プール枯渇対応、レプリカ切替、バックアップリストア |
    | **CDN/メディア配信障害** | キャッシュパージ、直接配信切替、代替CDN |
    | **決済システム障害** | メンテナンスバナー表示、未処理トランザクション整合 |
    | **認証基盤障害** | セッション管理フォールバック、緊急認証バイパス |
    | **クラウドリージョン障害** | マルチリージョンフェイルオーバー手順 |
    | **ランサムウェア感染** | §7.1プレイブック参照 |
    | **データ侵害** | §7.2プロトコル参照 |
    | **DDoS攻撃** | §7.3対応手順参照 |
    | **AI API障害/暴走** | Kill Switch発動、代替モデル切替 |
    | **ゼロデイ脆弱性** | §7.5対応手順参照 |
    | **規制報告** | §34報告タイムライン対応手順 |

-   **ランブック品質基準**:
    - **Executable（実行可能）**: 手順に曖昧さがなく、初見のエンジニアでも実行可能
    - **Tested（テスト済み）**: テーブルトップ演習またはGameDayで検証済み
    - **Current（最新）**: 四半期ごとにレビューし、アーキテクチャ変更を反映
    - **Accessible（アクセス可能）**: 本番障害時にアクセスできる場所（本番インフラに依存しない場所）に保管
    - **Versioned（バージョン管理）**: Git管理下に置き、変更履歴を追跡可能に

-   **Cross-Reference**: §5 (インシデント対応), §13 (DR戦略), §16 (テーブルトップ演習)

## Part IX: ポストインシデント

### §18. ポストモーテムと継続的改善

-   **Law（Blameless Post-Mortem Protocol）**: インシデント発生時、個人の責任を追及することを**厳禁**とします。「なぜ彼が間違えたか」ではなく「なぜシステムが彼に間違いを許したか」を問います。

-   **ポストモーテム実施基準**:

    | レベル | Post-Mortem | 期限 |
    |:------|:-----------|:-----|
    | **SEV-1** | 必須（全参加者） | 48時間以内 |
    | **SEV-2** | 必須（関係者） | 72時間以内 |
    | **SEV-3** | 推奨 | 1週間以内 |
    | **SEV-4** | 任意 | — |

-   **5 Whys（根本原因分析）**:
    - 「なぜ？」を5回繰り返し、真のRoot Causeに到達する
    - **「ヒューマンエラー」は原因ではなく、調査の出発点** に過ぎない
    - Root Causeは常に「システム・プロセス・ツールの欠陥」でなければならない

-   **ポストモーテム必須セクション**:

    | セクション | 内容 |
    |:---------|:-----|
    | **概要** | 何が起きたか（1-2文） |
    | **タイムライン** | 発生→検知→初動→解決の時系列（タイムスタンプ付き） |
    | **影響範囲** | 影響ユーザー数、影響時間、データ損失の有無 |
    | **根本原因** | 5 Whysの結果（コード変更、設定ミス、外部依存等） |
    | **検知の評価** | 検知方法と所要時間の分析（TTD: Time to Detect） |
    | **対応の評価** | 対応の適切性と所要時間の分析（TTM: Time to Mitigate） |
    | **改善アクション** | 再発防止策（具体的なタスク、担当者、期限） |
    | **教訓** | プロジェクト教訓ログに追記すべき内容 |
    | **規制対応の振り返り** | 規制報告義務の遵守状況（該当する場合） |

-   **完了条件（Systemic Fix Mandate）**:
    - ポストモーテムの完了条件は、再発を**物理的に防ぐ**ための以下のいずれかがPRとして提出されること:
      - **新しい自動テスト**の追加
      - **Lintルール**の追加
      - **監視アラート**の新設/閾値調整
      - **アーキテクチャ変更**の実施
    - **「気をつける」「マニュアルを修正する」は対策として認めない**

-   **メトリクス追跡**:

    | メトリクス | 定義 | 目標 |
    |:---------|:-----|:-----|
    | **MTTD（Mean Time to Detect）** | 障害発生から検知までの平均時間 | ≦ 5分 |
    | **MTTA（Mean Time to Acknowledge）** | 検知から初動までの平均時間 | ≦ 15分 |
    | **MTTR（Mean Time to Resolve）** | 障害発生から復旧までの平均時間 | ≦ 1時間（SEV-1） |
    | **MTBF（Mean Time Between Failures）** | 障害間の平均時間 | 増加傾向を維持 |
    | **Recurrence Rate** | 同一Root Causeの再発率 | 0%（再発ゼロ） |

-   **Cross-Reference**: `502_site_reliability.md` §5 (インシデント管理)

## Part X: ガバナンスと成熟度

### §19. 危機管理ガバナンス

-   **Law**: 危機管理は技術チームだけの責務ではなく、組織全体のガバナンスフレームワークとして運用してください。
-   **RACI マトリクス**:

    | 活動 | CTO/VP | Engineering Manager | SRE/DevOps | Product | Legal |
    |:-----|:------:|:------------------:|:---------:|:-------:|:-----:|
    | BIA策定 | A | R | C | C | I |
    | BCP策定/更新 | A | R | R | C | C |
    | インシデント対応 | I | A | R | I | C |
    | ポストモーテム | I | A | R | I | I |
    | 演習計画/実施 | A | R | R | I | I |
    | コンプライアンス | I | C | C | I | A/R |
    | 規制報告 | I | C | R | I | A |

    **凡例**: R=実行責任, A=説明責任, C=協議, I=情報共有

-   **予算と投資**:
    - 危機管理に関する年間予算を明確に確保
    - ツール（監視、バックアップ、DRaaS等）の投資はコスト回避の観点から正当化
    - **ROI計算**: 過去のインシデントのコスト（ダウンタイムコスト + レピュテーション影響）と対策コストの比較

-   **KPI**:

    | KPI | 目標 | レビュー頻度 |
    |:----|:-----|:-----------:|
    | BCP更新頻度 | 年1回以上 | 年次 |
    | テーブルトップ演習実施回数 | 年4回以上 | 四半期 |
    | ランブック網羅率 | 高リスクシナリオ100% | 四半期 |
    | MTTR（SEV-1） | ≦ 1時間 | 月次 |
    | MTTD | ≦ 5分 | 月次 |
    | ポストモーテム完了率 | SEV-1/2は100% | 月次 |
    | 改善アクション完了率 | 90%以上 | 月次 |
    | 規制報告期限遵守率 | 100% | 月次 |

-   **コンプライアンス要件**:
    - **ISO 22301**: BCMSの認証取得を検討（規模に応じて）
    - **SOC 2 Type II**: セキュリティ・可用性コントロールの監査証拠としてBCP/DRを含める
    - **GDPR/Global Privacy Laws**: データ侵害通知の法定期限の遵守（§7.2参照）
    - **EU DORA**: 金融セクター向けICTレジリエンス要件（§21参照）
    - **EU NIS2**: 重要・重要エンティティ向けセキュリティ要件（§32参照）
    - **EU CRA**: デジタル要素を含む製品の脆弱性報告（§33参照）

-   **CISO・経営層の法的責任（Management Liability）**:
    - **NIS2 Management Liability**: NIS2指令はサイバーセキュリティ措置に関する**経営層の個人責任**を明文化。怠慢が認定された場合、取締役個人の責任が問われる可能性
    - **SEC開示規則（米国）**: 重大サイバーインシデントは**4営業日以内**にForm 8-Kで開示義務（2023年12月施行）
    - **J-SOX（内部統制報告制度）**: IT全般統制の一部としてインシデント対応体制の整備が求められる
    - **取締役会への定期報告**: サイバーリスクの状況を四半期ごとに取締役会に報告するフレームワーク（§48参照）
    - **D&O保険**: サイバーインシデントに関連する取締役・役員責任保険の補償範囲の確認

-   **Cross-Reference**: §48 (CISO・経営層責任と取締役会報告)

### §20. 成熟度モデルと定期レビュー

-   **危機管理成熟度モデル（5段階）**:

    | レベル | 名称 | 特徴 |
    |:------|:-----|:-----|
    | **L1: 初期** | Ad Hoc | BCPなし。インシデント対応は場当たり的 |
    | **L2: 反復可能** | Reactive | 基本的なインシデント対応フローと連絡網が存在 |
    | **L3: 定義済み** | Proactive | BIA完了、BCP/DR文書化、ランブック整備、年1回以上の演習 |
    | **L4: 管理** | Managed | 定量メトリクス（MTTD/MTTR）で管理、四半期演習、ポストモーテム文化定着 |
    | **L5: 最適化** | Resilient | カオスエンジニアリング本番実施、サプライチェーンBCP統合、自動復旧確立、AIOps統合、規制報告自動化 |

    **Target**: 全プロジェクトは**L3以上**を最低基準とし、**L4**を目標としてください。

-   **定期レビュースケジュール**:

    | 頻度 | 対象 |
    |:-----|:-----|
    | **月次** | インシデントメトリクスのレビュー（MTTD/MTTR/再発率） |
    | **四半期** | リスクアセスメント更新、ランブックレビュー、テーブルトップ演習 |
    | **半年** | 緊急連絡網テスト、バックアップリストア訓練 |
    | **年次** | BIA更新、BCP全面レビュー、委託先BCP確認、規制コンプライアンス監査 |

-   **20大アンチパターン**:
    1.  ❌ BCPが存在しない / 作成後一度も更新されていない
    2.  ❌ テーブルトップ演習を一度も実施したことがない
    3.  ❌ バックアップを取得しているがリストアテストをしたことがない
    4.  ❌ ランブックが存在しない / 内容が古く実行不能
    5.  ❌ キーマンリスクが排除されていない（Bus Factor = 1）
    6.  ❌ ポストモーテムで個人の責任を追及している
    7.  ❌ Kill Switchが未実装（AI/決済の緊急停止手段がない）
    8.  ❌ 委託先障害時の対応フローが未定義
    9.  ❌ データ侵害時の法的通知手順が未確立
    10. ❌ インシデントメトリクス（MTTD/MTTR）を計測していない
    11. ❌ インシデント対応の自動化が一切なく手作業のみ
    12. ❌ 気候変動・自然災害リスクをBCPに含めていない
    13. ❌ サイバー保険に加入していない / 補償範囲を把握していない
    14. ❌ インシデント対応者のメンタルヘルスケアが未整備
    15. ❌ DORA/NIS2等の規制要件を危機管理フレームワークに統合していない
    16. ❌ ゼロデイ脆弱性対応プロトコルが未定義
    17. ❌ 規制報告テンプレートが事前準備されていない
    18. ❌ 量子コンピューティング脅威（HNDL攻撃）への備えがない
    19. ❌ OT/ICSセキュリティがBCPに含まれていない（該当業種）
    20. ❌ デジタルフォレンジクスのChain of Custodyが未確立

## Part XI: DORA/規制コンプライアンス統合

### §21. DORA（デジタルオペレーショナルレジリエンス法）統合

-   **Law**: EU金融セクターに関連するサービスを提供する場合、**DORA（Regulation (EU) 2022/2554）**に準拠したICTリスク管理フレームワークを実装しなければなりません。DORA は2025年1月17日に施行されました。

-   **DORA 5本柱と本ルールとの対応**:

    | DORA柱 | 要件概要 | 本ルール対応 |
    |:------|:--------|:-----------|
    | **ICTリスク管理** | 包括的なICTリスク管理フレームワーク | §4 リスクアセスメント, §19 ガバナンス |
    | **ICTインシデント報告** | 重大ICTインシデントの規制当局への報告 | §7.2 法的通知, §10 ステークホルダー通知 |
    | **デジタルオペレーショナルレジリエンステスト** | TLPT（脅威ベースペネトレーションテスト）含む | §16 テーブルトップ演習, `700_qa_testing.md` |
    | **ICTサードパーティリスク管理** | 重要ICT委託先の監督 | §12 委託先BCP |
    | **情報共有** | サイバー脅威情報の共有メカニズム | §18 ポストモーテム |

-   **ICTインシデント報告タイムライン（DORA準拠）**:

    | 報告種別 | 期限 | 内容 |
    |:--------|:-----|:-----|
    | **初期通知** | 重大インシデント分類後 **4時間以内** | インシデントの概要、影響範囲 |
    | **中間報告** | 初期通知後 **72時間以内** | 根本原因の分析、対応状況 |
    | **最終報告** | インシデント解決後 **1ヶ月以内** | 完全な分析、再発防止策、教訓 |

-   **TLPT（Threat-Led Penetration Testing）**:
    - 3年ごとに実施（規制要件）
    - TIBER-EUフレームワークに準拠
    - 独立した第三者テスト機関による実施
    - 結果を規制当局に報告

-   **ICTサードパーティ契約要件**:
    - 全重要ICT委託先との契約にDORA必須条項を含める
    - **監査権**: 委託先への直接監査アクセス権の確保
    - **出口戦略**: 委託先からの移行計画の事前策定
    - **データ保護**: データの所在地・処理場所の明確化
    - **インシデント報告**: 委託先からの遅滞なきインシデント通知義務

-   **DORA委任規則（2025年施行）**:
    - **Delegated Regulation (EU) 2025/301**: インシデント報告の内容・タイムライン・フォーマットの詳細規定
    - **Implementing Regulation (EU) 2025/302**: インシデント報告の標準フォーム・手順の規定
    - **ESA（欧州監督当局）監督プロセス**:
        - 2025年4月30日: ICT第三者委託情報の登録簿提出期限（管轄当局→ESA）
        - 2025年7月: ESAが重要ICTサードパーティサービスプロバイダの分類完了
        - 以降: ESAによる重要ICTプロバイダへの直接監督開始
    - **契約更新対応**: 既存のICT委託先契約にDORA必須条項が含まれていない場合、契約更新時に修正が必須

-   **Cross-Reference**: `601_data_governance.md` (DORA規制), `600_security_privacy.md` (セキュリティテスト)

## Part XII: AI駆動インシデント管理

### §22. AIOps（AI駆動インシデント管理）

-   **Law**: AI/MLを活用したインシデント管理の高度化（AIOps）を積極的に推進してください。ただし、AIによる判断は常に人間の最終確認を経ることを原則とします。

-   **AIOps適用領域**:

    | 領域 | AI活用方法 | 人間の関与 |
    |:-----|:---------|:---------|
    | **異常検知** | MLベースの動的ベースライン + 異常スコアリング | アラート確認・トリアージ判断 |
    | **予測分析** | 障害発生の事前予測（パターン分析・トレンド検知） | 予防措置の承認・実行判断 |
    | **自動トリアージ** | インシデントの自動分類・優先度付け・ルーティング | SEV-1/2の最終確認 |
    | **根本原因提案** | 過去インシデントDBからの類似事例・原因候補の提示 | 原因の確定判断 |
    | **自動修復** | 既知パターンの自動復旧（リスタート、スケーリング等） | 復旧結果の検証 |
    | **ポストモーテム支援** | タイムラインの自動生成、影響範囲の自動算出 | レビュー・補完 |
    | **規制報告支援** | 報告書テンプレートの自動入力、期限管理 | 最終確認・承認 |

-   **AIOps導入の3段階**:
    1.  **Phase 1（支援）**: AIがアラートノイズを削減し、関連アラートをグルーピング
    2.  **Phase 2（推奨）**: AIが対応手順を推奨し、ランブックへの自動リンクを提供
    3.  **Phase 3（自律）**: 低リスクパターンの自動修復（Guardrail付き）

-   **Guardrail（安全弁）**:
    - **自動修復の許可範囲**: SEV-3/4の既知パターンのみ自動修復を許可
    - **SEV-1/2禁止**: 重大インシデントの自動対応は**禁止**（人間の判断必須）
    - **ロールバック**: 自動修復が失敗した場合の即座のロールバック機構
    - **監査ログ**: AIによる全判断・アクションの完全な監査ログ記録
    - **説明可能性**: AIの判断根拠をオペレータが確認可能であること

-   **Agentic AI・AI SOC Co-pilot（2025-2026トレンド）**:
    - **AI SOC Co-pilot**: SOCアナリストを支援するAIコパイロットが主流化。大量データからの脅威優先度付け、修復提案の自動生成
    - **Agentic AI**: 特定セキュリティタスクにおいて自律的に行動するAIシステムの台頭（人間の最小限の介入で意思決定・緩和アクションを実行）
    - **Self-Healing Systems**: Gartner予測（2026年までに大企業の60%以上がAIOps形の自己修復システムを採用）
    - **導入の原則**: Agentic AIは「支援 → 推奨 → 自律」の段階を厳守し、各段階で十分な安全性検証を経てから次の段階に進むこと

-   **SOAR（Security Orchestration, Automation and Response）統合**:
    - **定義**: SOARをAIOpsのオーケストレーション層として統合し、検知→トリアージ→対応→復旧の自動ワークフローを構築
    - **SOAR Playbook設計原則**:
        - 検知ルールとPlaybookの1:1マッピング（アラートに対応するPlaybookが存在しないアラートは排除候補）
        - 各Playbookにhuman-in-the-loopのゲートを設置（完全自動化はSEV-3/4の既知パターンのみ）
        - Playbookのバージョン管理とテスト（CI/CDパイプラインでのPlaybook検証）
    - **SOAR活用メトリクス**:

        | メトリクス | 定義 | 目標 |
        |:---------|:-----|:-----|
        | **Playbook自動化率** | Playbook対応インシデント / 全インシデント | ≧ 60% |
        | **MTTR削減率** | SOAR導入前後のMTTR比較 | ≧ 50%削減 |
        | **False Positive削減率** | SOAR導入前後の誤検知率比較 | ≧ 70%削減 |
        | **アナリスト生産性** | アナリスト1名あたりの処理インシデント数 | 増加傾向 |

    - **AI生成インシデントレスポンス**: AI SOC Co-pilotによるリアルタイムの対応推奨:
        - 類似インシデントの過去事例検索と最適な対応手順の提案
        - 規制報告該当性の自動判定と報告書ドラフトの自動生成
        - ポストモーテムのタイムライン自動構築

-   **Alert Fatigue（アラート疲労）対策**:
    - **問題**: 大量のアラートがオペレータの注意力を分散させ、重要なアラートの見落としにつながる
    - **AIによるノイズ削減**: MLベースのアラート相関・グルーピングによるノイズ90%削減を目標
    - **メトリクス**: アラート削減率、誤検知率、アラート→インシデント変換率の定期レビュー
    - **詳細**: §45（インシデント疲労・アラート最適化）参照

-   **Cross-Reference**: `400_ai_engineering.md` (AI実装), `502_site_reliability.md` (可観測性), §45 (Alert Fatigue), §49 (Self-Healing)

## Part XIII: 気候レジリエンス

### §23. 気候レジリエンスとサステナビリティ

-   **Law**: 気候変動リスクをBCPと危機管理フレームワークに統合してください。
-   **気候リスク評価**:
    - データセンター/クラウドリージョンの所在地における気候リスク評価
    - 洪水、猛暑、暴風雨等の極端気象イベントの発生確率と影響度
    - データセンター冷却能力のヒートウェーブ耐性評価
-   **対策**:
    - 気候リスクの高い地域を避けたDR環境の配置
    - 電力供給の冗長化（UPS、発電機、再生可能エネルギー源の活用）
    - グリーンITの推進とエネルギー効率の最適化
    - 年次での気候リスクシナリオに基づくBCPレビュー

## Part XIV: 地政学リスク

### §24. 地政学リスクとソブリンクラウド

-   **Law**: 地政学的なリスクが事業継続に与える影響を評価し、データ主権要件に対応してください。
-   **地政学リスク評価**:
    - サービス提供国・クラウドリージョン所在国の政治的安定性評価
    - 経済制裁、輸出規制がサービスに与える影響のシナリオ分析
    - データローカライゼーション規制への対応（GDPR、中国CSL、ロシアPDL等）
-   **ソブリンクラウド戦略**:
    - データ主権要件に基づくクラウドプロバイダ・リージョンの選択
    - マルチクラウド戦略によるベンダーリスク・地政学リスクの分散
    - EU域内データ保持要件への対応（EU Sovereign Cloud Initiative）
-   **サプライチェーン分断対策**:
    - 半導体・ハードウェアサプライチェーンの代替ルート評価
    - ソフトウェアライセンスの地政学的リスク評価（制裁対象国の企業製品）
-   **Cross-Reference**: `800_internationalization.md` (データ主権), `601_data_governance.md`

## Part XV: パンデミックBCP

### §25. パンデミックBCP

-   **Law**: 感染症パンデミックによる長期的な事業継続リスクに対応するBCPを策定してください。
-   **パンデミック固有のBCP要素**:
    - 全従業員の完全リモートワークへの移行計画（72時間以内）
    - オフィス閉鎖時の物理セキュリティ維持計画
    - 同時欠勤率50%以上を想定した最低限の運用体制
    - メンタルヘルスサポートの強化（§29参照）
-   **テクノロジー準備**:
    - VPN/ゼロトラスト接続の全社同時接続テスト
    - オンラインコラボレーションツールの冗長化
    - ペーパーレス化の推進とデジタルワークフローの確立
-   **段階的対応計画**:

    | フェーズ | トリガー | アクション |
    |:--------|:--------|:---------|
    | **監視** | WHO警告/国内感染拡大初期 | リモートワーク準備、BCP確認 |
    | **部分移行** | 地域的感染拡大 | 該当地域のリモート化、出社制限 |
    | **全面移行** | パンデミック宣言/緊急事態宣言 | 全社リモート、オフィス閉鎖 |
    | **復帰** | 感染収束 | 段階的なオフィス再開 |

## Part XVI: インシデント自動化

### §26. インシデント自動化とChatOps

-   **Law**: インシデント対応の反復的な作業を自動化し、人間はより高度な判断に集中できる環境を構築してください。
-   **自動化対象**:

    | 領域 | 自動化例 | ツール例 |
    |:-----|:--------|:--------|
    | **検知** | 異常検知→自動アラート→Slackチャネル作成 | PagerDuty, OpsGenie, Datadog |
    | **トリアージ** | アラート分類→重大度判定→on-call呼び出し | PagerDuty Intelligent Alert Grouping |
    | **初動** | 診断情報の自動収集（メトリクス、ログ、デプロイ履歴） | Slack Bot, Rundeck |
    | **通知** | ステータスページ自動更新、ステークホルダー通知 | Statuspage API, FireHydrant |
    | **復旧** | 既知パターンの自動修復（ロールバック、スケーリング） | Kubernetes HPA, Lambda |
    | **記録** | タイムラインの自動記録、ポストモーテムテンプレート生成 | FireHydrant, Rootly |
    | **規制報告** | 報告書テンプレート自動生成、期限カウントダウン | カスタムツール |

-   **ChatOps（チャットベースのインシデント対応）**:
    - Slackコマンドでの即時アクション実行（ロールバック、Kill Switch等）
    - インシデントチャネルでの全対応ログの自動記録
    - ボットによる定期的なステータスリマインダー

-   **自動化の段階的導入**:
    1.  **Phase 1**: 通知・記録の自動化（低リスク）
    2.  **Phase 2**: 診断情報の自動収集（中リスク）
    3.  **Phase 3**: 既知パターンの自動修復（高リスク — Guardrail必須）

## Part XVII: サイバー保険

### §27. サイバー保険と財務レジリエンス

-   **Law**: サイバーインシデントの財務的影響を軽減するため、サイバー保険の加入と適切な補償範囲の確保を検討してください。
-   **2025-2026サイバー保険市場動向**:
    - **市場規模**: 2026年までにグローバル保険料は$16.4B〜$23Bに到達見込み
    - **リスクベースセグメンテーション**: 強固なサイバー防御を証明できる組織には有利な保険条件を提供する傾向（MFA導入、EDR/XDR運用、定期的なペネトレーションテスト実施が保険料割引の要件に）
    - **AI関連リスク**: AI利用に関連するインシデント（プロンプトインジェクション、AI生成コンテンツによる侵害等）の補償範囲が明確化されつつある
    - **システミックリスクの再評価**: クラウドアウテージ、サプライチェーン攻撃等の大規模・同時多発型インシデントに対する保険引受の厳格化
    - **NHI（非人間アイデンティティ）関連の除外強化**: APIキー・サービスアカウント侵害に起因するインシデントは、NHI管理の未整備を理由に補償除外されるケースが増加（§7.7参照）
-   **補償範囲のチェックリスト**:
    - ランサムウェア攻撃（身代金は含まない場合が多い点に注意）
    - データ侵害対応費用（法務、フォレンジック、通知費用）
    - 事業中断損失
    - 規制対応費用（罰金は通常含まない）
    - 第三者損害賠償
    - レピュテーション回復費用
    - サプライチェーン障害による間接損失
    - **AI関連インシデント費用**（新設カテゴリ: ポリシーによる）
-   **保険請求プロセスの事前準備**:
    - 保険会社指定のフォレンジック業者リストの確認
    - インシデント発生時の保険会社通知手順の文書化
    - 保険適用に必要なセキュリティ要件（MFA、EDR等）の維持
    - **証拠保全の徹底**: 保険請求に必要な証拠（ログ、タイムライン、影響評価）の保全手順の確認

-   **IRリテーナー契約（事前契約）**:
    - インシデント対応企業（IRベンダー）とのリテーナー契約の事前締結を推奨
    - リテーナー契約により、インシデント発生時の対応開始時間が大幅に短縮
    - 保険会社が指定するIRベンダーと自社選定のIRベンダーの関係を事前に整理
    - 詳細: §47（Ransomware Negotiation・IR Retainer）参照

-   **補償除外事項の確認**:
    - **一般的に除外される項目**: 戦争行為、国家支援攻撃（「戦争除外」条項）、既知の脆弱性の放置、内部不正
    - **規制罰金**: 通常は補償対象外（GDPR罰金等）
    - **身代金支払い**: 多くの保険ではランサム支払いは補償対象外
    - **NHI管理不備**: シークレットのハードコード、無期限トークン、ゾンビアカウント等のNHI管理不備に起因するインシデントは補償除外リスクが高い
    - **未パッチ脆弱性**: 公知の脆弱性を一定期間内にパッチ適用しなかった場合の除外条項の確認
    - 年次での保険契約内容の見直しと、セキュリティ体制の保険要件充足確認

-   **Cross-Reference**: §7.7 (NHIセキュリティ), §31 (危機管理FinOps), §47 (IR Retainer)

## Part XVIII: レピュテーション回復

### §28. レピュテーション回復計画

-   **Law**: 重大インシデント後のレピュテーション回復は、技術的復旧と同等の重要性を持ちます。
-   **回復フェーズ**:
    1.  **即時対応（24-72時間）**: 透明性のある初期コミュニケーション、影響を受けたユーザーへのプロアクティブな支援
    2.  **短期回復（1-4週間）**: 第三者監査の依頼、改善措置の進捗報告、ステークホルダーへの定期更新
    3.  **長期回復（1-6ヶ月）**: セキュリティ改善策の公開、業界ベストプラクティスの共有による信頼回復、継続的な改善報告
-   **回復メトリクス**:
    - ユーザーチャーン率の監視（インシデント前後の比較）
    - NPS（Net Promoter Score）の回復度
    - メディアセンチメント分析
    - ブランド検索トレンドの回復

## Part XIX: 人的要因

### §29. 人的要因とメンタルヘルス

-   **Law**: インシデント対応に関わる人員のメンタルヘルスと持続可能な働き方を確保してください。Burnoutした対応者は判断を誤り、インシデントを悪化させます。
-   **On-Call負荷管理**:
    - on-callローテーションの適切な設計（連続on-call期間の上限設定）
    - on-call中の睡眠妨害に対する補償（代休、手当）
    - on-call中のSEV-1対応後は最低8時間のクールダウンタイムを確保
-   **インシデント後のケア**:
    - 重大インシデント対応後のデブリーフィング（心理的安全性を確保した振り返り）
    - 必要に応じた外部カウンセリングリソースの提供
    - ポストモーテムでの Blameless 文化の厳格な維持
-   **持続可能なインシデント対応体制**:
    - 特定個人への依存排除（Bus Factor ≧ 2）
    - インシデント対応スキルの全チームへの分散教育
    - 定期的なストレスチェックの実施

## Part XX: 物理セキュリティ

### §30. 物理セキュリティとBCP

-   **Law**: デジタルセキュリティだけでなく、物理的なセキュリティもBCPに含めてください。
-   **物理セキュリティの考慮事項**:
    - オフィス/データセンターの入退管理とアクセスログ
    - 機密情報を含むデバイス（ノートPC、USBデバイス等）の管理と暗号化
    - クリーンデスクポリシーの実施
    - 災害時の物理メディア（バックアップテープ等）の保護
-   **リモートワーク環境のセキュリティ**:
    - 在宅勤務時の物理セキュリティ基準（画面ロック、プライバシーフィルター）
    - 機密情報の印刷・持ち出しルール
    - 個人デバイスの業務利用ポリシー（BYOD）

## Part XXI: 危機管理FinOps

### §31. 危機管理FinOps

-   **Law**: 危機管理活動のコストを可視化し、投資対効果を定量的に評価してください。
-   **コスト可視化**:
    - DR環境の運用コスト
    - バックアップストレージコスト
    - 監視・アラートツールのライセンスコスト
    - 演習・訓練にかかる人件費
    - サイバー保険の保険料
-   **コスト最適化戦略**:
    - DR環境のスケーリング（Pilot Light/Warm Standbyによるコスト削減）
    - バックアップの世代管理と自動ライフサイクル管理
    - 監視ツールの統合とアラートノイズ削減によるオペレーションコスト削減
-   **インシデントコスト追跡**:
    - インシデントごとの直接コスト（ダウンタイム、復旧作業、フォレンジック）
    - 間接コスト（レピュテーション損失、ユーザーチャーン、規制対応）
    - **コスト回避の定量化**: 予防措置によって回避できたインシデントのコスト試算
-   **DDoS攻撃時のコスト保護**:
    - オートスケーリングの上限設定（コスト爆発防止）
    - クラウドプロバイダのDDoS保護サービスの活用（AWS Shield Advanced等のコスト保護）
    - 異常コスト検知アラートの設定
-   **Cross-Reference**: `502_site_reliability.md` §9 (FinOps), `361_aws_cloud.md` (コスト管理)

## Part XXII: NIS2指令統合

### §32. NIS2（ネットワーク・情報セキュリティ指令2）統合

-   **Law**: EU域内で重要サービスを提供するか、テクノロジーサプライチェーンにおいて重要な役割を果たす場合、**NIS2指令（Directive (EU) 2022/2555）**に準拠したサイバーセキュリティ要件を実装しなければなりません。NIS2は2024年10月17日の国内法化期限を経て、各加盟国で順次施行中です。

-   **NIS2主要要件と本ルールとの対応**:

    | NIS2要件 | 内容 | 本ルール対応 |
    |:--------|:-----|:-----------|
    | **リスク管理措置** | 技術的・運用的・組織的セキュリティ措置 | §4 リスクアセスメント, §19 ガバナンス |
    | **インシデント報告** | 重大インシデントの段階的報告義務 | §34 インシデント報告国際調和 |
    | **サプライチェーンセキュリティ** | ICTサプライチェーンのセキュリティ評価 | §12 委託先BCP |
    | **事業継続管理** | BCM、バックアップ管理、災害復旧 | §11 BCP, §13-§15 DR |
    | **暗号・データ保護** | 暗号化ポリシー、データ完全性保護 | §15 データ保護, §35 PQC |
    | **アセット管理** | ICTアセットの識別・分類・管理 | `600_security_privacy.md` |

-   **NIS2インシデント報告タイムライン**:

    | 報告段階 | 期限 | 内容 |
    |:--------|:-----|:-----|
    | **早期警告（Early Warning）** | 重大インシデント認知後 **24時間以内** | 悪意ある行為の疑い有無、越境影響の有無 |
    | **インシデント通知** | 認知後 **72時間以内** | 重大度・影響の評価、侵害指標（IoC） |
    | **最終報告** | 初期通知後 **1ヶ月以内** | 完全な分析、根本原因、是正措置 |
    | **経過報告** | 未解決の場合 | 進捗状況の定期報告 |

-   **NIS2「重大インシデント」の定義**:
    - サービスに深刻な運用上の途絶を引き起こす、または引き起こし得るもの
    - 重大な財務損失を引き起こすもの
    - 他の自然人・法人に多大な物質的・非物質的損害を及ぼすもの

-   **OTセキュリティ要件（NIS2拡張スコープ）**:
    - NIS2はエネルギー、製造、輸送、医療等のOT依存セクターを明示的にスコープに含む
    - IT/OTネットワークの分離と監視
    - 産業用プロトコルの監視（Modbus、OPC UA等）
    - レガシーシステムの安全なバックアップ
    - 生産ダウンタイムを回避する脆弱性管理
    - §36（OT/ICS危機管理）と連携

-   **EU-CyCLONe（European Cyber Crisis Liaison Organisation Network）**:
    - 大規模インシデント時の加盟国間の連携と情報共有
    - 各加盟国のサイバー危機管理当局との連携体制構築

-   **Cross-Reference**: §34 (インシデント報告国際調和), §36 (OT/ICS), `600_security_privacy.md`, `601_data_governance.md`

## Part XXIII: EU CRA統合

### §33. EU CRA（サイバーレジリエンス法）統合

-   **Law**: デジタル要素を含む製品（ソフトウェア、ハードウェア、IoTデバイス等）を製造または販売する場合、**EU CRA（Regulation (EU) 2024/2847）**に準拠した脆弱性管理と報告体制を実装しなければなりません。CRAは2024年12月10日に発効し、**2026年9月11日から報告義務が適用**されます。

-   **CRA脆弱性報告義務**:

    | 報告段階 | 期限 | 内容 |
    |:--------|:-----|:-----|
    | **早期警告通知** | 積極的に悪用される脆弱性の認知後 **24時間以内** | 脆弱性の概要、影響を受ける製品 |
    | **脆弱性通知** | 認知後 **72時間以内** | 製品詳細、悪用の性質、緩和策 |
    | **最終報告** | 修正措置が利用可能になった時点で **14日以内** | 完全なインシデント詳細、重大度、是正措置 |

-   **報告先**:
    - 製造者の主要加盟国の**CSIRT（Computer Security Incident Response Team）**
    - **ENISA（European Union Agency for Cybersecurity）**
    - **ENISA SRP（Single Reporting Platform）**: 2026年9月運用開始予定の統一報告プラットフォーム

-   **CRAの危機管理への影響**:
    - **セキュリティ・バイ・デザイン**: 製品のライフサイクル全体にわたるセキュリティ確保
    - **脆弱性ハンドリング義務**: 製品の脆弱性を積極的に管理・修正する義務
    - **ユーザー通知義務**: 製品のセキュリティに影響するインシデントについてユーザーへの通知
    - **SBOM（Software Bill of Materials）**: 製品のソフトウェア部品表の維持と提供
    - **暗号アジリティ**: PQC移行に備えた暗号アルゴリズム切替対応（§35参照）

-   **DORA/NIS2/CRA報告義務の重複対応**:
    - 複数規制に同時に該当する場合、§34の統合報告ワークフローを使用
    - 最も厳しい期限を基準に対応（DORA: 4h > NIS2: 24h > CRA: 24h）

-   **Cross-Reference**: §34 (インシデント報告国際調和), §35 (PQC), `602_oss_compliance.md` (SBOM)

## Part XXIV: インシデント報告国際調和

### §34. インシデント報告の国際調和

-   **Law**: 複数の規制管轄に跨るインシデント発生時、各規制の報告義務を統合的に管理し、全ての法定期限を遵守しなければなりません。

-   **規制別インシデント報告タイムライン統合比較表**:

    | 規制 | 初期通知期限 | 中間/詳細報告 | 最終報告 | 適用条件 |
    |:----|:-----------|:------------|:--------|:---------|
    | **DORA** | 4時間以内 | 72時間以内 | 1ヶ月以内 | EU金融セクター重大ICTインシデント |
    | **NIS2** | 24時間以内 | 72時間以内 | 1ヶ月以内 | EU重要/必須エンティティ重大インシデント |
    | **EU CRA** | 24時間以内 | 72時間以内 | 14日以内 | 積極的に悪用される製品脆弱性 |
    | **GDPR** | 72時間以内 | — | — | 個人データ侵害 |
    | **Global Privacy Laws** | 速やかに（速報） | — | 30日以内（確報） | 個人データ漏洩 |
    | **CCPA/CPRA** | 遅滞なく | — | — | CA州消費者データ侵害 |
    | **APRA CPS 230（豪州）** | 重大度に応じて72時間以内 | — | 合理的に実行可能な限り速やかに | 豪州金融機関の重大なビジネスサービス障害 |
    | **PIPA（韓国）** | 72時間以内 | — | — | 10,000件以上の個人情報漏洩 |
    | **LGPD（ブラジル）** | 合理的な期間内（ANPD推奨: 2営業日以内） | — | — | 個人データ侵害 |

-   **マルチ規制対応ワークフロー**:
    1.  **インシデント発生**: 重大度判定 + 影響範囲特定
    2.  **適用規制の特定**: インシデントの内容・影響範囲から適用される全規制を特定
    3.  **最短期限の特定**: 適用規制のうち最も短い初期通知期限を採用（例: DORAの4時間）
    4.  **同時報告**: 可能な限り全規制当局への同時報告を行い、個別対応by-oneを回避
    5.  **期限管理**: 各規制の中間/最終報告期限をカレンダーに登録し、遵守を確認
    6.  **報告テンプレートの活用**: 各規制固有の報告テンプレートを事前準備（規制当局が提供するフォームを使用）

-   **規制報告の自動化支援**:
    - インシデント管理ツールとの統合による報告書テンプレートの自動入力
    - 規制別期限のカウントダウンアラート
    - 報告履歴のアーカイブと監査対応

-   **EU Digital Omnibus Package（Report Once原則）**:
    - 2025年11月に欧州委員会が提案した「デジタルオムニバスパッケージ」
    - **「Report Once, Share Many」原則**: ENISAが運営する単一報告窓口（Single Entry Point）を通じて、NIS2/DORA/GDPR/CER等の複数規制への一括報告を実現
    - **施行見込み**: 採択後18-24ヶ月で適用開始（2026年後半以降の見込み）
    - **対応準備**: 現時点では既存の個別報告体制を維持しつつ、統合報告への移行に備えて報告データの構造化を推進
    - 詳細: §42（EU Digital Omnibus Package統合）参照

-   **SEC・日本開示要件**:
    - **SEC（米国）**: 重大サイバーインシデントの4営業日以内のForm 8-K開示義務（§48参照）
    - **東京証券取引所**: 上場企業の適時開示規則に基づくインシデント開示判断

-   **Cross-Reference**: §7.2 (データ侵害), §21 (DORA), §32 (NIS2), §33 (CRA), §42 (Digital Omnibus), §48 (SEC開示), `601_data_governance.md`

## Part XXV: 量子コンピューティング脅威

### §35. 量子コンピューティング脅威とPQC（耐量子暗号）

-   **Law**: 量子コンピューティングの進化に伴う暗号解読リスクを評価し、耐量子暗号（Post-Quantum Cryptography: PQC）への移行準備を計画的に進めてください。

-   **HNDL（Harvest Now, Decrypt Later）攻撃**:
    - 攻撃者が現在の暗号化データを収集し、将来の量子コンピュータで解読する脅威
    - **長期機密データ**（政府データ、医療データ、知的財産、認証クレデンシャル）は即座にリスク評価対象
    - 現在から対策を開始する必要がある（量子コンピュータが実用化されてからでは遅い）

-   **CBOM（Cryptographic Bill of Materials）**:
    - 自社製品・サービスで使用する全暗号アルゴリズムの棚卸し
    - RSA、ECDSA、DH等の量子脆弱アルゴリズムの使用箇所の特定
    - PQC移行時の迅速な切替を可能にする暗号インベントリの維持

-   **暗号アジリティ（Crypto Agility）**:
    - 暗号アルゴリズムの切り替えがアプリケーションの変更なしに可能な設計
    - TLSライブラリ、鍵管理システム、バックアップ暗号化の暗号アジリティ確保
    - ハードコードされた暗号アルゴリズム依存の排除

-   **PQC移行ロードマップ（EU推奨に基づく）**:

    | フェーズ | 期限 | アクション |
    |:--------|:-----|:---------|
    | **棚卸し** | 2025-2026年 | CBOM作成、量子脆弱暗号の特定 |
    | **計画** | 2026年末まで | PQC移行計画の策定、優先順位付け |
    | **高リスク移行** | 2030年まで | 高リスクユースケースの量子安全化 |
    | **完全移行** | 2035年まで | 量子脆弱暗号からの完全移行 |

-   **NIST PQC標準**:
    - **ML-KEM（CRYSTALS-Kyber）**: 鍵カプセル化メカニズム
    - **ML-DSA（CRYSTALS-Dilithium）**: デジタル署名
    - **SLH-DSA（SPHINCS+）**: ハッシュベースデジタル署名
    - これらの標準が確立された暗号ライブラリでサポートされ次第、移行計画に組み込む

-   **Cross-Reference**: `600_security_privacy.md` (暗号化), §15 (バックアップ暗号化)

## Part XXVI: OT/ICS/SCADA危機管理

### §36. OT/ICS/SCADA危機管理

-   **Law**: 産業制御システム（ICS）、SCADA、OT環境を運用する場合、IT環境とは異なる固有の危機管理要件を実装しなければなりません。

-   **IT/OT統合インシデント対応**:
    - IT SOCとOT SOCの統合監視体制の構築
    - OTインシデントの重大度分類（安全性への影響を最優先）
    - IT/OTネットワーク間の**物理的セグメンテーション**の維持
    - OT環境固有のインシデント対応プレイブック（稼働停止リスクを考慮）

-   **OT固有の考慮事項**:
    - **安全性最優先**: OT環境ではセキュリティよりも安全性（Safety）が常に最優先
    - **レガシーシステム**: パッチ適用が困難なレガシーICSへの代替セキュリティ措置
    - **可用性要件**: OTシステムの24/7稼働要件とメンテナンスウィンドウの制約
    - **産業プロトコル**: Modbus、OPC UA、PROFINET等の固有プロトコルの監視
    - **物理的アクセス制御**: OT制御ルームへの厳格なアクセス管理

-   **準拠フレームワーク（OT固有）**:
    - **IEC 62443**: 産業通信ネットワークセキュリティ
    - **NERC CIP**: 電力セクター重要インフラ保護
    - **NIS2 OT要件**: エネルギー・製造・輸送セクターのOTセキュリティ

-   **OT環境のDR**:
    - PLC/HMI構成のバックアップと復元手順
    - OT環境のリカバリは**安全な状態への復帰**を最優先
    - IT環境のDRとは独立したOT-DR計画の策定

-   **Cross-Reference**: §32 (NIS2 OT要件), `600_security_privacy.md`

## Part XXVII: 法的準備態勢

### §37. 法的準備態勢とデジタルフォレンジクス

-   **Law**: インシデント発生時にデジタル証拠を法的に有効な形で保全し、証拠の完全性と追跡可能性を維持する体制を事前に構築してください。

-   **Chain of Custody（証拠管理連鎖）**:
    - デジタル証拠の収集・保管・移送・分析の全過程を記録
    - 各段階の担当者、日時、方法を明確に文書化
    - 証拠の改ざんを検出するためのハッシュ値（SHA-256以上）の記録
    - 法執行機関への引き渡し手順の事前定義

-   **フォレンジック準備態勢（Forensic Readiness）**:
    - ログの十分な保持期間の確保（最低90日、規制要件に応じて延長）
    - ログの改ざん防止（イミュータブルストレージ、集中ログ管理）
    - フォレンジック用ツールキットの事前準備（ディスクイメージング、メモリダンプ等）
    - フォレンジック対応可能な人材またはリテーナー契約の確保

-   **eDiscovery対応**:
    - 訴訟・規制調査に備えた電子データの保全義務（リーガルホールド）
    - データ保全の範囲と手順の事前定義
    - クラウド環境でのデータ保全方法の確立（プロバイダとの協力体制）

-   **クロスボーダー捜査対応**:
    - 国際的な法的管轄の考慮（MLAT、CLOUD Act等）
    - 異なる法域でのデータ提供要件の把握
    - 国際法執行機関との連携手順

-   **Cross-Reference**: §6 (証拠保全), §7 (サイバーインシデント), `601_data_governance.md`

## Part XXVIII: 高度演習プログラム

### §38. 高度演習プログラム

-   **Law**: テーブルトップ演習（§16）に加え、より高度で実践的な演習プログラムを段階的に導入し、組織のレジリエンスを継続的に向上させてください。

-   **Purple Team演習**:
    - **Red Team（攻撃チーム）** と **Blue Team（防御チーム）** が協力して実施
    - 攻撃手法とその検知・対応能力のギャップを特定
    - 四半期に1回の実施を推奨（成熟度L3以上）
    - 結果をランブック・監視ルールの改善に直結

-   **TLPT（Threat-Led Penetration Testing）詳細**:
    - TIBER-EU/TIBER-JPフレームワークに基づく脅威ベースのテスト
    - **Threat Intelligence Phase**: 脅威情報の収集と攻撃シナリオの策定
    - **Red Team Phase**: 実際の攻撃シミュレーション（通知された管理者限定）
    - **Blue Team Assessment**: 防御チームの検知・対応能力の評価
    - **Remediation Phase**: 発見された脆弱性と検知ギャップの是正
    - DORA対象組織は3年ごとの実施が義務

-   **シナリオ設計方法論**:
    - MITRE ATT&CKフレームワークに基づく攻撃シナリオの体系的設計
    - 脅威インテリジェンスに基づく現実的な攻撃シナリオの選定
    - マルチベクター攻撃シナリオ（例: DDoS + ランサムウェア + ソーシャルエンジニアリング）
    - 規制報告を含む統合シナリオ

-   **演習メトリクス**:

    | メトリクス | 測定対象 | 改善目標 |
    |:---------|:--------|:--------|
    | **検知率** | 注入された障害のうち検知できた割合 | 100% |
    | **検知時間** | 障害注入から検知までの時間 | ≦ MTTD目標 |
    | **対応時間** | 検知から初動までの時間 | ≦ MTTA目標 |
    | **復旧時間** | 障害注入からRTO以内に復旧できた割合 | 100% |
    | **コミュニケーション精度** | 通知内容の正確性・適時性 | 100% |
    | **規制報告遵守** | 演習内での規制報告期限遵守率 | 100% |

-   **Cross-Reference**: §16 (テーブルトップ演習), §21 (DORA TLPT), `700_qa_testing.md`

## Part XXIX: マルチクラウドDR

### §39. マルチクラウド・ハイブリッドDR

-   **Law**: 単一クラウドプロバイダへの依存リスクを評価し、必要に応じてマルチクラウドまたはハイブリッドクラウドのDR戦略を策定してください。

-   **マルチクラウドDR戦略**:
    - **プライマリ/セカンダリクラウド方式**: 本番環境とDR環境を異なるクラウドプロバイダに配置
    - **サービス別分散**: 機能ごとに最適なクラウドプロバイダを選択し、SPOF回避
    - **コスト/レジリエンスのトレードオフ評価**: マルチクラウドの運用コスト増 vs レジリエンス向上

-   **クラウド間フェイルオーバー設計**:
    - **抽象化レイヤー**: クラウドプロバイダ固有APIへの直接依存を最小化
    - **IaCの移植性**: Terraform/OpenTofuによるクラウド横断的なインフラコード管理
    - **データ同期**: クラウド間のデータ同期メカニズム（レイテンシ・整合性の考慮）
    - **DNSフェイルオーバー**: マルチクラウド対応のDNSベースフェイルオーバー

-   **ベンダーロックイン回避**:
    - クラウド固有サービスの利用判断フレームワーク（利点 vs ロックインリスク）
    - コンテナ化（Docker/Kubernetes）による移植性の確保
    - オープンスタンダード（OpenTelemetry、CloudEvents等）の採用
    - 定期的なクラウド間移行テストの実施

-   **ハイブリッドクラウドDR**:
    - オンプレミスとクラウドを組み合わせたDR環境の設計
    - VPN/専用線によるオンプレミス-クラウド間の安全な接続
    - データ分類に基づく配置ポリシー（機密データのオンプレミス保持等）

-   **Cross-Reference**: §13-§14 (DR戦略), `361_aws_cloud.md`

### §40. ゼロデイ脆弱性対応（詳細プロトコル）

-   **Law**: ゼロデイ脆弱性のライフサイクル全体にわたる対応プロトコルを確立し、発見から修正完了までの時間を最小化してください。

-   **ゼロデイ対応フェーズ**:

    | フェーズ | アクション | SLA |
    |:--------|:---------|:----|
    | **認知** | 脅威インテリジェンスフィード、CERT勧告、ベンダー通知の監視 | 即時 |
    | **トリアージ** | 自社環境への影響評価、CVSSスコア確認、攻撃面分析 | 2時間以内 |
    | **緩和** | 仮想パッチ（WAF/IPS）、機能無効化、ネットワーク制限 | 4時間以内 |
    | **パッチ適用** | ベンダーパッチのテスト → ステージング → カナリア → 全面展開 | 24時間以内（Critical） |
    | **検証** | パッチ後の正常性確認、攻撃指標（IoC）のモニタリング | パッチ後即時 |
    | **報告** | CRA対象の場合、ENISA/CSIRTへ24時間以内に報告 | §33準拠 |

-   **Threat Intelligence統合**:
    - **脅威フィード**: MITRE、NVD、JPCERT/CC、US-CERT等のフィードを自動監視
    - **IoC（Indicators of Compromise）**: 攻撃指標のSIEM/EDRへの自動反映
    - **脆弱性スキャン**: 新たなCVE公開時の自動スキャン実行
    - **情報共有**: ISAC等を通じた業界内の脅威情報共有

-   **脆弱性開示調整（CVD: Coordinated Vulnerability Disclosure）**:
    - 自社製品の脆弱性を発見した外部研究者との連携手順
    - 責任ある開示ポリシーの公開（security.txt標準の導入）
    - 修正・パッチ適用後の公開タイムラインの管理
    - EU CRA下での脆弱性報告義務との連携（§33参照）

-   **Cross-Reference**: §7.5 (ゼロデイ即時対応), §33 (CRA脆弱性報告), `600_security_privacy.md`

## Part XXXII: EU Digital Omnibus Package統合

### §42. EU Digital Omnibus Package統合

-   **Law**: EU Digital Omnibus Package（2025年11月提案）の「Report Once, Share Many」原則に基づく統合報告体制への移行を計画的に準備してください。

-   **概要**:
    - 欧州委員会が2025年11月19日に提案したデジタル規制簡素化パッケージ
    - 複数の重複するインシデント報告義務を統合・簡素化することが目的
    - **ENISA（欧州サイバーセキュリティ機関）**が運営する**単一報告窓口（Single Entry Point）**の設立

-   **統合対象規制**:

    | 規制 | 現行報告先 | 統合後 |
    |:----|:---------|:-------|
    | **NIS2** | 各加盟国CSIRT + 監督当局 | ENISA Single Entry Point |
    | **DORA** | 金融監督当局 | ENISA Single Entry Point |
    | **GDPR** | データ保護当局 | ENISA Single Entry Point |
    | **CER指令** | 重要エンティティ監督当局 | ENISA Single Entry Point |
    | **eIDAS規則** | 電子認証監督当局 | ENISA Single Entry Point |

-   **GDPR改正案（Digital Omnibus内）**:
    - 個人データ侵害通知期限を72時間→**96時間**に延長を提案
    - 通知閾値を高リスクケースのみに引き上げを提案
    - 最終確定は立法プロセスの進行に依拠

-   **組織の対応準備**:
    1.  **現行体制維持**: 統合報告体制が施行されるまで、既存の個別報告プロセスを維持

## Part XXXIII: 危機管理チーム教育・訓練体系

### §43. 危機管理チーム教育・訓練体系

-   **Law**: インシデント対応能力は個人のスキルに依存するためBus Factorリスクが高い。体系的な教育・訓練プログラムを確立し、組織全体のインシデント対応能力を底上げしてください。

-   **インシデントレスポンダー認定（3レベル）**:

    | レベル | 名称 | 要件 | 権限 |
    |:------|:-----|:-----|:-----|
    | **L1: Observer** | 見習い | テーブルトップ演習1回以上参加 | インシデント観察・記録 |
    | **L2: Responder** | 対応者 | L1 + 実インシデント対応2回以上、ランブック作成能力 | インシデント対応の実行、Ops Lead代行 |
    | **L3: Commander** | 指揮官 | L2 + SEV-1対応経験、リーダーシップ研修完了 | Incident Commander、エスカレーション判断 |

-   **教育ロードマップ**:
    1.  **基礎教育（全エンジニア）**: インシデント対応フロー、ポストモーテム文化、on-call心得
    2.  **中級教育（L2認定候補）**: ランブック作成、トリアージ判断、コミュニケーション実践
    3.  **上級教育（L3認定候補）**: 危機時の意思決定、メディア対応、規制報告実務
    4.  **専門教育**: フォレンジクス、サイバー攻撃分析、法的対応

-   **スキル維持要件**:
    - L2/L3認定の維持には、**年2回以上の演習参加**と**年1回以上の実インシデント対応**が必要
    - 12ヶ月以上インシデント対応に関与していない場合、認定レベルの再評価を実施
    - Deepfake・AI脅威等の最新脅威に関する年次アップデート研修の受講を義務化

-   **クロストレーニング**:
    - 特定個人への依存を排除するため、各サービスのインシデント対応能力を**最低2名以上**に分散
    - 四半期ごとにon-callローテーションの対象サービスをクロスアサイン
    - 他チームのランブックレビューへの参加を推奨

-   **Cross-Reference**: §5 (インシデント対応体制), §16 (テーブルトップ演習), §29 (人的要因)

## Part XXXV: インシデント疲労・アラート最適化

### §45. インシデント疲労・アラート最適化

-   **Law**: Alert Fatigue（アラート疲労）はインシデント見逃しの主要原因です。アラートの品質を継続的に改善し、対応者のCognitive Load（認知負荷）を適切に管理してください。

-   **Alert Fatigue対策フレームワーク**:

    | 対策 | 内容 | 目標 |
    |:-----|:-----|:-----|
    | **ノイズ削減** | 重複・低優先度アラートのフィルタリング | アラート総数を70%削減 |
    | **相関分析** | 関連アラートの自動グルーピング（AIOps） | 1インシデント=1アラートグループに集約 |
    | **動的閾値** | ML基づく動的ベースライン（固定閾値の排除） | 誤検知率を50%削減 |
    | **優先度付け** | ビジネスインパクトに基づく自動優先度付け | 最重要アラートの即時対応率100% |
    | **定期棚卸し** | 四半期ごとのアラートルール棚卸しと削除 | 不要アラートの継続的排除 |

-   **アラート品質メトリクス**:

    | メトリクス | 定義 | 目標 |
    |:---------|:-----|:-----|
    | **Signal-to-Noise Ratio** | アクション可能アラート / 全アラート | ≧ 80% |
    | **False Positive Rate** | 誤検知アラート / 全アラート | ≦ 5% |
    | **Alert-to-Incident Ratio** | インシデントに変換されたアラートの割合 | 把握・改善傾向 |
    | **MTTA（Mean Time to Acknowledge）** | アラート発報から確認までの時間 | ≦ 5分（SEV-1/2） |
    | **Alert Volume Trend** | 月次アラート総数の推移 | 減少傾向 |

-   **Cognitive Load管理**:
    - on-call中の同時処理アラート数の上限設定（推奨: 同時3件以下）
    - アラートの段階的通知（最初は関連チームのみ、エスカレーション時に拡大）
    - 情報過多を防ぐためのインシデントダッシュボードの設計ガイドライン
    - on-call後の振り返りでCognitive Load評価を含める

-   **Cross-Reference**: §22 (AIOps), §29 (人的要因・メンタルヘルス), `502_site_reliability.md` (可観測性)

## Part XXXVII: Ransomware Negotiation・IR Retainer

### §47. Ransomware Negotiation・IR Retainer

-   **Law**: ランサムウェア攻撃等の重大サイバーインシデントに備え、インシデント対応（IR）リテーナー契約と交渉プロトコルを事前に整備してください。

-   **IRリテーナー契約**:
    - **事前契約**: インシデント発生前に信頼できるIRベンダー（CrowdStrike、Mandiant、Secureworks等）とリテーナー契約を締結
    - **SLA**: リテーナー契約にインシデント発生後の対応開始SLAを明記（推奨: 2時間以内のリモート対応開始）
    - **スコープ**: フォレンジクス、マルウェア分析、ログ保全、復旧支援、法的助言の範囲を契約に明記
    - **保険会社との調整**: サイバー保険会社が指定するIRベンダーとリテーナー契約先が異なる場合の優先順位を事前に決定

-   **フォレンジクスベンダー事前選定**:
    - インシデント発生時にベンダー選定に時間を費やすことを防止
    - 秘密保持契約（NDA）の事前締結
    - ベンダーの対応能力（24/7対応可否、マルチリンガル対応等）の事前確認
    - 年次でのベンダー評価と契約更新判断

-   **ランサムウェア交渉プロトコル**:
    - **原則禁止**: 身代金支払いは**原則禁止**（§7.1参照）
    - **例外判断プロセス**: 支払いが検討される場合のエスカレーションフロー:
        1.  経営層（CEO/CTO）の承認
        2.  法務部門の法的リスク評価（制裁対象国のグループの場合は支払い自体が違法の可能性）
        3.  サイバー保険会社への通知と助言の取得
        4.  法執行機関への通報と協力
    - **交渉専門家**: 支払い判断と交渉は、専門のネゴシエーター（外部委託推奨）に委託し、技術チームは復旧に集中
    - **記録義務**: 交渉過程と判断根拠の完全な記録（法的証拠としての保全）

-   **Cross-Reference**: §7.1 (ランサムウェア), §27 (サイバー保険), §37 (法的準備態勢)

## Part XXXIX: Self-Healing・自律回復システム

### §49. Self-Healing・自律回復システム

-   **Law**: 自己修復システム（Self-Healing Systems）を段階的に導入し、既知の障害パターンに対する自動復旧能力を構築してください。ただし、Guardrailを必ず設け、人間の監督下で運用すること。

-   **Self-Healing成熟度モデル**:

    | レベル | 名称 | 内容 | 例 |
    |:------|:-----|:-----|:---|
    | **L1: 手動** | Manual | 全復旧オペレーションが手動 | オペレータがコマンドを実行 |
    | **L2: スクリプト** | Scripted | 復旧手順のスクリプト化 | ランブック実行ボタン |
    | **L3: 自動** | Automated | トリガーベースの自動復旧 | Kubernetes Self-Healing、HPA |
    | **L4: 適応** | Adaptive | AIによる障害予測と事前対応 | AIOps予測分析→プリエンプティブ修復 |
    | **L5: 自律** | Autonomous | 未知パターンへの自律的対応 | 完全自律復旧（将来目標・Guardrail必須） |

-   **Kubernetes Self-Healing（実装例）**:
    - **Liveness Probe**: コンテナのヘルスチェック→失敗時に自動再起動
    - **Readiness Probe**: トラフィック受付可否のチェック→準備未完了のPodにリクエストを送らない
    - **PodDisruptionBudget**: メンテナンス時の最低稼働Pod数の保証
    - **Horizontal Pod Autoscaler (HPA)**: CPU/メモリ使用率に基づく自動スケーリング
    - **Node Auto-repair**: ノード障害時の自動復旧とPodの再スケジュール

-   **Guardrail（必須）**:
    - **Blast Radius制限**: 自動修復の影響範囲を制限（1クラスタ・1サービス単位）
    - **Rate Limiting**: 単位時間あたりの自動修復回数の上限設定（無限ループ防止）
    - **Circuit Breaker**: 自動修復が連続失敗した場合は自動修復を停止し人間にエスカレーション
    - **Audit Trail**: 全自動修復アクションの完全な記録と事後レビュー
    - **SEV-1/2禁止**: 重大インシデントへの自動修復適用は**禁止**（§22 Guardrail準拠）

-   **Cross-Reference**: §22 (AIOps), §26 (インシデント自動化), `502_site_reliability.md` (カオスエンジニアリング)

## Part XL: 危機シミュレーションとWar Game

### §50. 危機シミュレーションとWar Game

-   **Law**: テーブルトップ演習（§16）やサイバーレンジ（§46）に加え、経営層を含むクロスファンクショナルな危機シミュレーション（War Game）を定期的に実施し、組織全体の意思決定能力を検証してください。

        - 大規模ランサムウェア攻撃 + メディア報道 + 規制当局からの問い合わせ

        - サプライチェーン攻撃 + 事業継続判断

-   **クロスファンクショナルシミュレーション**:
    - 技術チーム × 法務 × 広報 × カスタマーサポートの連携を検証
    - 規制報告プロセスの実地訓練（模擬報告書の作成と提出）
    - ステークホルダー通知のドライラン

-   **意思決定訓練**:
    - **Time-Pressured Decision Making**: 制限時間内での意思決定を模擬
    - **Fog of War**: 不完全・矛盾する情報下での判断力の訓練
    - **Ethical Dilemmas**: 身代金支払い判断、開示タイミング判断等の倫理的ジレンマシナリオ

-   **頻度と記録**:
    - 経営層War Game: **年2回以上**の実施を推奨
    - 訓練結果の記録と改善アクションの追跡
    - 訓練からの教訓を§19ガバナンスKPIに反映

-   **Cross-Reference**: §16 (テーブルトップ演習), §38 (高度演習プログラム), §43 (教育・訓練体系), §48 (取締役会報告)

## Part XLI: インサイダー脅威プログラム

### §51. インサイダー脅威プログラム

-   **Law**: インサイダー脅威は外部攻撃者が積極的にインサイダーを勧誘する形態が増加しています。CISA 2026ガイダンスに準拠したクロスファンクショナルなインサイダー脅威管理プログラムを確立してください。

-   **インサイダー脅威の分類**:

    | 種別 | 概要 | 検知難易度 |
    |:-----|:-----|:--------:|
    | **悪意ある内部者（Malicious Insider）** | 意図的なデータ窃取、システム妨害 | 高 |
    | **勧誘された内部者（Recruited Insider）** | 外部攻撃者に金銭等で勧誘された従業員 | 極高 |
    | **過失による内部者（Negligent Insider）** | セキュリティ意識の低さによるミス | 中 |
    | **侵害されたアカウント（Compromised Insider）** | 外部から認証情報を窃取された正当な従業員 | 高 |

-   **クロスファンクショナル管理体制（CISA 2026準拠）**:
    - **参加部門**: セキュリティ、HR、法務、IT、オペレーション、コンプライアンスの協働
    - **統合ポリシー**: 技術的監視と非技術的指標（行動変容、不満、経済的困窮）の統合
    - **透明性**: 監視プログラムの存在と目的を従業員に明示（信頼関係の維持）
    - **定期レビュー**: 四半期ごとのプログラム有効性評価

-   **検知手法**:
    - **UEBA（User and Entity Behavior Analytics）**: ベースラインからの逸脱検知
    - **DLP（Data Loss Prevention）**: 機密データの異常な外部送信の検知
    - **アクセスパターン監視**: 通常業務外の時間帯・システムへのアクセス検知
    - **OSINT**: オープンソースインテリジェンスによるリスク評価（CISA推奨）
    - **退職予定者の監視強化**: 退職予告〜最終出社日の期間のアクセスログの重点監視

-   **対応プロトコル**:
    1.  **調査段階**: 法務と連携した合法的な証拠収集（プライバシー法に準拠）
    2.  **封じ込め段階**: 段階的なアクセス権限の縮小（アカウント即時停止はリスクに応じて判断）
    3.  **法的対応**: 必要に応じた法執行機関への通報と法的措置
    4.  **事後対応**: 内部通報制度の見直し、セキュリティ文化の強化

-   **Cross-Reference**: §7.6 (アイデンティティベース攻撃), §29 (人的要因), §37 (フォレンジクス)

## Part XLII: クラウドネイティブインシデントレスポンス

### §52. クラウドネイティブインシデントレスポンス

-   **Law**: コンテナ、サーバーレス、マイクロサービスアーキテクチャ等のクラウドネイティブ環境は、従来のインシデント対応手法では十分に対応できない固有の課題を持ちます。クラウドネイティブ環境に特化したIRプロトコルを確立してください。

-   **クラウドネイティブIR固有の課題**:

    | 課題 | 従来環境 | クラウドネイティブ環境 |
    |:-----|:--------|:-------------------|
    | **Ephemeral（一時的）ワークロード** | 永続的なサーバー | コンテナは数秒〜数分で消滅、ログ・証拠も消失 |
    | **フォレンジクスの困難さ** | ディスクイメージ取得可能 | コンテナのファイルシステムは揮発性、ランタイムのメモリダンプが困難 |
    | **横展開の速度** | ネットワークセグメント単位 | マイクロサービス間のEast-West通信で高速伝播 |
    | **可視性のギャップ** | OS・ネットワーク層は監視可能 | コンテナランタイム・サービスメッシュ層の監視が不十分な場合あり |

-   **コンテナフォレンジクス**:
    - **ランタイムセキュリティ**: Falco、Sysdig、Aqua等によるコンテナランタイムの監視
    - **イメージ保全**: 侵害されたコンテナイメージのスナップショット保全（証拠として）
    - **ログの永続化**: コンテナログの外部ストレージへの即時転送（コンテナ消滅前に）
    - **サイドカーフォレンジクス**: フォレンジクス用サイドカーコンテナの事前配置（Kubernetes環境）

-   **サーバーレスIR**:
    - Lambda/Cloud Functions等の関数実行ログの保全
    - コールドスタート時の異常な初期化処理の検知
    - 関数レベルのIAM権限の監査（過剰権限の特定）
    - サーバーレス環境での横展開パスの可視化

-   **Kubernetes固有のIR**:
    - **Pod隔離**: NetworkPolicyによる侵害Podのネットワーク隔離
    - **ノード隔離**: Cordon + Drainによるノードレベルでの隔離
    - **RBAC監査**: ServiceAccount/ClusterRoleの権限エスカレーションの調査
    - **Audit Log**: Kubernetes Audit Logの保全と分析

-   **Cross-Reference**: §14 (クラウドネイティブDR), §37 (デジタルフォレンジクス), §49 (Self-Healing)

## Part XLIII: ポリクライシス（複合危機）対応

### §53. ポリクライシス（複合危機）対応フレームワーク

-   **Law**: 2025-2026年の危機環境は**ポリクライシス（Polycrisis）**— 複数の危機が同時に、または連鎖的に発生する状況 — が増加しています。個別インシデントへの対応だけでなく、同時多発的な危機に対するトリアージと資源配分のフレームワークを確立してください。

-   **ポリクライシスシナリオ例**:
    - ランサムウェア攻撃 + 同時期のクラウドプロバイダ障害（DR環境が使えない）
    - データ侵害 + パンデミックによる人員不足 + 規制当局の調査
    - DDoS攻撃 + サプライチェーン攻撃 + メディアスキャンダル
    - 自然災害 + サイバー攻撃（災害便乗型攻撃）

-   **複合危機トリアージマトリクス**:

    | 優先度 | 基準 | アクション |
    |:------|:-----|:---------|
    | **P0: Safety（安全）** | 人命・身体的安全に関わる危機 | 最優先で対応、全リソースを集中 |
    | **P1: Service（サービス）** | ミッションクリティカルサービスの停止 | Safety確保後、即時対応 |
    | **P2: Data（データ）** | データ侵害・損失のリスク | 封じ込めを優先、復旧は後回し可 |
    | **P3: Reputation（レピュテーション）** | ブランド・信頼への影響 | コミュニケーション対応、技術対応は並行 |
    | **P4: Compliance（コンプライアンス）** | 規制報告義務 | 法定期限の遵守、他の対応と並行 |

-   **資源配分の原則**:
    - **Incident Commander 2名体制**: 複合危機時は単一のICでは認知負荷が過大。主要インシデントごとにICを任命し、上位に統合ICを設置
    - **優先度に基づくリソース配分**: P0/P1に全専門家を集中、P2以下は最少人員で封じ込めに専念
    - **コミュニケーションの分離**: 各危機のコミュニケーション担当を分離し、情報の混同を防止
    - **不完全情報下の意思決定**: 完全な情報を待たずに「十分な情報」で意思決定する訓練（§50 Fog of War）

-   **Cross-Reference**: §5 (War Room), §50 (War Game), §45 (アラート最適化)

## Part XLIV: 情報操作・偽情報危機対応

### §54. 情報操作・偽情報（Disinformation）危機対応

-   **Law**: AI生成コンテンツの高度化により、企業に対する偽情報攻撃（ブランド攻撃型偽情報、Deepfakeを使った虚偽のスキャンダル等）が増加しています。情報操作に対する危機対応プロトコルを事前に準備してください。

-   **偽情報脅威カテゴリ**:

    | 脅威 | 概要 | 対応 |
    |:-----|:-----|:-----|
    | **ブランド攻撃型偽情報** | 製品・サービスに関する虚偽情報の意図的拡散 | モニタリング、迅速な事実提示、法的対応 |
    | **Deepfake PR危機** | 経営者のDeepfake動画・音声による虚偽発言 | 公式チャネルでの即座の否定、Deepfake検出証拠の提示 |
    | **フェイクレビュー/評価攻撃** | 大量の虚偽のネガティブレビュー投稿 | プラットフォームへの報告、正当なレビューの促進 |
    | **株価操作型偽情報** | 企業に関する虚偽情報による株価影響 | 法務・IR部門との連携、開示義務の確認 |
    | **インシデント便乗型偽情報** | 実際のインシデント発生時に誇張・虚偽情報を拡散 | 公式情報の迅速・頻繁な更新で対抗 |

-   **対応フレームワーク**:
    1.  **監視**: ソーシャルメディア・ニュースサイトのリアルタイム監視
    2.  **検証**: 偽情報の事実確認と証拠収集
    3.  **対抗**: 公式チャネルでの迅速な事実提示、SEOによる正確な情報の上位表示
    4.  **法的対応**: 名誉毀損・虚偽情報に対する法的措置
    5.  **事後分析**: 偽情報の発信源・拡散経路の分析

-   **AI偽情報対策ツールキット**:
    - Deepfake検出ツール（Microsoft Video Authenticator、Google SynthID等）
    - コンテンツ出所追跡（C2PA: Coalition for Content Provenance and Authenticity）
    - ソーシャルメディアモニタリングの自動化

-   **Cross-Reference**: §8 (Deepfake対策), §9 (危機コミュニケーション), §28 (レピュテーション回復)

## Part XLV: M&A・組織再編時の危機管理

### §55. M&A・組織再編時の危機管理

-   **Law**: M&A（合併・買収）や大規模な組織再編時は、IT環境の統合過程でセキュリティの脆弱性窓口が生じます。統合期間中の固有リスクに対応した一時的な強化措置を計画してください。

-   **M&A危機管理チェックリスト**:

    | フェーズ | リスク | 対策 |
    |:--------|:------|:-----|
    | **Due Diligence** | 被買収企業の未知の脆弱性・過去のインシデント | セキュリティ監査、インシデント履歴確認 |
    | **Day 1（統合初日）** | ネットワーク接続時の横展開リスク | ネットワーク隔離の維持、段階的統合 |
    | **統合期間** | Active Directory/IAM統合時のアクセス制御の一時的弱体化 | 統合専用の一時IAMポリシー、監視強化 |
    | **安定化期間** | レガシーシステムの残存、Shadow IT | 統合完了監査、シャドーITの発見・統合 |

-   **BCP統合プロセス**:
    - 被買収企業のBCPの評価と自社BCPへの統合
    - RTO/RPOの再定義（統合後の新しいサービス依存関係を反映）
    - 統合期間中の一時的な追加DR環境の検討
    - インシデント対応の指揮系統の一本化（Day 1〜）

-   **Cross-Reference**: §11 (BCP), §12 (委託先BCP), §19 (ガバナンス)

## Part XLVI: エッジ/IoTインシデント対応

### §56. エッジ/IoTインシデント対応

-   **Law**: エッジコンピューティングデバイスやIoTデバイスのフリートは、大規模なインシデントが発生した場合に従来のサーバー環境とは異なるチャレンジを提示します。大量デバイスの障害・侵害への対応プロトコルを確立してください。

-   **エッジ/IoT固有のIR課題**:
    - **スケール**: 数千〜数百万台のデバイスが同時に影響を受ける可能性
    - **接続性**: 断続的・低帯域な接続環境でのリモート対応
    - **リソース制約**: CPU・メモリ・ストレージの制約によるセキュリティエージェント導入の困難さ
    - **物理分散**: 世界中に分散したデバイスへの物理アクセスの困難さ

-   **大規模IoTインシデント対応**:
    1.  **影響範囲の特定**: デバイスフリート管理システムによる影響デバイスの特定
    2.  **リモート隔離**: ネットワークレベルでの影響デバイスの隔離（IoTゲートウェイ/VLANレベル）
    3.  **ファームウェア更新**: OTA（Over-The-Air）アップデートによるパッチ配布
    4.  **段階的リカバリ**: カナリアグループ → リージョン別 → 全体展開のローリング復旧
    5.  **物理回収**: リモート復旧不能な場合のデバイス物理回収プロセス

-   **予防策**:
    - デバイス証明書ベースの認証（シークレットキーのハードコード禁止）
    - セキュアブートの必須化
    - デバイスのライフサイクル管理（EOL/EOSデバイスの計画的更新）
    - ネットワークセグメンテーション（IoTネットワークとコーポレートネットワークの分離）

-   **Cross-Reference**: §36 (OT/ICS), §52 (クラウドネイティブIR)

## Part XLVII: 非技術的危機管理

### §57. 非技術的危機管理（法的/HR/環境危機）

-   **Law**: 危機はサイバーインシデントだけではありません。法的紛争、HR危機（ハラスメント、集団訴訟）、環境規制違反等の非技術的な危機にも対応するフレームワークを確立し、BCPと統合してください。

-   **非技術的危機カテゴリ**:

    | カテゴリ | 例 | 初動 |
    |:--------|:---|:-----|
    | **法的紛争** | 集団訴訟、知的財産侵害訴訟、規制当局の調査 | 法務即時対応、リーガルホールド発動 |
    | **HR危機** | ハラスメント告発、大量退職（Mass Resignation）、内部告発 | HR+法務の合同対応チーム |
    | **環境危機** | 環境規制違反、サステナビリティに関するメディアスキャンダル | コンプライアンス対応、広報対応 |
    | **財務危機** | 決算不正の発覚、重大な財務誤表示 | IR・法務即時対応、開示義務の確認 |
    | **製品安全** | 製品の安全性問題、リコール | 製品チーム・法務・広報の合同対応 |

-   **統合危機管理フレームワーク**:
    - 技術的危機と非技術的危機を**同一の指揮系統**で管理（重複する危機の統合対応が可能に）
    - 全種別の危機に対する**共通のエスカレーションパス**の定義
    - 非技術的危機の**BCP統合**: 法的紛争が長期化した場合の事業継続計画
    - **危機コミュニケーションの一本化**: §9の危機コミュニケーション管理を非技術的危機にも適用

-   **Cross-Reference**: §9 (危機コミュニケーション), §19 (ガバナンス), §28 (レピュテーション回復), §37 (法的準備態勢)
