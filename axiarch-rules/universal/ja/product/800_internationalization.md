## Part I: グローバル展開戦略基盤

### セクション 1: グローバル展開哲学・原則

- **i18n-First原則**: i18nはDay 1からアーキテクチャに組み込む。後付けは技術的負債の最大要因
- **L10n ≠ Translation**: ローカライゼーションは翻訳ではない。文化適応・法的適応・技術適応を含む包括的プロセス
- **優先順位の階層**:
  1. **機能的正確性**: データの正確な表示（通貨・日付・数値）
  2. **法的コンプライアンス**: 各国規制への準拠（GDPR・Global Privacy Laws・CCPA等）
  3. **文化的適切性**: 文化的に不適切な表現・デザインの排除
  4. **言語的品質**: 翻訳の自然さ・専門用語の正確性
- **Locale-Agnostic Code**: ビジネスロジックはロケールに依存しない設計。ロケール依存処理はUI層・フォーマット層に集約
- **AI-Ready i18n**: 翻訳リソースファイルはLLMが解析・処理しやすい構造で設計。コンテキストメタデータの付与・一貫した命名規則を徹底

### セクション 2: 市場ティア戦略・GTM

- **ティア分類**:

| ティア | 地域例 | 戦略 | 投資基準 |
|---|---|---|---|
| Tier 1（最優先） | 北米・日本・西欧・オーストラリア | フルカルチャライズ・マーケティング集中 | ARPU高・規制成熟 |
| Tier 2（成長市場） | 東南アジア・南米・インド | バイラル成長・軽量化・低帯域最適化 | ユーザー数成長・モバイルファースト |
| Tier 3（監視対象） | アフリカ・中東・中央アジア | 最低限のi18n・市場動向モニタリング | 将来ポテンシャル評価 |

- **Dynamic Pricing Metadata**: 国別の価格設定・機能制限は、コード内分岐ではなく**Stripe Metadata**で制御 → `100_product_strategy.md`
- **ティア昇格基準**: MAU成長率 > 20%/月、ARPU > グローバル平均の50%、規制環境の安定化

## Part II: 国際化（i18n）アーキテクチャ

### セクション 4: Day 1 i18nアーキテクチャ

- **ハードコード厳禁**: UI上のテキスト（ボタン・ラベル・プレースホルダー・エラーメッセージ・ツールチップ・aria-label含む）のハードコードは**絶対禁止**
- **外部化義務**: 全ての翻訳可能テキストはキーバリュー形式の外部リソースファイル（`en.json`, `ja.json`等）で管理
- **擬似ローカライズ（Pseudo-localization）**:
  - 開発中は擬似言語モード（例: `Account` → `[!!! Àççôûñţ !!!]`）でテスト
  - **検出対象**: ハードコード文字列・レイアウト崩れ（テキスト長30〜40%増への耐性）・文字化け
  - **CI統合**: 擬似ローカライズテストをCI/CDパイプラインに組み込み、ハードコードを自動検出

```javascript
// ❌ 禁止: ハードコード
const label = "名前を入力してください";

// ✅ 正しい: 外部化
const label = t('form.name.placeholder');
```

### セクション 5: 翻訳キー設計・Namespace管理

- **Namespace設計**: 機能/ページごとにNamespaceを分割（`common`, `auth`, `error`, `admin`, `settings`）
- **命名規約**: ドット区切り・最大3階層・英語スネークケース（`auth.login.submit_button`）
- **コンテキスト付加**: 同一テキストでもコンテキストが異なる場合は別キー
- **AI-Ready メタデータ**: 各キーにコンテキスト・最大文字数・スクリーンショットリンクをメタデータとして付加
- **禁止事項**: 翻訳テキスト自体をキーに使用しない。動的キー生成（`t(\`error.${code}\`)`）禁止
- **W3C String-Meta準拠**: 自然言語テキストを含む全フィールドに言語・方向メタデータを付与可能な構造設計

### セクション 6: ICU MessageFormat 2.0・CLDR 48/49統合

- **ICU MessageFormat 2.0採用**: CLDR 47 / ICU 77でStable化。CLDR 48.2（2026年3月17日 Stableリリース）で組合せフォーマットが安定化
- **CLDR活用義務**: 日付・時刻・数値・通貨のフォーマットは独自実装禁止。CLDRデータに準拠
- **CLDR 48/48.2 主要機能**:
  - 有理数フォーマット（例: `5½`）— Tech Preview
  - 相対日付+時刻の組合せフォーマット（例: `明日の12:30`）
  - 新単位: クーロン・ファラド・テスラ・フォートナイト・インペリアルパイント
  - タイムゾーン `usesMetazone` に `stdOffset`/`dstOffset` 属性追加
- **CLDR 49**（2026年4月リリース予定）:
  - **H24非推奨化**: H24は今後H23の挙動を示す。時刻フォーマットコードの確認義務
  - **pre-Meiji和暦削除**: 明治以前の和暦データ削除（グレゴリオ暦使用が慣例のため）。和暦関連コードの影響評価必須
  - **XML/JSONファイル再編**: より論理的なファイル分割。データフィルタリングとメンテナンス性向上
- **ICU 79**（2026年10月リリース予定）: CLDR 49 + Unicode 18.0対応。MF 2.0のJava実装がDraft→Stable昇格見込み
- **MF 2.0実装状況**:
  - **Java**: CLDR 48レベル。コアAPIがDraftステータス
  - **C++**: CLDR 47レベル。Tech Preview
  - **JavaScript**: Tech Preview実装利用可能
- **MF 1.0→2.0移行**: 新規コードはMF 2.0必須。既存コードは四半期ごとに移行計画を策定・実行

```
// MessageFormat 2.0 構文例
.local $count = {$itemCount :number}
.match $count
one {{カートに {$count} 個のアイテムがあります。}}
*   {{カートに {$count} 個のアイテムがあります。}}
```

### セクション 7: 複数形・性別・文法ルール

- **CLDR Plural Rules必須**: 複数形処理は独自ロジック禁止。CLDRの複数形カテゴリ（zero/one/two/few/many/other）を使用
- **言語別カテゴリ数**: 英語: 2、アラビア語: 6、日本語: 1、ポーランド語: 4
- **性別対応**: 人称を含むメッセージはMF 2.0の`.match`ディレクティブで性別分岐
- **序数対応**: 順位表示もCLDR ordinal rulesに準拠

### セクション 8: 日付・時刻・カレンダー体系

- **Intl.DateTimeFormat使用義務**: 日付・時刻のフォーマットはロケール自動適用
- **Temporal API — TC39 Stage 4達成（2026年3月）**:
  - **ステータス**: ECMAScript 2026仕様に**正式統合決定**。9年間の開発を経てStage 4到達
  - **ブラウザサポート**:

  | エンジン | バージョン | リリース時期 | 状態 |
  |---|---|---|---|
  | Firefox | 139+ | 2025年5月 | ✅ GA |
  | Chrome / Edge | 144+ | 2026年1月 | ✅ GA |
  | Safari | Technology Preview | — | ⚠️ 部分対応 |
  | Node.js | v26+ | 2026年予定 | ✅ GA予定 |
  | TypeScript | 6.0 Beta+ | 2026年2月 | ✅ 組み込み型定義 |

  - **義務化ロードマップ**:
    - **即時**: 新規コードでTemporal使用を**必須化**。レガシー`Date`の新規使用禁止
    - **2026 Q4目標**: 既存コードのTemporal移行完了
  - **Polyfill不要化**: 主要ブラウザネイティブ対応により、`@js-temporal/polyfill` の段階的除去を開始
- **ECMA-402 13th Edition（2026年版）**: ECMAScript 2026 Internationalization API仕様。Temporal統合に伴うIntl API拡張を活用
- **暦体系対応**: 和暦・仏暦・ヒジュラ暦・民国暦をサポート。CLDR 49でのpre-Meiji和暦削除に注意
- **相対時間**: `Intl.RelativeTimeFormat` を使用。CLDR 48の組合せフォーマット活用
- **週の開始日**: ロケールによる差異を考慮（日曜/月曜/土曜）

```javascript
// ✅ Temporal API（必須 — ECMAScript 2026標準・Stage 4）
const zdt = Temporal.ZonedDateTime.from({
  timeZone: 'Asia/Tokyo',
  year: 2026, month: 3, day: 28, hour: 17,
});
// ✅ Temporal.PlainDate（タイムゾーン不要の日付）
const date = Temporal.PlainDate.from('2026-03-28');
```

### セクション 9: 数値・通貨・測定単位フォーマット

- **通貨オブジェクト必須**: `{ amount: 1000, currency: 'JPY' }` — 金額と通貨を分離しない設計は禁止
- **Intl.NumberFormat使用義務**: 数値・通貨の表示フォーマットに使用
- **小数点処理**: 通貨ごとの小数桁数をCLDRに基づき処理（JPY: 0桁、USD: 2桁、BHD: 3桁）
- **測定単位**: `Intl.NumberFormat` の `unit` オプションでロケール対応
- **大きな数値**: `notation: 'compact'` でロケール別短縮表記

### セクション 10: 文字列連結禁止・コンテキスト付きメッセージ

- **文字列連結の絶対禁止**: 言語による語順の違い（SOV vs SVO）で翻訳不能になる
- **プレースホルダー方式**: 変数挿入は `{name}` で対応
- **HTML in 翻訳禁止**: リッチテキストはMF 2.0のマークアップ機能またはコンポーネント分割で対応

### セクション 11: リソースファイル管理・バンドル最適化

- **ファイル分割基準**: ページ/機能単位で分割。1ファイルあたり最大500キー目安
- **遅延ロード**: 画面遷移時に必要なNamespaceのみ動的ロード
- **フォールバックチェーン**: `ja-JP` → `ja` → `en`（デフォルト言語）
- **翻訳ファイルのイミュータビリティ**: キー削除はDeprecatedマーク後、2リリースサイクル経過後に実施
- **RSC統合**: React Server Componentでの翻訳はサーバーサイドで解決。クライアントバンドルに翻訳JSONを含めない
- **Tree-shaking**: 未使用Namespaceのビルド時自動除外を設定

### セクション 12: フレームワーク固有i18n実装

- **Next.js App Router**: `next-intl` v4.0+（2025年3月GA）を標準ライブラリとして採用
  - `app/[locale]/` サブパスルーティング
  - Middleware によるロケール検出・リダイレクト
  - Server Componentでの翻訳: クライアントバンドルゼロバイト
  - `createNextIntlPlugin` による設定統合
  - **v4.0新機能**:
    - **Strictly Typed Locales**: `hasLocale`関数でのロケール型絞り込み。`Locale`型のインポート
    - **Strictly Typed ICU Arguments**: ICU引数の型安全。IDE補完対応（オプトイン）
    - **ESM-only配信**: Tree-shaking向上（`next-intl/plugin`は例外）
    - **GDPR対応**: Cookie有効期限の適切な制御
    - **PPR/dynamicIO準備**: `NextIntlClientProvider`必須化、`await requestLocale`
- **React**: `react-intl`（FormatJS）またはi18nextエコシステム
- **Vue**: `vue-i18n` v10+
- **エッジi18n処理**: ロケール検出・リダイレクトをEdge Middleware/Cloudflare Workersで完結

```javascript
// Next.js Middleware によるロケール検出例

  const supportedLocales = ['ja', 'en', 'ko', 'zh'];
  const resolved = supportedLocales.includes(locale) ? locale : 'en';
  // ...リダイレクト処理
}
```

## Part III: ローカライゼーション（L10n）基盤

### セクション 13: 翻訳管理システム（TMS）統合

- **TMS導入基準**: 3言語以上の展開時はTMS導入を義務化
- **推奨TMS**:

| TMS | 強み | Git連携 | MF 2.0対応 | AI翻訳統合 |
|---|---|---|---|---|
| Crowdin | コミュニティ翻訳・大規模対応 | ✅ | ✅ | ✅ DeepL/Google/OpenAI |
| Lokalise | 開発者フレンドリー・REST API | ✅ | ✅ | ✅ DeepL/Google |
| Phrase | エンタープライズ・TMS+Strings統合 | ✅ | ✅ | ✅ 自社AI+外部 |

- **必須要件**: Gitリポジトリ直接連携・TM/Glossaryサポート・MF 2.0対応・CLI/CI統合・ブランチベースワークフロー
- **OTA翻訳更新**: モバイルアプリはアプリストア再申請なしでの翻訳反映設計を推奨

### セクション 14: 継続的ローカライゼーション（CI/CD連携）

- **ワークフロー**: 開発者がキー追加 → CI/CDが新規キー検出・TMS送信 → 翻訳者がTMS上で翻訳 → 承認済み翻訳がPR自動マージ
- **翻訳キー欠落検出**: ソースコード内のキーと翻訳ファイルを比較し、未翻訳キーをCIで自動検出・ビルド失敗
- **翻訳完了率ゲート**: Tier 1言語は100%必須、Tier 2言語は95%以上

```yaml
# CI/CDパイプライン例
- name: i18n-check
  run: |
    npx i18n-check --source src/ --locales locales/ --fail-on-missing
    npx i18n-coverage --threshold 95 --tier1-threshold 100
```

### セクション 15: 翻訳品質保証（LQA・MTPE）

- **機械翻訳のみでのリリース禁止**: 全翻訳テキストはネイティブスピーカーによるレビュー必須
- **MQM（Multidimensional Quality Metrics）**: エラー分類に基づく定量評価。合格基準: **98点以上**
- **MTPE**: Light PE（UIラベル短文）/ Full PE（マーケティング・法的文書）
- **レビュープロセス**: 翻訳者 → レビュアー → 技術検証の3段階

### セクション 16: 翻訳メモリ・用語集・スタイルガイド

- **TM活用**: 過去の翻訳を再利用し、一貫性とコスト効率を確保
- **用語集必須**: 言語別の技術用語・ブランド用語統一翻訳を定義。翻訳禁止ワードを明示
- **スタイルガイド**: 言語別のトーン・文体・敬語レベル定義
- **用語集Git管理・PRレビュー必須**

### セクション 17: AI翻訳ガバナンス・成熟度レベル

- **AI翻訳活用範囲**:

| 用途 | AI翻訳 | ポストエディット | ネイティブレビュー |
|---|:---:|:---:|:---:|
| UIラベル・ボタン | ✅ | ✅ Light PE | ✅ |
| ヘルプ・FAQ | ✅ | ✅ Full PE | ✅ |
| マーケティングコピー | ⚠️ 参考のみ | — | ✅ ゼロから執筆推奨 |
| 法的文書・利用規約 | ❌ 禁止 | — | ✅ 法務レビュー必須 |
| UGC | ✅ リアルタイム | — | ❌ 不要 |

- **マルチエンジンルーティング**:

## Part VI: タイムゾーン・暦・ロケール管理

### セクション 30: UTC保存・ローカル変換原則

- **UTC保存義務**: データベース・バックエンドでの時刻保存は**例外なくUTC**
- **ローカル変換タイミング**: ユーザーへの表示直前（UI層）でのみ変換
- **`timestamptz` 使用義務**: PostgreSQLでは `timestamptz` を使用。`timestamp without time zone` 禁止

```sql
-- ❌ 禁止
CREATE TABLE events (start_at TIMESTAMP);
-- ✅ 正しい
CREATE TABLE events (start_at TIMESTAMPTZ DEFAULT NOW());
```

### セクション 31: IANA Time Zone DB・DST対応

- **IANA TZ DB使用**: タイムゾーン識別子はIANA準拠（`Asia/Tokyo`, `America/New_York`）
- **固定オフセット禁止**: `UTC+9` での保存・処理禁止（DST対応不可）
- **DST対応テスト**: Spring Forward（存在しない時刻）・Fall Back（重複時刻）を必ずテスト
- **TZ DB更新追従**: 年2〜4回のIANA更新に自動追従するプロセス確立

### セクション 32: 物理ロケーション例外規則

- **物理店舗の営業時間**はユーザーの所在地ではなく**店舗の所在地タイムゾーン**で判定
- **「現在営業中」検索**: サーバーサイドで店舗所在地TZを使用して判定

### セクション 33: 暦体系対応

- **サポート対象**: グレゴリオ暦（標準）・和暦・仏暦・ヒジュラ暦・民国暦
- **Intl API活用**: `Intl.DateTimeFormat` の `calendar` オプション
- **週の開始日**: CLDR `firstDay` データに基づき処理
- **年度**: 会計年度・学年度は国により異なる（日本: 4月、米国: 9月/10月）

### セクション 34: ロケール検出・設定管理

- **ロケール優先度チェーン**: ユーザー設定 > Accept-Language > Geo-IP > デフォルト（en）
- **ロケール設定永続化**: サーバーサイド保存（Cookie + DB）

- **ヒジュラ暦**: イスラム暦サポート。`Intl.DateTimeFormat` の `calendar: 'islamic'`
- **アラビア語複数形**: 6形態（zero/one/two/few/many/other）対応。CLDR準拠必須
- **方言差異**: MSA（現代標準アラビア語）ベースだが、エジプト・レバノン・湾岸各方言の用語差異を考慮

### セクション 92: ヘブライ語固有要件

- **ニクード（母音記号）**: 技術文書ではニクード省略が一般的。必要時の正確配置保証
- **ヘブライ暦**: `Intl.DateTimeFormat` の `calendar: 'hebrew'`
- **数値・通貨**: 新シェケル（₪ NIS）。LTRで表示

### セクション 93: ペルシア語（ファルシー語）固有要件

- **ペルシア数字**: ۰۱۲۳（アラビア語東部数字とは異なるコードポイント）
- **イラン暦（ジャラーリー暦/太陽ヒジュラ暦）**: `Intl.DateTimeFormat` の `calendar: 'persian'`
- **ペルシア語キーボード**: アラビア語キーボードとの差異考慮（追加文字: پ・چ・ژ・گ）

### セクション 94: ウルドゥー語固有要件

- **ナスタアリーク書体**: ウルドゥー語の標準書体はナスタアリーク（Nastaliq）。ナスフ（Naskh）との使い分け
- **ウルドゥー語数字**: ペルシア語数字と同系統。`۰۱۲۳` 使用
- **パキスタンの規制**: PTA（パキスタン電気通信庁）コンテンツ規制対応

## Part XXI: 言語固有セクション — 南・東南アジア圏

### セクション 95: ヒンディー語・Indicスクリプト

- **デーヴァナーガリー**: 結合文字（conjuncts）・リガチャの正確なレンダリング
- **ヒンディー語数字**: `०१२३` デーヴァナーガリー数字。ユーザー設定で切替可能に
- **ヒンディー語フォント**: `Noto Sans Devanagari` プライマリ。サブセット最適化
- **インドの多言語**: 22公用語。`Intl.Segmenter` 対応状況モニタリング
- **DPDPA 2023準拠**: インドデジタル個人データ保護法。同意管理・データローカライゼーション

### セクション 96: タイ語

- **単語区切り無し**: タイ語にはスペース区切りが無い。`Intl.Segmenter` 必須
- **タイ語ソート**: タイ語辞書順は複雑。ICU Collation活用
- **仏暦**: 西暦+543。`Intl.DateTimeFormat` の `calendar: 'buddhist'`
- **タイ数字**: `๐๑๒๓` 使用可能（ユーザー設定で切替）
- **PDPA（タイPrivacy Laws）**: 2022年施行。同意管理義務

### セクション 97: ベトナム語

- **声調記号**: 5つの声調記号の正確なレンダリング。NFC正規化必須
- **ベトナム語フォント**: ラテンアルファベットベースだが声調記号対応フォント必要
- **ベトナム住所**: 省/直轄市 → 県/区 → 社/坊 → 番地の階層構造

### セクション 98: インドネシア語・マレー語

- **共通基盤**: インドネシア語とマレー語は高い類似性。ただし翻訳共用は非推奨（語彙・表記差異）
- **単名文化**: インドネシアでは姓を持たない文化が一般的。名前フォームの柔軟設計（姓フィールド任意）
- **ラテンアルファベット**: 特殊文字少なくフォント課題は軽微

## Part XXII: 言語固有セクション — ヨーロッパ言語

### セクション 99: ドイツ語固有要件

- **テキスト長拡張**: 英語比+30-40%。UIレイアウト設計で最重要テスト言語
- **複合名詞**: 極めて長い複合語が発生（例: Donaudampfschifffahrt）。`overflow-wrap: anywhere` 必須
- **大文字/小文字**: 名詞は常に大文字開始。`text-transform` 使用時の注意
- **エスツェット（ß）**: 大文字化時は`SS`。Unicode 5.1+ `ẞ`（大文字エスツェット）の処理
- **敬称**: Sie/Du の使い分け。B2C vs B2B での敬語レベル差
- **数値書式**: 小数点 `,`（コンマ）、千区切り `.`（ピリオド）。英語と逆

### セクション 100: フランス語固有要件

- **フランス語バリアント**: フランス本土（`fr-FR`）とカナダ（`fr-CA`）の語彙・表記差異
- **句読点前スペース**: `!`・`?`・`:` の前にナローノーブレークスペース（U+202F）。CSS `white-space` 対応
- **エリジオン/リエゾン**: 翻訳品質チェック項目
- **文法的性**: 製品名・機能名に付く冠詞の性一致

### セクション 101: スペイン語固有要件

- **スペイン語バリアント**: スペイン（`es-ES`）とラテンアメリカ（`es-419`/`es-MX`等）の大きな語彙差
- **逆疑問符/感嘆符**: `¿`・`¡` の正確な使用
- **Vos/Tú/Usted**: 地域別敬称使い分け。ラテンアメリカ内でも差異
- **LGPD準拠**: ブラジルポルトガル語圏でのデータ保護

### セクション 102: イタリア語・ポルトガル語・その他

- **イタリア語**: テキスト拡張+15-20%。冠詞と前置詞の縮約
- **ポルトガル語バリアント**: ポルトガル（`pt-PT`）とブラジル（`pt-BR`）の大きな差異。別言語として管理
- **オランダ語**: 複合語の処理。ベルギーオランダ語（`nl-BE`）との差異
- **ポーランド語**: 4つの複数形カテゴリ。格変化の翻訳品質

## Part XXV: 組織・プロセス・成熟度

### セクション 111: ローカライゼーション成熟度モデル

| レベル | 名称 | 特徴 | KPI基準 |
|:---:|---|---|---|
| 1 | **Ad-Hoc** | 場当たり的翻訳。ハードコード残存。TZ未考慮 | 翻訳カバレッジ < 50% |
| 2 | **Reactive** | キー外部化済。主要言語のみ。手動翻訳 | 50-80%、Missing Key > 10/月 |
| 3 | **Proactive** | TMS導入。CI/CD統合。擬似L10n。CLDR準拠 | 95%+、MQM 95+、Missing Key < 5/月 |
| 4 | **Strategic** | 全市場フルカルチャライズ。マルチリージョンインフラ。定量LQA | 100%、MQM 98+、Missing Key 0 |
| 5 | **Optimized** | AI翻訳+MTPE統合。Cross-lingual RAG。リアルタイムL10n。サステナブルi18n | 上記 + AI翻訳 COMET > 0.85 |

- **目標**: 全プロジェクトは**レベル3（Proactive）以上**を維持
- **評価頻度**: 四半期ごと
- **昇格トリガー**: 新市場参入時またはティア変更時

### セクション 112: グローバルチーム運営・コミュニケーション

- **タイムゾーン考慮**: 非同期コミュニケーション設計。Golden Hours活用
- **言語ポリシー**: チームの共通言語定義・自動翻訳ツール活用
- **文化的多様性**: 文化的背景を尊重するコミュニケーションガイドライン
- **i18n Champion**: 各チームにi18n推進担当者を配置

### セクション 113: i18n教育・ナレッジベース

- **オンボーディング**: i18nベストプラクティスガイドの新規メンバー向け提供
- **コードレビュー基準**: ハードコードテキスト・物理プロパティ・文字列連結を常にフラグ
- **インシデントDB**: i18n関連インシデント・改善事例の蓄積・共有
- **定期勉強会**: 四半期ごとのi18nトピック勉強会

### セクション 114: サステナブルi18n・Green Localization

- **フォントサブセット最適化**: 不要グリフ除外によるデータ転送量・CO2削減
- **翻訳キャッシュ**: 重複翻訳API呼び出し削減（API消費電力削減）
- **リージョン別右サイジング**: 過剰プロビジョニング回避
- **SCI（Software Carbon Intensity）**: i18n関連インフラのカーボンフットプリント計測
- → `720_cloud_finops.md` GreenOpsセクション参照
