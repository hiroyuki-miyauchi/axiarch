# 14. ブランド戦略 (Brand Strategy)

> [!CAUTION]
> **このファイルは Universal Rule（不変ルール）です。「憲法改正」の明示的指示がない限り編集禁止。**
> 改定日: 2026-04-06

> [!IMPORTANT]
> **Supreme Directive（最高指令）**
> 「ブランドは会社の内側から外側に滲み出るものであり、広告代理店が制作するものではない」
> ブランドは **「約束の集積」** である。ユーザーとのすべての接点がブランドを形成する。
> 一貫性なきブランドは信頼を破壊する。ブランドの構築は最も長期的な競争優位性の源泉である。
> **8パート・30セクション構成。**

## Part I. ブランドの哲学

### 1.1. ブランドとは何か

- **Rule 14.001**: ブランドを「ロゴ・色・フォント」として矮小化することを禁止する。ブランドとは **「ユーザーが持つ感情的な連想の総体」** である
- **Simon Sinek の Golden Circle**:

```
Why（なぜ存在するのか）← ブランドの核心
  ↓
How（どのように実現するか）
  ↓
What（何を提供するか）← ほとんどの会社はここから始める

Rule: ブランドコミュニケーションは常に「Why」から始める
```

### 1.2. Moat としてのブランド

- **Rule 14.002**: 強いブランドは以下の経済的価値を生む
  - **CAC削減**: 認知度向上によりインバウンドが増加し、CAC が低下する
  - **価格プレミアム**: 同等の機能でも高い価格を正当化できる（Apple vs 汎用PC）
  - **採用優位**: 優秀な人材が応募する確率が上がる
  - **パートナーシップ**: 信頼あるブランドにはパートナーが集まる

### 1.3. Brand Promise（ブランドの約束）

- **Rule 14.003**: ブランドの中核に **「Brand Promise」（ブランドの約束）** を定義する
  - Brand Promise は「ユーザーが毎回期待できること」を1文で表現する
  - Brand Promise はマーケティング部門ではなく、CEO・CPO が定義する
  - 例: "Amazon: 地球上で最も顧客を大切にする企業" / "Apple: 誰もが使える革新的なテクノロジー"

## Part II. ブランドアイデンティティの設計

### 2.1. Brand Pyramid（ブランドピラミッド）

- **Rule 14.010**: ブランドアイデンティティを以下の5層構造で定義する

```
    ┌─────────────────┐
    │   Brand Essence  │ （Why: 存在意義）
    ├─────────────────┤
    │ Brand Personality│ （人格・キャラクター）
    ├─────────────────┤
    │  Brand Values    │ （行動を制約する価値観）
    ├─────────────────┤
    │  Brand Promise   │ （ユーザーへの約束）
    ├─────────────────┤
    │  Brand Position  │ （市場における立ち位置）
    └─────────────────┘
```

### 2.2. Brand Personality（ブランドのキャラクター）

- **Rule 14.011**: ブランドのパーソナリティを **「もし人間だったら」** の視点で5つ以内の形容詞で定義する
  - 例（Stripe）: 洗練された / 技術者出身の / 率直な / 几帳面な / 革新的な
  - 例（Notion）: 遊び心のある / 最小主義的な / 包括的な / 思慮深い

### 2.3. Brand Values（ブランドの価値観）

- **Rule 14.012**: ブランドの価値観は「何をしないか」も定義する

```yaml
brand_values_template:
  values:
    - name: "Radical Simplicity"
      do: "複雑なことをシンプルに見せる"
      dont: "機能を追加してUIを複雑にする"
    - name: "Transparent by Default"
      do: "エラーメッセージも誠実に、人間的に伝える"
      dont: "「エラーが発生しました」のような無意味なメッセージ"
```

### 2.4. Mission / Vision / Values の整合性

- **Rule 14.013**: Mission（使命）・Vision（ビジョン）・Values（価値観）の3つが一貫しており、互いを強化していることを確認する

| 要素 | 定義 | 時間軸 |
|:----|:----|:------|
| **Mission** | なぜ存在するか / 現在の使命 | 現在進行形 |
| **Vision** | 目指す未来の姿 | 将来（5-10年） |
| **Values** | どのように行動するか | 不変 |

## Part III. ビジュアルアイデンティティの管理

### 3.1. ビジュアルシステムの構成要素

- **Rule 14.020**: ビジュアルアイデンティティは以下の要素で構成する

| 要素 | 管理ファイル | 更新頻度 |
|:----|:-----------|:--------|
| **Logo** | SVG（カラー・白・黒・マーク単独） | ブランド改訂時のみ |
| **Color Palette** | プライマリ・セカンダリ・アクセント | 年次レビュー |
| **Typography** | 見出し/本文/コードのフォント定義 | 年次レビュー |
| **Icon System** | 統一されたアイコンライブラリ | 機能追加時 |
| **Imagery Style** | 写真・イラスト・スクリーンショットのトーン | 年次レビュー |
| **Motion** | マイクロアニメーションの原則 | 年次レビュー |

### 3.2. ロゴ使用ガイドライン（Brand Safety）

- **Rule 14.021**: ロゴの使用に際して以下のルールを絶対遵守する

```
許可:
✅ 公式提供のSVGファイルをそのまま使用する
✅ 余白（クリアスペース）= ロゴの高さの×0.5以上を確保

禁止:
❌ ロゴのアスペクト比変更・拡大縮小の歪み
❌ ロゴに影・グラデーション・テクスチャを重ねる
❌ 承認されていないカラーバリエーションの使用
❌ 背景色との不十分なコントラスト（WCAG AA未満）
```

### 3.3. Design Token の階層管理

- **Rule 14.022**: すべてのデザイン要素は Design Token として管理し、コードとデザインツール（Figma）の単一ソース化を徹底する

```json
{
  "color": {
    "brand": {
      "primary":   { "value": "#6366F1", "description": "Indigo: メインブランドカラー" },
      "secondary": { "value": "#8B5CF6", "description": "Violet: アクセントカラー" }
    },
    "semantic": {
      "success": { "value": "{color.brand.primary}", "description": "同一でよい場合" },
      "error":   { "value": "#EF4444" }
    }
  }
}
```

## Part IV. ボイス・トーン・コピーライティング

### 4.1. Brand Voice（ブランドの声）

- **Rule 14.030**: Brand Voice を **「常に変わらない性格」** として定義し、Tone（状況に応じた調整）と区別する

```
Brand Voice（不変）:
- 率直で、余分な言葉がない
- 技術的に正確だが、専門用語で威圧しない
- 人間的で、ユーモアのセンスがある
- 自信があるが、傲慢ではない

Brand Tone（状況による）:
- エラー発生時: より共感的・サポート重視
- 新機能発表: より興奮・エネルギッシュ
- 利用規約: より正確・厳密
```

### 4.2. コピーライティングの原則

- **Rule 14.031**: すべての外部向けコピーに以下の5原則を適用する

| 原則 | ルール |
|:----|:----|
| **Clarity First** | 明確さは巧さより優先する。読んで即座に理解できるか？ |
| **Benefits over Features** | 機能ではなく、ユーザーが得るベネフィットを書く |
| **Active Voice** | 受動態より能動態を使う |
| **No Jargon** | 業界用語・社内用語をそのまま使わない |
| **Read Aloud Test** | 声に出して不自然でないか確認する |

### 4.3. Button Copy の標準原則

- **Rule 14.032**: CTAボタンのコピーは以下のルールで設計する

```
良いボタンのコピー:
✅ 動詞で始まる: 「始める」「無料で試す」「チームを招待」
✅ 具体的: 「送信」より「フィードバックを送る」
✅ ユーザー視点: 「私の○○を始める」形式の自己参照型

悪いボタンのコピー:
❌ 「はい」「OK」「送信」（何をするか不明）
❌ 「クリック」（操作の説明であり価値ではない）
❌ すべて大文字（攻撃的に見える）
```

### 4.4. Error Message 設計プロトコル

- **Rule 14.033**: エラーメッセージはブランドボイスが最も試される瞬間である。以下の構造で設計する

```
エラーメッセージの3要素:
1. 何が起きたか（What）: 状態を明確に伝える
2. なぜ起きたか（Why）: 可能な限り原因を示す
3. どうすればよいか（How）: 具体的な次のアクションを示す

良い例:
「この機能は Pro プランで利用できます。
 あなたは現在 Starter プランをご利用中です。
 [Pro プランへアップグレード] してこの機能を使いましょう。」

悪い例:
「エラーが発生しました（Error: 403）」
```

## Part V. ブランド体験（BX）の設計

### 5.1. Brand Experience Touchpoint Map

- **Rule 14.040**: ユーザーとブランドの全接点（Touchpoint）をマッピングし、各接点でのブランド体験を設計する

```
Touchpoint の例:
認知フェーズ: 広告・PR・SNS投稿・口コミ
検討フェーズ: LP・デモ動画・比較サイト掲載
体験フェーズ: Signup・Onboarding・最初の利用
継続フェーズ: メール通知・サポートチャット・インプロダクト体験
拡大フェーズ: 紹介プログラム・Upgrade フロー・イベント
```

### 5.2. Onboarding Experience としてのブランド

- **Rule 14.041**: Onboarding はブランド印象形成の最重要機会である
  - 最初の5分間でブランドの「Why」を体験させる設計とする
  - プロダクトの説明ではなく、**「ユーザーの成功ストーリーの始まり」** として演出する

### 5.3. 驚き（Delight）の設計

- **Rule 14.042**: ユーザーの期待を超える小さな驚き（Delight）をプロダクト内に意図的に設計する
  - 例: 100件目のタスク完了時の祝福アニメーション
  - 例: 誕生日メールの特別デザイン
  - 例: エラー404ページのユーモア・ブランドキャラクター
  - 例外: Delight の実装はコア機能の完成後に行う。未完成な状態での演出は逆効果

### 5.4. Community as a Brand Asset

- **Rule 14.043**: コミュニティ（Slack, Discord, Forum）をブランド体験の一部として設計する
  - コミュニティのトーンはブランドボイスに準拠させる
  - コミュニティでの公式アカウントは Brand Voice を体現する模範を示す
  - コミュニティの健全性（安全・歓迎・有益）を KPI として計測する

## Part VI. ブランド保護とガバナンス

### 6.1. Brand Governance（ブランドガバナンス）

- **Rule 14.050**: ブランドの一貫性を担保するため、以下の管理体制を構築する

| 役割 | 責務 |
|:----|:----|
| **Brand Owner** (CPO/CMO) | Brand Identity の承認・変更の最終決裁権 |
| **Brand Guardian** | ブランドガイドラインの維持・教育 |
| **Design Review** | 新しいビジュアルの承認プロセス |

### 6.2. Brand Guideline の整備

- **Rule 14.051**: ブランドガイドライン（Brand Book）を整備し、全チームがアクセスできる状態にする
  - 最低含める内容: Voice/Tone, Color, Typography, Logo Usage, Imagery
  - Figma の Brand Kit と連動させ、「最新版」を常に使えるようにする

### 6.3. 社外でのブランド使用管理

- **Rule 14.052**: パートナー・ユーザーによるブランド使用（Fan Art・紹介記事等）を管理するポリシーを公開する
  - **許可**: 非商用でのロゴ使用（Powered by ○○等）は歓迎する
  - **禁止**: 公式との混同を招く使用・商標権の侵害

## Part VII. ブランドの計測と管理

### 7.1. Brand Equity の計測指標

- **Rule 14.060**: ブランドの資産価値を以下の指標で四半期ごとに計測する

| 指標 | 計測方法 |
|:----|:---------|
| **aided Brand Awareness** | サーベイ: カテゴリ名を見てブランドを想起できるか |
| **unaided Brand Recall** | サーベイ: カテゴリの問いに対してブランドが上位に来るか |
| **Brand NPS** | 「このブランドを友人に薦めたいか？」 |
| **Share of Voice** | メディア掲載・SNS言及量 / 業界総量 |
| **Social Sentiment Score** | ポジティブ言及 / 全言及（ネガティブ含む） |

### 7.2. Brand Crisis 対応プロトコル

- **Rule 14.061**: 以下をブランドクライシスと定義し、24時間以内に公式対応を行う

```
ブランドクライシスの定義:
- ソーシャルでのネガティブ言及急増（24h内に前週比500%超）
- メジャーメディアの批判的報道
- データ漏洩・セキュリティ事故の公表
- 著名インフルエンサーからの批判

クライシス対応の5原則:
1. 迅速: 事実確認後24時間以内に公式見解
2. 誠実: 事実に基づく説明（隠蔽・矮小化禁止）
3. 共感: ユーザーへの影響を最優先に認識
4. 行動: 具体的な改善・対応策を示す
5. 追跡: 対応後の状況変化をモニタリングし続ける
```
