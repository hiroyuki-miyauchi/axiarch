> [!IMPORTANT]
> **Supreme Directive（最高指令）**
> 「ブランドは会社の内側から外側に滲み出るものであり、広告代理店が制作するものではない」
> ブランドは **「約束の集積」** である。ユーザーとのすべての接点がブランドを形成する。
> 一貫性なきブランドは信頼を破壊する。ブランドの構築は最も長期的な競争優位性の源泉である。
> AI時代において、ブランドは人間だけでなく **AIエージェントによっても「代行」** される。
> ブランドの一貫性はコード化・トークン化・自動検証されなければならない。

## Part I. ブランドの哲学

### 1.1. ブランドとは何か

- **Rule 14.001**: ブランドを「ロゴ・色・フォント」として矮小化することを禁止する。ブランドとは **「ユーザーが持つ感情的な連想の総体」** である
- **Simon Sinek の Golden Circle**:

```
Why（なぜ存在するのか）← ブランドの核心
  ↓
How（どのように実現するか）
  ↓
What（何を提供するか）← ほとんどの会社はここから始める

Rule: ブランドコミュニケーションは常に「Why」から始める
```

### 1.2. Moat としてのブランド

- **Rule 14.002**: 強いブランドは以下の経済的価値を生む
  - **CAC削減**: 認知度向上によりインバウンドが増加し、CAC が低下する
  - **価格プレミアム**: 同等の機能でも高い価格を正当化できる（Apple vs 汎用PC）
  - **採用優位**: 優秀な人材が応募する確率が上がる
  - **パートナーシップ**: 信頼あるブランドにはパートナーが集まる

### 1.3. Brand Promise（ブランドの約束）

- **Rule 14.003**: ブランドの中核に **「Brand Promise」（ブランドの約束）** を定義する
  - Brand Promise は「ユーザーが毎回期待できること」を1文で表現する
  - Brand Promise はマーケティング部門ではなく、CEO・CPO が定義する
  - 例: "Amazon: 地球上で最も顧客を大切にする企業" / "Apple: 誰もが使える革新的なテクノロジー"

### 1.4. AI時代のブランドの再定義

- **Rule 14.004**: AI時代において、ブランドは以下の3つの層で「体験」される。すべての層で一貫性を確保すること

```
┌──────────────────────────────────────────┐
│  Layer 1: Human-Delivered Brand          │
│  （人間が提供するブランド体験）              │
│  → 営業・サポート・イベント・社員の行動      │
├──────────────────────────────────────────┤
│  Layer 2: Product-Delivered Brand        │
│  （プロダクトが提供するブランド体験）         │
│  → UI/UX・マイクロコピー・通知・エラー       │
├──────────────────────────────────────────┤
│  Layer 3: AI-Delivered Brand             │
│  （AIが代行するブランド体験）                │
│  → Chatbot・AI Copilot・音声アシスタント    │
│  → AIエージェントによる自動応答・推薦        │
└──────────────────────────────────────────┘

Rule: Layer 3 は Layer 1・2 と矛盾してはならない。
AIが発する一言がブランドの約束を裏切る場合、
それは「バグ」ではなく「ブランド毀損事故」である。
```

- **Anti-Pattern**: AIチャットボットに「とりあえずフレンドリーに」と指示するだけで、Brand Voiceの定義書を参照させない → ブランド崩壊

### 1.5. ブランドの経済的価値フレームワーク

- **Rule 14.005**: ブランドの資産価値を以下のフレームワークで経営層に報告する

| 手法 | 説明 | 適用場面 |
|:----|:----|:--------|
| **Brand Strength Index** | 認知→検討→選好→ロイヤルティの4段階スコア | 四半期レビュー |
| **Brand Contribution** | 売上のうちブランドに帰属する割合の推定 | 年次事業計画 |
| **Brand-CAC Attribution** | オーガニック流入÷総流入 = ブランド寄与率 | マーケティング予算配分 |
| **Price Premium Analysis** | 競合同等品との価格差 × 販売数量 | 価格戦略レビュー |

- **Prohibition**: ブランド投資を「費用」として扱い、ROI計測なしに削減することは禁止。ブランドは減価する無形資産であり、継続投資が必要

### 1.6. ブランドの時間軸設計

- **Rule 14.006**: ブランド投資を以下の3つの時間軸で管理し、短期偏重を防止する

```
時間軸         投資配分目安   目的
─────────────────────────────────────────
短期（0-6ヶ月）   30%        Campaign: 認知獲得・リード創出
中期（6-24ヶ月）  40%        Positioning: カテゴリ内での差別化確立
長期（2-5年）     30%        Equity: ブランド資産としての信頼蓄積
```

- **Law**: 短期キャンペーンの成功指標（CPA, ROAS）のみでブランド投資を評価することを禁止する
- **Mandate**: 年次ブランド予算レビューでは、3時間軸の投資配分を明示し、経営層の承認を得ること

### 1.7. Brand Flywheel（ブランドフライホイール）

- **Rule 14.007**: ブランドを「キャンペーン」ではなく **自己強化ループ（Flywheel）** として設計する

```
Brand Flywheel:
┌─────────────────────────────────────────┐
│                                         │
│  ブランド信頼 → ユーザー獲得コスト低下   │
│       ↑                    ↓            │
│  ブランド強化  ← 成長 ← 有機的獲得      │
│                                         │
└─────────────────────────────────────────┘

1. Trust（信頼）: 一貫したブランド体験が信頼を蓄積
2. Acquisition（獲得）: 信頼がオーガニック流入を増加させCAC低下
3. Growth（成長）: 低CACが再投資可能な成長資金を生む
4. Reinforcement（強化）: 成長がブランド認知を拡大し信頼をさらに蓄積
```

- **Law**: Flywheelの各段階にKPIを設定し、どの段階がボトルネックかを四半期ごとに特定すること
- **Anti-Pattern**: ブランドを「一方通行の発信」として捉え、ループの設計を怠る → 広告費依存からの脱却不能

## Part II. ブランドアイデンティティの設計

### 2.1. Brand Pyramid（ブランドピラミッド）

- **Rule 14.010**: ブランドアイデンティティを以下の5層構造で定義する

```
    ┌─────────────────┐
    │   Brand Essence  │ （Why: 存在意義）
    ├─────────────────┤
    │ Brand Personality│ （人格・キャラクター）
    ├─────────────────┤
    │  Brand Values    │ （行動を制約する価値観）
    ├─────────────────┤
    │  Brand Promise   │ （ユーザーへの約束）
    ├─────────────────┤
    │  Brand Position  │ （市場における立ち位置）
    └─────────────────┘
```

### 2.2. Brand Personality（ブランドのキャラクター）

- **Rule 14.011**: ブランドのパーソナリティを **「もし人間だったら」** の視点で5つ以内の形容詞で定義する
  - 例（Stripe）: 洗練された / 技術者出身の / 率直な / 几帳面な / 革新的な
  - 例（Notion）: 遊び心のある / 最小主義的な / 包括的な / 思慮深い

### 2.3. Brand Values（ブランドの価値観）

- **Rule 14.012**: ブランドの価値観は「何をしないか」も定義する

```yaml
brand_values_template:
  values:
    - name: "Radical Simplicity"
      do: "複雑なことをシンプルに見せる"
      dont: "機能を追加してUIを複雑にする"
    - name: "Transparent by Default"
      do: "エラーメッセージも誠実に、人間的に伝える"
      dont: "「エラーが発生しました」のような無意味なメッセージ"
```

### 2.4. Mission / Vision / Values の整合性

- **Rule 14.013**: Mission（使命）・Vision（ビジョン）・Values（価値観）の3つが一貫しており、互いを強化していることを確認する

| 要素 | 定義 | 時間軸 |
|:----|:----|:------|
| **Mission** | なぜ存在するか / 現在の使命 | 現在進行形 |
| **Vision** | 目指す未来の姿 | 将来（5-10年） |
| **Values** | どのように行動するか | 不変 |

### 2.5. Brand Archetype（ブランド原型）

- **Rule 14.014**: ブランドのキャラクターを **12 Archetypes フレームワーク** に基づいて定義し、一貫したストーリーテリングの基盤とする

| カテゴリ | Archetype | キーワード | 例 |
|:--------|:---------|:---------|:---|
| **安定** | Caregiver | 思いやり・保護 | Johnson & Johnson |
| **安定** | Ruler | 統制・リーダーシップ | Mercedes-Benz |
| **独立** | Explorer | 自由・発見 | Patagonia |
| **独立** | Sage | 知恵・真実 | Google |
| **変革** | Hero | 勇気・達成 | Nike |
| **変革** | Magician | 変革・ビジョン | Apple |
| **帰属** | Everyman | 親しみ・共感 | IKEA |
| **帰属** | Jester | ユーモア・楽しさ | M&M's |

- **Law**: 選択したArchetypeはBrand Playbookに明記し、全コンテンツ・キャンペーンの「性格の軸」として一貫させること
- **Anti-Pattern**: キャンペーンごとにArchetypeを変更する → ブランドの「多重人格」化

### 2.6. Brand Story Framework（ブランドストーリー構造）

- **Rule 14.015**: ブランドストーリーを以下の3層構造で管理し、状況に応じて適切な層を使い分ける

```
┌──────────────────────────────────────┐
│  Origin Story（起源の物語）            │
│  → 創業者の原体験・解決したい課題      │
│  → 使用場面: About Us, 採用, 投資家   │
├──────────────────────────────────────┤
│  Customer Story（顧客の物語）          │
│  → ユーザーの変革ストーリー            │
│  → 使用場面: LP, Case Study, SNS     │
├──────────────────────────────────────┤
│  Future Story（未来の物語）            │
│  → ブランドが実現する未来像            │
│  → 使用場面: Vision, ロードマップ     │
└──────────────────────────────────────┘
```

- **Mandate**: 3層のストーリーはBrand Promiseと矛盾しないこと。整合性チェックを年次で実施する

### 2.7. Competitive Positioning Matrix（競合ポジショニング・マトリクス）

- **Rule 14.016**: 主要競合との差別化を **Perceptual Map（知覚マップ）** で可視化し、半期ごとに更新する

```
軸の選定基準:
1. ユーザーの購買決定に最も影響する2軸を選択する
2. 軸はブランド固有の強みが反映されるものを優先する

例:
  Y軸: 操作の直感性（高 ↑ 低）
  X軸: 価格帯（低← →高）

  自社 ★ は「高い直感性 × 適正価格」のポジションを獲得する
```

- **Law**: ポジショニングの変更（Repositioning）はCEO承認を必要とする戦略的意思決定であり、マーケティング部門の独断で実施してはならない

## Part III. ビジュアルアイデンティティの管理

### 3.1. ビジュアルシステムの構成要素

- **Rule 14.020**: ビジュアルアイデンティティは以下の要素で構成する

| 要素 | 管理ファイル | 更新頻度 |
|:----|:-----------|:--------|
| **Logo** | SVG（カラー・白・黒・マーク単独） | ブランド改訂時のみ |
| **Color Palette** | プライマリ・セカンダリ・アクセント | 年次レビュー |
| **Typography** | 見出し/本文/コードのフォント定義 | 年次レビュー |
| **Icon System** | 統一されたアイコンライブラリ | 機能追加時 |
| **Imagery Style** | 写真・イラスト・スクリーンショットのトーン | 年次レビュー |
| **Motion** | マイクロアニメーションの原則 | 年次レビュー |

### 3.2. ロゴ使用ガイドライン（Brand Safety）

- **Rule 14.021**: ロゴの使用に際して以下のルールを絶対遵守する

```
許可:
✅ 公式提供のSVGファイルをそのまま使用する
✅ 余白（クリアスペース）= ロゴの高さの×0.5以上を確保

禁止:
❌ ロゴのアスペクト比変更・拡大縮小の歪み
❌ ロゴに影・グラデーション・テクスチャを重ねる
❌ 承認されていないカラーバリエーションの使用
❌ 背景色との不十分なコントラスト（WCAG AA未満）
```

### 3.3. Design Token の階層管理

- **Rule 14.022**: すべてのデザイン要素は Design Token として管理し、コードとデザインツール（Figma）の単一ソース化を徹底する

```json
{
  "color": {
    "brand": {
      "primary":   { "value": "#6366F1", "description": "Indigo: メインブランドカラー" },
      "secondary": { "value": "#8B5CF6", "description": "Violet: アクセントカラー" }
    },
    "semantic": {
      "success": { "value": "{color.brand.primary}", "description": "同一でよい場合" },
      "error":   { "value": "#EF4444" }
    }
  }
}
```

### 3.4. Design Token Architecture 2.0

- **Rule 14.023**: Design Token は **DTCG (Design Token Community Group)** 標準に準拠し、以下の3層アーキテクチャで管理する

```
┌──────────────────────────────────────────┐
│  Layer 1: Global Tokens（原始値）          │
│  → color.blue.500: "#3B82F6"             │
│  → spacing.4: "16px"                     │
│  → font.family.sans: "Inter"             │
├──────────────────────────────────────────┤
│  Layer 2: Semantic Tokens（意味付け）      │
│  → color.brand.primary: "{color.blue.500}"│
│  → color.feedback.error: "{color.red.500}"│
├──────────────────────────────────────────┤
│  Layer 3: Component Tokens（部品固有）     │
│  → button.primary.bg: "{color.brand.primary}"│
│  → button.primary.text: "{color.white}"   │
└──────────────────────────────────────────┘
```

- **Law**: コンポーネント開発者はLayer 3（Component Token）のみを参照すること。Layer 1（Global Token）への直接参照は「トークンバイパス」として禁止
- **SSOT**: トークンのソース・オブ・トゥルースは `tokens.json` とし、Figma Variables と CSS Custom Properties に自動同期する

### 3.5. Multi-Theme & Dark Mode Protocol

- **Rule 14.024**: 以下のテーマバリエーションを標準サポートし、すべてのUI要素で一貫した表示を保証する

| テーマ | 要件 | 優先度 |
|:------|:----|:------|
| **Light Mode** | デフォルトテーマ | 🔴 必須 |
| **Dark Mode** | Semantic Tokenの値切替で実装 | 🔴 必須 |
| **High Contrast** | WCAG AAA準拠のコントラスト比 | 🟠 推奨 |
| **Reduced Motion** | `prefers-reduced-motion` 対応 | 🔴 必須 |

- **Law**: Dark Modeは「色反転」ではない。背景色・テキスト色・ボーダー色・シャドウ・画像の明るさすべてをSemantic Token層で個別に制御すること
- **Prohibition**: ハードコードされたカラーコード（`#ffffff`, `rgb(0,0,0)`等）のUI利用を禁止。全色はトークン経由で指定する

### 3.6. Accessibility-First Visual Design

- **Rule 14.025**: ブランドビジュアルは **WCAG 2.2 Level AA** を最低基準とし、以下の要件を満たすこと

| 要件 | 基準 | 検証ツール |
|:----|:-----|:---------|
| **テキストコントラスト** | 通常テキスト ≥ 4.5:1 / 大テキスト ≥ 3:1 | axe / Lighthouse |
| **非テキストコントラスト** | UI要素 ≥ 3:1 | Colour Contrast Analyser |
| **フォーカス表示** | 全インタラクティブ要素に可視フォーカスリング | 手動テスト |
| **テキストサイズ** | 本文 ≥ 16px（rem単位推奨） | CSS Audit |
| **色のみに依存しない** | エラー表示等で色以外の手がかり（アイコン・テキスト）を併用 | デザインレビュー |

- **Prohibition**: ブランドカラーのコントラスト比がWCAG AAを満たさない場合、そのカラーの使用用途を制限する（背景ではなくアクセント限定等）

### 3.7. Motion Design Language

- **Rule 14.026**: ブランドのモーション（アニメーション）を以下の3軸で定義し、統一された動きの「性格」を確立する

```yaml
motion_design_language:
  duration:
    instant: "100ms"    # フィードバック（ボタン押下等）
    fast: "200ms"       # 小さな状態変化（トグル、展開）
    normal: "300ms"     # 中規模の遷移（モーダル、ドロワー）
    slow: "500ms"       # 大規模な遷移（ページ遷移、ローディング完了）
  easing:
    standard: "cubic-bezier(0.2, 0, 0, 1)"   # 標準の加速・減速
    enter: "cubic-bezier(0, 0, 0, 1)"         # 画面に入る要素
    exit: "cubic-bezier(0.2, 0, 1, 1)"        # 画面から出る要素
  purpose:
    feedback: "ユーザー操作への即座の応答"
    orientation: "空間的な位置関係の理解を助ける"
    delight: "ブランドの性格を表現する微細な演出"
```

- **Law**: `prefers-reduced-motion: reduce` を検出した場合、`delight` カテゴリのアニメーションを即座に無効化すること。`feedback` と `orientation` は省略してはならない

### 3.8. Figma-Code SSOT Protocol

- **Rule 14.027**: デザインとコードの乖離を防ぐため、以下の一方向同期パイプラインを確立する

```
Figma Variables（設計の原本）
    ↓ [自動エクスポート: Tokens Studio / Style Dictionary]
tokens.json（DTCG形式）
    ↓ [ビルドパイプライン: Style Dictionary / Token Transformer]
CSS Custom Properties / Tailwind Config / etc.
    ↓ [ランタイム]
プロダクトUI
```

- **Law**: デザイナーがFigmaで変更 → トークンJSONが自動更新 → コードに反映、の単方向フローを遵守すること
- **Prohibition**: 開発者がCSS側で独自にカラー値を追加する「闇トークン」は禁止。必ずFigmaのVariablesに起源を持たせること
- **Audit**: 四半期ごとにCSS内の全カラー値・フォント値をスキャンし、トークン外の値が0件であることを検証する

## Part IV. ボイス・トーン・コピーライティング

### 4.1. Brand Voice（ブランドの声）

- **Rule 14.030**: Brand Voice を **「常に変わらない性格」** として定義し、Tone（状況に応じた調整）と区別する

```
Brand Voice（不変）:
- 率直で、余分な言葉がない
- 技術的に正確だが、専門用語で威圧しない
- 人間的で、ユーモアのセンスがある
- 自信があるが、傲慢ではない

Brand Tone（状況による）:
- エラー発生時: より共感的・サポート重視
- 新機能発表: より興奮・エネルギッシュ
- 利用規約: より正確・厳密
```

### 4.2. コピーライティングの原則

- **Rule 14.031**: すべての外部向けコピーに以下の5原則を適用する

| 原則 | ルール |
|:----|:----|
| **Clarity First** | 明確さは巧さより優先する。読んで即座に理解できるか？ |
| **Benefits over Features** | 機能ではなく、ユーザーが得るベネフィットを書く |
| **Active Voice** | 受動態より能動態を使う |
| **No Jargon** | 業界用語・社内用語をそのまま使わない |
| **Read Aloud Test** | 声に出して不自然でないか確認する |

### 4.3. Button Copy の標準原則

- **Rule 14.032**: CTAボタンのコピーは以下のルールで設計する

```
良いボタンのコピー:
✅ 動詞で始まる: 「始める」「無料で試す」「チームを招待」
✅ 具体的: 「送信」より「フィードバックを送る」
✅ ユーザー視点: 「私の○○を始める」形式の自己参照型

悪いボタンのコピー:
❌ 「はい」「OK」「送信」（何をするか不明）
❌ 「クリック」（操作の説明であり価値ではない）
❌ すべて大文字（攻撃的に見える）
```

### 4.4. Error Message 設計プロトコル

- **Rule 14.033**: エラーメッセージはブランドボイスが最も試される瞬間である。以下の構造で設計する

```
エラーメッセージの3要素:
1. 何が起きたか（What）: 状態を明確に伝える
2. なぜ起きたか（Why）: 可能な限り原因を示す
3. どうすればよいか（How）: 具体的な次のアクションを示す

良い例:
「この機能は Pro プランで利用できます。
 あなたは現在 Starter プランをご利用中です。
 [Pro プランへアップグレード] してこの機能を使いましょう。」

悪い例:
「エラーが発生しました（Error: 403）」
```

### 4.5. Multi-Context Tone Matrix（状況別トーンマトリクス）

- **Rule 14.034**: Brand Voiceは不変だが、Toneは **「状況 × チャネル × 感情」** の3軸マトリクスで設計する

| 状況 | アプリ内 | メール | SNS |
|:----|:--------|:------|:----|
| **成功・達成** | 祝福・シンプル | 詳細・次のステップ提示 | 共有可能・ビジュアル重視 |
| **エラー・障害** | 共感・即時解決 | 技術的・詳細説明 | 迅速・透明・誠実 |
| **新機能リリース** | インプロダクト案内 | ベネフィット訴求 | ティーザー・期待醸成 |
| **支払い・請求** | 明確・安心感 | 正式・証跡重視 | 使用しない |
| **セキュリティ警告** | 緊急・行動誘導 | 緊急・詳細手順 | 簡潔・公式リンク誘導 |

- **Law**: 新しいコミュニケーション種別を追加する際、このマトリクスへの登録を必須とする

### 4.6. Inclusive Language Protocol（包括的言語プロトコル）

- **Rule 14.035**: すべての公式コミュニケーションで以下の包括的言語ガイドラインを遵守する

| カテゴリ | 避けるべき表現 | 推奨表現 |
|:--------|:-------------|:--------|
| **性別** | 「彼/彼女」「主人/奥様」 | 「その方」「パートナー」 |
| **障害** | 「障害者」「健常者」 | 「障害のある方」「障害のない方」 |
| **技術レベル** | 「簡単です」「誰でもできます」 | 「次の手順で設定できます」 |
| **年齢** | 「お年寄り向け」 | 「すべての年齢の方に」 |
| **文化** | 文化固有のメタファー | ユニバーサルな表現 |

- **Prohibition**: 差別的・排他的な表現はブランド毀損として即時修正対象とする

### 4.7. Microcopy & UX Writing Standards

- **Rule 14.036**: プロダクト内のマイクロコピーを以下の分類ごとに標準化する

| UI要素 | 原則 | 例 |
|:------|:----|:---|
| **空状態（Empty State）** | 次のアクションを明示 | 「まだ投稿がありません。最初の投稿を作成しましょう」 |
| **ローディング** | 進行状況を伝える | 「データを読み込んでいます...」（不定形）/ 「3/5 完了」（定量形） |
| **ツールチップ** | 1文以内・動詞で開始 | 「クリックしてプロフィールを編集」 |
| **確認ダイアログ** | 結果を明示 | 「削除すると元に戻せません。本当に削除しますか？」 |
| **成功メッセージ** | 達成感を与える | 「保存しました ✓」 |

### 4.8. Multi-Language Copy Governance（多言語コピー管理）

- **Rule 14.037**: 多言語展開におけるコピーの品質を以下の基準で管理する

| 判定基準 | Translation（翻訳） | Transcreation（トランスクリエーション） |
|:--------|:------------------|:----------------------------------|
| **適用対象** | 技術文書、ヘルプ記事、法的文書 | タグライン、広告コピー、CTA |
| **目的** | 正確な意味の伝達 | 感情的なインパクトの再現 |
| **担当者** | 翻訳者 | コピーライター + ネイティブレビューア |
| **QA基準** | 用語統一・文法正確性 | ブランドトーン適合・文化適応 |

- **Law**: タグライン・Brand Promiseの翻訳は必ずTranscreationプロセスを経ること。直訳は禁止
- **Mandate**: 各言語の用語集（Glossary）を維持し、翻訳者間で統一すること

## Part V. AI-Native ブランドガバナンス

> **Supreme Directive**: AIがブランドを「代行」する時代において、AIの発言・生成物はすべてブランドの公式表現である。AI出力の品質管理はブランドガバナンスの中核責務となる。

### 5.1. AI生成コンテンツのブランド準拠ゲート

- **Rule 14.070**: AI生成コンテンツ（テキスト・画像・動画）は、公開前に以下の **Brand Compliance Scorecard** を通過すること

| チェック項目 | 基準 | 合格閾値 |
|:----------|:----|:--------|
| **Voice Match** | Brand Voiceの5属性との一致度 | 全属性で適合 |
| **Tone Accuracy** | 状況に応じたTone Matrixとの整合 | マトリクス準拠 |
| **Factual Accuracy** | 事実・数値の正確性 | エラー0件 |
| **Inclusive Language** | §4.6 包括的言語プロトコル準拠 | 違反0件 |
| **Visual Consistency** | ブランドカラー・タイポグラフィ準拠 | トークン準拠 |

- **Automation**: 可能な限り、LLMベースの自動チェックパイプラインを構築し、人的レビューの前段にゲートを設置すること

### 5.2. LLM Brand Persona Protocol

- **Rule 14.071**: AI Chatbot / AI Copilotのペルソナを以下の構造で定義する

```yaml
llm_brand_persona:
  name: "[ブランド名] AI Assistant"
  personality: "§2.2 Brand Personalityに準拠"
  voice: "§4.1 Brand Voiceに準拠"
  constraints:
    - 競合製品を批判しない
    - 未発表の機能・ロードマップを言及しない
    - 医療・法律・金融のアドバイスを提供しない
    - 個人情報を要求しない
    - 政治的・宗教的な立場を表明しない
  escalation:
    trigger: "ユーザーの感情が強くネガティブ / 安全に関わる相談"
    action: "人間のサポートチームへ即時エスカレーション"
```

- **Prohibition**: AIに「自分は人間です」と偽らせることは厳禁。AIであることを聞かれた場合は誠実に開示する

### 5.3. AI生成ビジュアルのブランド準拠

- **Rule 14.072**: AI画像生成をブランドアセット制作に使用する場合、以下のガードレールを遵守する

| ガードレール | 要件 |
|:----------|:----|
| **ブランドカラー強制** | プロンプトにブランドカラーパレットを明示的に指定 |
| **スタイル一貫性** | ブランドのImagery Styleガイドに準拠するプロンプトテンプレートを使用 |
| **人物表現** | 実在人物に酷似する画像の生成禁止 |
| **著作権リスク** | 既存アートワークの模倣を避けるネガティブプロンプト設定 |
| **メタデータ** | AI生成である旨をEXIF / C2PA で記録 |

### 5.4. Synthetic Media & Deepfake 防御

- **Rule 14.073**: ブランドの名前・ロゴ・創業者の肖像がAI生成のなりすまし（Deepfake）に悪用されるリスクに対し、以下の防御策を実施する

| 防御策 | 実施内容 | 頻度 |
|:------|:--------|:----|
| **モニタリング** | ブランド名+Deepfakeキーワードのソーシャル監視 | 日次 |
| **公式チャネル明示** | 公式SNSアカウント・Webサイトの一覧を公開 | 常時 |
| **C2PA署名** | 公式コンテンツにContent Credentials署名を付与 | 公開時 |
| **法的対応テンプレート** | DMCA / 各プラットフォームの報告テンプレート準備 | 事前準備 |

### 5.5. AI Agent Brand Delegation Protocol

- **Rule 14.074**: AIエージェントがブランドを代行して外部とコミュニケーションする場合、以下の権限マトリクスを遵守する

| アクション | AIの権限 | 人間の承認 |
|:---------|:--------|:---------|
| FAQ・ヘルプへの回答 | ✅ 自律可 | 不要 |
| 返金・補償の提案 | 🟠 提案のみ | 承認必須 |
| 公式声明の発信 | ❌ 禁止 | 人間が作成・発信 |
| SNSへの投稿 | 🟠 下書きのみ | 承認必須 |
| 契約・法的拘束力のある回答 | ❌ 禁止 | 人間が対応 |

- **Kill Switch**: AIエージェントの全外部コミュニケーションを30秒以内に停止できる緊急停止機構を必ず実装すること

### 5.6. Golden Dataset Protocol（ゴールデンデータセット管理）

- **Rule 14.075**: AI出力のブランド品質を担保するため、以下の **Golden Dataset（黄金基準データセット）** を管理・運用する

| データセット | 内容 | 更新頻度 |
|:----------|:----|:--------|
| **Voice Exemplars** | Brand Voiceを体現する承認済みコピー50件以上 | 四半期 |
| **Tone Samples** | 状況別（§4.5 Tone Matrix）の模範応答セット | 四半期 |
| **Visual References** | ブランド準拠の画像・デザイン参照セット | 半期 |
| **Anti-Examples** | ブランド違反の具体例（何がNGか）のセット | 随時追加 |

- **Law**: Golden DatasetはBrand Ownerの承認を経たものだけを格納する。未承認コンテンツの混入は「データ汚染」として即時修正対象
- **SSOT**: Golden Datasetのソース・オブ・トゥルースはバージョン管理されたリポジトリに格納し、AIシステムからの参照パスを一元化する

### 5.7. AI Brand Drift Detection（AIブランドドリフト検知）

- **Rule 14.076**: AIの出力がブランド基準から逸脱する「ドリフト」を以下のプロトコルで定量検知する

| 検知手法 | 内容 | 閾値 |
|:--------|:----|:-----|
| **Voice Consistency Score** | Golden Datasetとの類似度スコア（Embedding距離） | スコア < 0.85 でアラート |
| **Tone Drift Monitor** | 状況別Tone Matrixとの乖離率 | 乖離率 > 15% で自動エスカレーション |
| **Hallucination Rate** | ブランドに関する事実誤認の発生率 | 0.5% 以上で即時修正 |
| **Sentiment Deviation** | AI応答に対するユーザー感情の変化傾向 | ネガティブ率が前月比150%超 |

- **Automation**: ドリフト検知は日次バッチで自動実行し、閾値超過時はBrand Guardianに自動通知する
- **Remediation**: ドリフト検知時は①プロンプトの調整、②Golden Datasetの更新、③モデルのファインチューニングの順で対処する

### 5.8. Multi-Agent Brand Orchestration（マルチエージェントブランド統制）

- **Rule 14.077**: 複数のAIエージェントが同時にブランドを代行する場合、以下のオーケストレーションプロトコルを適用する

```
┌──────────────────────────────────────────┐
│  Brand Orchestration Layer               │
│  ┌────────────────────────────────────┐  │
│  │ Central Brand Config (SSOT)        │  │
│  │ → Brand Voice定義                  │  │
│  │ → Tone Matrix                      │  │
│  │ → 禁止事項リスト                    │  │
│  │ → Golden Dataset参照パス           │  │
│  └────────────┬───────────────────────┘  │
│               ↓                          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │Agent │ │Agent │ │Agent │ │Agent │   │
│  │Chat  │ │Email │ │Social│ │Voice │   │
│  └──────┘ └──────┘ └──────┘ └──────┘   │
└──────────────────────────────────────────┘
```

- **Law**: 各エージェントは独自のBrand Voice定義を持ってはならない。Central Brand Config（単一の設定ソース）から一律に読み込むこと
- **Anti-Pattern**: エージェントごとに個別のプロンプトでVoice/Toneを定義 → エージェント間の「人格の揺れ」が発生
- **Sync Protocol**: Central Brand Configの更新は全エージェントに15分以内に伝播すること

## Part VI. ブランド体験（BX）の設計

### 6.1. Brand Experience Touchpoint Map

- **Rule 14.040**: ユーザーとブランドの全接点（Touchpoint）をマッピングし、各接点でのブランド体験を設計する

```
Touchpoint の例:
認知フェーズ: 広告・PR・SNS投稿・口コミ
検討フェーズ: LP・デモ動画・比較サイト掲載
体験フェーズ: Signup・Onboarding・最初の利用
継続フェーズ: メール通知・サポートチャット・インプロダクト体験
拡大フェーズ: 紹介プログラム・Upgrade フロー・イベント
```

### 6.2. Onboarding Experience としてのブランド

- **Rule 14.041**: Onboarding はブランド印象形成の最重要機会である
  - 最初の5分間でブランドの「Why」を体験させる設計とする
  - プロダクトの説明ではなく、**「ユーザーの成功ストーリーの始まり」** として演出する

### 6.3. 驚き（Delight）の設計

- **Rule 14.042**: ユーザーの期待を超える小さな驚き（Delight）をプロダクト内に意図的に設計する
  - 例: 100件目のタスク完了時の祝福アニメーション
  - 例: 誕生日メールの特別デザイン
  - 例: エラー404ページのユーモア・ブランドキャラクター
  - 例外: Delight の実装はコア機能の完成後に行う。未完成な状態での演出は逆効果

### 6.4. Community as a Brand Asset

- **Rule 14.043**: コミュニティ（Slack, Discord, Forum）をブランド体験の一部として設計する
  - コミュニティのトーンはブランドボイスに準拠させる
  - コミュニティでの公式アカウントは Brand Voice を体現する模範を示す
  - コミュニティの健全性（安全・歓迎・有益）を KPI として計測する

### 6.5. Omnichannel Brand Consistency（オムニチャネル一貫性）

- **Rule 14.044**: 以下の全チャネルでブランド体験の一貫性を担保し、チャネル間の「人格の揺れ」を防止する

| チャネル | ブランド要素 | 検証頻度 |
|:--------|:----------|:--------|
| **Web** | 全Design Token準拠、Voice/Tone準拠 | リリースごと |
| **Mobile App** | プッシュ通知のトーン、アイコン・スプラッシュ画面 | リリースごと |
| **Email** | テンプレートのブランドカラー・フッター統一 | 月次 |
| **Support** | 返信テンプレートのVoice準拠 | 四半期 |
| **Physical** | 名刺・ノベルティ・イベントブース | イベント前 |

### 6.6. Product-Led Brand Experience

- **Rule 14.045**: プロダクト内でブランドを体験させる以下の「ブランドモーメント」を意図的に設計する

| モーメント | トリガー | ブランド表現 |
|:---------|:--------|:----------|
| **Welcome** | 初回ログイン | Brand Promiseを体現する歓迎体験 |
| **Milestone** | 利用回数・期間の節目 | 達成感の演出（§6.3連携） |
| **Recovery** | エラー・障害からの復帰 | 誠実さ・安心感の提供 |
| **Upgrade** | プランアップグレード | 感謝・新しい可能性の提示 |

### 6.7. Brand Ritual Design（ブランドリチュアル設計）

- **Rule 14.046**: ユーザーとの定期的な接触ポイント（Ritual）を設計し、ブランドの「存在感」を維持する

| リチュアル | 頻度 | 目的 |
|:---------|:----|:----|
| **Weekly Digest** | 週次 | 今週の活動サマリー・次のアクション提案 |
| **Monthly Insight** | 月次 | データに基づくパーソナライズされた洞察 |
| **Annual Review** | 年次 | 1年間の利用データの振り返り（Spotify Wrapped式） |

- **Law**: リチュアルは「価値提供」でなければならない。「売り込み」を主目的としたリチュアルはブランド信頼を毀損する

## Part VII. グローバルブランド・ローカライゼーション

> **Supreme Directive**: グローバルブランドは「画一的な翻訳」ではなく、「文化的な共鳴」を目指す。Global Core（不変の核）を保ちながら、Local Flex（地域適応）の余地を設計する。

### 7.1. Localization vs Transcreation Protocol

- **Rule 14.080**: コンテンツの種類に応じて、Translation（翻訳）とTranscreation（トランスクリエーション）を使い分ける

| コンテンツ種別 | 手法 | 判定基準 |
|:------------|:----|:--------|
| **法的文書・利用規約** | Translation | 正確性が最優先 |
| **ヘルプドキュメント** | Translation + ローカル例示 | 正確性 + 文化的文脈 |
| **UI/UXコピー** | Localization | 自然さ + 操作性 |
| **タグライン・広告** | Transcreation | 感情的インパクトの再現 |
| **Brand Promise** | Transcreation | ブランドの核心を文化的に再解釈 |

### 7.2. Cultural Adaptation Governance（文化的適応ガバナンス）

- **Rule 14.081**: 以下の文化的要素についてローカル市場ごとの禁忌リストを管理し、ブランド表現の事故を防止する

| 要素 | リスク | 対策 |
|:----|:-----|:----|
| **色彩** | 白=喪（東アジア）、緑=イスラム神聖色 | 地域別カラーコンテキストマップの維持 |
| **ジェスチャー** | サムズアップ=侮辱（中東一部） | ジェスチャーアイコンの地域別レビュー |
| **動物** | 犬=不浄（一部文化） | マスコット・イラストの文化的検証 |
| **数字** | 4=不吉（東アジア）、13=不吉（西洋） | プライシング・UIでの回避 |

### 7.3. Regional Brand Variation Policy

- **Rule 14.082**: グローバルブランドの地域適応は **「Global Core + Local Flex」** モデルで管理する

```
Global Core（変更不可）:
✅ Brand Promise
✅ Brand Values
✅ ロゴ・ブランドマーク
✅ プライマリカラー

Local Flex（地域適応可）:
🟠 セカンダリカラーの追加
🟠 イメージ写真のローカル差替え
🟠 トーンの文化的調整
🟠 キャンペーンのローカル企画
```

### 7.4. RTL & CJK Typography Protocol

- **Rule 14.083**: 右書き（RTL）言語およびCJK（中日韓）言語に対応する際、以下のタイポグラフィ要件を遵守する

| 要件 | RTL（アラビア語・ヘブライ語） | CJK（中国語・日本語・韓国語） |
|:----|:------------------------|:--------------------------|
| **レイアウト方向** | CSS `direction: rtl` 必須 | LTR維持（縦書き対応は任意） |
| **フォント** | Noto Sans Arabic等のRTL対応フォント | Noto Sans CJK / 游ゴシック等 |
| **行間** | 1.6〜1.8（diacritics対応） | 1.7〜2.0（漢字の可読性確保） |
| **改行** | 単語単位（word-break） | 文字単位（CSS `word-break: break-all` 禁止、`overflow-wrap` 使用） |

## Part VIII. ブランドアーキテクチャ & ポートフォリオ

> **Supreme Directive**: プロダクトラインの拡大に伴い、ブランド構造が「偶発的に」複雑化することを防止する。ブランドアーキテクチャは意図的に設計・管理される構造物である。

### 8.1. Brand Architecture Model（ブランド構造モデル）

- **Rule 14.090**: 組織のブランド構造を以下の4モデルから選択し、明文化する

| モデル | 構造 | 例 | 適用基準 |
|:------|:----|:---|:--------|
| **Branded House** | 単一ブランドで全製品を展開 | Google, FedEx | 製品間のシナジーが高い |
| **House of Brands** | 製品ごとに独立ブランド | P&G, Unilever | 製品ごとにターゲットが異なる |
| **Endorsed** | 親ブランドが子ブランドを保証 | Marriott Bonvoy | 信頼の移転が必要 |
| **Hybrid** | 上記の組み合わせ | Amazon | 事業の多角化 |

- **Law**: ブランド構造の変更はCEO・CPO承認の戦略的意思決定であり、年次レビューの議題とする

### 8.2. Sub-Brand & Product Line Naming（サブブランド命名規則）

- **Rule 14.091**: 新プロダクト・機能の命名は以下のプロセスを経ること

```
命名プロセス:
1. 候補生成: 3-5案を作成
2. 商標チェック: 既存商標との抵触確認
3. ドメイン/ハンドル確認: .com / SNSハンドルの取得可能性
4. 言語チェック: 主要言語での不適切な意味がないか確認
5. Brand Architecture適合: 親ブランドとの関係性が明確か
6. 承認: Brand Owner (CPO/CMO) の最終承認
```

### 8.3. M&A Brand Integration Protocol

- **Rule 14.092**: 企業買収後のブランド統合は以下の4戦略から選択し、12ヶ月以内にロードマップを策定する

| 戦略 | 説明 | 適用基準 |
|:----|:----|:--------|
| **Absorb** | 被買収ブランドを廃止し、自社ブランドに完全統合 | 機能統合が目的の買収 |
| **Endorse** | 「by [親ブランド]」形式で保証 | 被買収ブランドに一定の認知がある |
| **Maintain** | 被買収ブランドを独立維持 | ターゲット顧客が異なる |
| **Retire** | 段階的に被買収ブランドを終了 | 重複製品の整理 |

### 8.4. Feature Branding Protocol（機能ブランディング）

- **Rule 14.093**: プロダクト内の個別機能に固有名を付与する場合、以下の基準を満たすこと

| 基準 | 要件 |
|:----|:----|
| **差別化** | 競合の同等機能と明確に区別できる名称 |
| **発音容易性** | 主要言語で容易に発音・記述できる |
| **ブランド整合** | 親ブランドのArchetype・Personalityと整合 |
| **拡張性** | 将来の機能拡張時に名称が制約にならない |

- **Anti-Pattern**: すべての機能に固有名を付与する「名前インフレ」→ ユーザーの認知負荷増大

### 9.4. IP & Trademark Protection Protocol

- **Rule 14.053**: ブランド名・ロゴ・タグラインの知的財産を以下の戦略で保護する

| **ドメイン防衛** | .com / .co / 国別TLDの確保 | 🔴 必須 |

| **類似ドメイン監視** | タイポスクワッティング検知 | 🟠 推奨 |

### 9.5. Co-Branding & Partnership Governance

- **Rule 14.054**: パートナーとのブランド共演は以下の承認プロセスを経ること

| ステップ | 内容 | 承認者 |

### 9.6. Platform-Specific Brand Policy

- **Rule 14.055**: 各プラットフォームのブランド表示基準を個別に管理する

| プラットフォーム | 固有要件 |
|:-------------|:--------|
| **App Store** | アイコン・スクリーンショット・App名のガイドライン準拠 |
| **Google Play** | Feature Graphic・短縮説明文のブランド一貫性 |
| **SNS各種** | プロフィール画像・バナー・Bio文の統一 |

### 9.7. Brand Audit Protocol（ブランド監査プロトコル）

- **Rule 14.056**: 年次ブランド監査を実施し、以下のチェックリストで一貫性スコアを算出する

| 監査項目 | 合格基準 |
|:--------|:--------|
| ロゴ使用の正確性 | §3.2 ガイドライン100%準拠 |
| カラー・トークンの一貫性 | 闇トークン0件 |
| Voice/Tone の一貫性 | §4.1 Brand Voice適合率 ≥ 90% |
| AI生成コンテンツのブランド準拠 | §5.1 Scorecard通過率 ≥ 95% |

### 9.8. Brand Security Operations Center（ブランドSOC）

- **Rule 14.057**: ブランドに対する脅威をリアルタイムで監視・対応する **Brand Security Operations Center（BrandSOC）** を設計する

| 監視領域 | 脅威 | 対応 |
|:--------|:----|:-----|
| **Impersonation** | 偽公式アカウント / フィッシングサイト | 検知後24h以内にプラットフォーム報告 |
| **Counterfeit** | 偽造品 / 非公認リセラー | 法務チーム連携で停止措置 |
| **Data Leak** | ブランドアセット（未公開ロゴ等）の流出 | 流出経路特定と封じ込め |
| **Reputation Attack** | 組織的なネガティブキャンペーン | PR + 法務の合同対応チーム招集 |
| **AI Abuse** | ブランドを悪用したAI生成コンテンツ | §5.4 Deepfake防御プロトコル連携 |

- **Mandate**: BrandSOCは少なくとも月次でインシデントサマリーを経営層に報告すること
- **Escalation**: §12.3 Crisis Severity Escalation Matrix と連動し、自動エスカレーションを実行する

## Part XI. ブランドとサステナビリティ / 倫理

> **Supreme Directive**: ブランドの社会的責任は「飾り」ではなく、Brand Promiseの一部である。サステナビリティの主張には、必ず検証可能な根拠を伴わせること。

### 11.1. Brand Purpose & ESG Alignment

- **Rule 14.110**: ブランドパーパスとESG（環境・社会・ガバナンス）戦略の整合性を年次で検証する
  - パーパスが具体的なESGアクションに結びついていること
  - 「言うだけ」のパーパスウォッシングを防止する具体的KPIを設定すること

### 11.2. Anti-Greenwashing Protocol

- **Rule 14.111**: サステナビリティに関するブランドメッセージは以下の基準を満たすこと

| 基準 | 要件 |
|:----|:----|
| **具体性** | 「環境に優しい」等の曖昧な表現禁止。定量データで裏付け |
| **検証可能性** | 第三者認証またはデータソースを明記 |
| **全体像** | 一部の改善だけを強調し全体の負の影響を隠さない |
| **法的準拠** | EU Green Claims Directive / FTC Green Guides 準拠 |

### 11.3. Ethical Advertising Standards

- **Rule 14.112**: 広告・マーケティングにおいて以下の倫理基準を絶対遵守する

| 禁止事項 | 理由 |
|:--------|:----|
| **Dark Pattern** | ユーザーの意図に反する誘導はブランド信頼を破壊 |
| **脆弱層の搾取** | 子ども・高齢者・障害者をターゲットにした搾取的マーケティング |
| **虚偽の希少性** | 「残り1個」等の虚偽のFOMO演出 |
| **隠れたコスト** | 最終価格に含まれない追加費用の隠蔽 |

### 11.4. DEI in Brand Representation

- **Rule 14.113**: ブランドビジュアル・コンテンツにおいて多様性・包括性を確保する

| 要件 | 基準 |
|:----|:----|
| **人物表現** | 年齢・性別・民族・体型の多様性を反映 |
| **アクセシビリティ** | 障害のある方の表現を自然に含める |
| **ステレオタイプ回避** | 性別・文化に基づくステレオタイプ的表現の禁止 |
| **レビュープロセス** | DEI視点のレビューをクリエイティブ承認プロセスに組み込む |

### 11.5. AI Ethics in Brand（AIブランド倫理）

- **Rule 14.114**: AIがブランドを代行する際、以下の倫理基準を絶対遵守する

| 倫理原則 | 要件 | 検証方法 |
|:--------|:----|:--------|
| **透明性** | AI生成であることの開示義務（EU AI Act準拠） | コンテンツメタデータに自動記録 |
| **公平性** | AI出力における人種・性別・年齢等のバイアス排除 | 四半期ごとのバイアス監査 |
| **説明責任** | AI判断の根拠を人間が説明できること | 判断ログの保存義務（90日間） |
| **非操作性** | AIによる心理的操作（ダークパターン）の禁止 | UXレビューでの倫理チェック |
| **データ最小化** | ブランド体験に不要な個人データの収集禁止 | プライバシー影響評価（PIA）の実施 |

- **Law**: AIブランド倫理違反は、技術的バグではなく **ブランドガバナンス違反** として扱い、Brand Ownerへの即時報告を義務化する
- **Cross-Reference**: `000_ai_guidelines`（AI利用ガイドライン）、§5 AI-Nativeブランドガバナンスと連携

## Part XII. ブランドの計測と管理

### 12.1. Brand Equity の計測指標

- **Rule 14.060**: ブランドの資産価値を以下の指標で四半期ごとに計測する

| 指標 | 計測方法 |
|:----|:---------|
| **aided Brand Awareness** | サーベイ: カテゴリ名を見てブランドを想起できるか |
| **unaided Brand Recall** | サーベイ: カテゴリの問いに対してブランドが上位に来るか |
| **Brand NPS** | 「このブランドを友人に薦めたいか？」 |
| **Share of Voice** | メディア掲載・SNS言及量 / 業界総量 |
| **Social Sentiment Score** | ポジティブ言及 / 全言及（ネガティブ含む） |

### 12.2. Brand Crisis 対応プロトコル

- **Rule 14.061**: 以下をブランドクライシスと定義し、24時間以内に公式対応を行う

```
ブランドクライシスの定義:
- ソーシャルでのネガティブ言及急増（24h内に前週比500%超）
- メジャーメディアの批判的報道
- データ漏洩・セキュリティ事故の公表
- 著名インフルエンサーからの批判

クライシス対応の5原則:
1. 迅速: 事実確認後24時間以内に公式見解
2. 誠実: 事実に基づく説明（隠蔽・矮小化禁止）
3. 共感: ユーザーへの影響を最優先に認識
4. 行動: 具体的な改善・対応策を示す
5. 追跡: 対応後の状況変化をモニタリングし続ける
```

### 12.3. Crisis Severity Escalation Matrix

- **Rule 14.062**: クライシスの深刻度を3段階で分類し、対応レベルを自動決定する

| レベル | 定義 | 対応チーム | 初動期限 |
|:------|:----|:---------|:--------|
| **L1 (Minor)** | 個別の苦情・小規模ネガティブ言及 | Brand Guardian | 24時間 |
| **L2 (Major)** | メディア報道・インフルエンサー批判 | Brand Owner + PR | 6時間 |
| **L3 (Critical)** | データ漏洩・法的訴訟・CEO関与が必要 | CEO + 全経営陣 | 2時間 |

### 12.4. AI-Powered Brand Monitoring

- **Rule 14.063**: AIを活用した以下のリアルタイムブランド監視システムを構築する

| 監視項目 | ツール/手法 | アラート基準 |
|:--------|:----------|:----------|
| **Sentiment急変** | NLP感情分析 | ネガティブ率が前日比200%超 |
| **Brand Mention量** | ソーシャルリスニング | 前週比300%超の急増 |
| **Fake Review検知** | パターン分析 | 不自然なレビュー集中 |
| **Logo不正使用** | 画像認識AI | ガイドライン違反の使用検知 |

## Part XIII. GEO & E-E-A-T ブランド信頼戦略

> **Supreme Directive**: AI検索エンジン（Generative Engine）時代において、ブランドの信頼性は「検索結果に引用されるか」で決まる。E-E-A-T（Experience, Expertise, Authoritativeness, Trustworthiness）はSEOの指標ではなく、ブランド戦略の基盤である。

### 13.1. GEO (Generative Engine Optimization) Protocol

- **Rule 14.120**: AI検索エンジンにブランドが正しく引用されるため、以下の最適化を実施する

| 施策 | 内容 |
|:----|:----|
| **Structured Data** | Schema.org マークアップでブランド情報を機械可読化 |
| **Answer-First Content** | 質問に対する明確な回答をコンテンツ冒頭に配置 |
| **Citation Worthiness** | 独自データ・調査・専門家見解を含むコンテンツ制作 |
| **LLMs.txt** | AI検索エンジン向けのブランド情報サマリーページを公開 |

### 13.2. E-E-A-T Brand Signal Design

- **Rule 14.121**: Google品質評価ガイドラインのE-E-A-Tシグナルをブランド戦略に組み込む

| E-E-A-T要素 | ブランド施策 |
|:----------|:----------|
| **Experience** | 実際のプロダクト使用体験に基づくコンテンツ制作 |
| **Expertise** | 業界エキスパートの寄稿・監修を明示 |
| **Authoritativeness** | 権威あるメディア・機関からの被リンク獲得 |
| **Trustworthiness** | SSL・プライバシーポリシー・著者情報の透明性 |

### 13.3. Brand Knowledge Graph Management

- **Rule 14.122**: Google Knowledge Graph・Wikidata でのブランド情報を正確に管理する
  - Google Business Profileの最新化
  - Wikipedia記事の正確性監視（直接編集ではなく、情報提供を通じた修正）
  - Wikidata エントリの構造化データ管理

### 13.4. Share of Model Audit Protocol（AIモデル内ブランド認知監査）

- **Rule 14.123**: 主要LLMにおけるブランドの認知状態を以下のプロトコルで定期監査する

| 監査項目 | 手法 | 頻度 |
|:--------|:----|:-----|
| **ブランド認知正確性** | 「[ブランド名]とは何ですか？」のプロンプトテスト | 月次 |
| **カテゴリ内引用率** | 「[カテゴリ]のおすすめは？」での言及率 | 月次 |
| **競合比較精度** | 「[ブランド] vs [競合]」の比較回答の正確性 | 四半期 |
| **ハルシネーション検知** | ブランドに関する事実誤認の有無 | 月次 |

- **対象モデル**: GPT-4o, Gemini, Claude, Llama, Mistral（主要5モデル以上）
- **Remediation**: ハルシネーション検知時は、公式Webサイト・プレスリリース・構造化データの更新により「正しい情報」をAIの学習ソースに浸透させる
- **Cross-Reference**: Part XVIII Share of Model & AI Discoverability と連携

## Part XIV. ブランドの進化 & リブランド管理

> **Supreme Directive**: ブランドは生き物である。市場環境・競合状況・自社の成長に応じて「進化」する必要がある。ただし進化は「計画的」でなければならない。

### 14.1. Brand Evolution vs Rebrand

- **Rule 14.130**: ブランドの変更を以下の3段階で分類し、適切なプロセスを適用する

| 変更レベル | 説明 | 承認者 | 例 |
|:---------|:----|:------|:---|
| **Refresh** | 微調整（フォント変更、カラー微修正） | Brand Guardian | Slack 2019 |
| **Evolution** | 中規模変更（ロゴ刷新、Voice更新） | Brand Owner | Instagram 2022 |
| **Rebrand** | 全面刷新（ブランド名・アイデンティティ変更） | CEO + 取締役会 | Meta 2021 |

### 14.2. Rebrand Risk Assessment

- **Rule 14.131**: リブランドを検討する際、以下のリスク評価を実施する

| リスク項目 | 評価内容 |
|:---------|:--------|
| **既存認知の喪失** | 現在のブランド認知度の定量化 |
| **SEO影響** | ドメイン変更・URL変更によるSEO損失予測 |
| **顧客離反** | 既存顧客のブランド愛着度調査 |
| **コスト** | 全タッチポイントの変更に必要な総コスト算出 |
| **法的リスク** | 新ブランド名の商標クリアランス |

### 14.3. Brand Sunset Protocol

- **Rule 14.132**: ブランド・サブブランドの廃止は以下のプロセスで管理する

```
Brand Sunset プロセス:
1. 告知期間: 廃止の6ヶ月前から段階的に告知
2. 移行支援: 代替製品・サービスへの移行パスを明示
3. アーカイブ: ブランドアセットのアーカイブ保存
4. リダイレクト: WebサイトのURLリダイレクト設定（最低2年維持）
5. 事後モニタリング: 廃止後の顧客反応を3ヶ月間モニタリング
```

### 14.4. Brand Evolution Playbook for AI Era（AI時代のブランド進化戦略）

- **Rule 14.133**: AI時代におけるブランドの進化を以下の3フェーズで計画する

```
Phase 1: AI-Assisted（現在）
→ AIはブランド表現を「支援」する
→ 人間がすべてのブランド判断を行い、AIは実行を支援

Phase 2: AI-Augmented（中期: 1-2年）
→ AIがブランド表現を「拡張」する
→ AIが定型的なブランド判断を自律実行、人間は例外対応に集中

Phase 3: AI-Native（長期: 3-5年）
→ AIがブランド体験の「主要チャネル」となる
→ ブランドの大半がAI経由で体験され、人間は戦略・クリエイティブに特化
```

- **Mandate**: 各フェーズでの移行基準（AIの品質スコア、ブランド準拠率、インシデント率）を事前に定義し、基準を満たさない限り次フェーズに移行しないこと
- **Anti-Pattern**: AIの技術的能力だけで移行判断し、ブランド品質の検証を怠る → 「AI化したがブランドが崩壊した」状態
- **Law**: フェーズ移行はBrand Owner + CTO の共同承認を必要とする戦略的意思決定である

## Part XV. Sonic & Sensory Branding（ソニック・感覚ブランディング）

> **Supreme Directive**: 視覚偏重のブランド設計はもはや不完全である。音声AI・ポッドキャスト・空間オーディオの普及により、ブランドは「聴覚」を含む多感覚で設計されなければならない。

### 15.1. Sonic Identity Architecture（ソニックアイデンティティ設計）

- **Rule 14.140**: ソニックブランドDNAを以下の4層で定義する

| 層 | 要素 | 用途 |
|:--|:----|:-----|
| **Sonic Logo** | 3秒以内の認識可能なサウンドマーク | アプリ起動、動画エンディング |
| **Brand Music System** | ブランドの音楽的DNA（テンポ、キー、楽器） | BGM、CM、イベント |
| **UI Sound Design** | 操作フィードバック音・通知音・完了音 | プロダクト内 |
| **Brand Silence** | 意図的な「無音」の設計 | 集中モード、瞑想機能 |

- **Law**: Sonic Logoは視覚ロゴと同等のブランド資産として商標登録を検討すること
- **Anti-Pattern**: UI音を「おまけ」として後付け設計 → ブランドの聴覚体験が統一されない

### 15.2. Voice UI & Conversational Brand（音声UI・会話型ブランド）

- **Rule 14.141**: 音声アシスタント・IVR・ポッドキャスト向けのブランドボイスを以下で設計する

```
Voice Physical Attributes:
- Pitch（声の高さ）: ブランドの性格に合致する音域
- Tempo（テンポ）: 発話速度の基準値
- Tone（声色）: 温かみ / 権威性 / 親しみやすさのバランス
- Pause（間）: 情報消化のための間の設計

TTS Brand Validation:
1. TTS出力をBrand Voice属性と照合
2. 自然さスコア ≥ 4.0/5.0（ユーザー評価）
3. 感情認識テストでの意図一致率 ≥ 85%
```

### 15.3. Podcast & Audio Content Governance（音声コンテンツガバナンス）

- **Rule 14.142**: ブランデッドオーディオコンテンツの品質基準

| 要素 | 基準 |
|:----|:-----|
| **イントロ/アウトロ** | Sonic Logoを含む統一フォーマット |
| **BGM** | Brand Music Systemに準拠した楽曲選定 |
| **ナレーション** | Brand Voice属性に合致した話者選定 |
| **音質** | 最低ビットレート128kbps、ノイズフロア-60dB以下 |

### 15.4. Multi-Sensory Brand Experience（多感覚ブランド体験）

- **Rule 14.143**: 視覚以外の感覚チャネルを統合した多感覚ブランド体験を設計する

| 感覚 | ブランド表現 | 適用場面 |
|:----|:----------|:--------|
| **聴覚** | Sonic Identity（§15.1） | デジタル・イベント |
| **触覚** | ハプティクスパターン（振動フィードバック） | モバイルアプリ |
| **嗅覚** | ブランドの香り設計 | 物理店舗・イベント |
| **味覚** | ブランデッドリフレッシュメント | イベント・福利厚生 |

- **Mandate**: 各感覚チャネルのブランド表現は、Brand Personalityの5属性（§2.2）と一貫していること

## Part XVI. Spatial Computing & Immersive Brand（空間コンピューティング・没入型ブランド）

> **Supreme Directive**: AR/VR/MR空間において、ブランドは「画面」から「空間」へ拡張される。2Dのガイドラインだけではブランドを保護できない。

### 16.1. Spatial Brand Architecture（空間ブランド設計）

- **Rule 14.150**: AR/VR/MR空間におけるブランドプレゼンスを以下の原則で設計する

| 原則 | 内容 |
|:----|:-----|
| **Spatial Consistency** | 3Dブランドアセットが2Dと同一のブランドDNAを維持 |
| **Context Awareness** | 物理環境に応じたブランド表現の適応（明るさ、広さ） |
| **Non-Intrusive** | ユーザーの空間を侵害しない礼儀あるブランド配置 |
| **Interaction Paradigm** | 視線・ジェスチャー・音声によるブランドインタラクション設計 |

- **Law**: 3Dロゴ・空間UIの承認はBrand Guardianの事前レビューを必須とする

### 16.2. Immersive Brand Experience Design（没入型ブランド体験）

- **Rule 14.151**: メタバース・バーチャルイベント空間でのブランド体験ガイドライン

| 要素 | ガイドライン |
|:----|:----------|
| **Virtual Space Design** | ブランドカラー・マテリアルの3D空間への変換ルール |
| **Avatar Branding** | 公式アバターのデザインガイドライン |
| **Spatial Audio** | §15.1 Sonic Identity の3D空間適用 |
| **Interaction Design** | ブランド世界観に沿ったインタラクション設計 |

### 16.3. Context-Aware Brand Adaptation（コンテキスト適応型ブランド）

- **Rule 14.152**: ユーザーの行動・環境に基づくリアルタイムブランド表現の適応

```
Context Signal → Brand Adaptation:
- 時間帯 → ライト/ダークテーマの自動切替
- 活動状態 → 集中時は通知抑制、休憩時はDelight表示
- 感情信号 → エラー後はToneをより共感的に調整
- 環境光 → AR表示のコントラスト自動調整
```

- **Law**: コンテキスト適応は常にユーザーのプライバシーを尊重し、§11.5 AI Ethics in Brand に準拠すること
- **Prohibition**: ユーザーの感情状態を利用した購買誘導は§11.3 倫理的広告基準に違反する

## Part XVII. Community-Led Brand & Web3（コミュニティ駆動型ブランド & Web3）

> **Supreme Directive**: ブランドは企業が「発信」するものから、コミュニティと「共創」するものへと進化している。コミュニティの力をブランド資産として戦略的に設計・管理する。

### 17.1. Community Flywheel Design（コミュニティフライホイール設計）

- **Rule 14.160**: コミュニティを核としたブランド成長のフライホイールを設計する

```
Community Flywheel:
┌──────────────────────────────────────────┐
│  Member → Contributor → Advocate         │
│    ↑                         ↓           │
│  Value ← Brand Growth ← Co-Creator      │
└──────────────────────────────────────────┘

段階:
1. Member: 参加・消費する受動的メンバー
2. Contributor: コンテンツ・回答を提供する能動的メンバー
3. Advocate: 外部にブランドを推薦するアンバサダー
4. Co-Creator: ブランドの方向性に影響を与える共創者
```

- **Law**: 各段階の移行トリガーとインセンティブを設計し、自然な上昇を促進すること

### 17.2. UGC & Co-Creation Governance（UGC・共創ガバナンス）

- **Rule 14.161**: ユーザー生成コンテンツ（UGC）のブランド準拠ガバナンス

| 項目 | ルール |
|:----|:------|
| **品質基準** | UGCがブランドと並列表示される場合の最低品質基準を定義 |
| **IP権利** | UGC利用時の権利帰属・ライセンスを明確化（利用規約に明記） |
| **モデレーション** | ブランド毀損リスクのあるUGCの検知・対応プロトコル |
| **クレジット** | コクリエイターへの適切なアトリビューション |

- **Anti-Pattern**: UGCを無許可で公式コンテンツに流用 → 法的リスク + コミュニティ信頼の崩壊

### 17.3. Token-Gated Brand Experience（トークンゲート型ブランド体験）

- **Rule 14.162**: Web3技術を活用したブランドロイヤルティ・メンバーシップの設計

| 手法 | 用途 | ガバナンス |
|:----|:-----|:---------|
| **NFT Membership** | 限定コンテンツ・機能へのアクセス権 | 投機目的の抑制策を必ず設計 |
| **SBT（Soul Bound Token）** | 譲渡不可の実績・認定バッジ | コミュニティ貢献の可視化 |
| **Reputation Score** | オンチェーン透明性のある評価 | スコアリング基準の公開義務 |

- **Law**: Web3施策はブランドの信頼性を「強化」するものでなければならない。投機・FOMO煽動への利用は§11.3 倫理基準に違反する

### 17.4. DAO-Adjacent Brand Governance（コミュニティ参加型ブランドガバナンス）

- **Rule 14.163**: コミュニティがブランドの方向性に参加できるガバナンスフレームワーク

| 参加領域 | コミュニティの権限 | 最終決定権 |
|:--------|:--------------|:---------|
| **機能優先順位** | 投票による優先度提案 | プロダクトチーム |
| **コンテンツテーマ** | テーマ提案・投票 | Brand Guardian |
| **ブランドイベント** | イベント企画への参加 | Brand Owner |
| **Brand Identity変更** | フィードバック提供 | CEO（最終承認権は不可譲） |

## Part XVIII. Share of Model & AI Discoverability（AIモデル内ブランド認知戦略）

> **Supreme Directive**: AI時代において「LLMがブランドをどう認識しているか」は、検索エンジンでの順位と同等以上に重要である。Share of Model（AIモデル内でのブランドシェア）は新たなブランド戦略KPIである。

### 18.1. Share of Model Audit Framework（SoM監査フレームワーク）

- **Rule 14.170**: 主要LLMにおけるブランド認知を定量的に監査する

```
Share of Model (SoM) 監査プロセス:
1. プロンプトテストセットの定義（50問以上）
   - ブランド直接質問: 「[ブランド]とは？」
   - カテゴリ質問: 「[カテゴリ]のおすすめは？」
   - 比較質問: 「[ブランド] vs [競合]」
2. 主要LLM（GPT-4o, Gemini, Claude, Llama, Mistral）で実行
3. 回答の正確性・引用率・感情をスコアリング
4. 月次レポートとしてBrand Ownerに報告
```

- **KPI**: SoM Score = (正確な引用数 / 総テスト数) × 100
- **目標**: SoM Score ≥ 70%（カテゴリリーダーの基準）

### 18.2. Machine-Readable Brand Assets（機械可読ブランドアセット）

- **Rule 14.171**: AIエージェントが正しくブランドを理解するための機械可読アセットを整備する

| アセット | 内容 | フォーマット |
|:--------|:----|:----------|
| **LLMs.txt** | AIエンジン向けブランド情報サマリー | テキスト |
| **Brand Schema** | Schema.org準拠の構造化ブランドデータ | JSON-LD |
| **Brand API** | プログラマティックなブランド情報提供 | REST/GraphQL |
| **Brand Knowledge Base** | FAQ・用語集の構造化データ | Markdown + JSON |

### 18.3. AI Agent Commerce Readiness（AIエージェント商取引対応）

- **Rule 14.172**: AIエージェント（A2A / MCP）がブランドの製品を自律的に推薦・購入する時代への準備

| 準備項目 | 内容 |
|:--------|:-----|
| **Machine-Readable Pricing** | 構造化された価格データの公開 |
| **Product Catalog API** | AIエージェントが参照可能な製品カタログ |
| **Agent-Facing Brand Brief** | AIエージェント向けの簡潔なブランドガイドライン |
| **Trust Signal** | 信頼性を示すデジタル証明（C2PA、Verified Org等） |

### 18.4. Brand Narrative Seeding for LLMs（LLMブランドナラティブ浸透戦略）

- **Rule 14.173**: LLMの学習データに正確なブランドナラティブを浸透させる戦略

| 施策 | 手法 |
|:----|:-----|
| **Authority Source Publishing** | 権威あるメディアでの一貫したブランド情報発信 |
| **Knowledge Graph Management** | Wikipedia / Wikidata / Crunchbase の情報正確性管理 |
| **Structured Content** | Answer-First形式の公式コンテンツ（§13.1 GEO連携） |
| **Citation Worthiness** | 原典データ・独自リサーチの定期公開 |

- **Cross-Reference**: §13 GEO & E-E-A-T ブランド信頼戦略と連携

## Part XIX. Data-Driven Brand Intelligence 2.0（データ駆動型ブランドインテリジェンス 2.0）

> **Supreme Directive**: 従来の四半期サーベイによるブランド測定は「遅すぎる」。AIとリアルタイムデータを活用した「常時接続型」ブランドインテリジェンスへ移行し、ブランドをプロダクトと同じ速度で最適化する。

### 19.1. Continuous Brand Tracking（連続ブランド追跡）

- **Rule 14.180**: 四半期サーベイから「常時接続型」ブランド追跡への移行

| 手法 | 内容 | 頻度 |
|:----|:-----|:-----|
| **Always-On Survey** | マイクロサーベイ（3問以下）の常時配信 | 日次 |
| **Social Listening AI** | NLP感情分析によるリアルタイム感情追跡 | リアルタイム |
| **Search Signal** | ブランド検索ボリューム・トレンド分析 | 日次 |
| **Product Signal** | プロダクト内行動データからのブランド推定 | リアルタイム |

### 19.2. 3-Layer Brand Metrics Architecture（3層ブランド指標）

- **Rule 14.181**: ブランド指標を以下の3層アーキテクチャで管理する

```
3-Layer Brand Metrics:
┌──────────────────────────────────────────┐
│  Layer 1: Behavioral（行動指標）          │
│  → エンゲージメント率 / チャーン率        │
│  → 利用頻度 / 機能採用率                  │
├──────────────────────────────────────────┤
│  Layer 2: Perceptual（知覚指標）          │
│  → ブランド信頼スコア / 感情スコア        │
│  → Share of Voice / Share of Model       │
├──────────────────────────────────────────┤
│  Layer 3: Financial（財務指標）           │
│  → 価格プレミアム / CLV / CAC削減効果     │
│  → ブランド起因オーガニックトラフィック比率 │
└──────────────────────────────────────────┘
```

- **Law**: 3層すべてを統合したBrand Equity Indexを月次で算出し、経営KPIとして報告すること

### 19.3. Brandformance Framework（ブランドフォーマンス）

- **Rule 14.182**: ブランディングとパフォーマンスマーケティングの統合測定フレームワーク

| 原則 | 内容 |
|:----|:-----|
| **Unified Attribution** | 全キャンペーンを「認知」と「行動」の両軸で追跡 |
| **Brand Lift + Conversion** | ブランドリフトとコンバージョンを同一ダッシュボードで可視化 |
| **Long-term Incrementality** | 短期ROAS + 長期ブランドリフトの複合ROI算出 |
| **Cross-Channel Integration** | チャネル横断でのブランド貢献の定量化 |

- **Anti-Pattern**: ブランディング予算を「効果測定不能」として放置 → ブランド投資の正当性が説明できない

### 19.4. Competitive Brand Intelligence（競合ブランドインテリジェンス）

- **Rule 14.183**: 競合ブランドのリアルタイム監視と自動アラート

| 監視項目 | 手法 | アラート閾値 |
|:--------|:----|:----------|
| **競合ポジショニング変更** | メッセージング・価格変更の追跡 | 主要変更を検知次第 |
| **競合SoV変動** | Share of Voice の週次比較 | ±20%以上の変動 |
| **競合ブランドキャンペーン** | 広告出稿・PR活動の監視 | 大型キャンペーン検知 |
| **レビュー・評判変動** | 競合製品のレビュースコア変動 | ±0.5pt以上の変動 |

### 19.5. Predictive Brand Analytics（予測ブランド分析）

- **Rule 14.184**: 予測分析によるブランドヘルスの先行指標管理

| 先行指標 | 予測対象 | 手法 |
|:--------|:--------|:----|
| **感情トレンドの傾き** | 今後30日のブランド感情変化 | 時系列分析 |
| **検索トレンド** | ブランド認知度の変動予測 | Google Trends + 内部データ |
| **エンゲージメント低下** | チャーンリスクの早期検知 | 行動パターン異常検知 |

- **Law**: 先行指標がネガティブ傾向を示した場合、14日以内に原因分析と対策案をBrand Ownerに提示すること

## Part XX. Brand Compliance Automation & CI/CD（ブランド準拠の自動化）

> **Supreme Directive**: ブランド準拠チェックを人的レビューのみに依存することは、スケーラビリティとスピードの両面で限界がある。CI/CDパイプラインにブランドガバナンスを組み込み、自動化する。

### 20.1. Brand Linting Pipeline（ブランドリンティングパイプライン）

- **Rule 14.190**: CI/CDパイプラインに以下のブランド準拠チェックを統合する

| チェック項目 | 手法 | ゲート |
|:----------|:----|:------|
| **Color Token Validation** | CSSから非トークンカラー値を検出 | 0件でPass |
| **Typography Check** | 未承認フォントの使用検出 | 0件でPass |
| **Copy Voice Score** | AI分析によるBrand Voice適合スコア | ≥ 0.85 でPass |
| **Alt Text Compliance** | 画像のalt属性の存在・品質チェック | 100%でPass |
| **Inclusive Language** | §4.6 準拠の自動用語チェック | 0違反でPass |

- **Law**: Brand Lintingの失敗はCIパイプラインをブロックする（Warning扱い不可）

### 20.2. Design System Governance Automation（デザインシステムガバナンス自動化）

- **Rule 14.191**: デザインシステムの変更管理を自動化する

| 自動化領域 | 内容 |
|:---------|:-----|
| **Token Change Impact Analysis** | トークン変更時の影響範囲を自動分析 |
| **Visual Regression Testing** | スクリーンショット比較によるUI変更検知 |
| **Component Usage Audit** | 非推奨コンポーネントの使用を自動検知 |
| **Figma-Code Drift Detection** | §3.8 SSOT Protocol違反の自動検出 |

### 20.3. AI-Powered Brand Review Bot（AIブランドレビューボット）

- **Rule 14.192**: PRレビュー時のブランド準拠を自動チェックするボットの実装

```
Brand Review Bot のスコープ:
1. マイクロコピー / エラーメッセージのVoiceスコア
2. UIスクリーンショットのブランドカラー準拠
3. 新規アイコン / 画像のスタイルガイド適合
4. アクセシビリティ基準（§3.6）の自動検証
5. Inclusive Language（§4.6）の自動スキャン
```

- **Mandate**: Brand Review Botの指摘はPRの必須レビュー項目とし、未対応のままマージすることを禁止する

### 20.4. Brand Compliance SLA（ブランド準拠SLA）

- **Rule 14.193**: ブランド準拠違反の修正SLAを定義する

| 重要度 | 定義 | 修正期限 |
|:------|:----|:--------|
| **Critical** | Brand Identity の毀損（ロゴ改変、カラー逸脱） | 24時間以内 |
| **Major** | Voice/Tone の逸脱、トークン違反 | 1スプリント以内 |
| **Minor** | 軽微なコピー修正、スペーシング調整 | 次回リリース |

- **Escalation**: Critical違反が2件以上同時発生した場合、Brand Guardianが全チームに緊急通知を発行する

## Part XXI. Brand & Platform Economy（ブランドとプラットフォームエコノミー）

> **Supreme Directive**: API・SDK・マーケットプレイスを通じてブランドが体験される場合、従来のUI/UXガイドラインだけではブランドを保護できない。開発者体験（DevBX）を含む拡張されたブランドガバナンスが必要である。

### 21.1. API-First Brand Experience（API駆動型ブランド体験）

- **Rule 14.200**: API / SDK 経由でブランドが体験される場合のガバナンス

| 領域 | ブランドガバナンス |
|:----|:--------------|
| **開発者ドキュメント** | Brand Voice準拠の文体・トーン |
| **エラーメッセージ** | §4.4 Error Message Protocol準拠 |
| **SDKログ出力** | ブランド名・用語の一貫性 |
| **Changelog** | Brand Voiceに沿った変更履歴の記述 |

### 21.2. Marketplace & Ecosystem Brand Governance（マーケットプレイスブランドガバナンス）

- **Rule 14.201**: サードパーティがプラットフォーム上でブランドを利用する際のガバナンス

| 項目 | ルール |
|:----|:------|
| **パートナーブランドガイドライン** | ロゴ・名称使用の詳細ガイドを策定・配布 |
| **品質スコアリング** | プラットフォーム上のブランド表現品質を定期スコアリング |
| **違反検知** | 自動画像認識によるロゴ・カラー不正使用の検知 |
| **是正プロトコル** | 違反検知から是正完了までのSLA定義 |

### 21.3. White-Label & OEM Brand Protocol（ホワイトラベル・OEMプロトコル）

- **Rule 14.202**: ホワイトラベル / OEM提供時のブランドガバナンス

| シナリオ | ブランド表現 | ガバナンス |
|:--------|:----------|:---------|
| **Powered by** | 控えめなバッジ表示 | バッジサイズ・配置の厳密な規定 |
| **Co-branded** | 両ブランドの並列表示 | §9.5 Co-Branding規則に準拠 |
| **Full White-Label** | 自社ブランド非表示 | 技術的品質基準の維持義務 |

- **Law**: ホワイトラベル提供であっても、製品品質の低下は自社ブランドの毀損につながるため、品質基準の遵守を契約に明記すること

### 21.4. Developer Brand Experience (DevBX)（開発者ブランド体験）

- **Rule 14.203**: 開発者向けブランド体験の設計

| タッチポイント | ブランド準拠 |
|:------------|:----------|
| **Developer Portal** | Brand Voice + デザイントークン準拠 |
| **API Documentation** | 一貫したコード例・用語 |
| **Status Page** | クリアで誠実なコミュニケーション |
| **Developer Community** | §6.4 Community規則に準拠 |

- **Mandate**: 開発者はブランドの「アンバサダー候補」である。DevBXの品質はエンドユーザー向けBXと同等以上の注力を要する
