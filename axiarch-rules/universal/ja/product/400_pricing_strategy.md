# 13. 価格戦略 (Pricing Strategy)

> [!CAUTION]
> **このファイルは Universal Rule（不変ルール）です。「憲法改正」の明示的指示がない限り編集禁止。**
> 改定日: 2026-04-06

> [!IMPORTANT]
> **Supreme Directive（最高指令）**
> 「価格はマーケティングの最重要メッセージである。値付けを間違えることはポジショニングを間違えることと同義である」
> 価格は「コスト積上げ」ではなく **「価値の対価」** である。値下げは最後の手段であり、価値伝達の失敗のシグナルである。

## Part I. 価格戦略の哲学

### 1.1. 価格設定の3つの誤り

- **Rule 13.001**: 以下の価格設定アプローチを不適切なものとして定義する

| 誤ったアプローチ | 問題点 |
|:-------------|:------|
| **コストプラス価格** | 顧客価値を無視する。低付加価値の烙印を押す |
| **競合追従価格** | 自社の差別化を否定する。価格競争スパイラルに入る |
| **なんとなく設定** | データなき確信。機会損失か過度な離脱かを判断できない |

### 1.2. Value-Based Pricing（価値基準価格）

- **Rule 13.002**: 適切な価格設定の原則は **顧客が得る価値の一部を対価として受け取る** こと

```
価格設定の正しい起点:
1. 顧客が自社製品で達成する価値を定量化する
   例: 月10時間の工数削減 × 時給5,000円 = 5万円/月の価値
2. その価値の 10〜30% を価格とする
   例: 5万円 × 20% = 月額1万円
3. 競合との比較でポジショニングを調整する
```

### 1.3. Willingness to Pay（支払い意欲）の計測義務

- **Rule 13.003**: 価格を決定する前に **WTP（Willingness to Pay）** を定量的に測定する
- **Van Westendorp Price Sensitivity Meter（PSM）** の実施を標準とする

```
PSM の4設問:
1. 「いくらなら高すぎて買わないか？」（Too Expensive）
2. 「いくらなら高いと感じるか？」（Expensive but Consider）
3. 「いくらなら安いと感じるか？」（Cheap but Accept）
4. 「いくらなら安すぎて品質を疑うか？」（Too Cheap）

最適価格帯 = Q2とQ3の交点付近
```

## Part II. 価格モデルの設計

### 2.1. SaaS 価格モデルの種類

- **Rule 13.010**: 以下の価格モデルからプロダクトの特性に合ったものを選択する

| モデル | 説明 | 向いているケース |
|:------|:----|:------------|
| **Flat Rate** | 一律の月額料金 | シンプルなプロダクト、早期PMF確認 |
| **Per Seat** (User-based) | ユーザー数に応じた課金 | コラボ・コミュニケーション系 |
| **Usage-based** | 使用量に応じた課金 | API、クラウド、AI系 |
| **Tiered** | 機能・容量で段階的な料金 | SaaS多数 |
| **Hybrid** | 基本料金 + 使用量課金 | Enterprise SaaS |
| **Freemium** | 基本無料 + 有料Premium | コンシューマ、PLG |

### 2.2. Usage-Based Pricing（UBP）の設計プロトコル

- **Rule 13.011**: UBP を採用する場合、以下を設計する

```
Value Metric の選定基準（1つに絞ること）:
□ 顧客が価値を感じるに比例して増加するか？
□ 顧客が理解・予測できるか？
□ 自社でコントロール・計測できるか？
□ 競合と比較してわかりやすいか？

Good な Value Metric の例:
- Twilio: SMSメッセージ数（行動に直連結）
- Snowflake: クエリ処理量（価値に比例）
- Stripe: 決済金額（顧客収益に比例）
```

### 2.3. Freemium の設計プロトコル

- **Rule 13.012**: Freemium を採用する場合、フリープランの制限設計を以下の原則で行う

```
Freemium の制限設計原則:
1. 価値を"体験"できるが"十分"ではない状態を作る
2. 制限によりユーザーが「もっと使いたい」と感じる設計
3. チームや組織で使う場合にアップグレードが必要になる設計

良い制限例:
- プロジェクト数（3つまで無料、無制限は有料）
- ストレージ容量（2GB無料、追加は有料）
- チームメンバー数（3人まで無料）
- API コール数（月1000回まで無料）

悪い制限例:
- コア機能の制限（製品の本質価値が体験できない）
- 過度な広告表示（UXを著しく損なう）
```

## Part III. 価格設定方法論

### 3.1. 価格設定の5ステップ

- **Rule 13.020**: 新製品・新プランの価格設定は以下の5ステップで行う

```
Step 1: バリュープロポジション（価値）の定量化
→ 顧客インタビュー+ROI計算で経済価値を算出

Step 2: WTP（支払い意欲）の測定
→ Van Westendorp PSM / Conjoint Analysis / Direct Survey

Step 3: セグメント別WTPの分析
→ ICP別、規模別に分離して価格帯を把握

Step 4: 競合価格帯との比較
→ Tier / Feature 比較でのポジショニング確認

Step 5: Pricing Page の A/Bテスト（§6参照）
→ 実際の転換率でデータ検証
```

### 3.2. Conjoint Analysis（コンジョイント分析）

- **Rule 13.021**: 大規模サービス（ARR 1億円以上）では Conjoint Analysis を実施し、属性別の支払い意欲を把握する
  - 複数の製品属性（機能、サポートレベル、SLA等）の組み合わせに対してユーザーが選択を行い、各属性の価値を逆算する
  - 最低 100 回答以上を確保する

### 3.3. ベンチマーク価格の使い方

- **Rule 13.022**: 競合価格は **上限の参考** として使用する。競合より低い価格を目指すことを原則禁止する
- **価格設定の目安**:
  - 競合と同等機能: 競合price × 90-110%
  - 競合より明確に優れている: 競合price × 120-200%
  - 全く新しいカテゴリ: Value-Based で設定（競合なし）

## Part IV. パッケージング・ティア設計

### 4.1. ティア設計の原則

- **Rule 13.030**: 価格ティアは必ず **奇数** で設計する（2択は意思決定を困難にし、4択は情報過多になる）
- **Rule 13.031**: 最も多くの顧客が選択するティアが **中央ティア** になるよう設計する（The Goldilocks Principle）

```
推奨ティア構成:
Starter / Pro / Enterprise（3層）または
Free / Growth / Business / Enterprise（4層・但し3択に見せる）
```

### 4.2. デコイ価格（Decoy Pricing）の設計

- **Rule 13.032**: 中央ティアを魅力的に見せるため「デコイ」として機能するティアを設計してよい
  - デコイは上のティアへの誘導ではなく、「中央ティアが合理的」と感じさせる役割

### 4.3. Anchor Pricing（アンカリング）

- **Rule 13.033**: 価格ページでは **高い価格を先/左に表示** し、中央ティアの価格感を下げる
- **Rule 13.034**: 年払いオプションでは月額換算を表示し、「月あたり○円」の割安感を強調する

### 4.4. Feature Packaging のルール

- **Rule 13.035**: 機能を各ティアに割り当てる際のルール

```
Starter に含めるべき機能:
- コアバリューを体験できる最低限の機能
- 個人・小規模チームで完結する機能

Pro に含めるべき機能:
- チームコラボ・権限管理機能
- 高度なカスタマイズ・API アクセス
- 優先サポート

Enterprise に含めるべき機能:
- SSO / SAML
- カスタム SLA / 専任CSM
- 高度なセキュリティ・監査
- カスタム契約・請求
```

## Part V. 価格変更プロトコル

### 5.1. 価格変更の意思決定基準

- **Rule 13.040**: 価格変更は以下の条件が揃った場合にのみ実施する

| トリガー | 判断基準 |
|:--------|:--------|
| **値上げの正当化** | NPS ≥ 50 / NRR ≥ 110% / 競合との価格差 ≥ 30% |
| **値下げの禁止** | キャンペーン以外の恒久的値下げを本質的には禁止する |
| **ティア追加** | 特定セグメントのWTPが現行ティアと大きくずれている |

### 5.2. 既存顧客への価格変更通知プロトコル

- **Rule 13.041**: 既存顧客への値上げは以下の手順で行う

```
必須手順:
1. 変更60日前にメール通知（件名に価格変更を明記）
2. 値上げの理由を具体的に説明する（機能追加・品質向上）
3. 移行期間を設ける（90日間は旧価格を適用）
4. 年払いユーザーは更新時まで旧価格を保証する
5. カスタマーサクセスが影響が大きい顧客に個別連絡
```

### 5.3. Grandfather Clause（既得権保護）

- **Rule 13.042**: 価格変更時、既存顧客に一定期間（最低6ヶ月）現行価格を保証する「Grandfather Clause」を適用することを検討する
- **適用の例外**: セキュリティ・規制対応が必要な場合は例外とし、影響を最小化した上で顧客に事前説明する

## Part VI. 価格実験（A/Bテスト）

### 6.1. 価格 A/Bテストの倫理原則

- **Rule 13.050**: 価格の A/Bテストにおいて以下を厳守する
  - 同一ユーザーが異なる価格を同時に見ることを禁止する（Geo分割またはコホート分割で実施）
  - テスト期間中に「なぜ価格が違うのか」という問い合わせに正直に答える
  - 個人情報に基づく価格差別（人種・性別etc）を完全禁止する

### 6.2. 価格実験の設計プロトコル

- **Rule 13.051**: 価格実験を行う場合、以下を事前定義する

```yaml
pricing_experiment:
  hypothesis: "月額3,980円から4,980円への値上げでコンバージョン率が5%以内の下落に収まる"
  primary_metric: "Trial to Paid Conversion Rate"
  secondary_metrics:
    - "MRR per Signup"
    - "Refund Rate"
    - "NPS"
  sample_size: 最低各グループ 200 Signup
  duration: 最低 4 週間
  statistical_significance: 95%
```

### 6.3. 価格実験の禁止ケース

- **Rule 13.052**: 以下の状況では価格実験を禁止する
  - CMO/CEO の承認なく単独で実施する
  - サンプルサイズが統計的有意性に足りない状態で判断する
  - 複数の変数（機能+価格）を同時に変更する

## Part VII. エンタープライズ価格交渉

### 7.1. エンタープライズ価格設定の原則

- **Rule 13.060**: エンタープライズ（Enterprise）ティアの価格は以下の原則で管理する
  - **Anchor High**: 最初の提示額は高めに設定する。顧客からの値下げ圧力に備えたバッファを持つ
  - **Price Floor**: どんな交渉でも下回ってはならない最低価格（Price Floor）を社内で事前定義する
  - **Discount Authority**: 割引権限をレベル別に明確化する（SDR: 0%, AE: ≤10%, VP: ≤20%, CEO: ≤30%）

### 7.2. 値引き要求への対応プロトコル

- **Rule 13.061**: 顧客からの値引き要求は以下の手順で対応する

```
Step 1: 値引きの代わりに「何を提供できるか」を提示する
  例: より長い年払いへの誘導（一括払いなら15%割引）
  例: 成功共有型（ROIが出たら追加課金）

Step 2: 値引きをする場合は必ず「何かと交換」にする
  例: 6ヶ月の先払い / Case Study への協力 / Beta機能への参加

Step 3: 値引きは単発ではなく更新条件として交渉する
  例: 「今期の値引きは来期の更新を約束いただく場合に限る」

禁止: 理由なき値引き（「いいですよ」だけの合意）
```

### 7.3. Multi-Year Contract の推奨プロトコル

- **Rule 13.062**: 3年以上の複数年契約を推奨する場合、以下の特典構造を設計する

```
契約期間別割引体系（例）:
月払い: 定価（100%）
年払い: 17%割引（＝2ヶ月分無料に相当）
2年払い: 25%割引
3年払い: 30%割引

注意: 割引率のコミットは前払い条件として設計する
```

## Part VIII. 価格KPIとモニタリング

### 8.1. 価格ヘルスの主要指標

- **Rule 13.070**: 以下の指標を月次でモニタリングし、価格戦略の健全性を測定する

| 指標 | 目標値 | 警戒ライン |
|:----|:------|:--------|
| **Average Revenue Per Account (ARPA)** | 前月比 ≥ 0% | 連続2ヶ月マイナス |
| **Discount Rate** | ≤ 15% of Deals | > 30% of Deals |
| **Net Revenue Retention (NRR)** | ≥ 110% | < 100%（収縮） |
| **Price-to-Value Ratio** | NPS 9-10が自発的に「安い」と言及 | 「高い」という言及が増加 |
| **Expansion MRR** | ≥ 20% of Total MRR Growth | ≤ 5% |

### 8.2. Price-to-Churn 相関分析

- **Rule 13.071**: 価格ティア別のチャーン率を月次で分析し、価格と解約の相関を監視する

```
分析例:
Starter: Churn 8%/月
Pro: Churn 3%/月
Enterprise: Churn 0.5%/月

→ Starter のチャーンが高い場合:
  a) Starter ユーザーのOnboardingを改善する
  b) Starter の価格を下げてライフタイムを伸ばす
  c) Starter を廃止して Pro に一本化する
のいずれかを選択する
```

### 8.3. 競合価格モニタリング

- **Rule 13.072**: 競合の価格変更を四半期ごとにモニタリングし、以下を評価する
  - 競合が値上げした場合: 自社の相対的割安感が増加 → 値上げの機会
  - 競合が値下げした場合: 「価格競争」か「価値をもっと伝える必要がある」かを判断する
  - 競合が新ティアを追加した場合: セグメント動向を評価する
