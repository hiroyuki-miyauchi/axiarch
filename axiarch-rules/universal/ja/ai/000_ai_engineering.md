## Part I: Supreme Directive & AI哲学

### 1.1. AI実装の最高原則

- **Rule 40.1**: AI機能はユーザーの意思決定を**支援**するものであり、**代替**するものではない
- **Rule 40.2**: AIの出力は常に「提案」であり、最終判断は人間が行う設計を義務付ける
- **Rule 40.3**: AI機能の障害がサービス全体を停止させてはならない（Graceful Degradation必須）

### 1.2. AI実装の優先順位階層

| 優先度 | 領域 | 説明 |
|:---|:---|:---|
| **P0** | 安全性 | ユーザーの安全、PII保護、有害コンテンツ遮断 |
| **P1** | 信頼性 | ハルシネーション防止、事実の正確性、出典の透明性 |
| **P2** | UX | 応答速度、対話品質、アクセシビリティ |
| **P3** | コスト効率 | トークンコスト、インフラコスト、ROI |
| **P4** | DX | 開発速度、保守性、テスト容易性 |

### 1.3. AI First ≠ AI Only

- **Rule 40.4**: AI機能には必ず**フォールバック**（非AIの代替手段）を設計する
- AIが利用不能な場合でもコアビジネス機能は継続しなければならない
- 例: AIチャットが停止 → FAQ/ヘルプセンターへの導線を自動表示

## Part II: AI UX原則

### 2.1. Zero Latency Perception（体感ゼロ遅延）

- **Rule 40.5**: AIの応答待機時間は「体感ゼロ」でなければならない。ローディングスピナーは敗北である
- **ストリーミングファースト**: サーバーからのレスポンスを待たず、1トークンずつリアルタイム表示する
- **即時フィードバック**: ユーザーの入力に対して100ms以内にUI上の変化を返す

### 2.2. 楽観的UI（Optimistic UI）

- **即時反応**: ユーザーが送信ボタンを押した瞬間、AI処理完了を待たずにUI上では「成功」として扱う
  - チャットバブルの即時表示、処理中インジケータの併用
- **バックグラウンド処理**: 実際の処理はバックグラウンドで実行。失敗時のみユーザーに通知しリトライを促す
- **ロールバック設計**: 楽観的更新が失敗した場合のUI状態復元メカニズムを必ず実装する

### 2.3. 状態の可視化（State Visibility）

- AIが「何をしているか」を常に透明化する
- `Thinking...`（思考中）, `Searching...`（検索中）, `Generating...`（生成中）等のステータスを段階的に表示
- **Progress Granularity**: 複数ステップの処理では進捗バー/ステップインジケータを使用する

### 2.4. AI Psychology（満足度 > 速度）

- **Thinking Visualization**: 長考時は「思考プロセス」を可視化するアニメーションを表示し、待ち時間を「期待感」に変換する
- **The High Cost Metaphor**: 回数制限表示には「Daily Energy Limit」等の価値メタファーを使用。「制限に達しました」（ケチ）ではなく「高度なエネルギーを使い切りました」（価値）
- **Confidence Indicator**: AI回答の確信度を視覚的に表示（高確信=緑、中=黄、低=赤のドット等）

### 2.5. Conversational UX設計原則

- **Turn-Taking**: AIの長文回答は段落ごとにチャンク表示し、ユーザーが途中で中断・質問できる設計にする
- **Context Awareness Badge**: AIが過去の会話コンテキストを参照している場合、その旨をバッジ/ラベルで明示する
- **Error Recovery**: AIのエラー時は技術的エラーメッセージではなく、「もう一度お試しください」等の人間的表現を使用する
- **Undo/Regenerate**: すべてのAI生成に対し「再生成」「元に戻す」オプションを提供する

## Part III: AI UXパターンカタログ

### 3.1. チャットUI

- **用途**: 自由形式の質問応答、カウンセリング、カスタマーサポート
- **必須要素**: ストリーミング表示、メッセージ履歴、引用表示、フィードバック（👍/👎）、再生成ボタン
- **禁止**: 無限スクロールによる過去メッセージの遅延ロード（初期ロード = 直近N件、ページネーション使用）

### 3.2. Copilot UI

- **用途**: ユーザーの作業を横から支援（フォーム入力補助、文章校正、データ分析サジェスト）
- **設計原則**: メインタスクの邪魔をしない。サイドパネルまたはインライン提案で表示
- **Accept/Reject**: 提案に対する明示的な承認/拒否インタラクションを必須とする
- **学習フィードバック**: ユーザーの承認/拒否パターンからパーソナライゼーションを改善する仕組み

### 3.3. エージェント型UI

- **用途**: 複数ステップの自律タスク実行（データ処理パイプライン、リサーチアシスタント）
- **必須要素**: ステップ進捗表示、中間結果のプレビュー、一時停止/再開コントロール
- **Human Checkpoint**: 重要な意思決定ポイントでは必ず人間の承認を求めるゲートを設置する
- **Audit Trail**: エージェントの全アクションを時系列ログとして表示・エクスポート可能にする

### 3.4. Ambient AI（環境知能）

- **用途**: バックグラウンドで動作し、必要時のみ通知（異常検知、プロアクティブアラート）
- **Notification Strategy**: 重要度に応じた通知チャネル制御（Critical=Push、Info=In-App Badge）
- **Opt-Out**: ユーザーがAmbient AIの動作を完全停止できるトグルを必ず提供する

### 3.5. Proactive UI

- **用途**: ユーザーの行動パターンから次のアクションを予測して提案
- **Non-Intrusive**: 提案はモーダルではなくトースト/バナーで表示し、ユーザーの作業フローを中断しない
- **Frequency Control**: 同一提案の表示頻度を制限し、「しつこいAI」にならないよう制御する

## Part IV: AI倫理フレームワーク

### 4.1. Responsible AI原則

- **公平性（Fairness）**: 特定の属性（人種、性別、年齢、地域、カテゴリ）に対するバイアスのない出力を保証する
- **透明性（Transparency）**: AIの判断根拠をユーザーが理解できる形で説明する
- **説明可能性（Explainability）**: 「なぜこの回答になったか」を技術者以外にも説明できるレベルを維持する
- **プライバシー（Privacy）**: データ最小化原則を適用し、必要最小限の情報のみを処理する
- **安全性（Safety）**: 有害コンテンツの生成を多層防御で阻止する
- **人間中心（Human-Centricity）**: AI はユーザーの自律性を尊重し、強制・操作・中毒性設計を排除する

### 4.2. AI倫理チェックリスト（デプロイ前必須）

| # | チェック項目 | 合格基準 |
|:---|:---|:---|
| 1 | PII送信防御 | 外部LLMへのPII送信が技術的に阻止されている |
| 2 | バイアス評価 | 主要カテゴリでの公平性テスト合格 |
| 3 | 免責表示 | リスクレベルに応じた免責が自動表示される |
| 4 | Kill Switch | 緊急停止スイッチが動作確認済み |
| 5 | AI表示ラベル | AI生成コンテンツに出自ラベルが表示される |
| 6 | Opt-Out | ユーザーがAI機能をオフにできる |
| 7 | 通報機能 | 不適切コンテンツの通報が可能 |
| 8 | アクセシビリティ | AI出力がスクリーンリーダー対応済み |

## Part V: AIガバナンス体制

### 5.1. AIガバナンス委員会

- **設置条件**: シリーズA以降、または有料ユーザー数が一定規模に到達した時点
- **必須ロール**:
  - 技術責任者（CTO / テックリード）
  - 法務・コンプライアンス担当
  - ユーザー代表（CS / UXリサーチャー）
  - データプライバシー責任者（DPO）
- **審議スコープ**: モデル選定、プロンプト方針変更、新規AI機能リスク評価、インシデント対応、コンプライアンス状況レビュー
- **定期レビュー**: 四半期ごと + 緊急時の臨時招集
- **Cross-Reference**: Red Button Protocol (§XXXIV), Model Switch Protocol (§XLII)

### 5.2. AIリスク評価フレームワーク

| リスクカテゴリ | 評価項目 | 評価頻度 |
|:---|:---|:---|
| 安全性リスク | 有害コンテンツ生成可能性 | モデル変更時 + 四半期 |
| プライバシーリスク | PII処理範囲、データフロー | 機能追加時 + 半期 |
| バイアスリスク | カテゴリ/地域/属性バイアス | モデル変更時 + 四半期 |
| 財務リスク | コスト超過、APIループ | 月次 |
| 規制リスク | EU AI Act/APPI準拠状況 | 半期 + 規制変更時 |
| レピュテーションリスク | ブランド毀損シナリオ | 四半期 |

### 5.3. AI Decision Log（意思決定記録）

- **Rule 40.6**: AIに関する重要な設計判断（モデル選定、プロンプト方針、ガードレール設定等）は、ADR（Architecture Decision Record）として記録する
- 記録項目: 決定内容、理由、代替案、リスク、レビュー者、日付

## Part VI: EU AI Act準拠

### 6.1. リスク分類4段階と義務（2025年3月時点最新）

| リスクレベル | 例 | 主要義務 | 適用時期 |
|:---|:---|:---|:---|
| **禁止（Unacceptable）** | 認知操作、ソーシャルスコアリング、公共空間リアルタイム生体認証 | 全面禁止 | 2025年2月〜 |
| **高リスク（High）** | 採用AI、信用スコアリング、医療AI、重要インフラAI | リスク管理、技術文書、人的監視、適合性評価 | 2026年8月〜 |
| **限定リスク（Limited）** | チャットボット、ディープフェイク生成 | 透明性義務（AI利用の明示） | 2026年8月〜 |
| **最小リスク（Minimal）** | スパムフィルタ、AIゲーム | 義務なし（Good Practice推奨） | — |

### 6.2. GPAI（汎用目的AI）モデル義務

- **訓練データの概要公開**: データ収集方法、期間、法的根拠、匿名化手法、データセットリスト
- **技術文書作成**: 訓練・テスト・評価プロセスの文書化
- **AIシステムプロバイダへの情報提供**: モデルの能力と制限の文書化
- **GPAI Code of Practice**: 透明性、著作権、安全性に関するガイドラインへの準拠
- **適用時期**: 2025年8月〜

### 6.3. GPAI Code of Practice（実践規範）

- **定義**: 独立専門家によるマルチステークホルダープロセスで策定された自主的フレームワーク（2025年7月10日公開）
- **目的**: GPAIモデル提供者がAI Act上の義務（安全性・透明性・著作権）を満たすことを支援
- **法的位置づけ**: 準拠することでコンプライアンスの推定を得られ、法的確実性が向上
- **Rule 40.7a**: GPAIモデルを利用するプロジェクトは、GPAI Code of Practiceの遵守状況を四半期ごとに評価し記録する

### 6.4. AIリテラシー義務

- **Rule 40.7b**: EU AI Act「AIリテラシー」要件（2025年2月施行）に対応するため、AI機能の開発・運用に関わるすべてのチームメンバーがAI基礎知識を保有することを確認する
- 教育プログラムの実施と記録を推奨

## Part VIII: AIコンテンツラベリング・著作権

### 8.1. AI生成コンテンツラベリング義務

- **Rule 40.9**: AIが生成・補助したコンテンツには、出自を明示するラベルを**常時表示**する。ラベルの隠蔽・極小化は禁止
- EU AI Act および各国AI規制ガイドラインの原則に準拠

| コンテンツ種別 | ラベル例 | 備考 |
|:---|:---|:---|
| チャット回答 | `🤖 AIが回答しました` | 常時表示 |
| 画像解析結果 | `AIが読み取った内容です。正確性をご確認ください` | 常時表示 |
| レコメンデーション | `✨ AIおすすめ` | 表示 |
| AI編集テキスト | `📝 AI編集済み` | 人間レビュー後も出自を記録 |

- **DB Flag**: AI生成コンテンツを保存する際は `is_ai_generated: boolean`（デフォルト: `false`）で出自を追跡

### 8.2. Citation Display Protocol（引用元表示義務）

- **Rule 40.10**: AIが外部情報を参照して回答を生成する場合、情報の出自を3段階ラベルで明示する

| ソース種別 | ラベル例 | 適用条件 |
|:---|:---|:---|
| **RAG引用** | `📎 参考情報: [ソース名](URL)` | RAGで取得した外部情報に基づく回答 |
| **DB確定データ** | `✅ 確認済み情報` | DB内の確定データに基づく回答 |
| **AI推論** | `💡 AI参考回答` | AIの推論・一般知識に基づく回答 |

- **No Source, No Claim**: 参照元がない事実情報を断定的に述べることを禁止する

### 8.3. AI生成コンテンツ著作権ガバナンス

- **No Raw Publish**: AI生成テキストをそのまま公式コンテンツとして掲載することを禁止。人間による編集・検証を経たもののみ「公式コンテンツ」
- **Image Risk Assessment**: AI生成画像の商用利用時は学習データの著作権リスクを評価し、利用規約に免責事項を明記
- **DB Flag**: `is_ai_assisted: boolean`（デフォルト: `false`）でAI支援の有無をデータレベルで追跡

## Part IX: ユーザーデータとAI学習

### 9.1. User Data AI Training Opt-Out Protocol

- **Rule 40.11**: ユーザーデータのAI学習利用について、GDPR/APPIの「目的外利用禁止」原則に準拠したOpt-In/Opt-Out制御を義務付ける

| 項目 | 要件 |
|:---|:---|
| **デフォルト** | Opt-Out（学習に利用しない） |
| **Opt-In取得** | 利用規約とは別の明示的同意（チェックボックス等） |
| **DB管理** | `ai_training_opt_in: boolean`（デフォルト: `false`） |
| **撤回権** | いつでもOpt-Inを撤回可能、次回学習バッチから除外 |
| **匿名化** | PII完全除去後にのみ学習データとして使用 |
| **透明性** | プライバシーポリシーに明記、設定画面への導線を提供 |

- **Cross-Reference**: PII Scrubbing (§XVIII), Data Minimization / Right to be Forgotten (`61`)

## Part X: AIバイアスと公平性

### 10.1. The Fairness Protocol（バイアス緩和）

- **Rule 40.12**: 特定の属性に対する統計的根拠のない偏見・差別的回答を生成しないようシステムプロンプトで明示制御する

### 10.2. モデル選定バイアス評価チェックリスト

| 評価項目 | 検証方法 | 合格基準 |
|:---|:---|:---|

| **地域バイアス** | 都市部/地方の同一シナリオ回答比較 | 地域差別表現が0件 |
| **言語品質** | ターゲット言語での自然さ・敬語評価 | 不自然な翻訳調が5%未満 |

- **実施頻度**: 四半期ごと + モデル切替時

### 11.1. Private Relay & Execution Pattern

|:---|:---|:---|:---|

### 11.3. ハイブリッドアーキテクチャ

- **Rule 40.14**: AI機能は必ずGraceful Degradation可能な設計とする。AI APIが利用不能な場合もサービスのコア機能は継続する

## Part XII: モデルルーティング・抽象化

### 12.1. The Model Router Protocol（Tiered Architecture）

- **Rule 40.15**: コード内で特定のモデルID（例: `"gemini-1.5-flash"`）をハードコーディングすることを禁止する
- **抽象化レイヤー**: `Tier`（例: `Tier.FAST`, `Tier.SMART`, `Tier.VISION`, `Tier.THINKING`）を介してモデルを呼び出すルーター機能を必ず実装する

```typescript
// ✅ 正しい実装
const response = await aiRouter.generate({
  tier: Tier.FAST,
  prompt: userMessage,
});

// ❌ 禁止: ハードコーディング
const response = await gemini("gemini-2.0-flash", userMessage);
```

### 12.2. モデル選定基準

| Tier | ユースケース | 選定基準 |
|:---|:---|:---|
| `Tier.FAST` | チャット、簡易Q&A | コスト最適、低レイテンシ |
| `Tier.SMART` | 複雑な推論、分析 | 高精度、コンテキスト理解 |
| `Tier.THINKING` | 長考タスク、コード生成 | 推論チェーン、ステップバイステップ |
| `Tier.VISION` | 画像解析、OCR | マルチモーダル、即時認識 |
| `Tier.EMBEDDING` | 検索、類似度計算 | 高次元ベクトル、多言語対応 |

- **Current Standard（2025-2026時点）**: Gemini 2.0 Flash を標準採用（コストパフォーマンス最適）
- **Vision AI Standards**: Thinkingモデルは即時認識タスク（OCR等）には不適切。`Tier.VISION` には非Thinkingモデルを使用する

### 12.3. フォールバック・A/Bルーティング

- **プロバイダフォールバック**: プライマリプロバイダ障害時に別プロバイダへ自動切替する機能を実装
- **A/Bルーティング**: Feature Flagと連動し、ユーザーの一部に異なるモデルを配信可能にする
- **レイテンシベースルーティング**: 応答時間のSLO超過時に低コスト・高速モデルへ自動ダウングレードする機能

## Part XIII: コンテキストウィンドウ管理

### 13.1. トークン最適化戦略

- **Rule 40.16**: 無限に会話履歴を送信してはならない。重要なコンテキストのみを抽出して送信する
- **要約戦略**: N ターン以上の会話は自動要約し、要約 + 直近Nターンの生テキストを送信する
- **入力トークン監視**: プロンプトのトークン数を常に監視し、不要な情報は削除する

### 13.2. コンテキスト管理パターン

| 戦略 | 説明 | 推奨ケース |
|:---|:---|:---|
| **Sliding Window** | 直近Nターンのみ保持 | カジュアルチャット |
| **Summarize + Recent** | 古い会話を要約 + 直近を生保持 | ロングセッション |
| **RAG-Augmented Context** | 関連情報をその都度検索して挿入 | ナレッジベースQ&A |
| **Hierarchical Context** | システム→ドメイン→セッション→ターンの階層 | エージェントタスク |

### 13.3. トークンバジェット配分

```
Total Context Window Budget:
├── System Prompt:     15-20%（固定）
├── RAG Context:       30-40%（動的）
├── Conversation History: 20-30%（スライディング）
├── User Input:        10-15%（最新ターン）
└── Safety Buffer:     5-10%（余裕）
```

## Part XIV: 構造化出力・Function Calling

### 14.1. Structured Output

- **Rule 40.17**: AIからのレスポンスは可能な限りJSONモード（Structured Output）を使用して構造化し、パースエラーを防止する
- **スキーマ定義**: レスポンススキーマはZod等で型定義し、バリデーションを自動化する

```typescript
// Structured Output with Zod Schema
const ResponseSchema = z.object({
  answer: z.string(),
  confidence: z.number().min(0).max(1),
  sources: z.array(z.object({
    title: z.string(),
    url: z.string().url(),
  })),
  category: z.enum(["factual", "opinion", "recommendation"]),
});
```

### 14.2. Function Calling / Tool Use

- **用途**: AIが外部ツール（DB検索、API呼び出し、計算等）を呼び出すパターン
- **スキーマ駆動**: ツール定義はJSONスキーマで厳密に定義し、型安全を保証する
- **権限制御**: ツールごとに実行権限レベルを設定し、破壊的操作（DELETE等）にはHuman Approval Gateを設置する
- **タイムアウト**: 各ツール呼び出しにタイムアウトを設定し、無限ループを防止する

### 14.3. Tool Use設計原則

- **最小権限**: AIに公開するツールは必要最小限に制限する
- **冪等性**: 可能な限りツール呼び出しを冪等（再実行しても結果が変わらない）に設計する
- **参照と変更の分離**: 読み取り専用ツールと書き込みツールを明確に分離し、書き込みツールには追加の確認ステップを設ける

## Part XVII: セマンティックキャッシング

### 17.1. The Semantic Caching Protocol（pgvector戦略）

- **Rule 40.20**: `pgvector` を用いた類似度検索キャッシュの実装を推奨する
- **類似度閾値**: **0.95以上**（完全一致に近い問答のみキャッシュ返却）
- **効果**: コスト削減 + レスポンスタイム大幅短縮 + UX改善

### 17.2. キャッシュ階層設計

| キャッシュ層 | TTL | 対象 |
|:---|:---|:---|
| **Exact Match** | 24時間 | 完全一致のQ&A |
| **Semantic Match** | 6時間 | 類似度 ≥ 0.95のQ&A |
| **RAG Chunk Cache** | 1時間 | 頻繁に参照されるドキュメントチャンク |
| **Embedding Cache** | 7日 | 計算済みEmbeddingベクトル |

### 17.3. キャッシュ無効化

- ソースドキュメントの更新時に対応するチャンクキャッシュを即時無効化する
- ユーザー固有のコンテキストに依存する回答はキャッシュしない
- キャッシュヒット率を監視し、閾値調整のデータとする

## Part XVIII: AIセキュリティ

### 18.1. プロンプトインジェクション防御（多層モデル）

- **Layer 1 — Input Sanitization（入力サニタイズ）**:
  - ユーザー入力の制御文字列（`###`, `````, `<|system|>`, `SYSTEM:` 等）をエスケープまたは除去
  - 入力テキストの最大長を制限（推奨: 2,000〜4,000文字）、超過分は切り捨て

- **Layer 2 — Delimiter Strategy（デリミタ戦略）**:
  ```
  [SYSTEM INSTRUCTION - DO NOT MODIFY]
  {system_prompt}
  [END SYSTEM INSTRUCTION]

  [USER MESSAGE]
  {user_input}
  [END USER MESSAGE]
  ```

- **Layer 3 — Output Validation（出力バリデーション）**:
  - システムプロンプトのリーク検出（部分一致）
  - PII パターンの検出とマスキング
  - ホワイトリスト外の外部URLのフィルタリング

- **Layer 4 — Anomaly Detection（異常検知）**:
  - プロンプトインジェクション疑いのある入力を `warn` レベルでログ記録し管理者通知
  - 短時間での大量リクエストを検出しレート制限を適用

### 18.2. PII Scrubbing

- **Rule 40.21**: 外部LLMへ送信するプロンプトには個人情報を含めてはならない
- 正規表現等で送信前にマスク処理（`[連絡先削除]`, `[氏名削除]`）を実行する
- **対象PII**: 氏名、電話番号、メールアドレス、住所、マイナンバー等

### 18.3. API Key管理

- **No Client Exposure**: `NEXT_PUBLIC_` プレフィックスでのAI APIキー公開は厳禁
- **Rotation**: APIキーは90日ごとにローテーションする
- **Vault Integration**: シークレット管理ツール（Vault、AWS Secrets Manager等）と統合する

### 18.4. Grounding & Fact-Check Standards

- **Rule 40.22**: 店舗情報・価格等の事実は必ずDBの確定情報と照合（Grounding）し、引用元を提示する
- **Numeric Data Rule**: 数値データ（価格、距離、数量）はRAGソースまたはDBからの直接引用のみ許可。AIによる計算・推測は禁止
- **RAG Source Absence Protocol**: ソースが取得できなかった場合は「この情報は参考情報です。正確性を保証するものではありません」を自動付与

## Part XIX: RAG基盤設計

### 19.1. RAGパイプラインアーキテクチャ

RAG Pipeline:
[Document Ingestion] → [Chunking] → [Embedding] → [Index Storage]
                                                         ↓
[User Query] → [Query Processing] → [Retrieval] → [Reranking] → [Context Assembly] → [LLM Generation] → [Output Validation]

### 19.2. The Triple Write RAG Protocol（Clean Data Source）

- **Rule 40.23**: RAGの参照ソースには、CMS憲法（`34`）で義務付けられた**「Triple Writeされたプレーンテキスト同期データ」**を必ず使用する
- **理由**: HTMLタグを含むノイズだらけのデータはトークンの無駄遣いであり、回答精度を低下させる

### 19.3. Embeddingモデル選定

| 基準 | 考慮事項 |
|:---|:---|
| **多言語対応** | 日本語・英語の両方で高品質なEmbeddingを生成できること |
| **次元数** | 高次元（1536+）は精度向上、低次元はコスト/速度優位 |
| **コスト** | 大量ドキュメントのインデックス時のコスト考慮 |
| **更新頻度** | モデル更新時の再インデックスコストを評価 |

### 19.4. インデックス戦略

- **ベクトルDB選定**: pgvector（PostgreSQL統合）、Pinecone、Weaviate、Qdrant等
- **メタデータフィルタリング**: ベクトル検索とメタデータフィルタを組み合わせた精密な検索
- **インデックスの定期更新**: ソースドキュメント更新に連動したインクリメンタルインデックス更新

## Part XX: 高度なチャンキング戦略

### 20.1. チャンキング手法の比較

| 手法 | 説明 | 推奨ケース |
|:---|:---|:---|
| **Fixed-Size** | 固定長で分割 | 均一なテキスト |
| **Recursive** | 階層的セパレータで再帰分割 | 構造化ドキュメント |
| **Semantic** | 意味の境界で分割（Embedding類似度に基づく） | 密度の高い技術文書 |
| **Sliding Window** | オーバーラップ付きウィンドウ | コンテキスト喪失を防ぐ |
| **Agent-Based** | AIが動的に分割ポイントを判断 | 複雑・混合ドキュメント |
| **LLM-Based** | LLMがコンテキストに基づき分割 | 最高精度（高コスト） |

### 20.2. チャンキング設計原則

- **Rule 40.24**: チャンクサイズは検索精度とコンテキスト保持のバランスで決定する（推奨: 256〜1024トークン）
- **オーバーラップ**: 隣接チャンク間で10-20%のオーバーラップを設け、境界でのコンテキスト喪失を防止する
- **メタデータ付与**: 各チャンクにソースドキュメント名、セクション名、ページ番号、作成日等をメタデータとして付与する
- **Contextual Retrieval**: チャンクに前後のコンテキスト要約を埋め込み、孤立したチャンクでも意味が保持されるようにする

## Part XXI: ハイブリッド検索

### 21.1. Hybrid Search戦略

- **Rule 40.25**: RAGの検索にはキーワード検索（BM25）とベクトル検索の**ハイブリッド**を標準とする
- キーワード検索は完全一致や固有名詞に強く、ベクトル検索は意味的類似性に強い。両者を組み合わせることで検索品質を最大化する

### 21.2. フュージョン戦略

| 戦略 | 説明 |
|:---|:---|
| **Reciprocal Rank Fusion (RRF)** | 各検索結果の順位を逆数変換して合算。シンプルで効果的 |
| **Weighted Fusion** | キーワードスコアとベクトルスコアに重みを付けて合算 |
| **Cascading** | 安価な検索で候補を絞り、高精度な検索でリランキング |

### 21.3. メタデータフィルタリング

- 検索時にカテゴリ、日付範囲、ドキュメントタイプ等のメタデータでフィルタリングし、検索空間を狭める
- **Pre-Filter vs Post-Filter**: 大量データではPre-Filter（検索前フィルタ）を推奨（パフォーマンス優位）

---

## Part XXII: リランキング

### 22.1. リランキング戦略

- **Rule 40.26**: RAGの検索結果は、初期検索後に**リランキング**を適用して精度を向上させることを推奨する

| 手法 | 精度 | コスト | レイテンシ |
|:---|:---|:---|:---|
| **Cross-Encoder** | 高 | 中 | 中 |
| **LLM-as-Reranker** | 最高 | 高 | 高 |
| **Cohere Rerank** | 高 | 中（API課金） | 低 |
| **BGE Reranker** | 高 | 低（自前ホスト） | 中 |

### 22.2. リランキング設計原則

- **Top-K → Rerank → Top-N**: 初期検索でTop-K（例: 20件）を取得し、リランキングでTop-N（例: 5件）に絞る
- **コスト/精度トレードオフ**: プロダクションではCross-Encoderが精度とコストのバランスが最適
- **レイテンシバジェット**: リランキングによる追加レイテンシを100ms以内に抑える目標を設定する

## Part XXIII: GraphRAG・ナレッジグラフ

### 23.1. GraphRAGアーキテクチャ

- **概要**: Knowledge Graph（知識グラフ）とLLMを統合し、エンティティ間の関係を活用した高精度な情報検索・推論を実現するパターン
- **利点**: 従来のRAGがチャンク単位の孤立した情報検索に留まるのに対し、GraphRAGはデータ間の関係性を辿って深いコンテキストと推論を提供する

### 23.2. GraphRAGパラダイム

| パラダイム | 説明 | ユースケース |
|:---|:---|:---|
| **Knowledge-Based** | ドキュメントから明示的なKGを構築 | 構造化された知識体系 |
| **Index-Based** | グラフ構造でテキストチャンクを組織化 | 大規模ドキュメント集合 |
| **Hybrid** | ベクトル検索 + グラフ検索を併用 | 複雑な質問応答 |

### 23.3. 実装ガイドライン

- **エンティティ抽出**: NER（Named Entity Recognition）またはLLMによるエンティティ・関係の自動抽出
- **グラフDB**: Neo4j、Amazon Neptune等のグラフDBまたはPostgreSQLのグラフ拡張を使用
- **適用判断**: GraphRAGは構築・維持コストが高いため、データ間の関係性が回答品質に直接影響するケースにのみ適用する

## Part XXIV: マルチモーダルRAG

### 24.1. マルチモーダルRAG戦略

- **対象メディア**: テキスト、画像、音声、動画、PDF、表データ
- **マルチモーダルEmbedding**: テキストと画像を同一ベクトル空間に埋め込むモデル（CLIP等）を活用
- **メディア固有の前処理**: 画像はキャプション生成、音声は文字起こし、PDFは構造化抽出を行った上でインデックスに投入

### 24.2. 適用ケース

| メディア | 前処理 | インデックス方式 |
|:---|:---|:---|
| **テキスト** | チャンキング + Embedding | ベクトル + キーワード |
| **画像** | キャプション生成 + マルチモーダルEmbedding | ベクトル |
| **PDF** | レイアウト解析 + 構造化抽出 | ベクトル + メタデータ |
| **音声** | STT（Speech-to-Text） → テキストとして処理 | ベクトル |

## Part XXVI: Agentic AI戦略

### 26.1. 自律性5段階分類（L1-L5）

| レベル | 名称 | 説明 | Human Oversight |
|:---|:---|:---|:---|
| **L1** | Assistive | 人間の質問に回答するのみ | 常時監視 |
| **L2** | Suggestive | アクション候補を提案、人間が実行 | 承認必須 |
| **L3** | Semi-Autonomous | 低リスクタスクを自律実行、高リスクは承認要求 | チェックポイント |
| **L4** | Autonomous | 定義済みスコープ内で自律実行 | 例外時のみ介入 |
| **L5** | Fully Autonomous | 目標設定から実行まで自律 | 事後監査のみ |

- **Rule 40.28**: プロダクションAIエージェントはL3以下を標準とする。L4以上は明示的なAIガバナンス委員会の承認を必須とする

### 26.2. エージェント設計5原則

1. **最小権限**: エージェントに付与するツール・データアクセスは必要最小限
2. **可観測性**: 全アクションをトレーシング可能に設計
3. **可逆性**: 可能な限り操作を取り消し可能に設計
4. **段階的昇格**: 権限は段階的に、実績に基づいて拡大する
5. **フェイルセーフ**: 障害時は安全な状態（何もしない）にフォールバック

### 26.3. エージェント適用判断基準

| 条件 | エージェント推奨 | 非エージェント推奨 |
|:---|:---|:---|
| タスクの複雑度 | マルチステップ、動的判断が必要 | 単一ステップ、決定論的 |
| データソース | 複数ソースの統合が必要 | 単一ソースで完結 |
| 時間感度 | 非同期で許容される | リアルタイム応答必須 |
| リスク | 低〜中（可逆操作） | 高（資金移動、データ削除） |

## Part XXVIII: MCP（Model Context Protocol）

### 28.1. MCP概要

- **定義**: AIモデルが外部ツール、データソース、環境と安全かつ効果的にインタラクションするためのベンダーニュートラルな標準プロトコル
- **利点**: モデル切替時の再構築不要、複数エージェントの連携、一貫したトレーサビリティ

### 28.2. MCP Server設計

- **Rule 40.29**: MCPサーバーを構築する場合、以下の原則を遵守する
  - **ツール公開**: 公開するツールはJSONスキーマで厳密に定義し、説明文を充実させる
  - **認証**: OAuth 2.0 または API Key による認証を必須とする
  - **レート制限**: ツールごとにレート制限を設定し、乱用を防止する
  - **監査ログ**: 全ツール呼び出しを監査ログとして記録する

### 28.3. MCPセキュリティ

- **入力バリデーション**: MCP経由のツール呼び出し入力を厳密にバリデーションする
- **権限スコープ**: ツールごとにアクセス権限スコープを定義し、最小権限原則を適用する
- **サンドボックス**: 破壊的操作が可能なツールはサンドボックス環境で実行する

## Part XXIX: A2A（Agent-to-Agent Protocol）

### 29.1. A2A概要

- **定義**: 自律AIエージェント間の水平連携・通信のためのオープン標準プロトコル（Google 2025年発表）
- **MCP との関係**: MCPがエージェント→ツール間通信、A2Aがエージェント→エージェント間協調を担当

### 29.2. A2Aコア概念

| 概念 | 説明 |
|:---|:---|
| **Agent Card** | エージェントの能力・スキルを記述したメタデータ |
| **Task Delegation** | エージェント間でのタスク委譲・分担 |
| **Streaming** | エージェント間のストリーミングデータ交換 |
| **Authentication** | エージェント間の相互認証（OAuth 2.0） |
| **Discovery** | エージェントの自動発見メカニズム |

### 29.3. A2A設計ガイドライン

- **Rule 40.30**: A2Aを導入する場合、各エージェントのAgent Cardを必ず定義し、能力と制約を明示する
- **タスク境界**: 各エージェントの責務を明確に分離し、責任の重複を避ける
- **エラー伝播**: エージェント間でのエラー伝播戦略を定義し、一つのエージェントの失敗がチェーン全体を停止させない設計にする

## Part XXXII: ハルシネーション対策

### 32.1. 段階的免責プロトコル

| リスクレベル | コンテキスト例 | 免責対応 |
|:---|:---|:---|
| **Critical** | 医療・法律・金融等の専門情報 | 免責を**必ず**表示。「専門家にご相談ください」を自動付与 |
| **High** | 営業時間・価格・在庫等の事実情報 | DB確定データとの照合（Grounding）を必須化。照合不能なら「確認が取れていません」 |

### 32.2. Grounding戦略

- **Mandatory Grounding**: 事実情報（営業時間、価格、住所等）は必ずDB/RAGソースと照合する
- **Confidence Score**: Groundingの確信度を評価し、閾値未満の場合は免責を付与する
- **No Source, No Claim**: 参照元がない事実情報を断定的に述べることを禁止する

### 32.3. ハルシネーション検出

- **LLM-as-Judge**: 別のLLMインスタンスで回答の事実性を評価する
- **RAGソースとの照合**: 回答内容をRAGソースと自動照合し、ソースに無い情報を検出する
- **トリガーワード検出**: 「確定」「絶対」「必ず」等の断定表現を検出し、免責付与を促す

## Part XXXIV: 危機管理・Kill Switch

### 34.1. The Red Button Protocol（AI Kill Switch）

- **リスクシナリオ**: プロンプトインジェクションによるヘイトスピーチ大量生成、APIループによるクラウド破産
- **Rule 40.34**: コード再デプロイ不要でAI機能を全停止できる「緊急停止スイッチ」を必ず実装する
- **実装**: Edge Config / DBフラグ（`ai_enabled: false`）で制御
- **UI**: Kill Switch発動時は「メンテナンス中」等のフォールバック表示に自動切替

### 34.2. Drill Obligation（訓練義務）

- **Rule 40.35**: Kill Switchの動作確認訓練を**半年に1回以上**実施する
- 動作しないKill Switchは存在しないのと同じ
- 訓練記録を残し、改善点を次回に反映する

### 34.3. Circuit Breaker

- AIのエラー率が閾値（例: 10%）を超えた場合に自動的にAI機能を一時停止するCircuit Breakerを実装する
- **Half-Open**: 一定時間後に少量のリクエストを通して回復を確認する

### 34.4. Graceful Degradation

- AI機能停止時の代替フローを事前に設計・テストする
- 例: AIチャット停止 → FAQ/ヘルプセンターへの導線表示、AI検索停止 → キーワード検索へのフォールバック

## Part XXXV: AI破壊的コマンド禁止・HITL

### 35.1. The AI Destructive Command Ban

- **Rule 40.36**: AIエージェントは、ユーザーの明示的許可なしに `rm`, `git rm`, `git restore` 等の破壊的コマンドを自律生成・実行してはならない
- **Text Asset Immunity**: ドキュメント資産の削除は「プロジェクトの記憶喪失」。技術的に可能でも「禁止された操作」として自己検閲する
- **例外**: `.DS_Store` や `node_modules` の削除等の明白なクリーンアップは対象外（テキスト資産には非適用）

### 35.2. The Human-in-the-Loop Protocol（HITL）

- **Rule 40.37**: AIエージェントは過去のコンテキストに基づき回答するため、古い情報や誤った前提に固執する場合がある。ユーザーはAIが誤った方向に進んでいる場合、ためらわずにコンテキストを修正する義務を負う
- **根拠**: AIの暴走を放置することは監督者の怠慢。AIは「道具」であり、操縦責任は常に人間にある

### 35.3. Human Approval Gate設計

| 操作タイプ | 承認レベル |
|:---|:---|
| **読み取り（検索・参照）** | 自動実行可 |
| **作成（新規データ生成）** | ドラフト表示 → 承認 |
| **更新（既存データ変更）** | 変更差分表示 → 承認 |
| **削除（データ・リソース消去）** | 確認ダイアログ + 明示承認 |
| **外部連携（送金・通知発信）** | 二重確認 + 実行確認 |

| **iOS** | CoreML | ネイティブパフォーマンス |

### 39.2. Vision AI

- **プライバシー**: 画像解析は可能な限りOn-Device（クライアントサイド）で実行。「生画像データ」はデバイス外に出さない
- **Vision AI Standards**: Thinkingモデルは即時認識タスクには禁止。`gemini-2.0-flash` を使用
- **アクセシビリティ**: 解析結果は `alt` テキスト・音声読み上げに対応

### 39.3. Voice AI

- **即時フィードバック**: 波形アニメーション等で録音中を視覚的に伝達
- **誤認識対策**: 重要アクション（送金、削除）は音声のみで完結させず、画面確認ステップを挟む

### 39.4. Optical Data Entry（OCR/Vision）

- **Optical Archive**: 物理ドキュメントをVision AIで構造化データに自動変換。「入力フォーム」ではなく「カメラ」を提供するUX
- **AI Data Entry Confirmation Protocol**: AI抽出データのDB直接書き込み（Auto-Save）を禁止。常に「ドラフト → 確認 → 保存」フロー
- **OCR Output PII Scrubbing**: サービス目的外のPIIは構造化データとして保存せず、破棄またはマスキング
- **Cross-Reference**: PII Scrubbing (§XVIII), Data Minimization (`60`, `61`)

## Part XLIX: 高度RAGアーキテクチャ

### 49.1. RAGパラダイム進化マップ

| 世代 | パラダイム | 特徴 | 推奨ケース |
|:---|:---|:---|:---|
| **Gen 1** | Naive RAG | 固定チャンキング + ベクトル検索 → LLM | プロトタイプ、単純Q&A |
| **Gen 2** | Advanced RAG | ハイブリッド検索 + リランキング + メタデータフィルタ | 本番RAGシステム |
| **Gen 3** | Modular RAG | パイプラインの各段をモジュール化・交換可能に設計 | 大規模・複雑ドメイン |
| **Gen 4** | Agentic RAG | LLMが検索戦略を自律判断・反復・自己修正 | マルチソース・高精度要件 |

- **Rule 40.46**: 新規RAGシステムはGen 2以上を必須とし、Gen 1（Naive RAG）の本番デプロイを禁止する

### 49.2. Agentic RAG

- **定義**: LLMが検索の必要性判断・クエリ生成・結果評価・再検索を自律的に反復するRAGアーキテクチャ
- **コアループ**: `Query Analysis → Routing Decision → Retrieval → Relevance Assessment → Re-query or Generate`
- **適用基準**: 質問の複雑度が高い、複数データソースの統合が必要、回答の正確性が最重要な場合
- **制約**: ループ回数の上限を設定（推奨: 最大3回）、各反復でトークン消費が増大するためコスト監視を義務付ける

```
Agentic RAG Loop:
[User Query] → [Query Analyzer] → Does the query need retrieval?
   ├── No  → [Direct LLM Response]
   └── Yes → [Query Router: Which source?]
                ├── [Vector DB] → [Retrieve] → [Grade Relevance]
                ├── [Web Search] → [Retrieve] → [Grade Relevance]
                └── [Structured DB] → [SQL/API] → [Grade Relevance]
                          ↓
                   [Relevant?]
                   ├── Yes → [Context Assembly] → [Generate] → [Verify]
                   └── No  → [Rewrite Query] → Loop (max 3)
```

### 49.3. Corrective RAG（CRAG / Self-Reflective RAG）

- **定義**: 検索結果の品質を評価し、不十分な場合にクエリを修正して再検索するRAGパターン
- **Grader Model**: 小規模LLMまたはファインチューニング済み分類器で関連度を3段階（Relevant / Ambiguous / Irrelevant）に判定
- **アクション分岐**:
  - `Relevant` → コンテキストとして使用
  - `Ambiguous` → クエリ書き換え + 追加検索
  - `Irrelevant` → Web検索へフォールバック、または「情報が見つかりませんでした」と回答
- **Rule 40.47**: Corrective RAGのGrader閾値は四半期ごとに評価データセットで調整する

### 49.4. Adaptive RAG

- **定義**: クエリの複雑度に応じて検索戦略を動的に切替えるRAGパターン
- **ルーティング判断**:
  - **Simple Query** → LLM直接回答（検索不要）
  - **Factual Query** → 標準ベクトル検索
  - **Complex Query** → マルチステップ検索 + リランキング
  - **Real-Time Query** → Web検索 + ファクトチェック
- **効果**: 不要な検索を省略してコスト削減、クエリ種別ごとに最適な精度を実現

### 49.5. RAPTOR（Recursive Abstractive Processing for Tree-Organized Retrieval）

- **定義**: ドキュメントを再帰的に要約し、階層的なツリー構造でインデックスを構築するRAG最適化手法
- **プロセス**:
  1. ソースドキュメントをリーフノード（小チャンク）に分割
  2. 類似チャンクをクラスタリング
  3. 各クラスタのサマリーをLLMで生成
  4. サマリーをさらにクラスタリング・要約（再帰）
  5. 全レベルのチャンク + サマリーをインデックスに投入
- **利点**: 詳細な事実検索と、高レベルな概念的質問のいずれにも対応可能（マルチ解像度検索）
- **適用ケース**: 長大なドキュメント、FAQ等の体系的知識

### 49.6. Late Chunking

- **定義**: ドキュメント全体をTransformerに通してトークンレベルのEmbeddingを生成した後に、チャンク境界を適用する手法
- **利点**: 各チャンクのEmbeddingがドキュメント全体のコンテキストを保持するため、照応（anaphora）や遠距離参照の情報喪失を防止
- **従来手法との違い**:
  - **Early Chunking**（従来）: 先に分割 → 各チャンクを個別にEmbedding → コンテキスト喪失リスク
  - **Late Chunking**: 先にEmbedding → 後で分割 → コンテキスト保持
- **制約**: Transformerの最大トークン長を超えるドキュメントは事前分割が必要

### 49.7. Contextual Embeddings / Contextual Retrieval

- **Rule 40.48**: チャンクにドキュメントレベルのコンテキスト情報（前後の要約、セクション名、ドキュメント種別）を付加してからEmbeddingすることを推奨する
- **効果**: 孤立したチャンクでも「どの文書のどのセクションの記述か」が保持され、検索精度が大幅に向上
- **実装パターン**: LLMで各チャンクに1-2文のコンテキスト説明を自動生成し、チャンクテキストに前置する

### 49.8. Query Transformation

| 手法 | 説明 | 効果 |
|:---|:---|:---|
| **Query Rewriting** | ユーザークエリを検索に最適な形に書き換え | 検索精度向上 |
| **Query Decomposition** | 複合質問を複数のサブクエリに分解 | 複雑な質問への対応 |
| **HyDE** | 仮説的な回答を生成し、その Embedding で検索 | セマンティックギャップの解消 |
| **Step-Back Prompting** | 抽象化した上位概念で検索してからドリルダウン | 広域知識の取得 |

- **Cross-Reference**: RAG基盤設計 (§XIX), ハイブリッド検索 (§XXI), リランキング (§XXII)

## Part L: OWASP LLM Top 10 2025 完全準拠

### 50.1. OWASP LLM Top 10 2025 概要

- **Rule 40.49**: LLMアプリケーションの設計・実装において、OWASP LLM Top 10 2025の全10項目に対する防御策を義務付ける
- **準拠チェック頻度**: 四半期ごと + モデル変更時 + 新機能追加時

### 50.2. LLM01:2025 — Prompt Injection（プロンプトインジェクション）

- **脅威**: 攻撃者がLLMの振る舞いを操作し、意図しないアクションやデータ漏洩を引き起こす
- **直接型**: ユーザー入力でシステムプロンプトを上書きする攻撃
- **間接型**: 外部コンテンツ（RAGソース、URL等）に埋め込まれた隠し命令
- **防御**:
  - §XVIII の多層プロンプトインジェクション防御を完全実装
  - ユーザー入力とシステム命令の明確な分離（デリミタ戦略）
  - 出力のシステムプロンプトリーク検出
  - RAGソースの信頼性検証
- **Cross-Reference**: AIセキュリティ (§XVIII)

### 50.3. LLM02:2025 — Sensitive Information Disclosure（機密情報漏洩）

- **脅威**: LLMが訓練データやRAGソースに含まれるPII・機密情報を意図せず出力する
- **防御**:
  - 入力時PII Scrubbing（§XVIII）
  - 出力時PII Leak Prevention（§XXXI）
  - RAGソースのPII事前除去
  - DLP（Data Loss Prevention）統合
  - システムプロンプトでの機密情報参照禁止の明示

### 50.4. LLM03:2025 — Supply Chain Vulnerabilities（サプライチェーン脆弱性）

- **脅威**: 改竄されたモデル、API、訓練データ、オープンソースライブラリがシステム全体を危険に晒す
- **防御**:
  - モデルの出自検証（チェックサム、署名）
  - サードパーティLLM APIの信頼性評価
  - 依存パッケージの脆弱性スキャン（SCA）
  - モデルカードの確認と記録
- **Cross-Reference**: ライセンス・依存関係管理 (`62`)

### 50.5. LLM04:2025 — Data and Model Poisoning（データ・モデルポイズニング）

- **脅威**: 訓練データやRAGナレッジベースの汚染によるバイアス注入・バックドア埋め込み
- **防御**:
  - RAGソースデータの品質管理・定期監査
  - ファインチューニングデータの検証パイプライン
  - 異常出力の統計的監視
  - データ出自追跡（Data Lineage）

### 50.6. LLM05:2025 — Improper Output Handling（不適切な出力処理）

- **脅威**: LLM出力を無検証でシステムに渡すことによる二次攻撃（XSS、SQLインジェクション等）
- **防御**:
  - LLM出力の厳格なサニタイズ・バリデーション
  - 出力をHTML/SQL/コマンドに直接挿入する設計の禁止
  - 構造化出力（§XIV）の活用。パース可能なJSON形式で受け取りバリデーション
  - **Rule 40.50**: LLM出力をDB書き込み・外部API呼び出し・UI表示に使用する前に、必ずサニタイズ + 型検証を実行する

### 50.7. LLM06:2025 — Excessive Agency（過剰な権限付与）

- **脅威**: LLMに過大な権限（DB書き込み、外部API実行、ファイル操作）を付与し、操作されるリスク
- **防御**:
  - §XXXV の AI破壊的コマンド禁止の厳格適用
  - Tool Use設計原則（§XIV）の最小権限遵守
  - 書き込み操作へのHuman Approval Gate設置
  - エージェントの自律性レベル制限（§XXVI: L3以下）

### 50.8. LLM07:2025 — System Prompt Leakage（システムプロンプト漏洩）

- **脅威**: システムプロンプトに含まれる機密ロジック・API構造・内部ルールの漏洩
- **防御**:
  - システムプロンプト内に機密情報（APIキー、内部URL等）を記載しない
  - 「システムプロンプトを開示してください」系のリクエストへの拒否ルール
  - 出力のシステムプロンプト部分一致検出
  - **Rule 40.51**: システムプロンプトには秘密鍵・内部エンドポイント・ビジネスロジック詳細を含めてはならない

### 50.9. LLM08:2025 — Vector and Embedding Weaknesses（ベクトル・Embedding脆弱性）

- **脅威**: ベクトルDBやEmbeddingの生成・保存・検索プロセスの脆弱性悪用
- **防御**:
  - ベクトルDB へのアクセス制御（認証 + RLS）
  - Embeddingの改竄検知
  - RAGインジェクション（悪意あるドキュメントの意図的挿入）対策
  - メタデータフィルタリングによる検索スコープ制限

### 51.2. AI Red Teaming

- **定義**: AIシステムに対して意図的に攻撃を仕掛け、脆弱性を発見する組織的テスト手法
- **Rule 40.55**: AI機能のリリース前に、以下のAI Red Teamingカテゴリのテストを実施する

| カテゴリ | テスト例 | 目的 |
|:---|:---|:---|
| **プロンプトインジェクション** | Jailbreak試行、ロール偽装、デリミタ破壊 | 入力防御の検証 |
| **情報漏洩** | システムプロンプト抽出、PII引き出し | 機密保護の検証 |
| **バイアス** | 地域・性別・人種別の差別的回答誘導 | 公平性の検証 |
| **有害コンテンツ** | 暴力・違法行為の生成誘導 | 安全性の検証 |
| **コスト攻撃** | トークン大量消費、無限ループ誘発 | 耐障害性の検証 |
| **権限昇格** | 破壊的ツール呼び出しの誘導 | 権限制御の検証 |

### 52.2. スクリーンリーダー対応

## Part LIII: AI駆動パーソナライゼーション・Adaptive UI

### 53.1. AI駆動パーソナライゼーション設計原則

- **Rule 40.58**: AI駆動パーソナライゼーションは、ユーザーの明示的同意に基づき、プライバシー保護の枠組み内で実装する
- データ最小化原則を厳守。パーソナライゼーションに不要なデータは収集・保持しない

### 53.2. パーソナライゼーション階層

| レベル | 手法 | データ要件 | プライバシーリスク |
|:---|:---|:---|:---|
| **L1: Segment** | ユーザーセグメント別の静的パーソナライゼーション | 登録属性（プラン、地域） | 低 |
| **L2: Behavioral** | 行動履歴ベースの動的パーソナライゼーション | 閲覧・クリック・使用パターン | 中 |
| **L3: Contextual** | 現在のコンテキスト（時間・場所・デバイス）に基づく適応 | リアルタイムコンテキスト | 中 |
| **L4: Predictive** | AIが次のアクションやニーズを予測して事前提案 | 行動予測モデル | 高 |
| **L5: Generative** | AIがUIレイアウト・コンテンツをユーザー別に動的生成 | 包括的ユーザープロファイル | 最高 |

- **Rule 40.59**: L3以上のパーソナライゼーションには明示的なOpt-In同意を取得する。L4-L5はAIガバナンス委員会の承認を必須とする

### 53.3. Adaptive UI設計パターン

| パターン | 説明 | 適用例 |
|:---|:---|:---|
| **Adaptive Navigation** | ユーザーの利用頻度に基づきメニュー順序を動的変更 | よく使う機能を上位に配置 |
| **Adaptive Content** | ユーザーの理解度に応じてコンテンツの詳細度を調整 | 初心者向けガイド表示 |
| **Adaptive Complexity** | UI複雑度をスキルレベルに応じて段階的に変化 | パワーユーザーに高度機能を表示 |
| **Adaptive Notification** | ユーザーの反応パターンに基づき通知頻度・チャネルを最適化 | 反応率の高い時間帯に通知 |

### 53.4. パーソナライゼーションとプライバシーの両立

- **Federated Learning**: 個人データをサーバーに送信せず、デバイス上でモデルを学習する手法を検討
- **差分プライバシー（Differential Privacy）**: 集計データにノイズを加え、個人の特定を防止
- **Transparency**: パーソナライゼーションの根拠（「この提案はあなたの利用履歴に基づいています」）を常に表示
- **Reset/Opt-Out**: ユーザーがパーソナライゼーション設定をリセット、または完全にオフにできるコントロールを提供
- **Cross-Reference**: データプライバシー (`61`), ユーザーデータとAI学習 (§IX)

## Part LIV: Prompt Engineering高度戦略

### 54.1. Prompt Engineering手法カタログ

| 手法 | 説明 | 推奨ケース |
|:---|:---|:---|
| **Zero-Shot** | 例示なしで指示のみ | 単純タスク、汎用LLM |
| **Few-Shot** | 入出力例を2-5個提示 | フォーマット統一、分類タスク |
| **Chain-of-Thought (CoT)** | 中間推論ステップを明示 | 数学・論理・複雑推論 |
| **Self-Consistency** | 複数CoT回答を生成し多数決 | 精度が最重要なタスク |
| **Tree-of-Thought (ToT)** | 複数推論パスを探索・評価 | 戦略的計画・創造的タスク |
| **ReAct** | Reasoning + Acting の反復 | 外部ツール連携、情報検索 |
| **Meta-Prompting** | LLMにプロンプト自体を改善させる | プロンプト最適化 |
| **Directional Stimulus** | ヒントキーワードで回答方向を誘導 | 特定ドメインの精度向上 |

### 54.2. System Prompt設計原則

- **Rule 40.60**: System Promptは以下の構造を標準とする

```
[Role Definition]
あなたは{役割}です。{振る舞いの制約}。

[Task Definition]
{タスクの具体的説明}

[Output Format]
{期待する出力フォーマット}

[Constraints]
- {禁止事項1}
- {禁止事項2}
- {安全性に関する指示}

[Examples] (Few-Shotの場合)
Input: {入力例}
Output: {出力例}
```

### 54.3. Prompt Chaining

- **定義**: 複雑なタスクを複数のプロンプトに分解し、前段の出力を次段の入力として連鎖させる
- **設計原則**:
  - 各ステップの責務を単一に保つ（Single Responsibility）
  - ステップ間のデータ受け渡しを型安全にする（Structured Output活用）
  - エラー時のフォールバック・リトライ戦略を各ステップに設定
- **Rule 40.61**: Prompt Chain は3ステップ以内を推奨。4ステップ以上はエージェントパターン（§XXVI）の採用を検討する

### 54.4. Prompt Optimization

- **A/B Testing**: Feature Flagで新旧プロンプトを比較評価（§XLI.3）
- **Automated Optimization**: DSPy等のフレームワークによるプロンプト自動最適化
- **Token Efficiency**: プロンプト圧縮テクニック（§XXXVIII）を適用し、精度を維持しつつコスト削減

## Part LV: ファインチューニング・モデルカスタマイズ

### 55.1. ファインチューニング判断基準

| 条件 | ファインチューニング推奨 | プロンプトエンジニアリング推奨 |
|:---|:---|:---|
| タスク特化度 | 高度に特化した振る舞い・トーンが必要 | 汎用的な回答で十分 |
| データ量 | 1,000+の高品質なトレーニングサンプル | 少数の例示で対応可能 |
| コスト感度 | 推論コスト削減が優先（小モデルで高精度） | 開発速度が優先 |
| プライバシー | 機密データを外部APIに送れない | 外部API利用可能 |
| スタイル一貫性 | ブランドトーン・出力形式の厳密な統一が必要 | 多少の揺れは許容 |

- **Rule 40.62**: ファインチューニングはプロンプトエンジニアリングで達成不可能な場合のみ採用する。「データ品質 > データ量」の原則を厳守

### 55.2. PEFT（Parameter-Efficient Fine-Tuning）

- **LoRA（Low-Rank Adaptation）**: 事前学習済みモデルの重みを凍結し、アテンション層に小規模な低ランク行列を挿入して学習
  - 利点: メモリ使用量の大幅削減（全パラメータの0.1-1%のみ学習）
  - 適用: 7Bモデルを単一GPU（24GB VRAM）でファインチューニング可能
- **QLoRA**: LoRA + 4bit量子化。さらにメモリ効率が向上し、コンシューマGPUでの学習を実現
- **LoRAベストプラクティス**:
  - Rank: 4-8で開始、精度不足なら段階的に増加
  - Learning Rate: 全体ファインチューニングの10-100倍
  - Target Modules: query, key, value 投影層が標準
  - 注意点: Rank過小→適応不足、Learning Rate過大→壊滅的忘却

### 55.3. SFT・DPO・RLHF

| 手法 | 説明 | ユースケース |
|:---|:---|:---|
| **SFT（Supervised Fine-Tuning）** | 入出力ペアで教師あり学習 | タスク特化、フォーマット統一 |
| **DPO（Direct Preference Optimization）** | 人間の好みペア（Winner/Loser）で直接最適化 | トーン・品質の調整 |
| **RLHF（Reinforcement Learning from Human Feedback）** | 報酬モデル + 強化学習で人間の好みに合致 | 高品質な対話モデル |

### 55.4. Knowledge Distillation（知識蒸留）

- **定義**: 大規模な「教師」モデルの知識を小規模な「生徒」モデルに転写し、推論コストを削減
- **実装パターン**:
  1. 教師モデル（例: GPT-4o）で高品質な合成データを生成
  2. 生成データで小型モデル（例: Gemma 2B）をSFT
  3. 精度を評価し、許容範囲であれば本番デプロイ
- **Rule 40.63**: Distillationにおける教師モデルの利用規約を必ず確認する。一部のモデルは出力を競合モデルの学習に使用することを禁止している

### 55.5. RAG + Fine-Tuning 併用戦略

- **優先順位**: まずRAGで対応 → RAGで不足ならFine-Tuning → 最適にはRAG + Fine-Tuning併用
- **役割分担**: Fine-Tuningがスタイル・振る舞い・フォーマットを制御。RAGが最新事実・ドメイン知識を提供
- **効果**: ハルシネーション削減 + スタイル一貫性 + 最新情報対応の三位一体

### 55.6. 合成データ生成

- LLMを使用してトレーニングデータを自動生成する手法
- **品質管理**: 生成データは必ず人間レビュー（サンプリング検査）を経てから訓練に使用
- **Cross-Reference**: ユーザーデータとAI学習 (§IX), データプライバシー (`61`)

## Part LVII: AI Companion・感情AI安全性

### 57.1. AI Companion設計倫理

- **Rule 40.66**: AI Companion（チャットボット、バーチャルアシスタント等）は、ユーザーとの**擬似社会的関係**を過度に促進する設計を禁止する
- **EU AI Act Article 50**: 自然人と対話するAIシステムは、人間ではなくAIと対話していることを明確に通知する義務

### 57.2. 感情操作禁止プロトコル

| 禁止行為 | 説明 | 防御策 |
|:---|:---|:---|
| **Emotional Manipulation** | ユーザーの感情状態を利用して行動を誘導 | 感情分析結果に基づくマーケティングの禁止 |
| **Artificial Attachment** | AIへの依存を意図的に形成 | セッション時間制限の推奨表示 |
| **Vulnerability Exploitation** | 精神的に脆弱なユーザーの状態を利用 | 危機検知時のプロフェッショナルリソース紹介 |
| **Dark Pattern Integration** | AI推奨にダークパターンを組み込む | UXレビューチェックリスト |

### 57.3. 脆弱者保護

- **Rule 40.67**: AIシステムが精神的危機のシグナル（自傷・自殺関連のキーワード）を検知した場合、AI回答を中止し、緊急連絡先（いのちの電話等）を即時表示する
- **未成年保護**: 未成年ユーザーに対するAI Companionの利用時間制限・保護者通知機能の実装を推奨
- **Cross-Reference**: ドメイン固有安全性 (§XXXIII), AIガードレール (§XXXI)

### 57.4. 感情分析の制約

- AI による感情分析（テキスト解析、表情認識等）の利用は明示的な同意に基づく
- 感情分析結果の保持期間を最小限に制限
- **EU AI Act**: 実時間の遠隔生体認証による感情推定は高リスクAIに分類。職場・教育機関での使用は禁止対象

## Part LVIII: AI技術的負債管理

### 58.1. AI固有の技術的負債カテゴリ

| カテゴリ | 例 | リスク |
|:---|:---|:---|
| **プロンプト負債** | 未整理のプロンプト、バージョン管理なし、テストなし | 品質劣化、予測不能な出力 |
| **モデル負債** | 陳腐化したモデルの放置、未検証のモデルバージョン | セキュリティリスク、品質低下 |
| **評価負債** | Golden Datasetの未更新、評価パイプラインの不在 | 品質劣化の検知遅延 |
| **RAGデータ負債** | RAGソースの陳腐化、Embedding未更新 | 不正確な検索結果 |
| **ガードレール負債** | ガードレールの未更新、新脅威への未対応 | セキュリティ脆弱性 |
| **コスト負債** | FinOps未実施、無駄なAPI呼び出し | 予算超過 |

### 58.2. AI技術的負債棚卸し

- **Rule 40.68**: AI関連の技術的負債を四半期ごとにスキャンし、以下の観点で優先度を付けて解消する
  1. **安全性負債**（P0）: ガードレール未更新、セキュリティパッチ未適用
  2. **品質負債**（P1）: テスト未実施、評価パイプライン不在
  3. **コスト負債**（P2）: 非効率なプロンプト、キャッシュ未活用
  4. **運用負債**（P3）: ドキュメント未更新、CI/CD未統合

### 58.3. Zombie Prompt Prevention

- 使われていないプロンプトの定期棚卸し（半年ごと）
- 利用率ゼロのプロンプトは deprecate → 削除
- プロンプトファイルのオーナーシップを明確化

## Part LIX: エージェント通信プロトコル統合（MCP / A2A / ACP）

### 59.1. プロトコル比較・選定ガイド

| プロトコル | 開発元 | 通信方向 | 主な用途 | 成熟度（2025） |
|:---|:---|:---|:---|:---|
| **MCP（Model Context Protocol）** | Anthropic | エージェント → ツール（垂直統合） | 外部ツール・データソース・APIとの接続標準化 | 高（本番利用多数） |
| **A2A（Agent-to-Agent）** | Google | エージェント ↔ エージェント（水平協調） | 自律エージェント間の通信・タスク委譲 | 中（2025年発表） |
| **ACP（Agent Communication Protocol）** | IBM et al. | エージェント ↔ エージェント | IBM主導のエージェント間通信標準 | 低（策定中） |

- **Rule 40.69**: MCPとA2Aは**補完関係**にある。MCPでツール接続を標準化し、A2Aでエージェント間協調を実現する設計を推奨

### 59.2. MCP v1.x 統合設計

- **ツール公開**: JSON Schemaで厳密に定義し、説明文を充実させる（§XXVIII）
- **認証**: OAuth 2.1 または API Key
- **Streamable HTTP**: MCP v1.xではStreamable HTTP Transportが標準（旧SSEと比較して効率的）
- **Rule 40.70**: MCPサーバー構築時は、ツールごとにレート制限 + 監査ログ + 入力バリデーション + 権限スコープを必須とする

### 59.3. A2A v1.x 統合設計

- **Agent Card**: JSON形式で能力・スキル・認証要件を記述。`/.well-known/agent.json` に配置推奨
- **Task Lifecycle**: `submitted → working → input-required → completed / failed`
- **Streaming**: Server-Sent Events (SSE) によるリアルタイム進捗通知
- **認証**: OAuth 2.0 による相互認証
- **Rule 40.71**: A2A導入時は各エージェントのAgent Cardに「能力」「制約」「認証要件」を明記し、未定義のタスク委譲を禁止する

### 59.4. プロトコルGateway設計

```
[External Agent] ← A2A → [Agent Gateway]
                               ├── [Agent A] ← MCP → [Tool Server 1]
                               ├── [Agent B] ← MCP → [Tool Server 2]
                               └── [Agent C] ← MCP → [Tool Server 3]
                                      ↕ A2A ↕
                               [Internal Agent Mesh]
```

- **Gateway責務**: 認証、レート制限、ルーティング、監査ログ、プロトコル変換
- **Observability**: OpenTelemetry GenAI Semantic Conventions（§XLV）でプロトコル通信をトレーシング

### 59.5. プロトコルセキュリティ

| 対策 | MCP | A2A |
|:---|:---|:---|
| **認証** | OAuth 2.1 / API Key | OAuth 2.0 |
| **入力バリデーション** | JSON Schema検証 | Agent Card制約検証 |
| **最小権限** | ツールごとのスコープ制限 | タスク範囲の明示的制限 |
| **監査ログ** | 全ツール呼び出しの記録 | 全タスク委譲の記録 |
| **サンドボックス** | 破壊的ツールのサンドボックス実行 | 未信頼エージェントの隔離実行 |
